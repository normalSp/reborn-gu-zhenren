## P1-1 战斗系统：核心架构

### 决策 1：杀招释放模式

**背景**：蛊真人原著中，杀招是蛊虫组合使用的战斗技。玩家战斗时如何释放杀招，决定了战斗的策略深度和开发复杂度。

**方案 A：纯预设杀招（从 JSON 加载）**

玩家无法自由组合，只能在已知杀招列表中选择。杀招定义在 `killer-moves.json` 中，每个杀招明确：
- 所需蛊虫列表（如：月光蛊 + 月光蛊 + 月光蛊 → 月牙斩）
- 消耗真元量
- 基础伤害公式
- 流派属性

优点：实现简单，战斗平衡可控，AI 判定逻辑清晰，适合快速上线。
缺点：策略深度受限，玩家无法体验"创造杀招"的乐趣，原著的自由组合特性丢失。

预估工时：3天

**方案 B：纯动态组合（玩家自由搭配蛊虫）**

玩家每回合选择1-3只激活的蛊虫，系统根据蛊虫类型（攻击/防御/治疗/控制）和流派互斥规则，动态计算组合效果。

优点：策略深度高，符合原著"蛊师创造杀招"设定，重玩性强。
缺点：平衡极难控制，AI 判定需要大量规则引擎，易出 bug，开发周期长。

预估工时：7-10天

**方案 C：混合模式（预设杀招 + 自由槽位）**

分两层——
1. 杀招槽：预设杀招，从 `killer-moves.json` 加载，玩家可在非战斗时配置（类似巫师3的法印槽）
2. 自由蛊虫槽：每回合额外可选 1只蛊虫做自由操作（如使用治疗蛊、控制蛊、逃跑蛊）

优点：兼顾策略深度与平衡性，预设杀招保证下限，自由槽位提供变数。开发成本适中。
缺点：UI 需要两套选择逻辑。

推荐方案：**C（混合模式）**。这是蛊真人世界观最自然的映射——杀招是经过千锤百炼的组合，自由蛊虫是随机应变的手段。

| 方案 | 策略深度 | 开发成本 | 平衡难度 | 原著还原度 |
|------|---------|---------|---------|-----------|
| A 纯预设 | ★★ | 低 (3天) | 低 | ★★★ |
| B 纯动态 | ★★★★★ | 高 (10天) | 极高 | ★★★★★ |
| **C 混合** | **★★★★** | **中 (5天)** | **中** | **★★★★** |

### 决策 2：战斗节奏模式

**背景**：回合制 vs ATB（Active Time Battle，速度条）是战斗体验的核心分水岭。

**方案 A：传统回合制**
- 玩家回合 → AI 判定 → 敌方回合 → 结果检查 → 循环
- 每个阶段玩家有无限思考时间

优点：实现简单，与 PIPE 管线天然契合（每回合一次 state_update + AI请求），适合文本叙事为主的游戏。
缺点：节奏偏慢，缺乏紧迫感。

**方案 B：ATB 速度条系统**
- 双方有速度属性，决定行动顺序
- 速度条随时间填充，满格行动
- 可以插入"反应型操作"

优点：战斗更有节奏感，速度属性的策略价值高。
缺点：实现复杂，与 PIPE 管线异步冲突（AI 响应延迟导致速度条逻辑混乱），AI 文本叙事与实时节奏矛盾。

推荐方案：**A（传统回合制）**。本游戏核心是 AI 驱动的文本叙事，传统回合制与 PIPE 管线天然兼容。ATB 需要等待 AI 响应，会破坏速度条节奏。战斗节奏可通过叙事文案和特效增强紧凑感。

### 决策 3：逃跑机制

**背景**：是否允许玩家在战斗中选择逃跑？蛊真人世界观中"留得青山在"是常见策略，但也重视"狭路相逢勇者胜"。

**方案 A：无条件允许逃跑**
- 逃跑成功率 = 100%
- 代价：随机丢失 1-2 只蛊虫 或 掉落一定元石

优点：降低挫败感，符合方源"活着才有机会"的价值观。
缺点：可能鼓励玩家频繁逃跑规避难度。

**方案 B：条件逃跑 + 代价递增**
- 逃跑成功率基于速度/烟雾蛊/环境
- 成功则丢失少量资源，失败则损失更大（蛊虫死亡、重伤）
- 连续逃脱增加"怯战" flag，影响 NPC 态度和声望

优点：有策略深度，代价有明显感知。
缺点：开发成本增加。

推荐方案：**B（条件逃跑+代价递增）**。

## P1-2 战斗系统：数据结构与 UI

### 决策 4：战斗伤害公式

**背景**：伤害计算需要同时兼顾境界、道痕、蛊虫 tier、杀招威力，且结果要能被 AI 合理叙事化。

**推荐公式（基于世界规则 world-rules.json 的「道痕战力模型」）**：

```
基础攻击力 = 杀招基础威力 × (1 + 相关道痕/10000)
实际伤害 = 基础攻击力 × (攻击方境界系数/防御方境界系数) × 流派相克系数 × 暴击/闪避随机因子
```

境界系数参考：
| 境界 | 攻击系数 | 防御系数 |
|------|---------|---------|
| 一转 | 1.0 | 1.0 |
| 二转 | 1.5 | 1.3 |
| 三转 | 2.2 | 1.8 |
| 四转 | 3.5 | 2.5 |
| 五转 | 5.5 | 3.5 |

流派相克（基于 world-rules.json 互斥表）：克制 → 伤害 ×1.5；被克 → 伤害 ×0.7

## P1-3 NPC 交互系统

### 决策 6：对话风格

**背景**：当前 CharacterPanel 只读。需要让玩家能与 NPC 自由对话，同时保持世界观一致性。

**方案 A：完全自由对话**
- 玩家可以输入任意文本
- AI 以 NPC persona + 世界观约束回应
- System prompt 注入 `personality + currentMood + relationshipWithPlayer`

优点：沉浸感最强，符合 AI 文字游戏的优势。
缺点：可能跑题严重，NPC 描写不受控，需要 C13 守卫强力兜底。

**方案 B：话题选项制（推荐）**
- 预设 3-5 个话题选项（基于章节上下文动态生成）
- 每个话题下 AI 自由生成回复
- 话题结构：
  ```
  1. 打探情报（"关于狼潮..."）
  2. 交易蛊虫（"你这儿有什么好东西？"）
  3. 闲聊（"今天天气不错..."）
  4. 【特定章节话题】
  5. 【好感度达标解锁话题】
  ```

优点：既保留 AI 自由生成的优势，又控制对话方向。话题选项本身就是玩法设计工具。
缺点：实现稍复杂，需要话题生成逻辑。

**方案 C：情感轮盘（类似 SIMS/Persona）**
- 固定情感动作选择（友善/敌意/调侃/请求/威胁）
- AI 根据情感动作 + NPC 性格回应

优点：操作极简。
缺点：策略深度不足，不适合文字重心游戏。

推荐方案：**B（话题选项制）**。这是 AI 文字游戏的最佳平衡——通过话题框定范围，AI 在范围内自由发挥。

### 决策 7：NPC 感情线

**背景**：蛊真人原著中感情线相对克制（商心慈对方源的感情、诸多后宫角色的存在），但作为游戏是否需要感情系统？

**方案 A：不支持感情线**
- NPC 关系仅限「态度」数值（-100 至 +100）
- 态度影响交易折扣、任务解锁、情报分享
- 没有恋爱/婚姻机制

优点：避免 AI 崩坏风险（AI 可能生成不当恋爱内容），开发简单。
缺点：少了角色深度，原著中商心慈、白凝冰等角色的感情线无法体现。

**方案 B：轻度感情线（推荐）**
- 基于态度 + 特定剧情触发「特殊关系」flag
- 触发条件严格限制（需完成该 NPC 专属任务线 + 道心值匹配）
- 特殊关系仅影响部分叙事文案，不开放婚姻系统
- C13 守卫强化：检测 NPC 关系是否超过 canon 允许范围

优点：有角色深度但不失控，守卫可兜底。
缺点：需要额外设计任务线和关系 flag。

**方案 C：完整感情系统**
- 约会/礼物/婚姻/子女

优点：角色沉浸感最强。
缺点：极度容易崩坏，AI 可能生成不当内容，金丝雀守卫难以全面覆盖，与原著严肃基调冲突。

推荐方案：**B（轻度感情线）**。用任务线+flag机制严格控场，不追求完整恋爱模拟。

### 决策 8：任务刷新机制

**背景**：NPC 任务如何生成和刷新？

**方案 A：剧情驱动（推荐）**
- 任务随章节推进自动解锁，与 `chapter.json` 绑定
- 每章固定 3-5 个主线任务 + 2-3 个支线任务
- 任务完成则永久完成，不做定期刷新

优点：叙事连贯，任务质量可控，减少重复感。
缺点：内容消耗快，需要大量手工设计任务。

**方案 B：定期刷新**
- 每 N 回合随机生成一批新任务
- 模板化生成（从 `quests.json` 模板池随机组合）

优点：无限内容，避免"做完所有任务后无事可做"。
缺点：任务质量不稳定，叙事断裂感强。

**方案 C：混合（剧情主线 + 刷新支线）**
- 每章固定主线任务（不可刷新）
- 商会/NPC随机发布刷新支线任务（fetch/defeat/deliver 模板）

推荐方案：**C（混合）**。主线保证叙事质量，支线保证重复可玩性。

## P1-4 章节叙事弧光

### 决策 9：章节触发条件

**背景**：玩家如何从「青茅山篇」推进到「商队篇」？

**方案 A：纯 flags 驱动（自动化）**
- 关键事件 flag 全部达成 + 境界达标 → 自动触发转章叙事
- 玩家无法控制转章时机

优点：叙事流畅，玩家不必手动选择。
缺点：玩家可能想多待一会探索支线却被强制推进。

**方案 B：flags + 手动确认（推荐）**
- 达成条件后弹出「前方有新的机缘...是否踏上新的旅程？」确认框
- 玩家可选择「再等等」暂缓转章（但转章事件不会消失）

优点：给玩家主动权，同时避免忘记推进。
缺点：需要额外 UI。

**方案 C：完全手动**
- 玩家在「远行」面板中选择目的地，主动触发转章

优点：完全自由。
缺点：新手可能不知道何时该推进。

推荐方案：**B（flags + 手动确认）**。这是最尊重玩家自主权的设计。

### 决策 10：P1 章节规划数量

推荐 P1 先做 3 章：

| 章节 ID | 名称 | 境界范围 | 关键事件 | 预估叙事量 |
|---------|------|---------|---------|-----------|
| qingmao | 青茅山篇 | 一转-二转 | 开窍大典、狼潮来袭、山寨覆灭 | 15-20轮 |
| caravan | 商队篇 | 二转-三转 | 加入商队、首次远行、商心慈初遇 | 15-20轮 |
| nansha | 南疆初探 | 三转-四转 | 商家内斗、南疆势力网、晋升心机 | 20-30轮 |

三章完成后，玩家已经熟悉完整游戏循环，后续章节可作为 P2/P3 扩展。

### 决策 11：章节目标失败处理

**背景**：如果玩家没有达成章节目标（如青茅山篇的「在狼潮中存活」），怎么办？

**方案 A：必须完成（死循环）**
- 目标未达成则章节不推进
- 后果：玩家可能卡关，需要设计保底机制

**方案 B：允许跳过但丢失奖励（推荐）**
- 章节有「推进门槛」（最低条件）和「完美达成」（全部目标）
- 达到门槛即可推进，但未达成的目标永久失去奖励
- 狼潮中死亡 → 触发「山寨覆灭」叙事，以幸存者身份进入商队篇

优点：不会有卡关，失败的叙事本身也是故事。
缺点：需要设计多条分支叙事（增加设计成本）。

推荐方案：**B**。蛊真人原著的魅力之一就是「失败本身也是故事的一部分」。

## P2-1 音频系统

### 决策 12：音效资源策略

**背景**：项目目前无音频文件，需要决定音频资源来源。

**方案 A：使用免费音效库（推荐）**
- 音效：Freesound.org（CC0/CC BY 许可证）
- BGM：ccMixter（CC BY，已有 bgm-library skill 支持）
- 风/水/火等环境音效可找到大量免费资源

优点：零成本，版权合规，快速上线。
缺点：音质参差不齐，需要筛选。

**方案 B：用户自行提供**
- 仅提供音频框架，用户自行放入音频文件

优点：尊重用户偏好。
缺点：开箱体验差，大部分用户不会配置。

推荐方案：**A**。先用免费库填充默认音效，同时提供「自定义音频」文件夹供高级用户替换。

### 决策 13：音量控制

**方案 A：独立音量控制条**
- BGM / SFX / Ambient 三通道独立滑块
- 设置面板中的常驻选项

优点：精细控制，尊重不同偏好。
缺点：UI 稍复杂。

**方案 B：简单静音按钮**
- 全局静音开/关

优点：实现极简。
缺点：不够灵活。

推荐方案：**A**。三个独立滑块所占 UI 空间很小，但体验提升明显。

## P2-2 成就系统

### 决策 14：隐藏成就

**方案 A：有隐藏成就**
- 不公开触发条件的成就（如：在特定场景说出特定对话）
- 成就列表中显示为「???」直到解锁

优点：增加探索乐趣和惊喜感。
缺点：设计成本增加。

**方案 B：全部公开**
- 所有成就条件可见

优点：目标明确，玩家有追求方向。
缺点：少了神秘感。

推荐方案：**A**。建议隐藏成就占 20%（约 5-8 个），在成就列表底部标注「??? 未发现的秘密」。

### 决策 15：成就奖励

**方案 A：纯收集（推荐）**
- 仅展示成就计数和完成度百分比
- 无游戏内奖励

优点：设计简单，不会影响游戏平衡。
缺点：对部分玩家缺乏驱动力。

**方案 B：有奖励**
- 每个成就提供小额元石/蛊虫/经验
- 完成度累计解锁特殊称号

优点：更多正反馈循环。
缺点：需要调平衡。

推荐方案：**B**。给每个成就提供微量元石奖励（如 10-50 元石），完成度解锁特殊称号是很好的社交货币。

## P2-3 新手引导

### 决策 16：引导触发策略

**方案 A：仅首次触发**
- 检测 localStorage 中 `tutorial_complete` flag
- 老玩家重新开始也不会再看到引导

优点：不打扰回锅玩家。
缺点：忘记操作时没有回顾途径。

**方案 B：每次新游戏都触发但可跳过（推荐）**
- 默认弹引导
- 右上角有明显的「跳过引导」按钮
- 引导完成后在设置中可重新查看

优点：兼顾新手和老玩家，可回顾。
缺点：每次开始多一步操作。

推荐方案：**B**。

## P3 远期规划

### 决策 17：P3 启动时机

P3（宝黄天拍卖/蛊仙升炼/多人联网）建议在 P1+P2 全部完成且玩家反馈稳定后启动。预估时间线：

- P1 全部完成：3-4 周
- P2 全部完成：2-3 周
- 内部测试 & 迭代：1-2 周
- P3 启动：约 2 个月后

P3 的最终形态与前期玩家反馈强相关，不建议现在做过多详细设计。当前大纲保留架构方向即可。

### 决策 18：优先开发顺序

在 P1 内部的模块执行顺序，建议：

**顺序 A（推荐）：章节弧光 → NPC交互 → 战斗系统**

理由：
1. 章节弧光改动最小（新增 `chapter.json` + 修改 `context-builder`）、效果最大（立即给玩家推进感）
