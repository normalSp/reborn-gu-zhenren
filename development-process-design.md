## 1. 文档目的

本文档定义蛊真人世界·人生重来模拟器从当前状态到项目完工的完整开发流水线。本流程遵循专业游戏开发的标准阶段模型，同时适配本项目的三个特殊性：AI驱动的动态叙事、纯前端技术架构、以及基于31次API调用已验证的技术决策。

本文档面向项目开发者和协作者，回答「下一步做什么、怎么做、做到什么程度算完成」这三个核心问题。

## 3. 阶段详解

### 阶段 -1: 概念验证 (已完成)

**什么是概念验证**

在投入大量编码之前，先验证最大的技术风险是否可控。对本项目而言，最大的风险是「DeepSeek V4 Pro能否稳定生成符合蛊真人世界观的AI叙事，并且输出格式可被前端可靠解析？」

**做了什么**

进行了三回合共31次API调用的递增测试。第一回合建立基线（窄关键词，15次调用，一致性71%），发现两个问题——NPC叙事偏温和、方源人设描写不足。第二回合注入Few-shot示例和自检协议做A/B对比（8次调用），发现核心洞察：窄关键词检查系统性误判了AI的叙事质量。第三回合升级评测体系为宽词簇语义评分（8次调用），一致性推至88%，同时确认自检文本指令对LLM无效。详细过程见 `prompt-optimization-ceiling-guide.md`。

**决策门禁 — 是否通过？**

通过。AI引擎的关键指标全部达标：JSON合规率100%（零解析失败）、记忆持久性100%（5轮后准确回溯）、世界观一致性88%（Prompt层天花板）。确认了「Prompt优化到88%后应停止，转入代码层四层策略」的技术路线。

**对应 game-dev-text 工作流阶段**

概念验证处于 game-dev-text 的「游戏类型识别」和「技术栈确认」阶段。本项目的游戏类型被确定为蛊真人专属模式（SIM核心+VN叙事+RPG成长），技术栈选定React+TypeScript。

### 阶段 0: 概念设计 (已完成)

**什么是概念设计**

从「这个能做」到「做成什么样」的阶段。输出GDD（概要设计文档），定义产品范围、核心系统架构、数据模型、UI范式和分阶段路线图。

**做了什么**

编写了完整的GDD（`game-design-document.md`），覆盖：产品定义与5项可衡量成功指标、四层核心循环设计、12个子系统的完整架构总览、AI叙事集成方案（AINarrativeEngine接口、三层记忆架构、双通道输出协议、四层一致性策略）、蛊虫GuSpec/GuInstance双表数据模型、账簿式UI设计范式、技术选型与6项关键架构决策、5阶段12-16周迭代路线图。

同时编写了两个System Prompt文件：Layer 1 世界观规则（含5组Few-shot示例）和 Layer 2 输出格式协议（含JSON Schema和禁止事项）。

**决策门禁 — 是否通过？**

通过。12个子系统全部有明确定义的接口、数据模型和实现路径。没有任何子系统需要「摸着石头过河」——每个系统都可以从 game-dev-text Skill 复用60-90%的底层代码。

**对应 game-dev-text 工作流阶段**

概念设计对应 game-dev-text 的「按能力域加载参考指南」阶段。蛊真人模式要求四份参考文档全部加载：narrative-design（AI叙事集成）、number-balance（蛊虫/战斗数值）、interaction-logic（FSM状态机+事件驱动）、game-type-patterns（蛊真人专属系统模板）。

### 阶段 1: 技术设计 (当前阶段)

**什么是技术设计**

在GDD确定了「做什么」之后，TDD确定「怎么做」。不重复GDD中已经说清楚的功能定义，而是深入到实现级细节——精确的TypeScript类型、状态机转移规则、公式参数、缓存策略。

**本阶段的产出物**

- `development-process-design.md` (本文档)：开发流程全景设计
- `technical-design-document.md` (TDD)：三份核心技术设计
  - AINarrativeEngine 技术设计：Zod Schema字段级定义、响应管道状态机、Layer 3评分算法、Layer 2触发逻辑
  - Zustand Store 架构设计：12 slice完整类型、persist配置、依赖矩阵、middleware链
  - 数据流与错误处理架构：时间序数据流、React Query缓存键策略、UI三态管理

**决策门禁**

TDD必须回答以下问题才能通过：

其一，AINarrativeEngine的响应管道是否覆盖了所有异常路径？JSON解析失败 → Zod校验失败 → Layer 3语义不通过 → Layer 2裁判不通过 → API限流/网络错误 → 降级模式。每一条异常路径必须有明确的处理策略。

其二，Zustand Store的12个slice之间的依赖关系是否清晰？当一个slice更新时，哪些slice需要同步刷新？persist中间件的白名单是否排除了纯UI状态（如当前Tab、加载动画状态）？

其三，React Query的缓存键设计是否能避免「AI叙事还在加载中，但缓存命中了旧数据」的错误？

**估算耗时**: 0.5周

### 阶段 2: 预生产/原型验证 (第1-3周)

这是编码阶段的起点。目标不是在3周内完成12个子系统，而是实现「最小可玩核心循环」——从开局到AI叙事到玩家选择到状态更新到面板刷新的完整闭环。

**阶段2A: 项目脚手架 (第1周)**

搭建React+TypeScript+Vite项目骨架，配置Tailwind CSS和shadcn/ui，实现12个Zustand slice的空壳，封装DeepSeek API调用函数（含重试和错误处理），实现API Key输入界面。

里程碑：运行 `npm run dev`，看到账簿式标题画面，输入API Key后调用DeepSeek返回「Hello World」。

**阶段2B: 核心游戏循环 (第2-3周)**

这是项目最关键的两周。实现从玩家点击选择按钮到AI生成叙事到状态更新到面板刷新的完整闭环。

工作内容：构建三层上下文（System Prompt + 玩家状态JSON + 叙事上下文）、实现AINarrativeEngine的响应管道（JSON解析 → Zod校验 → Layer 3语义打分的简化版）、基础属性系统（四大属性+境界管理+生命真元）、天赋选择界面（6-8个精简天赋）、叙事面板+选项面板+打字机效果、前端语义规则引擎Layer 3的基础版（核心规则）、Zustand persist中间件的基础存档。

里程碑：可以完整游玩开窍到青茅山的章节，所有数值跟随AI叙事变化。

**game-dev-text React模板复用策略**

阶段2B是复用 game-dev-text React模板最密集的阶段。以下模块可直接或适配使用：

- `useGameState.ts` → 80%直接复用，作为响应管道状态机的基础骨架。其dispatch/guard/onEnter/onExit模式完美对应管道的状态转移。需要扩展的是：新增「retrying」「degraded」两个状态，以及对应API调用失败后的转移规则。
- `useAttribute.ts` → 90%直接复用，其Modifier系统（add/multiply/set/cap_min/cap_max/percent_add/percent_set）完整覆盖蛊师属性的修正需求。需要新增的是境界维度和道心四维。
- `TextRenderer.tsx` → 95%直接复用，打字机效果组件。唯一需要适配的是将叙事text的渲染与Layer 3语义规则引擎的校验状态联动（通过/不通过的颜色标记）。
- `ChoicePanel.tsx` → 70%适用，需要新增风险等级的颜色编码（rg-blood红色高风险、rg-jade绿色低风险）和risk_note tooltip。
- `DialogueBox.tsx` → 60%适用，需要适配DeepSeek返回的JSON驱动格式（非预设ScriptNode格式）。

总体估算：game-dev-text React模板的13个文件中，7个可高度复用（>75%），4个需中等适配（50-75%），2个需大幅修改。约节省40%的初期代码量。

**决策门禁**

是否能完整玩一场从开窍到青茅山覆灭的游戏？AI是否能稳定输出符合JSON Schema的响应？Layer 3语义引擎的基础规则是否能捕获明显的世界观违规？

**阶段3A: 可视化系统 (第4-6周)**

## 4. game-dev-text 工作流与本项目的映射

game-dev-text 技能定义了一个三步工作流：识别游戏类型 → 确认技术栈 → 按能力域加载参考指南。本项目的执行路径如下：

**游戏类型**: 蛊真人模式。从需求中检测到「蛊真人」「蛊虫」「杀招」等关键词，触发蛊真人专属系统模板（GuSpec/GuInstance数据模型、九转境界体系、48流派+道痕互斥）。底层架构采用混合模式——SIM核心提供AP/时间/事件循环，VN叙事层提供AI驱动的动态文本生成，RPG成长系统提供属性/境界/Build/战斗。

**技术栈**: React+TypeScript。选择理由：12个可视化面板需要组件化独立更新，TypeScript的类型系统适合数据模型定义，Vite的HMR适合快速迭代，Zustand的persist中间件直接对接localStorage存档。

**参考指南加载**: 蛊真人模式要求四份参考文档全部加载，对应本项目各阶段的需求：narrative-design（阶段1-2，AI叙事引擎）、number-balance（阶段3，战斗/仙窍数值）、interaction-logic（阶段2，FSM状态机+事件驱动）、game-type-patterns（阶段0-3，蛊虫/流派/境界系统模板）。

## 7. 结论与下一步

当前处于阶段1（技术设计）。完成TDD之后，下一步是阶段2A（项目脚手架），这是一个编码密集但方向明确的阶段——所有技术决策已经在TDD中做出，脚手架代码是TDD的直接翻译。

项目预计在12-16周内完成从当前状态到Beta发布的全流程，共5个可玩里程碑，每个里程碑都有明确的验证标准。
