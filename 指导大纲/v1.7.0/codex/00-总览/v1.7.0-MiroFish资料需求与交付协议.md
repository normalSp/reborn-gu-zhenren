## 何时升级 blocking

以下任一项出现，MiroFish 需求升级为 `blocking`：

- 正式 topic-slice 要进入 a2 intake。
- 需要命名地点、命名 NPC、正式势力规则。
- 触碰 hidden-adjacent、方源公开证据、后期高阶事实。
- 任意内容准备进入知识索引、测试样本、rule draft 或 canon draft。
- DeepSeek 可见摘要或 prompt context 候选。

## 双仓自动检查

v1.7 起使用只读检查：

```powershell
npm run check:mirofish-dual-repo-pipeline -- --target-version=v1.7.0 --topic=southern_border_low_rank_outer_edge_life_slice --stage=a1
```

阶段：

- `a1`：检查基础包、双仓路径、制度和 MiroFish export 工具入口。
- `request`：检查 RebornG request 已写明 topic。
- `export`：检查 RebornG 收到 export-ready 包、handoff、coverage，且 JSON 无原文字段。
- `intake`：检查 intake review 与晋升分流。
- `complete`：检查测试矩阵或规则草案引用链。

脚本只读，不复制、不修复、不晋升 runtime canon、不修改 DeepSeek context。

## 子代理提速规则

v1.7-a2 第一条样板稳定前，不启用子代理写文件。样板稳定后，由主线程向用户提交“只读/分析型子代理”的收益和风险，再由用户决定是否启用。

允许的子代理范围只能是只读扫描 source pointer、coverage gap、request 覆盖度和 intake/test 引用链；禁止子代理直接写 runtime、`src/canon`、DeepSeek context、同一个 request/intake 文件或执行 stage/commit/push。
