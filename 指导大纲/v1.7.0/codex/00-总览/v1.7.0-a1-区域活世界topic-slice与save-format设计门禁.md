## 当前架构判断

当前 RebornG 已经具备“同开局叙事表达可能不同”的基础：

- DeepSeek 使用 `deepseek-v4-flash`，叙事温度比战斗更开放。
- prompt 会注入玩家状态、近期事件、历史摘要、位置、蛊虫、势力和行动回流事实。
- `livingWorldState`、`routeLocationState`、`survivalEconomyState`、社会/区域/冲突 projection 已经能承接公开证据。
- World Intent Engine 能把自由输入降级为目标、前置、阻断、压力或候选。

但当前还没有真正完成“同开局、同初始路线也能长期玩出不同低阶人生”的系统能力。原因是：为了防止剧情漂移和 DeepSeek 越权，v1.1-v1.6 大量系统都保守地走本地确定性与 projection-first。这个选择是正确的，但 v1.7 开始需要把“差异度”纳入长期设计，而不是只让 DeepSeek 换一种写法。

## 推荐 topic-slice

推荐 slice id：

`southern_border_low_rank_outer_edge_life_slice`

玩家可见范围：

- 山路外缘公开压力。
- 临时商队接触窗口。
- 散修落脚与身份风险。
- 城门外缘门槛。
- 低阶补给、询价、拒绝、绕路、被盘问、欠人情等非胜利推进。

读取来源：

- `routeLocationState`
- `survivalEconomyState`
- `livingWorldState`
- local action ledger / narrative return context
- v1.3 social projection
- v1.4 region projection
- v1.5 conflict projection
- v1.6 report-only 工具输出与 MiroFish source-pointer inventory

允许输出：

- 区域生活态势 projection。
- 今日压力 / 近期风险 / 下一步候选。
- 低阶失败推进说明。
- 可测试的 event deck / pressure deck / replay sample。

禁止输出：

- 正式地点解锁。
- 正式区域状态。
- 正式商队、散修、城门、势力身份写入。
- 正式交易成交、价格表、库存、奖励。
- NPC 生死、捕获、背叛、永久伤势。
- hidden/private body 或方源私密因果。
- DeepSeek visible full-book summary / RAG。

## save-format 评估

| 方案 | 内容 | 专家团建议 |
|---|---|---|
| A | 保持 `SAVE_FORMAT_VERSION = 24`，b1 projection-first | 推荐 |
| B | 新增 `regionalLifeState` / `areaLivingState`，bump v25 | 不推荐进入 b1，除非 D-171 后用户单独批准字段契约 |
| C | 新增 per-save `runFingerprint` / `worldSeed` | 远期可能需要；v1.7 先设计测试，不直接进存档 |
| D | 用 DeepSeek 自由生成差异 | 禁止作为事实差异来源 |

推荐理由：

- v1.7 的首要目标是让 v1.1-v1.6 地基合流，而不是立刻扩 save schema。
- 当前已有三类持久地基：路线、低阶生存、活世界公开证据。b1 可以先做只读组合态势。
- 同开局差异度需要先证明“差异不导致事实漂移”，再决定是否持久化 seed 或事件账本。

## 同开局可重玩差异度

定义：

同一开局、同一初始身份、同一公开事实边界下，多次开始游戏时，玩家应看到不同的低阶生活压力组合、候选入口和叙事质感；但正式事实、原著边界、身份、奖励、地点、hidden 和 NPC 生死必须稳定。

v1.7-a1 将它纳入三层设计：

| 层级 | v1.7 做法 | 未来演进 |
|---|---|---|
| 测试层 | 新增 V17-A1-DIVERSITY 样本，同开局多次 probe 检查差异与稳定边界 | v1.8 起形成固定 replayability eval |
| runtime 设计层 | b1/b3 若获批，可设计本地 event deck / pressure deck；不直接写持久字段 | v1.8-v1.9 评估 per-save `runFingerprint` |
| 活世界层 | v1.7 只验证一个小 slice 的差异 | v2.0 区域活世界应支持 300+ 总轮次 long-run，含 live/mixed 差异检查 |

差异允许来自：

- 本地公开压力权重。
- 事件牌组顺序。
- 低阶失败推进类型。
- NPC/势力注意窗口的公开表现。
- 补给、绕路、盘问、交易拒绝、同行风险等候选。
- DeepSeek 对已给定公开压力的叙事表达。

差异禁止来自：

- DeepSeek 自行创造正式奖励、地点、阵营、NPC 生死。
- hidden/private 泄露。
- 未经 intake 的 MiroFish 内容。
- 随机改写玩家身份、境界、库存、路线事实。
- 让同一存档内前后矛盾。

## 测试与 live probe 设计

最低建议：

| 阶段 | 门禁 |
|---|---|
| a1 | 文档门禁、测试矩阵更新、MiroFish need level、save-format 决策项 |
| a2 | MiroFish / knowledge checker + intake review；只晋升 RebornG-owned test sample / rule draft |
| b1 | focused unit、old-save、hidden/DeepSeek authority、30 轮 Player Advocate、T0 7 日 deterministic replay、标准 `deepseek-v4-flash` live smoke |
| b3 | 若加入事件牌组/失败推进，增加同开局差异度 replay 与 live samples |
| process-2 | 复核 `seeded_runtime_probe` 是否必须工具化；至少记录执行方式或豁免理由 |
| rc | 100+ 轮 Player Advocate；`deepseek-v4-flash` 40-60 轮 live，高风险升 80-120 |

关于 `seeded_runtime_probe`：

- 当前已有 `api_sample_probe` 与 replay/deterministic 骨架。
- 通用 seeded runtime runner 还没有完全工具化。
- 专家团建议：v1.7 不把它作为 a1 前置，但应在 b1/process-2 前决定是否做最小工具化。若要测试“同开局可重玩差异度”，seeded runtime probe 会比纯 API sample 更可靠。
