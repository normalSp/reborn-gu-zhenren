# 长线叙事漂移测试制度

日期：2026-05-20
状态：项目级制度；v1.1-process-2 建立第一版
用途：防止 RebornG 在长回合、长文本、长世界线中出现剧情逻辑衰退、世界观漂移、DeepSeek 越权、玩家目标遗失或原著边界污染。

## 背景

长线 AI 跑团常见失败不是单纯因为模型上下文不够大，而是因为世界事实、角色记忆、玩家愿望、摘要、大纲和模型补全混成同一层权威。RebornG 的基本解法是：

- 本地 canon / engine / store 保管世界事实、奖励、地点、战斗、NPC 生死、隐藏事实和存档。
- DeepSeek 只写叙事、候选、线索、传闻、请求和压力。
- MiroFish 只提供 quote-redacted 候选材料，必须经过 intake review 和 RebornG 转写。
- 长线测试必须检查“本地事实是否稳定”和“叙事是否尊重本地事实”，不能只看 UI 是否能打开。

## 适用范围

以下任一情况触发本制度：

- runtime 小版本触碰路线、地点、区域、NPC、势力、经济、战斗后果、自由意图或 DeepSeek 叙事。
- 阶段中出现新的原著/IF 边界样本、DeepSeek 漂移样本、Player Advocate 困惑或 MiroFish intake 风险。
- rc、公开发布前、v2.0 区域活世界前。
- 用户要求评估长线剧情质量、世界观稳定或原著贴合度。

纯文档、CI、Git、安全清单阶段可豁免，但必须在阶段记录中写明“本阶段不触发长线叙事漂移测试”的原因。

## 测试分层

| 层级 | 名称 | 是否 live DeepSeek | 建议轮次 | 目标 |
|---|---|---:|---:|---|
| T0 | deterministic soak | 否 | 100-300 | 本地状态、路线、账本、UI 摘要和禁写边界不漂 |
| T1 | short live probe | 是，标准档默认使用 `deepseek-v4-flash` | 20-40 | 检查 DeepSeek JSON、术语、越权和隐藏事实泄漏 |
| T2 | long narrative soak | 可 live 或 replay；高风险 rc 默认 mixed/live | 80-150 | 每 20/25 回合审查剧情连续性、世界状态一致性和目标保留 |
| T3 | region life soak | 可 live 或 replay；v2.0 前必须 mixed/live | 300+ | v2.0 前验证一个区域活世界长期运行 |

默认 CI 只能跑 deterministic / replay。当前标准 live 模型固定为 `deepseek-v4-flash`；标准档成本不再作为阻断理由，但仍必须记录模型、样本、轮次、token/缓存命中、费用和报告路径。任何模型切换、超出标准档的采样规模扩大、DeepSeek 可见知识摘要或 RAG 接入，都必须单独停下来让用户决策。

## Checkpoint 审查项

每个 checkpoint 至少记录：

| 项 | 检查问题 | 失败例 |
|---|---|---|
| 世界状态 | 当前路线、地点、区域、目标、压力是否与本地 store 一致 | 文本说已到商家城，但 store 仍是候选路线 |
| 权威边界 | DeepSeek 是否写了正式奖励、地点、阵营、NPC 生死、隐藏事实 | 叙事直接授予仙蛊或判定白凝冰死亡 |
| 原著边界 | 是否出现凡人宝黄天交易、九转蛊、盗天传承直达、方源隐藏因果泄漏 | 一转玩家获得宝黄天正式入口 |
| 连续性 | 玩家目标、最近行动、公开后果是否被保留 | 前 20 轮逃离青茅，后 40 轮忽然回到族学无解释 |
| 口感 | 是否空转、重复、机械、爽点错位或不像蛊真人 | 连续多轮只有泛泛“继续修炼” |
| 可读性 | 玩家能否知道下一步、风险、缺口和可执行入口 | 只给诗意文本，没有行动缺口 |

## 通过标准

小版本通过条件：

- P0/P1 漂移为 0。
- P2 漂移必须有修复、阻断、降级或测试矩阵分流。
- 新增样本必须按 `测试矩阵演进规则.md` 三分流。
- live probe 若未跑，必须说明未跑原因，不能把未验证项写成已验证。

rc 通过条件：

- T0 必跑。
- T1 标准档 live 默认跑，模型为 `deepseek-v4-flash`。
- T2 是否 live / mixed 由当前版本风险决定：普通 rc 可 40-60 轮 live；高风险 rc 建议 80-120 轮 live/mixed。
- 若 runtime rc 不跑 live，必须写明豁免原因，并用 replay / fixed eval 覆盖 DeepSeek 越权样本。
- Player Advocate 记录必须包含“叙事/世界观漂移检查”栏或引用本制度的 checkpoint 记录。

## 工程化要求

v1.1-process-2 建立制度和记录格式。v1.1-b1 起逐步补脚本接口：

- `check:long-narrative-drift-record`：验证记录是否包含 checkpoint、分流表、live 成本口径和结论。
- `run-long-narrative-drift`：v1.6 内容/长测工厂再正式实现，可生成 replay/live 报告。

脚本未实现前，阶段完成记录必须人工按本制度列清 checkpoint。脚本实现后，阶段完成必须跑脚本或写明豁免。

## 与其他制度关系

- Player Advocate：检查玩家是否看懂；本制度检查长线事实和叙事是否漂移。两者互补。
- 测试矩阵演进：本制度发现的漂移样本必须三分流。
- MiroFish intake：MiroFish 候选只能成为样本或规则草稿，不能直接作为叙事真相。
- 全书知识库治理：知识库提供审查索引，不直接喂给 DeepSeek。

## 版本节奏

- v1.1-process-2：制度、模板、知识库骨架、脚本接口。
- v1.1-b1：T0 route/location soak。

- v1.1-rc：T0 必跑；T1/T2 live 需用户批准。
- v1.2-v1.5：按经济、NPC、区域、冲突增长漂移样本。
- v1.6：长测工厂工具化。
- v2.0 前：T3 区域活世界 300+ 回合硬门。
