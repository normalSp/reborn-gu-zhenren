# 前置批量审批与例外停机制度

日期：2026-05-23
状态：项目级流程制度；从 v2.5.0-a0 起试运行

## 目标

把“每个小版本都需要用户审批”升级为“启动会或 a1 一次性审批全版本授权包，后续只在例外停机条件触发时回到用户”。目标是降低用户被流程碎片打断的频率，同时保留 RebornG 对 save、DeepSeek、原著、MiroFish、外部依赖、后端和公开发布的硬边界。

本制度不减少阶段记录、测试、Skill 同步、Git/CI 或 handoff。它只改变用户审批的时机和颗粒度：能前置批量讲清楚的，前置一次性审批；真正改变方向或越过边界的，仍必须停机。

## 适用范围

适用于从 v2.5.0 开始的每个大版本、小版本和 standalone 专项，尤其是：

- 专家团启动会。
- a0 范围冻结。
- a1 设计门禁。
- 用户明确希望“后续没有新决策就一个 `/goal` 跑完”的版本。

纯问答、临时解释、已经明确要求“先不用动手”的讨论，不强制进入本制度。

## 前置授权包

每个版本的专家团启动会或 a1 设计门禁必须给出一个“前置授权包”。授权包至少包含：

| 项 | 必填内容 |
|---|---|
| 全版本阶段路线 | a0/a1/a2/b1/b2/b3/process/rc 分别做什么、哪些可省略、哪些必须保留 |
| 预计决策项 | 按 ID 列出建议批准、建议暂不批准、必须未来另批的事项 |
| 自动继续范围 | 只要不越过授权包，Codex 可在 `/goal` 中继续完成设计、实现、测试、文档、提交、推送和 CI |
| 例外停机清单 | 触发后必须停止自动推进并向用户说明 |
| 负授权说明 | 明确“本次批准不等于批准什么” |
| 测试和证据计划 | Player Advocate、live probe、runner、CI、扫描、MiroFish checker、文档自检等 |
| Git 与交接计划 | 分支名、提交边界、推送、CI、handoff、skill sync audit |

授权包必须具体到工程动作和边界，不得只写“按计划做”“继续开发”这类抽象口径。

## 自动继续规则

用户批准前置授权包后，Codex 可以自动继续完成授权包内的后续阶段，条件是：

1. 当前工作仍在授权包声明的范围内。
2. 没有触发例外停机条件。
3. 所有新增样本已进入测试矩阵演进规则。
4. Skill 同步审计按阶段记录。
5. Git 分支、显式 stage、提交、推送和 CI 按制度执行。
6. 每个阶段仍产出对应记录；只是无需每个阶段再次向用户要审批。

如果用户在中途提出新方向，以用户最新消息为准；必要时重开授权包。

## 例外停机清单

以下情况不能被前置批量审批吞掉，必须停下来让用户决策：

| 类别 | 停机条件 |
|---|---|
| Save / 存档 | 新 save field、`SAVE_FORMAT_VERSION` bump、迁移策略改变、正式存档兼容风险 |
| DeepSeek | prompt/context/model/authority 扩大，DeepSeek visible knowledge/RAG，或 runtime 模型切换 |
| live probe | 超出已批准档位，成本/样本/轮次明显变化，或出现 P0/P1 |
| MiroFish / 原著 | 真实原著事实、命名 NPC、hidden-adjacent、方源公开证据、正式 lore 结论、hidden/private body |
| 基础设施 | backend/BFF/service、private canon service、eval archive service、job queue、cloud save |
| 外部能力 | 外部依赖、framework PoC、vendored subset、read-only scan、patch artifact、subagents、外部 agent 文件/命令/git 权限 |
| 正式游戏事实 | 正式地点、阵营、身份、奖励、NPC 生死、canon promotion |
| 质量门 | CI/测试连续失败且需要改变方案，或必须降低测试覆盖才能继续 |
| 发布和合规 | 成本异常、公开发布、部署、版权/法律边界、EdgeOne 自动部署 |
| 用户体验方向 | 玩法口感、叙事方向、难度、节奏、世界观取舍出现明显新分歧 |

触发停机时，Codex 必须给出：发生了什么、为什么越过授权包、可选方案、推荐方案、风险收益和需要用户拍板的具体 D 项。

## 不需要停机的情况

以下情况只要仍在授权包内，可以自动处理并记录：

- 文档措辞微调、入口同步、handoff 更新。
- 纯 report-only schema / checklist / risk model 的补全。
- 已批准范围内的测试矩阵补样本。
- 不改变方案的 P2 文案/术语/可读性警告。
- CI 因临时网络、依赖源 5xx、缓存等非项目问题失败，且无需改方案即可重跑或记录。
- Skill 同步中 `no_update_needed` 或按授权包更新 current override。

## 阶段记录要求

批量审批不等于省略阶段记录。每个阶段仍需按实际情况更新：

- 当前版本 README。
- 需求决策池或授权包记录。
- 测试矩阵。
- 真相源索引。
- Git 提交与推送计划。
- Player Advocate / live probe / MiroFish / Skill sync 相关记录。
- 项目仪表盘、PROJECT-STATE、AGENTS。
- 最新 handoff。

## D 项写法

从 v2.5 起，建议把决策池拆成三类：

| 状态 | 含义 |
|---|---|
| `approved_in_frontloaded_pack` | 用户已在启动会/a1 授权包中批量批准，后续可自动继续 |
| `future_gate_required` | 已识别但不批准；触发时必须重新找用户 |
| `blocked_or_rejected` | 明确不做，除非用户后来反转 |

历史阶段可以继续保留旧 D 项；新阶段优先使用这三类状态。

## `/goal` 执行口径

用户说“完成 vX.Y”或进入 `/goal` 后，Codex 应按以下顺序执行：

1. 读取当前版本授权包和例外停机清单。
2. 确认当前需求在授权包内。
3. 从当前未完成阶段继续推进，不重新问已批事项。
4. 每完成一个阶段记录证据和状态。
5. 触发例外停机时立即暂停并汇报。
6. 未触发例外时完成到 rc、提交、推送、CI 和 handoff。

## 试运行与复盘

v2.5.0 是本制度试运行版本。v2.5 rc 必须复盘：

- 审批次数是否减少。
- 是否出现误自动推进。
- 例外停机是否足够清晰。
- 用户是否仍能掌握方向。
- 是否需要把制度升级为硬门或调整停机清单。

复盘通过后，本制度继续作为 v2.6+ 默认流程。

## 硬边界

本制度不授权任何具体 runtime、backend、DeepSeek、MiroFish、save、外部依赖、正式地点/阵营/奖励/NPC 生死、canon promotion 或 EdgeOne 部署。它只是审批流程制度。任何实体能力仍必须由当前版本授权包或用户单独决策批准。
