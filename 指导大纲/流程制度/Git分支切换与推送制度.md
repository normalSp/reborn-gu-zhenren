# Git 分支切换与推送制度

日期：2026-05-20
状态：项目级制度；用户已批准“每次进入新的小版本先切分支”

## 目标

让每个小版本、专项和资产晋升都有清晰分支语义、提交边界和回滚点。用户只需要提出需求或批准方向，Codex 必须在开工前把分支、脏区、提交和推送路径整理好。

## 分工

- 本制度负责：开工前主动切语义分支、阶段内明确提交边界、阶段末推送和记录。
- `Git脏区收束制度.md` 负责：已经乱了之后如何分流、清废弃物、按归属收束。

## 触发条件

出现以下任一情况，先检查是否需要新分支：

1. 进入新的小版本或阶段，例如 `a1`、`b2`、`process-1`、`rc`。
2. 进入独立专项，例如 BGM runtime 晋升、MiroFish intake、知识库建设、质量门禁工具。
3. 当前分支名和即将开发的阶段语义不一致。
4. 上一阶段已经提交或推送，下一阶段会写 runtime、docs、资产或测试。
5. 用户明确说“开始开发”“进入下一步”“做专项”。

可以不切新分支的例外：

1. 同一小版本内的紧急修补、测试补证或 handoff 小提交。
2. 用户明确要求沿用当前分支。
3. 当前分支就是本阶段语义分支，且尚未完成/推送。

例外必须写入当前版本 Git 计划或 handoff。

## 分支命名

默认格式：

`codex/v<major><minor><patch>-<phase>-<short-topic>`

示例：

- `codex/v120-b2-survival-ledger-bgm-runtime`
- `codex/v120-rc-quality-closure`
- `codex/v160-stale-entrypoint-checker`

要求：

1. 分支名必须包含版本和阶段。
2. 主题短语必须能看出主范围。
3. 资产专项、制度专项和 runtime 小版本可以共用一个分支，前提是用户同一回合批准且提交边界分开。

## 开工流程

1. 读取当前版本入口、项目仪表盘、PROJECT-STATE、最新 handoff 和当前版本 Git 计划。
2. 执行 `git status --short --branch`，确认当前分支和脏区。
3. 如果有无关脏区，先按 `Git脏区收束制度.md` 分流；不要在错误分支继续堆功能。
4. 从最新已批准基线创建或切换语义分支。
5. 在当前版本 Git 计划记录分支名、基线、阶段范围和禁止混入项。
6. 开始 runtime/docs/assets/test 工作。

## 提交与推送

每个阶段至少需要一个可回滚提交；较大阶段可拆为：

1. docs/design gate。
2. runtime/store/UI。
3. tests/evidence。
4. assets/runtime promotion。
5. process/skill/handoff sync。

提交规则：

1. 禁止 `git add -A`。
2. 只 stage 明确路径。
3. 每个 commit message 写清版本、阶段和边界。
4. 阶段完成后推送当前分支，除非用户明确要求不推送或远端不可用。
5. 不自动部署 EdgeOne，不改 branch protection，不自动合并。

## 主线合并与版本分支制度补丁

日期：2026-05-25
状态：项目级 Git 制度补丁；用户已批准写入

### 定位

每个版本、小版本或专项使用 `codex/v<version>-<phase>-<topic>` 语义分支是 RebornG 的默认工作方式；它负责隔离开发、保留审计证据、降低回滚成本。远端 `main` 是稳定主线，不是日常试错分支。

本补丁补齐“版本分支完成后如何回到稳定主线”的规则，避免只持续开新分支而没有归宿，导致 `main`、当前完成版本、发布基线和新分支基点逐渐脱节。

### 分支职责

1. `main`：稳定主线。只接收已完成、已验证、已记录证据的版本或专项合并；不得直接在 `main` 上做日常开发。
2. `codex/vXYZ-*`：版本、小版本或专项工作分支。承载设计、实现、测试、文档、证据和 CI；每个分支必须有清楚的版本/阶段/主题。
3. `develop` / `integration`：暂不设为默认分支。只有当多人并行、多个长期 runtime 分支互相依赖、或 `main` 合并节奏无法承载集成压力时，才由用户另批。
4. tag：可用于 `vX.Y.Z-rc`、`vX.Y.Z-complete`、正式发布候选或回滚锚点；默认不强制，除非当前版本 Git 计划要求。

### 主线合并准入

版本分支准备合并回 `main` 前，必须同时满足：

1. 当前版本/专项 scope 已完成或明确只是 docs/process-only 补丁。
2. 当前版本 Git 计划、项目仪表盘、PROJECT-STATE、AGENTS 和必要 skill sync audit 已更新。
3. 所有应跑测试已通过；被豁免的测试必须写明原因。
4. 远端 CI 已通过，或远端不可用原因已写入当前版本 Git 计划。
5. 工作区无无关脏文件；若有，先按 `Git脏区收束制度.md` 分流。
6. 没有未解决的 `future_gate_required` 被误当成已开放。
7. 没有自动部署 EdgeOne、自动改 branch protection、自动公开发布或自动合并。

### 合并流程

默认流程：

1. 在版本分支完成并推送后，记录 commit、CI run、测试证据和剩余风险。
2. 由 Codex 给出“是否建议合并 main”的简短结论；若合并会改变发布、公开测试、法律/版权、部署或 branch protection，必须先问用户。
3. 用户批准合并后，优先使用 GitHub PR 或非交互式 merge；保持 commit 历史可追溯。
4. 合并后同步 `main` 的最新 commit / CI / tag 状态到项目仪表盘、PROJECT-STATE、AGENTS 和当前版本 Git 计划。
5. 后续新版本分支默认从最新稳定 `main` 或用户指定的已验证分支拉出；如果从未合并的版本分支继续，必须写明原因。

### 禁止项

1. 不在未获批准时自动合并到 `main`。
2. 不用 `git reset --hard`、force push、history rewrite 解决分支混乱，除非用户明确要求并理解后果。
3. 不把失败 CI、未完成测试、未收束 D/F 决策、未更新文档的分支并入 `main`。
4. 不把 `main` 当作“最新实验态”；实验态只留在语义分支。
5. 不用 `develop` 掩盖缺少 mainline 合并制度的问题。

### RebornG 当前推荐

当前项目保持：

- 日常开发：继续每个版本/专项开 `codex/vXYZ-*` 分支。
- 稳定归宿：把 `main` 定义为“当前已验证稳定主线”。
- 暂不新增 `develop`。
- v3.7 及后续版本完成时，在 rc/质量收束阶段增加“是否建议合并 main”的显式检查；合并本身仍需用户批准或 PR 审核。

## 完成定义

一个小版本或专项完成时必须能回答：

1. 当前分支名是什么，是否符合阶段语义。
2. 本阶段基线 commit 或上游分支是什么。
3. 哪些文件属于本阶段，哪些明确不属于。
4. 运行了哪些验证，哪些被记录豁免。
5. 是否已提交、是否已推送，若没有，原因是什么。
6. 下一阶段是否需要新分支。
7. 本版本/专项是否建议合并回 `main`，若不合并，原因是什么。

完成后同步：

- 当前版本 `Git提交与推送计划.md`
- `指导大纲/项目仪表盘.md`
- `PROJECT-STATE.md`
- 最新 `codex上下文信息/*.md`
- 必要时同步 `AGENTS.md` 与相关 skill
