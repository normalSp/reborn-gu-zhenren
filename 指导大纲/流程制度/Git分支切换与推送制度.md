# Git 分支切换与推送制度

日期：2026-05-20
状态：项目级制度；用户已批准“每次进入新的小版本先切分支”

## 目标

让每个小版本、专项和资产晋升都有清晰分支语义、提交边界和回滚点。用户只需要提出需求或批准方向，Codex 必须在开工前把分支、脏区、提交和推送路径整理好。

## 分工

- 本制度负责：开工前主动切语义分支、阶段内明确提交边界、阶段末推送和记录。
- `Git脏区收束制度.md` 负责：已经乱了之后如何分流、清废弃物、按归属收束。

## 触发条件

出现以下任一情况，先检查是否需要新分支：

1. 进入新的小版本或阶段，例如 `a1`、`b2`、`process-1`、`rc`。
2. 进入独立专项，例如 BGM runtime 晋升、MiroFish intake、知识库建设、质量门禁工具。
3. 当前分支名和即将开发的阶段语义不一致。
4. 上一阶段已经提交或推送，下一阶段会写 runtime、docs、资产或测试。
5. 用户明确说“开始开发”“进入下一步”“做专项”。

可以不切新分支的例外：

1. 同一小版本内的紧急修补、测试补证或 handoff 小提交。
2. 用户明确要求沿用当前分支。
3. 当前分支就是本阶段语义分支，且尚未完成/推送。

例外必须写入当前版本 Git 计划或 handoff。

## 分支命名

默认格式：

`codex/v<major><minor><patch>-<phase>-<short-topic>`

示例：

- `codex/v120-b2-survival-ledger-bgm-runtime`
- `codex/v120-rc-quality-closure`
- `codex/v160-stale-entrypoint-checker`

要求：

1. 分支名必须包含版本和阶段。
2. 主题短语必须能看出主范围。
3. 资产专项、制度专项和 runtime 小版本可以共用一个分支，前提是用户同一回合批准且提交边界分开。

## 开工流程

1. 读取当前版本入口、项目仪表盘、PROJECT-STATE、最新 handoff 和当前版本 Git 计划。
2. 执行 `git status --short --branch`，确认当前分支和脏区。
3. 如果有无关脏区，先按 `Git脏区收束制度.md` 分流；不要在错误分支继续堆功能。
4. 从最新已批准基线创建或切换语义分支。
5. 在当前版本 Git 计划记录分支名、基线、阶段范围和禁止混入项。
6. 开始 runtime/docs/assets/test 工作。

## 提交与推送

每个阶段至少需要一个可回滚提交；较大阶段可拆为：

1. docs/design gate。
2. runtime/store/UI。
3. tests/evidence。
4. assets/runtime promotion。
5. process/skill/handoff sync。

提交规则：

1. 禁止 `git add -A`。
2. 只 stage 明确路径。
3. 每个 commit message 写清版本、阶段和边界。
4. 阶段完成后推送当前分支，除非用户明确要求不推送或远端不可用。
5. 不自动部署 EdgeOne，不改 branch protection，不自动合并。

## 完成定义

一个小版本或专项完成时必须能回答：

1. 当前分支名是什么，是否符合阶段语义。
2. 本阶段基线 commit 或上游分支是什么。
3. 哪些文件属于本阶段，哪些明确不属于。
4. 运行了哪些验证，哪些被记录豁免。
5. 是否已提交、是否已推送，若没有，原因是什么。
6. 下一阶段是否需要新分支。

完成后同步：

- 当前版本 `Git提交与推送计划.md`
- `指导大纲/项目仪表盘.md`
- `PROJECT-STATE.md`
- 最新 `codex上下文信息/*.md`
- 必要时同步 `AGENTS.md` 与相关 skill
