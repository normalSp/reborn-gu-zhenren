# 系统连续性与孤儿系统审计制度

日期：2026-05-26
状态：项目级流程制度；v4.0.0-process-1 建立第一版

## 目标

RebornG 已经从单点功能开发进入长期系统演化：活世界、MiroFish、知识库、Agent Lab、transient AgentProposal、Auto-Theater Combat、HeavenWill/Fate 宏观压力都会跨多个大版本递进。本制度补齐一个横向门禁：每个大版本不能只证明“本版本做完了”，还必须证明“上一阶段产物被正确承接、替代、归档或明确废弃”，避免出现孤儿代码、孤儿脚本、孤儿制度、孤儿测试或只写 TODO 的系统。

本制度不替代前置授权、例外停机、Skill 同步、Git、Player Advocate、长线漂移、MiroFish 或知识库制度。它负责检查这些制度之间的接续关系。

## 适用范围

以下情况必须执行本制度：

1. 新大版本启动会或 a0 范围冻结。
2. 从 report-only / design-gate 进入 runtime first cut 前。
3. 新增跨版本制度、长期路线、checker、eval runner、UI tab、engine layer 或存档字段时。
4. rc 质量收束，尤其是准备进入下一个大版本前。
5. 用户询问“路线有没有接上”“代码有没有漏需求”“有没有孤儿系统或 TODO”。

纯解释问答可豁免；但如果结论进入版本路线、制度或授权包，仍要落审计记录。

## 审计对象

| 类型 | 审计问题 |
|---|---|
| 版本路线 | 当前版本是否明确承接上个版本完成态、长期路线和用户 D/F 决策 |
| runtime / engine | 新 helper、engine、store、UI 是否被当前入口引用；若不再 active，是否标明 historical / report-only / remove candidate |
| scripts / package | `package.json` script 是否指向真实文件；新增 runner 是否有样本、报告、文档引用 |
| tests / fixtures | 新测试是否进入 focused gate、long e2e、PA 或测试矩阵；旧 fixture 是否仍有用途 |
| docs /制度 | 新制度是否进入流程 README、仪表盘、PROJECT-STATE、AGENTS、当前版本路线图和 Skill audit |
| Skill | 相关 skill 是否 current override 同步；是否把历史状态堆进 skill 导致选择失败 |
| artifacts / reports | 报告是否有当前文档指针；是否被错误当成 runtime 权威 |
| future gates | 未批准能力是否仍保持 `future_gate_required`，没有被新代码绕开 |

## 分类标签

每个跨版本产物应能归入以下一种：

| 标签 | 含义 | 允许状态 |
|---|---|---|
| `active_runtime` | 当前游戏运行时正在使用 | 必须有 source/import/UI/store/test 证据 |
| `active_gate` | 当前或下一版本门禁正在使用 | 必须有 script/test/doc/矩阵证据 |
| `report_archive` | 已完成 report-only 证据 | 必须有报告路径和历史入口，不得被当成 runtime |
| `planning_baseline` | 长期路线或设计基线 | 必须写明不授权 runtime/save/DeepSeek 等 |
| `historical_reference` | 历史来源或旧版本证据 | 必须有 historical-index 或当前入口指针 |
| `superseded_with_pointer` | 已被新文件替代 | 必须写替代文件、影响范围和不删除原因 |
| `remove_candidate` | 可清理候选 | 必须确认无 runtime/test/docs/handoff 依赖，并由用户另批删除 |
| `blocking_orphan` | 无入口、无测试、无归属且可能误导 | 不允许阶段完成 |

禁止让产物停在“大家记得它有用”的状态。

## 最小审计清单

每次审计至少回答：

1. 当前版本的上游输入是什么，是否有 commit / CI / report / PA / long-route 证据。
2. 当前新增产物分别属于 `active_runtime`、`active_gate`、`report_archive`、`planning_baseline` 还是其他分类。
3. `package.json` script 是否全部指向存在的文件；新增 script 是否有文档与测试矩阵引用。
4. 旧 engine/helper/test 是否被 import、script、e2e、文档或 historical-index 承接。
5. 当前文档是否还有已过期状态，例如 `pending_user_decision`、`需批准后`、`TODO`、`待补`。
6. 新制度是否进入 `指导大纲/流程制度/README.md`、项目仪表盘、PROJECT-STATE、AGENTS 和 Skill audit。
7. 是否存在未获批 future gate 被代码、脚本、文案或长期路线偷开。
8. 是否需要用户决策：删除、合并 main、开放 runtime、引入依赖、扩大 DeepSeek/MiroFish/save/backend 权限。

检查 package scripts 引用：

```powershell
@'
const fs = require('fs');
const pkg = JSON.parse(fs.readFileSync('package.json','utf8'));
const missing = [];
for (const [name, cmd] of Object.entries(pkg.scripts)) {
  const matches = [...cmd.matchAll(/(?:node|vitest run|playwright test(?: --workers=\d+)?)\s+([^\s&|]+\.(?:mjs|ts|tsx|spec\.ts))/g)];
  for (const match of matches) {
    const file = match[1].replace(/^['"]|['"]$/g, '');
    if (!fs.existsSync(file)) missing.push({ script: name, file });
  }
}
console.log(JSON.stringify({ checkedScripts: Object.keys(pkg.scripts).length, missing }, null, 2));

```

## 阶段记录位置

审计记录应落在当前版本 `codex/00-总览/` 下，推荐命名：

`vX.Y.Z-process-N-开发流程与系统连续性审计.md`

若审计导致新增或修改项目级制度，还必须同步：

1. `指导大纲/流程制度/README.md`
2. `指导大纲/项目仪表盘.md`
3. `.codex/skills/reborn-expert-council/references/PROJECT-STATE.md`
4. `AGENTS.md`
5. 当前版本 `README.md`、路线图、测试矩阵和 Skill 同步审计记录
6. 必要时外部 skill current override

## 通过标准

阶段通过必须满足：

1. 未发现 `blocking_orphan`。
2. 过期状态文本已修正，或写明为何保留为历史。
3. 新增制度已进入所有当前入口。
4. 未批准 future gate 没有被误开。
5. report-only、planning、runtime 三类权威没有混层。
6. 若存在可疑孤儿，只能标为 `remove_candidate` 或 `superseded_with_pointer`，不得静默删除。

## 与减法的关系

本制度允许做减法，但减法必须是正收益：

- 先建替代入口，再标旧入口 historical / superseded。
- 保留证据链，不删除历史报告、PA、CI、MiroFish intake 或 eval artifact，除非用户另批。
- 删除代码或测试前必须证明无 import、无 script、无 doc、无 runtime 和无 handoff 依赖。
- 对长线系统，优先收入口和加索引，不做“看起来旧就删”的清理。

## 硬停

以下情况必须停止自动推进：

1. 发现 package script 引用不存在的文件。
2. 当前版本声称完成某系统，但没有 source/test/doc/PA/checker 任一证据。
3. 新 runtime 代码绕过了已冻结的 WorldCore / save / DeepSeek / MiroFish / future gate 边界。
4. 旧系统被新系统替代但没有替代指针，导致用户或未来 Codex 可能读错入口。
5. 需要删除历史证据、引入依赖、扩大 runtime 权限或合并 main。
