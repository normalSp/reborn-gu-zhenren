MiroFish 基础包是 RebornG 的“全书后勤仓库”和“source-pointer 地图”，不是知识库本体，更不是 runtime 或 DeepSeek 的脑子。

## 三层使用方式

| 层级 | 目录 | v1.6 用法 | 禁止 |
|---|---|---|---|
| 档案层 | `指导大纲/vMiroFish/基础包/` | inventory、coverage、source-pointer、topic-slice 后勤 | 不进 runtime、不进 DeepSeek、不整包导入 |
| 知识索引层 | `指导大纲/知识库/蛊真人/` | RebornG-owned 摘要、visibility、promotionStatus、allowedUses、forbiddenUses | 不存原文、不存 hidden body 可见摘要 |
| runtime canon 层 | `src/canon/*.json` | 未来当前版本门禁批准后的极小安全子集 | 不从基础包直接复制、不批量晋升 |

## v1.6 推荐使用路线

### a1：先定 schema

先定义知识条目、canon 草案、测试样本、intake review 引用链的字段，而不是先吸收内容。

重点字段：

- `id`
- `kind`
- `summary`
- `sourcePointers`
- `visibility`
- `promotionStatus`
- `allowedUses`
- `forbiddenUses`
- `mirofishRefs`
- `testSampleRefs`
- `lastReviewedVersion`

当前状态：D-160/D-161/D-162 均已批准并落地。v1.6 已实现基础包 inventory checker、知识索引边界 checker、intake 晋升链 checker、长测 replay dry-run 和 stale-entrypoint checker，全部保持 report-only，不批量扫描吸收基础包内容，不创建知识索引条目，不进入 runtime canon。

### a2：只做基础包 inventory 门禁

检查：

- 章节包数量和连续性。
- manifest / coverage_report 是否存在。
- 是否有 forbidden text keys，例如 `quote`、`originalText`、`excerpt`、`verbatim`。
- source pointer 数量和坏指针数量。
- hidden/private 字段分布。
- review.status 是否仍为 `needs_review`。

输出 report，不吸收内容。

### b1：基础包 inventory 检查

已落地 `npm run check:mirofish-base-pack-inventory`，只证明 archive/source-pointer inventory 结构通过，不代表知识库/canon/DeepSeek 可见通过。

### b2：知识索引边界检查

检查知识库条目是否具备最小字段，是否把 hidden/private 条目误标为 player_visible，是否把 forbiddenUses 漏掉。

### b3：intake 晋升链检查

检查每个被引用的知识索引或测试样本是否能追溯到：

`基础包/source pointer -> topic-slice intake review -> RebornG-owned summary/rule/test -> 当前版本门禁`

追溯断裂时只能报告，不自动晋升。

### b4：长测样本工厂

从基础包和 intake review 中抽取的材料只能进入：

- deterministic drift sample
- Player Advocate checkpoint
- DeepSeek eval adversarial sample
- future_sample_pool

不能直接进入 live prompt。

### b5：过期入口检查

用基础包无关的项目入口检查脚本，扫描旧 `active draft`、`待批准`、过期版本当前入口等残留。先 report-only。

## 什么时候可以用基础包

允许：

- 查 source pointer 覆盖缺口。
- 生成主题切片 request 草案。
- 生成测试样本候选。
- 为知识索引写 RebornG-owned 摘要草案。
- 检查 hidden/private 风险。

不允许：

- 整包导入知识库。
- 整包喂给 DeepSeek。
- 让 runtime 直接读取基础包。
- 把 hidden/private body 写入 UI、prompt、公开文案或测试可见文本。
- 用基础包直接批准正式地点/阵营/奖励/NPC 生死。

## 什么时候必须停下来问用户

- 要把任何基础包内容晋升到 runtime canon。
- 要让 DeepSeek 看到基础包派生摘要。
- 要把 hidden/private 内容转为玩家可见。
- 要请求新的 MiroFish quote-redacted export。
- 要把工具脚本加入 CI 硬门。
- 要删除或重写基础包原始交付档案。

## 专家团建议

v1.6 应先把基础包“可查、可审、可拒绝、可追溯”做好。等工具和 schema 绿了，后续 v1.7-v2.0 再围绕第一个区域活世界做小范围、强审查、强测试的内容晋升。
