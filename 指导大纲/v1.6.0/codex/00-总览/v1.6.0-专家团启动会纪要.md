# v1.6.0 专家团启动会纪要

状态：已召开；D-160 已批准，当前进入 a1
日期：2026-05-21
主题：内容生产、canon schema 与长测工厂

## 开会背景

v1.1-v1.5 已经把路线、经济、社会、区域和冲突后果都做成了 projection-first / bounded authority 的第一层。现在最大风险不再是“有没有入口”，而是长期内容生产、知识吸收和长线测试会不会重新失控：

- MiroFish 全书基础包已经有 2340 章 source-pointer 档案，但目前只能作为 archive inventory。
- 知识库骨架已经建立，但还缺机械检查。
- 长线漂移制度已经存在，但 replay/eval、样本晋升和报告归档还没有工厂化。
- 旧版本入口已经压缩，但过期 active draft / 待批残留仍靠人工扫描。

v1.6 应该把这些制度变成工程齿轮，而不是急着堆新 runtime 玩法。

## 专家角色意见

| 角色 | 意见 |

| Canon/Knowledge Curator | 全书基础包必须继续分层：vMiroFish 档案层、知识索引层、runtime canon 层。v1.6 第一优先级是 schema、promotionStatus、allowedUses/forbiddenUses 和引用链检查。 |

## 专家团结论

建议 v1.6 主线为：

`内容生产、canon schema 与长测工厂`

推荐完成口径：

- 把全书基础包登记、覆盖、字段边界和 quote-redacted 状态做成可检查 inventory。
- 把知识索引条目的最小 schema 和 forbidden uses 做成机械检查。
- 把 intake review 到知识索引、测试矩阵、runtime canon 的晋升链做成可审计报告。
- 把长线叙事漂移样本、replay、eval、Player Advocate 检查点做成可复用工厂骨架。
- 把历史入口过期残留做成 report-only 检查。

## 不做什么

- 不做全书整包知识库导入。
- 不做 DeepSeek 全书上下文或全书 RAG 接入。
- 不做 runtime canon 批量晋升。
- 不新增存档字段。
- 不开放正式地点、阵营、奖励、NPC 生死、隐藏事实可见化。
- 不删除历史证据文档。
- 不自动部署 EdgeOne。

## MiroFish 基础包一句话

基础包是“后勤仓库”和“source-pointer 地图”，不是游戏运行时的大脑。v1.6 应先让它可审计、可切片、可追踪、可拒绝，再允许任何局部内容进入知识索引或测试样本。
