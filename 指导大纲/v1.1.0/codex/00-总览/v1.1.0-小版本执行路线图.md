## 执行顺序

1. a0 只做文档和讨论，不改 runtime。
2. a1 先审字段和存档，不直接 bump。
3. a2 先 review MiroFish 包，不直接吸收为 runtime truth。
4. b1 只有在用户批准字段方案后才做。
5. b2/b3/b4 只在 b1 地基稳定后推进。
6. process-1 穿插处理旧档、预览和回滚风险。
7. rc 统一收束质量证据。

## Player Advocate 轮次建议

- a0/a1/a2：纯文档/设计阶段可豁免，但需记录豁免。
- b1：若新增存档字段，至少 50 轮，覆盖新档/旧档/极端意图。
- b2/b3/b4：每阶段至少 30 轮。
- rc：至少 150 轮；如果 b1 实际 bump save format，建议提升到 180 轮。

## Git 计划

- a0 启动包单独 commit。
- a1 设计门禁单独 commit。
- a2 MiroFish intake 单独 commit。
- b1/b2/b3/b4/process-1/rc 均单独 commit。
- 不使用 `git add -A`，只 stage 当前阶段明确文件。
- 每个 runtime commit push 后等待 GitHub Actions 通过并补录 run id。
