# v1.1.0 总体开发大纲

日期：2026-05-20
状态：v1.1 本地开发完成；D-001 至 D-024 已批准并落地；D-025 小规模 live probe 已执行并记录

## 主题

`路线、地点与区域状态地基`

## 版本目标

v1.1 的目标不是扩张地图规模，而是建立“正式路线/地点/区域状态”的最小可信地基。

具体来说，v1.1 要解决：

- 玩家从青茅山离开后，系统如何记录路线与地点。
- 旧档如何迁移，不把旧玩家错误传送到新区域。
- UI 如何让玩家理解当前位置、路线压力和下一步。
- DeepSeek 如何叙事，但不拥有地点、路线、阵营、奖励或 NPC 生死权威。
- MiroFish 候选材料如何进入规则草稿和测试样本，而不是直接变成 canon/runtime 权威。
- 长线叙事如何在几十到几百回合后仍不丢目标、不漂移世界观、不让 DeepSeek 写正式事实。
- 全书 MiroFish 基础包如何作为档案层和主题切片来源，而不是直接变成知识索引、runtime canon 或 DeepSeek 大上下文。
- 历史制度和旧版本文档如何通过 `historical-index.md` 导航，而不是塞满当前入口。

## 设计原则

1. 先设计门禁，再写 runtime。
2. 先 route/location 权威表，再 store 字段。
3. 先旧档迁移矩阵，再 `SAVE_FORMAT_VERSION` bump。
4. 先小范围青茅 -> 南疆早期，不开完整南疆。
5. 先玩家可读位置面板，不做全地图。
6. DeepSeek 只写叙事、候选、传闻、压力说明。
7. 正式地点、阵营、奖励、NPC 生死、路线成功/失败必须由本地 engine/store 决定。
8. 长线叙事漂移检查必须进入 runtime 阶段门禁；live DeepSeek probe 需用户另批成本。
9. 知识库实行 `vMiroFish -> 指导大纲/知识库 -> src/canon` 三层晋升链。
10. 历史资料实行 `historical-index.md -> 原始版本目录` 的导航链，不删除原始证据。

## 范围内

- route/location/currentRegion 或等价字段方案评估。
- 单一聚合 `routeLocationState` 方案与 `SAVE_FORMAT_VERSION = 23` runtime 实装。
- 旧档迁移/default/normalization/test 方案。
- 青茅 -> 南疆早期路线状态第一刀。
- 区域状态面板和玩家可读位置说明。
- 前期账本对路线/补给/追踪/身份压力的回流。
- 南疆早期外缘行动承接，不开放完整商家城。
- MiroFish v1.1 包 intake 与候选吸收。
- 全书基础包登记与使用边界：`指导大纲/vMiroFish/基础包/` 只作档案层和 source pointer 后勤。
- Player Advocate 升级轮次和旧档迁移走查。
- `v1.1.0-process-2` 长线叙事漂移与知识库门禁。
- 项目级长线叙事漂移测试制度和全书知识库治理制度。
- `historical-index.md` 历史入口和当前入口展示压缩。
- v1.6 过期入口自动检查脚本规划。
- 知识库骨架，不作为 runtime authority。
- b1 T0 deterministic soak 方案，不执行 live DeepSeek probe。

## 范围外

- 完整南疆、完整商家城、五域大地图。
- 蛊仙期、宝黄天正式交易、尊者线、任意时代开局。
- 正式阵营转移、通缉结论、招揽成功、NPC 生死。
- 正式奖励、掉落池、完整市场/黑市/委托经济。
- BFF/backend、云存档、账号系统。
- 自动 EdgeOne 部署或公开发布承诺。
- DeepSeek 权限扩大。
- live DeepSeek drift probe 默认执行或进入 CI。
- D-025 小规模 live probe；需等 b1/b2 稳定后由用户单独批准成本、模型、样本和轮次。
- 全书知识库直接进入 DeepSeek visible context。
- 全书基础包整包导入知识库、prompt、runtime 或 DeepSeek。
- 删除历史制度/文档，或废弃仍被当前门禁引用的流程入口。

## 版本成功标准

- 玩家能明确看懂“我在哪里、我为什么在这里、我下一步能做什么、有什么风险”。
- 旧档迁移和新档默认状态稳定。
- route/location 不与 `livingWorldState`、行动账本、NPC/势力反应、低阶 life loop 冲突。
- 极端意图仍会被正确降级或阻断。
- MiroFish 内容只进入候选/规则/测试，不进入 runtime 权威。
- 长线叙事漂移样本能进入测试矩阵、Player Advocate 或 checkpoint 记录，不再只靠记忆。
- 知识库材料有 visibility、promotionStatus、allowedUses、forbiddenUses，不混入 runtime canon。
- 全量质量门、关键 e2e、生产预览 smoke 和 Player Advocate gate 通过。

## 落地结果

- 存档格式已提升到 `SAVE_FORMAT_VERSION = 23`，并新增唯一持久地基字段 `routeLocationState`。
- `routeLocationState` 只覆盖青茅到南疆早期外缘的路线/地点/区域 scope，不包含奖励、阵营、NPC 生死、完整商家城或完整南疆开放。
- 写入链路为 local canon/rules -> engine -> store；DeepSeek、UI、MiroFish 和知识库均不拥有路线/地点正式写入权威。
- 已补齐 default、migration、normalization、store action、UI 投影、focused tests、e2e、T0 deterministic soak 与 180 轮 Player Advocate gate。
- 包版本和公开展示口径仍保持 `1.0.0`，直到用户单独批准 v1.1 公开发布文案。
- D-025 小规模 live DeepSeek drift probe 已执行；C27 后 clean-final 阻塞门通过，runtime 已加 L4 C27 防护。

## 主要风险

| 风险 | 说明 | 对策 |
|---|---|---|
| 存档字段过早膨胀 | route/location 字段会影响后续所有系统 | a1 先做字段表和迁移矩阵 |
| 路线候选误变正式地点 | 玩家可能以为已进入完整南疆/商家城 | UI 文案和 engine 输出必须区分路线状态、区域外缘、正式地点 |
| DeepSeek 越权 | 叙事可能写成“你已进入商家城并获得身份” | DeepSeek contract、copy scan、e2e 和 Player Advocate 拦截 |
| 原著事实不足 | 南疆早期外缘和商队/散修压力需要事实支撑 | v1.1 MiroFish 包必须 intake review |
| 旧档误迁移 | 旧档可能缺少路线信息 | migration default 必须保守，旧档默认仍在原状态 |
| 长线叙事漂移 | 多回合后目标遗失、路线前后不一致或文风空转 | `长线叙事漂移测试制度.md`，T0/T1/T2/T3 分层 checkpoint |
| 知识库混层 | 全书基础包或索引条目被误当 runtime/canon/DeepSeek 权威 | `全书知识库治理制度.md`、基础包入库使用计划、三层晋升链和 forbiddenUses |
| 历史入口漂移 | 旧版本流程被误当当前门禁，或当前入口重新堆满旧阶段清单 | `historical-index.md` 和 v1.6 `check:doc-entrypoints` 规划 |
| 流程冗余失控 | 制度越来越多但入口不清 | `流程制度/README.md` 作为入口；减法先收入口、标历史、加替代指针 |

## 与长期路线的关系

v1.1 是 v1.x 的第一块地基。它不负责让世界变大，而是让世界有稳定的位置和路线概念。后续 v1.2 经济、v1.3 社会关系、v1.4 区域扩展、v1.5 冲突深化，都需要依赖 v1.1 的路线/地点状态。

外部活世界参考见 `指导大纲/长期路线/RebornG-外部活世界参考映射.md`。v1.1 主要吸收 ReAct、Reflexion、Voyager 的思想：

- ReAct：自由意图必须转为本地裁决和行动协议，不直接变事实。
- Reflexion：MiroFish intake、Player Advocate 和 bug 经验必须回流测试矩阵。
- Voyager：专家团需求池和测试样本矩阵要像能力库一样持续成长。
