# v1.1.0 Git 提交与推送计划

日期：2026-05-20
状态：v1.1 本地开发已完成；D-025 小规模 live probe 已执行并记录；本轮 commit/push 需显式 stage 当前文件，不能 `git add -A`

## 原则

- 每个小版本独立 commit。
- runtime 与证据/文档可分 commit，但不得混入无关历史脏文件。
- 不使用 `git add -A`。
- 只 stage 当前阶段明确文件。
- 每个 runtime commit push 后等待 GitHub Actions 通过并补录 run id。

## 本轮实际本地变更边界

本轮为了完成 v1.1 本地开发，a2/b1/b2/b3/b4/process-1/rc 在同一工作区连续完成。若提交，建议拆成不超过三组显式 stage：

1. `a2/process/docs`：v1.1 a2、路线图、测试矩阵、仪表盘、PROJECT-STATE、AGENTS、handoff。
2. `runtime-v110-route-location`：`src/canon/v110-route-location-state-rules.json`、`src/engine/v110-route-location-state.ts`、types/store/UI/e2e harness/tests。
3. `rc-evidence`：v1.1 180 轮 Player Advocate 记录和验证补录。

不纳入本轮 commit：

- `指导大纲/vMiroFish/基础包/` 原始基础包。
- 既有美术候选、zip、artifacts、public asset 脏文件。
- 任何 EdgeOne 部署产物。

## Handoff 必填

每个阶段完成后更新：

- `指导大纲/项目仪表盘.md`
- 当前阶段文档
- `PROJECT-STATE.md`
- 最新 handoff/context 文件
- Git commit hash、push 状态、Actions run id
- 如使用长线叙事漂移或知识库制度，记录 checkpoint / 知识库层级 / live 成本是否获批
- a2 完成前不得创建 b1 runtime commit；b1 若执行 live probe 必须另有用户批准记录
- 如压缩历史入口，记录 `historical-index.md` 指针和原始证据未删除

## 禁止

- 不在 v1.1 commit 中夹带历史 art、MiroFish 原包、zip、外部分析目录或无关 dirty files。
- 不自动部署 EdgeOne。
- 不设置 branch protection 或 auto-merge，除非用户另行批准。
- 不把 live DeepSeek drift probe 成本、模型、样本或轮次混入默认 CI。
- 不把 v1.6 过期入口检查脚本提前加入当前 CI 硬门。
- 不删除历史制度或文档；流程减法先标历史和加替代指针，删除需用户另批。
