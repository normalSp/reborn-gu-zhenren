# v2.5.0 专家团启动会纪要

日期：2026-05-23
状态：completed；D-250-001 至 D-250-012 已获用户批准。
分支：`codex/v250-startup-private-canon-visibility`

## 会议主题

v2.5 建议主题：

`private canon / knowledge visibility 试验设计`

本版本承接：

- v2.0 第一个区域活世界与 T3 证据。
- v2.1-v2.3 Agent Lab / eval farm 的 report-only 与 offline 证据。
- v2.4 薄 BFF / private canon / eval archive 边界评估。
- v2.5-a0 前置批量审批与例外停机制度。

本启动会的核心目标不是实现 private canon service，而是把“哪些知识能被谁看见、什么时候能进入候选、什么时候必须拒绝、未来服务边界怎么审计”设计成可测试、可复盘、可进入 v2.6/v3.0 的工程门禁。

## 专家团意见

### Project Lead

v2.5 不应急着建服务，也不应为了长期 agent 愿景把知识直接喂给模型。

本版本最有价值的产物是前置授权包和 visibility/schema 门禁：让以后 private canon、knowledge index、eval archive、DeepSeek visible context、AgentProposal、MiroFish intake 之间不再靠临场记忆判断。

### Systems Architect

推荐将 v2.5 保持为 docs/report-only/schema-first：

- 定义 visibility level。
- 定义 knowledge item lifecycle。
- 定义 redacted visibility envelope。
- 定义 WorldCore post-check 对知识流的拒绝规则。
- 输出 v2.6 是否需要 runner / archive / service 的 go-no-go。

本阶段不实现 private canon service、BFF/backend、job queue、cloud save 或 eval archive service。

### AI Pipeline Architect

DeepSeek 当前仍是 `deepseek-v4-flash`，但 v2.5 默认不调用 live DeepSeek，不改 prompt/context/model/authority。

v2.5 应把“DeepSeek 不可见内容”写成明确协议：

- hidden/private body 不可见。
- source pointer 可作为本地审计证据，但不可作为 DeepSeek visible lore。
- DeepSeek 可见摘要必须未来单独 gate。
- prompt hash / evaluator version / redaction policy 是未来 eval archive 的必要字段。

### Canon / Knowledge Curator

全书 MiroFish 基础包仍是 archive/source-pointer inventory，不是 runtime canon，不是 DeepSeek RAG。

v2.5 可以设计知识可见性，不应写知识库正文；可以定义 promotion chain，不应做 canon promotion。

如果任何阶段需要真实原著事实、命名 NPC、hidden-adjacent、方源公开证据或正式 lore 结论，MiroFish need level 立即升级为 `blocking`。

### Lore Guard

v2.5 必须特别防止两个误区：

1. 把 private canon 当成“给模型更多上下文”的包装。
2. 把 source pointer 当成玩家可见事实。

本版本应只使用 synthetic markers、source-pointer classes 和 redacted envelope schema。不得引入真实 hidden/private body、原著正文或命名 NPC 结论。

### QA / Eval Guardian

v2.5 是纯文档 / schema / report-only 版本，Player Advocate 和 live narrative probe 默认豁免，但要在每个阶段记录豁免理由。

需要保留的质量门：

- `git diff --check`
- 敏感边界扫描。
- D-250 授权包状态扫描。
- 文档入口扫描。
- Skill sync audit。

不运行 runtime 单测/e2e，除非后续阶段实际改源码。

### Dev Environment / Git Steward

本版本必须遵守 v2.5-a0 新流程：

- 启动会给完整前置授权包。
- 用户批准后，后续 `/goal` 可以自动跑授权包内阶段。
- 每个阶段仍写记录、显式 stage、提交、推送、查 CI。
- 不使用 `git add -A`。
- 不自动部署 EdgeOne。

### Skill Sync Steward

startup 触发 Skill 同步审计。当前建议：

- `reborn-expert-council`：no_update_needed，v2.5-a0 已含前置授权制度。
- `game-dev-text`：no_update_needed，v2.5-a0 已含自动继续与硬停口径。
- `reverend-insanity-lore`：no_update_needed，v2.5-a0 已含 lore/MiroFish/hidden/canon promotion 硬停。
- `reborn-combat-motion`：no_update_needed，本版本不触碰战斗表现。
- `mirofish-reborng-export`：no_update_needed，本版本初始 MiroFish need level 为 `not_needed`。

## 启动结论

专家团建议 v2.5 采用保守但有实际收益的路线：先建立 private canon / knowledge visibility 的规则、schema、拒绝条件和未来准入报告，不做服务、不接 DeepSeek、不写 runtime。

本启动会已输出完整前置授权包。用户已批准 D-250-001 至 D-250-012，并确认 F-250-001 至 F-250-010 为 `future_gate_required`；后续 v2.5 文档 / schema / report-only 阶段已在该授权范围内完成，未触发例外停机。
