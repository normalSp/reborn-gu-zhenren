## 背景

用户指出：当前每个小版本都需要审批，安全但过碎。目标是让用户在提出需求和方向后，Codex 能像精确流水线一样执行，不需要用户持续干预细碎阶段。

专家团结论：可以优化，但不能牺牲 RebornG 的硬边界。正确做法不是取消审批，而是把审批前置、批量化、结构化，并保留例外停机。

## 新流程摘要

从 v2.5 起，每个大版本启动会或 a1 设计门禁必须输出“前置授权包”：

1. 全版本阶段路线。
2. 全部预计决策项。
3. 自动继续范围。
4. 例外停机清单。
5. 负授权说明。
6. 测试和证据计划。
7. Git 与交接计划。

用户批准授权包后，Codex 可以在 `/goal` 中连续完成后续阶段，不再每个小版本反复索要同类审批。

## 例外停机

以下事项仍必须停机：

- 新 save field / `SAVE_FORMAT_VERSION` bump。
- DeepSeek prompt/context/model/authority 扩大。
- live probe 超出已批档位，或出现 P0/P1。
- MiroFish blocking：真实原著事实、命名 NPC、hidden-adjacent、方源证据、正式 lore 结论。
- backend/BFF/service、外部依赖、PoC、subagents、patch artifact。
- 正式地点、阵营、身份、奖励、NPC 生死、canon promotion。
- CI/测试连续失败且需要改变方案。
- 成本、公开发布、部署、法律/版权边界变化。
- 玩法口感、叙事方向、难度、节奏、世界观取舍出现明显新分歧。

## 对 v2.5 的直接影响

v2.5 专家团启动会不能再只给下一阶段 D 项。它必须一次性给出全 v2.5 的授权包，并把事项分为：

- `approved_in_frontloaded_pack`
- `future_gate_required`
- `blocked_or_rejected`

如果用户批准 v2.5 授权包，后续可用一个 `/goal` 从 a1 跑到 rc。只有触发例外停机时才回到用户。

## 本次不授权

本 a0 只是流程制度补丁，不授权任何：

- runtime/source/UI/store/prompt/save 变更。
- private canon service、backend/BFF、eval archive service、runner、artifact。
- live DeepSeek、DeepSeek visible lore/RAG。
- external framework PoC、dependency、vendored subset、read-only scan、patch artifact、subagents。
- MiroFish export/intake。
- 正式地点、阵营、身份、奖励、NPC 生死或 canon promotion。
