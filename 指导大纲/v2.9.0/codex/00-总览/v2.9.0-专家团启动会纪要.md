# v2.9.0 专家团启动会纪要

日期：2026-05-24
状态：startup completed locally；等待用户审批 D-290 / F-290
主题：`v3.0 前安全收束：权限、prompt、eval、Player Advocate、MiroFish 边界统一复核`

## 启动背景

用户要求开 v2.9 专家团启动会，并重点检查：

- 长期路线是否按期实现。
- 代码是否漏需求。
- 是否存在“文档写了但代码实现不同”。
- 是否存在只写 TODO、没有真正工程化落地的情况。

## 专家团共同结论

v2.9 可以启动，但不能把 v2.8 的 runtime admission 通过误读为 v3.0 runtime agent 已获批准。

当前路线总体是稳的：v2.0 完成第一个区域活世界；v2.1-v2.3 完成 Agent Lab 与 eval farm；v2.4-v2.6 完成基础设施/private canon/eval archive 的边界研究；v2.7-v2.8 完成多 NPC 样板和 runtime admission 预检。v2.9 应该做最后的安全收束和 v3.0 go/no-go，而不是急着上 runtime agent。

## Project Lead

v2.9 的主线不是“再做一个功能小版本”，而是把 v3.0 能不能开始变成可判定问题：

- 哪些 L2/L3 能力能进入 v3.0 小区域试点。
- 哪些还必须留在 Agent Lab。
- 哪些能力一旦需要，就必须停机找用户拍板。

建议 D-290 前置授权包一次性给全版本范围；后续 `/goal` 可以连续推进，但任何 F-290 触发都必须停止。

## Systems Architect

当前代码没有出现“文档承诺 runtime agent 已实现但代码没做”的问题。反而相反：v2.1-v2.8 大多数代码都非常克制，严格停在 self-owned、zero-dependency、report-only/offline tooling。

v2.9 应把以下合同机械化：

- `AgentProposal -> WorldCore post-check -> rejected/accepted/report`。
- agent 不写 save、不写 canon、不写正式事实。
- L5 只给 pressure，不当 NPC，也不宣判结局。
- v3.0 若要接 runtime，必须先有 rollback、old-save、Player Advocate、live probe 与 failure taxonomy 对照。

## AI Pipeline Architect

v2.9 不应改 DeepSeek prompt/context/model/authority，也不应实跑 live probe。正确做法是：

- 写 v3.0 runtime agent prompt/contract 草案。
- 写 live probe 档位、样本、成本、接受标准。
- 明确任何 prompt/context 扩权都不是 D-290 自动批准范围。

## Lore & World Designer

MiroFish 当前是 `not_needed`。v2.9 只做安全收束和 go/no-go，不需要新增原著事实。

一旦 v3.0 触及命名 NPC、真实势力、hidden-adjacent、方源公开证据、L4/L5 原著锚点、正式 lore 结论或知识库正文，MiroFish 必须升级为 `blocking`，先 request/export/intake review，再谈 runtime。

## QA / Test Engineering Guardian

v2.9 必须补两道质量门：

- 长期路线兑现审计：逐版本核对承诺、脚本、报告和禁止项。
- 文档代码一致性与 TODO 审计：把误报、旧功能占位、真实缺口分开。

启动审计已完成初扫：未发现 v2.x Agent Lab / runtime admission 已批准 scope 的阻塞性缺口。后续 b1/b2 可把这些结果转成 report-only checker 或 go/no-go 文档。

## Player Advocate

v3.0 若接 runtime agent，必须增强 Player Advocate 记录：

- 是否调用 live DeepSeek。
- 模型、样本、轮次、成本、报告路径。
- 每个 agent 候选是否让玩家理解“这是候选/表达”，而不是正式世界事实。
- 玩家是否能看见每一步选择对后续 NPC/势力/压力的真实影响。

v2.9 可以先冻结模板，不做实跑。

## Git / Rollback Steward

v2.9 已按制度从 v2.8 clean branch 切出：

`codex/v290-startup-pre-runtime-safety-closure`

本 startup 只提交文档、审计和入口同步；不混入 runtime、scripts 或 artifacts。后续若 D-290 批准，再按阶段提交。
