## 专家团共识

### Project Lead

v2.4 不应急着实现后端，也不应把 Hermes、LangGraph、Mastra 或任何外部框架接进 runtime。

正确动作是先回答：

- 什么必须留在本地 WorldCore。
- 什么可以未来放进薄 BFF。
- 什么只能作为 report-only / offline / eval archive。
- 什么必须永远走用户决策门。

### Systems Architect

薄 BFF 的候选职责可以讨论：

- API proxy / key protection。
- private canon / hidden body 可见性服务。
- agent job queue。
- cost / cache / retry / prompt hash 观测。
- replay / eval archive。
- cloud save 远期预研。

但 BFF 不能接管 WorldCore，不能写 save，不能批准奖励、NPC 生死、地点、身份、canon。

### AI Pipeline Architect

DeepSeek 当前仍保持 `deepseek-v4-flash`。v2.4 默认不调用 live DeepSeek，不改 prompt/context/model/authority。

v2.4 应重点设计观测项：

- prompt hash。
- evaluator version。
- retry/recovery。
- token/cache/cost。
- report archive。
- failure taxonomy link。

### Canon / Knowledge Curator

private canon 和 hidden/private body 不是“给模型更多上下文”的借口。

v2.4 可以设计 visibility service，但不得：

- 做全书 RAG。
- 把 MiroFish 基础包导入 DeepSeek visible context。
- 把 hidden/private body 下发到公开前端。
- 把 source pointer 误当 runtime canon。

### Lore Guard

本阶段不新增原著事实、正式地点、阵营、身份、奖励、NPC 生死、canon promotion。

如果后续 a1/a2 触及命名 NPC、hidden-adjacent、方源公开证据、正式 lore 结论，MiroFish need level 升级为 blocking。

### QA / Eval Guardian

v2.4 是 docs/report-only 规划阶段，Player Advocate 与 live probe 默认豁免，但必须写明豁免理由。

如果 b1-b3 增加 report-only scripts，则需要：

- schema/unit 检查。
- `git diff --check`。
- 敏感词边界扫描。
- 不改变 runtime/source/UI/store/prompt/save 的断言。

### Dev Environment / Git Steward

本阶段必须继续：

- 使用语义分支。
- 显式 stage。
- 提交推送。
- 不自动部署 EdgeOne。
- 不让外部 agent 执行命令或 git。

### Skill Sync Steward

本轮已触发 expert skill 更新；game-dev/lore/combat/mirofish-export 记录为 `no_update_needed`，除非 v2.4 后续 a1/a2 进入 runtime、lore、combat 或 MiroFish。

专家团建议 v2.4 先做边界评估，不实现后端，不接 runtime agent。
