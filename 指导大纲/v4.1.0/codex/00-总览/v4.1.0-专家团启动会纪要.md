# v4.1.0 专家团启动会纪要

状态：completed locally；D-410 已获用户批准并执行，F-410 全部继续 `future_gate_required`。
日期：2026-05-26。
主线建议：`Auto-Theater Contract / Schema / Checker 加固`。

## 背景

`v4.0.0` 已完成 Auto-Theater Combat、Auto-Theater Lite、battle theater、killer-move stack、Combat Ledger、HeavenWill/Fate/L5 宏观压力的设计门禁与 `v400` readiness checker。v4.0 的结论清楚：纯自走棋不是高阶主路线，平面棋盘也不足以承载蛊仙级战斗；Auto-Theater 是更合适的招牌方向，但必须由 WorldCore / Combat Core 保持最终裁决。

`指导大纲/长期路线/v4.0-v5.0-AutoTheater到高阶世界Runtime总体大纲.md` 把 v4.1 定位为 `Auto-Theater contract v1`。专家团一致认为：v4.1 的正确动作不是“开始做 UI”，而是先把 contract/schema/evidence/checker 做硬，让 v4.2 凡阶 Lite 映射、v4.3 高阶战场语义、v4.4 杀招栈与 Combat Ledger、v4.7 runtime 准入都使用同一套语言。

## 专家意见

Project Lead：
v4.1 应成为 v4.x 的“接口宪法”。如果没有统一 contract，后面的 Lite、Full Theater、杀招栈、宏观压力、UI、DeepSeek 表达都会各说各话。v4.1 应少而硬：字段、lane、authority、evidence、negative fixtures。

Systems Architect：
建议冻结 `AutoTheaterContractV1`，将 preparation、theater layer、killer-move stack、Combat Ledger、expression candidate、WorldCore evidence chain 放进同一份 envelope。不要把 schema 写进 save，也不要过早做 Zod/runtime types；先用文档门禁和 report-only checker 验证边界。

Gameplay/Numerics Architect：
Auto-Theater 的关键不是自动播放，而是把准备、风险、反制、反噬、成本、撤退阈值和战后复盘做成玩家能理解的系统。v4.1 的 contract 必须支持“为什么成功/失败/被反制/只是候选”，否则后续战斗会变成看不懂的播片。

Combat/Motion Lead：
表现层需要一个明确输入：哪些是事实，哪些是候选，哪些是压力，哪些是拒绝，哪些必须等用户决策。v4.1 必须定义 `ExpressionAuthorityV1`，防止 UI、动效、美术资产把候选画成事实，把 pressure 画成宿命裁决，把仙蛊屋或环境破坏画成已发生。

Lore & World Designer：
v4.1 只应使用 synthetic/generic 样本。真实原著战役、命名 NPC、方源、尊者、hidden-adjacent、HeavenWill/Fate 真实锚点都不应进入 contract body。若未来需要真实 source pointer，必须升级 MiroFish 为 blocking。

AI Pipeline Architect：
DeepSeek 后续可以参与战斗旁白或候选意图，但 v4.1 contract 必须提前规定 DeepSeek 只接收 WorldCore 已批准的表达材料或 report-only 候选，不能生成命中、伤害、结算、奖励、地点变化、NPC 生死、宿命结果。

QA/Test Engineering Guardian：
v4.1 的主要测试不是 e2e UI，而是 schema/checker 的负样本覆盖。必须覆盖：候选伪装事实、pressure 伪装裁决、UI 私有结算、DeepSeek 越权、动效暗示正式奖励、ledger 泄露 hidden body、theater layer 缺 evidence、凡阶 runtime 迁移误开。

Player Advocate：
建议 v4.1 后续 b3 仍做 40-60 轮 PA 或等价读者走查。玩家必须能理解 Auto-Theater 合同的核心含义：准备不是胜负、候选不是事实、ledger 是证据不是万能旁白、天道/宿命是压力不是结局按钮。

Git/Rollback Steward：
v4.1 已切到 `codex/v410-startup-auto-theater-contract-hardening`。启动包完成后提交推送并等待 CI；v4.1 完成时仍不自动合并 main。

Skill Sync Steward：
v4.1 触发 `reborn-expert-council`、`game-dev-text`、`reverend-insanity-lore`、`reborn-combat-motion`。`mirofish-reborng-export` 当前 `no_update_needed`。

## 路线比较

| 路线 | 内容 | 收益 | 风险 | 专家建议 |
|---|---|---|---|---|
| A | Auto-Theater contract/schema/checker hardening | 最稳，直接承接 v4.0-v5.0 总纲；为 v4.2-v4.9 提供共同接口 | 玩家短期看不到新 UI | 推荐 |
| B | 先做凡阶 Auto-Theater Lite 映射 | 更贴近现有战斗，容易验证 | contract 未稳会导致 Lite 映射返工 | 暂缓到 v4.2 |
| C | 先做 theater UI 原型 | 可视化冲击强 | 触发 UI/素材/runtime 多重 future gate | 不推荐 |
| D | 直接做高阶战斗 runtime | 短期刺激 | save、lore、MiroFish、DeepSeek、数值、表现层全部高风险 | 禁止，除非未来另批 |

## 结论

专家团建议采用路线 A。v4.1 只做 Auto-Theater contract/schema/checker 加固，冻结工程合同和负样本门禁，不开放任何 runtime、save、UI、DeepSeek、MiroFish、后端、外部框架或素材权限。用户已审批 D-410，可以一个 `/goal` 跑完整个 v4.1；F-410 继续作为未来门禁。
