## 执行顺序

1. 先完成 a0/a1/a2，冻结 v1.0 要承诺什么、不承诺什么。
2. 等 MiroFish continuity 包通过 intake，再进入 b1 runtime absorption。
3. b1 已采用保守吸收：不写正式 route/location persistent fields；后续若要写入正式 route/location/currentRegion，仍必须停手让用户批准 save-format bump。
4. b2 已采用保守吸收：只写 v1.0 life loop 验收事实、公开压力和账本，不写正式经济/库存/路线/地点字段。
5. b4 不做未经批准的新图大规模生成，只整理已有候选和必要文案边界。
6. process-1 可穿插，但不自动部署。
7. rc1 之后由用户决定是否进入公开发布准备。

## Player Advocate 轮次建议

v1.0 是公开候选，建议比历史版本提高：

- a0/a1/a2 纯文档/威胁模型：可豁免，但必须记录豁免原因。
- b1/b2/b3 玩家可见 runtime：每阶段至少 30 轮。
- b4 公开素材/文案：至少 20 轮，重点看玩家是否被误导。
- rc1：至少 150 轮，分层覆盖新档、旧档、移动端、极端意图、路线承接、低阶 life loop、公开文案。
- 若 rc1 发现 P1 以上玩家理解问题，修复后需追加 30-50 轮回归。

## Git 计划

- a0/a1/a2 可合并为一个启动审查 commit，但如果变更较大应拆分。
- b1/b2/b3/b4/process-1/rc 均单独 commit。
- 每个 runtime 小版本 push 后等待 GitHub Actions 通过并补录 run id。
- 不使用 `git add -A`；只 stage 本阶段明确文件。
