## 接手经验

RebornG 的工作节奏不要变快到失控：

1. 先读当前事实，再动代码。
2. 每个阶段都按 `canon -> types/schema -> engine -> store -> UI -> tests/docs` 推进。
3. 遇到公开承诺、原著边界、存档字段、DeepSeek 权限、正式奖励/地点/阵营/NPC 生死、MiroFish 阻塞包，必须停下来问用户。
4. 不使用 `git add -A`；只 stage 本轮明确改过的文件。
5. 保留历史脏文件，不要清理、不回滚、不顺手纳入提交。
6. 每个小版本必须更新文档、测试、仪表盘和 Git 状态。
7. 玩家可见/runtime 改动要走 Player Advocate gate；高风险或 rc 按版本规则加轮次。
8. MiroFish 包只能先做 intake review，再转写为 RebornG-owned candidate/rule/test，不直接进入玩家可见文本或 DeepSeek context。

用户手动 EdgeOne 发布后，先验收：

1. 预览页不黑屏。
2. 首屏显示 `RebornG v1.0《青茅之后，活世界初成》` 和 `v1.0.0`。

4. Network 无 JS/CSS/PNG 404，尤其是三张 hero 图。
