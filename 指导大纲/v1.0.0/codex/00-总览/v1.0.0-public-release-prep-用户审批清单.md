# v1.0.0 public release prep 用户审批清单

日期：2026-05-20
状态：用户已批准，发布标识与 hero 绑定已进入落地验证
用途：把真正发布前需要用户拍板的事项拆开，避免 Codex 擅自形成公开承诺。

## 使用方式

每项只需要用户回复：批准、驳回、延期、调整。

未经批准前，以下事项不得进入正式发布动作：

1. 改版本号。
2. 发布公告。
3. 绑定 hero 图。
4. 部署或公开 EdgeOne。
5. 对外使用截图/短录屏。

## D-PUB-001 版本标识是否改为 v1.0.0

批准前状态：

- `package.json` 仍是 `0.9.0`。
- 标题页显示 `v0.9.0`。
- 游戏页底部显示 `v0.9.0`。
- `v1.0.0` 是本地开发里程碑完成，不是当前 UI 显示版本。

建议：

如果这次要以 `v1.0.0` 对外发布，应批准把包版本、标题页、游戏页、公开测试说明和发布文档口径统一成 `v1.0.0`，并重跑完整发布验证。

风险：

- 正面影响：玩家不会看到 v1.0 公告打开后仍写 v0.9.0，发布可信度更高。
- 负面影响：改版本显示会扩大发布验证范围，必须重跑 build/e2e/production smoke。

用户决策：批准。落地方向为统一 `package.json`、标题页、游戏页和公开测试说明为 `v1.0.0`，并重跑完整发布验证。

## D-PUB-003 公告正文口径

批准正文：

RebornG v1.0 把前期多个系统收束成一段可连续体验的低阶蛊师人生。玩家可以从青茅山后续压力出发，看到南疆早期路线候选、补给与修行压力、NPC/势力反应、公开事件摘要和自由目标裁决。DeepSeek 仍只负责叙事、候选、线索、传闻和压力；正式奖励、地点、阵营、NPC 生死和世界事实由本地系统决定。

本版本不是完整蛊界，也不是“AI 想写什么就是什么”的跑团壳。它验证的是一个更长期的方向：玩家可以提出目标，世界会给出前置、风险、代价、拒绝或降级，而不是被一句话改坏。

必须保留的边界：

1. 不承诺完整南疆、完整商家城、五域大地图。
2. 不承诺任意时代开局、任意人物附身。
3. 不承诺仙蛊、九转蛊、盗天魔尊传承、尊者线。
4. 不说 route/location/currentRegion 已正式持久化。

用户决策：批准正文口径。后续公告不得越过本文“必须保留的边界”。

## D-PUB-005 截图/短录屏方向

建议展示四类：

1. 自由意图收束：展示目标如何被裁决、降级、转前置。
2. 低阶 life loop：展示修行、补给、炼养用、交易窗口、路线压力。
3. 青茅到南疆承接：展示路线压力、候选承接、身份/补给风险。
4. NPC/势力回流：展示公开事件、关系压力、后续候选。

禁止展示：

1. 九转蛊、仙蛊、盗天传承已经成功。
2. 关键 NPC 生死已经被玩家直接决定。
3. 完整商家城、完整南疆或任意时代已经开放。
4. DeepSeek 直接写奖励、地点、阵营或 NPC 生死。

用户决策：批准四类截图/短录屏方向。实际对外素材仍不得展示禁止项。

## D-PUB-006 EdgeOne 发布动作

用户决策：EdgeOne 发布动作由用户执行，Codex 不自动部署。

## 发布前必须复验

任何 D-PUB-001、D-PUB-004、D-PUB-006 被批准并落地后，必须复验：

1. `npx tsc --noEmit --pretty false`
2. `npm test -- --reporter=dot`
3. `npm run build`
4. `npm run check:runtime-assets`
5. `npm run check:qingmao-assets`
6. `npm run check:player-visible-copy`
7. `npm run check:v019-content-governance`
8. `npm run test:e2e`
9. `npm run test:e2e:long`
10. `npm run check:production-preview`

若只是审批文案而不改 runtime，可先跑轻量门：`check:player-visible-copy`、`check:v019-content-governance`、`check:production-preview`。
