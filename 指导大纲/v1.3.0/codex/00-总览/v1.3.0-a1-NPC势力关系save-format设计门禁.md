# v1.3.0-a1 NPC 势力关系 save-format 设计门禁

日期：2026-05-21
状态：设计门禁已起草，等待用户拍板 D-131-001 至 D-131-007
性质：文档/架构门禁；不改 runtime、不 bump save、不新增 DeepSeek 权限

## 0. 本门禁要回答什么

v1.3 的主题是“NPC、势力关系与社会反应骨架”。a1 只处理进入 runtime 前的 save-format 与权威边界问题：

1. b1 是否需要把 `SAVE_FORMAT_VERSION` 从 24 bump 到 25。
2. 是否现在新增持久字段，例如 `socialRelationState`。
3. 现有 `livingWorldState.npcMemories`、`livingWorldState.factionPressure`、`livingWorldState.actionConsequences` 能不能承载 b1。
4. 旧字段如 `npcRelations`、`standings`、`npcContacts`、`dynamicNPCs` 能不能被当成 v1.3 权威。
5. MiroFish 基础包在 NPC/势力关系里何时是 preferred、何时是 blocking。
6. v1.3 的 Player Advocate、长线漂移、rc live probe 应该怎么定强度。

门禁结论先给出：专家团建议 **v1.3-b1 不 bump v25、不新增社会关系持久字段，先做 projection-only 社会反应层**。如果 b1/b2 证明需要长期关系账本，再单独进入 `SAVE_FORMAT_VERSION = 25` 小门禁。

## 1. 输入与当前事实

已确认输入：

- `SAVE_FORMAT_VERSION = 24`，v1.2 已加入 `survivalEconomyState`。
- `livingWorldState` 已有 `npcMemories`、`factionPressure`、`actionConsequences`、`knownFacts`、`hiddenFactRefs`、`ifDeviations` 等社会反应可读证据。
- `LivingNpcMemoryEntry` 已有 `npcId`、`turn`、`regionId`、`actionId`、`publicSummary`、`privateRefId`、`attitudeDelta`、`weight`、`tags`、`expiresTurn`。
- `LivingFactionPressureEntry` 已有 `factionId`、`pressureType`、`delta`、`reason`、`turn`、`visibility`。
- 项目内存在历史/旧系统字段，如 `npcRelations`、`characterRelations`、`npcContacts`、`standings`、`dynamicNPCs`，但它们不是 v1.3 的已审核社会关系权威。
- v1.3-a0 已批准 D-130-001 至 D-130-009，其中 rc live probe 原则批准，模型固定 `deepseek-v4-flash`，具体成本、样本、轮次到 rc 前再拍板。

## 2. 权威边界表

| 对象 | v1.3-a1 结论 | 说明 |
| --- | --- | --- |
| NPC 正式好感、仇恨、信任等级 | 不开放 | 不新增正式数值，不让 DeepSeek 写结论。 |
| 势力正式阵营、通缉、招揽、封锁、奖励 | 不开放 | b1 只展示压力、传闻、前置条件或风险，不落正式裁决。 |
| NPC 生死、命运、隐藏事实正文 | 不开放 | 仍由 canon/engine/user 决策控制。 |
| `livingWorldState.npcMemories` | 可作为投影证据 | 可读、可排序、可解释，但不升级成正式关系图。 |
| `livingWorldState.factionPressure` | 可作为投影证据 | 可显示“谁在关注/怀疑/给机会”，但不等于通缉或阵营状态。 |
| `livingWorldState.actionConsequences` | 可作为投影证据 | 用于“因何而来”的可读链路。 |
| 旧字段 `npcRelations`/`standings` 等 | 暂不作为 v1.3 权威 | 需要单独 normalize/review 后才能晋升。 |
| DeepSeek | 只写叙事、候选、传闻、压力、请求 | 不能写数值事实、奖励、地点结论、正式关系结论。 |
| MiroFish | 只做 candidate/source pointer/test seed | 不能直接成为 runtime/canon/DeepSeek visible context。 |

## 3. 方案比较

### 方案 A：b1 projection-only，暂不 bump v25

做法：

- 新增只读 projection/helper，从现有 `livingWorldState`、`routeLocationState`、`survivalEconomyState`、局部 canon rule 中汇总“社会压力视图”。
- UI 只展示玩家能理解的社会反应：相关人物/势力、公开证据、压力类型、下一步风险或机会。
- 不新增存档字段，不迁移旧档，不把旧字段晋升为权威。

收益：

- 最小风险进入 v1.3 runtime。
- 不制造 v25 兼容负担。
- 能快速验证玩家是否真的需要更长期的 NPC/势力关系账本。
- 避免早期设计把“关系”写死成好感度或阵营数值，保留蛊真人味的阴谋、利益、证据链和不完全信息。

风险：

- b1 的关系变化更多是“可读投影”，不是长期可审计账本。
- 若玩家跨很多回合反复与同一 NPC/势力互动，可能需要 b2/b3 补持久聚合。
- projection 逻辑如果命名不谨慎，容易被误认为正式关系系统。

专家团建议：**采用**。这是 v1.3-b1 的默认建议。

### 方案 B：直接 bump v25，新增 `socialRelationState`

做法：

- `SAVE_FORMAT_VERSION = 25`。
- 新增单一聚合对象 `socialRelationState`，保存社会关系证据账本、压力摘要、可见性、source refs。

收益：

- 为长期 NPC/势力关系提供稳定承载。
- 更适合后续 b2/b3 做长期社会反馈、跨区域追踪、Player Advocate 长测。
- 有利于将来做审计与 replay。

风险：

- 现在需求还未被 b1 验证，容易过早固化字段。
- 迁移、默认值、兼容测试、旧字段隔离都会增加开发成本。
- 一旦字段命名或结构过拟合，v1.4-v1.6 会背历史包袱。

专家团建议：**保留为 b2/b3 备选，不在 b1 默认启用**。

### 方案 C：扩写 `livingWorldState` 语义，把它直接当社会关系系统

做法：

- 不 bump save，但把 `npcMemories`、`factionPressure` 解释为正式关系权威。

收益：

- 表面上最省事。

风险：

- 语义污染。记忆/压力证据不等于关系结论。
- 会模糊 DeepSeek、engine、canon、store 的责任。
- 长期会造成“玩家以为已经通缉/结仇/被招揽，但系统并没有稳定裁决”的错位。

专家团建议：**拒绝**。

### 方案 D：复用旧字段 `npcRelations`、`standings` 等作为 v1.3 权威

做法：

- 直接读取历史字段，作为 NPC/势力关系展示或判断基础。

收益：

- 看似减少新增工作。

风险：

- 旧字段来源、语义、测试覆盖、与当前 living-world/route/economy 状态关系不清。
- 容易绕过 v1.3 新门禁，把未审核遗产字段变成新系统地基。
- 兼容和迁移行为很难解释。

专家团建议：**拒绝。旧字段只能作为 future normalization 候选，不作为 b1 权威**。

## 4. 如未来需要 v25，字段应怎样长

如果 b1/b2 证明必须加入持久社会关系聚合，建议只新增一个保守对象，而不是到处加散字段：

```ts
type SocialRelationState = {
  version: 1;
  status: 'empty' | 'active';
  ledger: SocialRelationLedgerEntry[];
  summaries: SocialRelationSummary[];
  sourceRefs: string[];
  lastUpdatedTurn?: number;
};

type SocialRelationLedgerEntry = {
  id: string;
  turn: number;
  scope: 'npc' | 'faction' | 'public_event' | 'hidden_boundary';
  subjectId: string;
  category:
    | 'suspicion'
    | 'favor'
    | 'hostility'
    | 'opportunity'
    | 'fear_hint'
    | 'trust_hint'
    | 'interest_hint';
  evidenceRefs: string[];
  visibility: 'player_visible' | 'system_only' | 'hidden_ref_only';
  pressureScoreDelta?: number;
};
```

必须明确不放入：

- `affinity` 这类正式好感度。
- `wantedLevel`、`recruitmentStatus`、`factionMembership`。
- `reward`、`locationUnlock`、`npcAlive`。
- 隐藏事实正文、原著未来真相、未授权 canon 结论。

如果进入 v25，迁移原则为：

- v24 -> v25 只添加空对象/default，不从旧字段自动推导正式关系。
- 旧字段只登记为 `sourceRefs` 或 future normalization 候选，不直接改写进 ledger。
- migration/defaults/tests 与 save bump 同一提交完成。
- 需要新增旧档加载、非法字段清洗、DeepSeek 越权回滚、Player Advocate、长线漂移样本。

## 5. b1 推荐 runtime 切口

b1 如果按本门禁建议走，应做“社会反应投影层”：

- 读取 `livingWorldState.npcMemories`、`factionPressure`、`actionConsequences`。
- 汇总出玩家可理解的“谁注意到了你、为什么、风险是什么、可能的机会是什么”。
- 输出 projection-only view model，例如 `socialPressureCards` 或 `socialReactionProjection`。
- UI 只展示公开摘要、压力类型、证据来源、下一步提示，不展示隐藏事实和正式关系数值。
- DeepSeek 只允许润色叙事/传闻/请求，不允许写入正式状态。
- 如果需要 store，最多写 UI 临时态或可重算缓存，不写 save。

建议 b1 完成标准：

- 新档/旧档都可进入，不需要 migration。
- 没有 `SAVE_FORMAT_VERSION` 变化。
- 没有新 DeepSeek authority。
- 至少覆盖 NPC 记忆、势力压力、公开事件、隐藏事实保护、旧字段不越权五类样本。
- Player Advocate 30 轮，外加 T0-lite deterministic drift。

## 6. MiroFish 使用判定

v1.3-a1 本身不需要新的 MiroFish 交付才能完成，因为它只是 save-format/权威门禁。

v1.3-a2 应做 preferred topic slice：

- 优先复用 v0.13 三包：NPC memory/motive、faction pressure、public event chronicle。
- 从全书基础包中只取 source pointer、覆盖缺口、候选样本，不整包导入知识库。
- 产物仍是 intake review、候选样本、边界规则，不是 canon/runtime 权威。

升级为 blocking 的触发条件：

- 正式命名 NPC allowlist。
- 正式通缉、招揽、封锁、势力任务或奖励规则。
- 触碰方源公开证据的长期后果。
- 触碰隐藏事实、原著未来事实、NPC 生死或命运。
- 要把某条 MiroFish 摘要晋升进 `src/canon/*.json`。

## 7. 测试与门禁强度建议

### b1 projection-only

- Unit/type：projection helper 与 view model。
- Save compatibility：新档、v24 旧档、缺失 living-world 子字段默认值。
- Boundary tests：DeepSeek 不得写正式关系/阵营/奖励/通缉。
- Player Advocate：30 轮。
- Long drift：T0-lite deterministic drift，重点看重复空转、关系错认、隐藏事实泄露、旧字段越权。

### b2/b3 如果新增持久社会关系字段

- `SAVE_FORMAT_VERSION = 25` 与 migration/defaults/tests 同提交。
- Player Advocate：至少 40 轮。
- Long drift：T0 deterministic soak。
- 增加 ledger replay/audit 样本。

### rc

- 如果 v1.3 全程不新增社会关系持久字段：建议 rc Player Advocate 100 轮。
- 如果 v1.3 任一阶段新增 `socialRelationState`、正式命名 NPC allowlist、或长期社会 ledger：建议 rc Player Advocate 120 轮。
- rc live probe 原则已批准，模型 `deepseek-v4-flash`；具体成本、样本、轮次在 rc 前再次给用户确认。
- 临时建议口径：无持久社会字段时 `6 samples x 4 rounds`；有持久社会字段时 `8 samples x 5 rounds`。
- live probe 通过线建议：P0=0，P1=0，P2<=2，accepted>=90%，隐藏事实泄露=0，正式关系越权=0。
