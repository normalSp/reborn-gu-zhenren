# v1.8.0-process-1 只读分析型子代理提速评估

状态：已完成
日期：2026-05-21

## 背景

用户此前批准：v1.7-a2 第一轮双仓流水线由主控跑通；样板稳定后，再给出风险和收益，让用户决策“只读/分析型子代理”是否参与提速。

v1.7 与 v1.8 的双仓样板均已跑通，但本轮 v1.8 completion 未启用任何子代理。

## 正面收益

- 可让只读 agent 并行检查 MiroFish source pointer coverage。
- 可让只读 agent 并行审阅测试矩阵是否漏掉 high-risk samples。
- 可让只读 agent 并行扫描文档入口是否陈旧。
- 对大规模基础包 topic-slice 有速度收益。

## 风险

- 只读 agent 也可能误读 hidden/private 或 source pointer 边界。
- 多 agent 结论可能出现冲突，主控仍必须做最终 intake review。
- 若边界不严，容易把 producer/export 任务和 RebornG governance 任务混在一起。
- writable agent 风险更高，会破坏 runtime/canon/git 边界。

## v1.8 结论

- 不启用子代理。
- 不把子代理写入默认开发流程。
- 若后续用户批准，只能先从 `只读/分析型` 开始。
- 允许任务仅限：coverage 摘要、重复样本发现、文档入口检查、候选风险列表。
- 禁止任务：修改 RebornG runtime、canon、store、DeepSeek context、MiroFish export、intake review 结论、Git stage/commit/push。

## 后续建议

v1.9 或 v2.0 前，如 topic-slice 数量明显增多，可单独开一个 `只读子代理试运行专项`，先用 1 个 agent 做 coverage 对照报告，再由主控审阅是否值得制度化。
