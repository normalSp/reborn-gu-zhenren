# v4.0-v5.0 Auto-Theater 到高阶世界 Runtime 总体大纲

日期：2026-05-26
状态：长期路线总纲；承接 `v3.0-v4.0-RuntimeAgent到高阶世界总体大纲.md` 与 `v4.0.0` 启动包；不授权当前 runtime。

## 定位

这份总纲回答一个问题：`v4.0` 把 Auto-Theater Combat、杀招栈、Combat Ledger、HeavenWill/Fate/L5 宏观压力设计出来之后，怎样稳稳推进到 `v5.0` 的第一个高阶世界可玩试点。

结论先写清楚：

- `v4.0` 是高阶世界设计地基，不是高阶 runtime。
- `v4.x` 应先把 Auto-Theater 的合同、证据链、视觉验收、凡阶 Lite 映射、杀招栈、高阶战场语义、天道/宿命压力和 MiroFish/lore intake 门禁逐步做稳。
- `v5.0` 的目标不是完整蛊仙开放、五域两天全开放或方源/尊者 live agent，而是一个边界清楚、可回滚、可测试的高阶世界第一刀。

本路线是项目长期规划，不替代 `v4.0.0-前置授权包.md`。用户已批准 `D-400-001` 至 `D-400-012`，并确认 `F-400-001` 至 `F-400-012` 全部继续 `future_gate_required`。

更远期的终局路线见 `v4.0-v12.0-RebornG终局形态总体大纲.md`。本文件只负责 v4.0 到 v5.0 的高阶世界第一刀，不声称覆盖终局。

## 核心原则

### Auto-Theater 是招牌，但不是自走棋皮肤

Auto-Theater Combat 的价值是把准备、态势、杀招栈、自动结算、复盘证据和叙事表达合成一个可理解的战斗体验。它可以吸收自走棋的准备和自动演算优点，但不能变成单纯的站位棋盘或不可交互播片。

### WorldCore / Combat Core 拥有最终裁决权

命中、伤害、消耗、反噬、环境破坏、NPC 生死、奖励、地点变化、身份变化和存档写入必须来自 WorldCore / Combat Core / canon gate。DeepSeek、UI、动效和视觉素材只能表达事实或提出候选。

### 凡阶先 Lite，高阶再 Full

凡阶/中阶战斗不应在 v4.0 直接迁移。路线应先做 `Auto-Theater Lite` 的映射和验收，让现有棋盘/行动卡逐步获得 Combat Ledger、失败原因、准备意图和局部 theater 镜头能力。高阶战斗再进入 Full Auto-Theater。

### 天道/宿命是压力系统，不是结局按钮

HeavenWill/Fate/L5 只输出 pressure / constraint / risk / era anchor class，不直接裁决玩家结局。任何真实原著锚点、隐藏因果、方源证据、尊者布局或 formal lore conclusion 都必须升级 MiroFish blocking。

### v5.0 是第一刀，不是终局

v5.0 只应追求一个受控、高质量、高可解释的高阶世界试点。它可以证明 Auto-Theater 与宏观压力能进入玩家体验，但不能声称已经实现完整蛊仙、五域两天、所有 NPC、所有蛊虫、所有杀招或完整天道/宿命。

## v4.0：设计地基与 readiness checker

目标：

- 完成 Auto-Theater Combat、Auto-Theater Lite、battle theater、killer-move stack、Combat Ledger、HeavenWill/Fate/L5 pressure 的设计门禁。
- 建立 `v400` report-only readiness checker。
- 验证玩家是否能理解“候选、压力、事实、拒绝、需要用户决策”的区别。

完成口径：

- 可以说项目拥有高阶世界第一版工程门禁。
- 可以说 Auto-Theater Combat 是 RebornG 后续战斗表现招牌。
- 不能说已经实现高阶战斗、theater UI、凡阶迁移、天道/宿命裁决或素材生产。

## v4.1：Auto-Theater contract v1

目标：

- 把 v4.0 的概念整理成工程合同，而不是散落的设计词。
- 定义 preparation、theater layer、killer-move stack、Combat Ledger、expression candidate 和 evidence chain 的统一 envelope。
- 让后续每个 v4.x 小版本都能用同一份 schema 说话。

建议产物：

- `AutoTheaterContractV1` 设计门禁。
- `CombatLedgerEntryV1` 字段草案。
- `TheaterLayerV1`：air / ground / underground / water / aperture / formation / domain / house / environment。
- `ExpressionAuthorityV1`：事实表达、候选表达、拒绝表达、需要用户决策表达。
- negative fixtures：UI/DeepSeek/动效试图把候选画成事实必须被拒。

不做：

- 不新增 runtime UI。
- 不新增素材。
- 不把 schema 写入 save。

## v4.2：Auto-Theater Lite 凡阶映射

目标：

- 复盘现有凡阶棋盘战斗，把它映射成 Lite transcript，而不是直接推倒重做。
- 证明低阶战斗可以保留棋盘/行动卡，同时拥有更强的 Combat Ledger 和 theater 解释。

建议产物：

- 当前凡阶战斗 action -> Lite preparation -> Lite ledger 的 report-only mapping。
- 月光蛊、白玉防御、酒虫支援、低阶兽战、失败原因的 Lite 样本。
- reduced-motion 与移动端可读性验收标准。

不做：

- 不迁移现有凡阶 runtime。
- 不改变玩家低阶战斗操作方式。
- 不新增战斗数值。

## v4.3：高阶战场语义实验室

目标：

- 证明高阶战斗不能只靠平面棋盘表达。
- 把空域、地面、地下、水域、洞天、阵法、领域、仙蛊屋和环境破坏拆成可测试的 theater layer。

建议产物：

- `HighRankTheaterSemanticsV1`。
- synthetic/generic 高阶压力样本，不使用真实原著战役。
- terrain / domain / formation / house / environment damage 的 boundary samples。
- checker 判断：任何地点毁灭、奖励、NPC 生死、仙蛊屋归属变化都必须需要 WorldCore evidence 或用户决策。

不做：

- 不开放真实高阶地点。
- 不使用命名原著 NPC。
- 不把仙蛊屋做成装备或奖励。

## v4.4：杀招栈与 Combat Ledger 硬化

目标：

- 把杀招从“文本招式”提升为可解释的 stack：准备、打断、反制、连锁、反噬、资源成本和结果证据。
- 让 Combat Ledger 成为玩家理解战斗的核心，不只是日志。

建议产物：

- `KillerMoveStackV1`。
- `CounterWindowV1`。
- `BacklashCostV1`。
- Combat Ledger 分 lane：facts、candidates、rejected、needs_user_decision、pressure。
- Player Advocate 样本：玩家能说清为什么某个杀招成功、失败、被反制或只是候选。

不做：

- 不开放完整杀招库。
- 不让 DeepSeek 创造杀招结果。
- 不把 ledger 变成隐藏数值泄露面板。

## v4.5：HeavenWill / Fate / L5 宏观压力实验室

目标：

- 把天道、宿命、时代大势从“剧情词”变成受限压力系统。
- 检查它不会变成万能失败按钮，也不会替玩家决定结局。

建议产物：

- `MacroPressureContractV1`。
- Fate state 仍只允许 `intact` / `fractured` / `destroyed`。
- HeavenWill pressure 只允许 pressure / constraint / risk / opportunity bias。
- Causality debt 用于记录偏离后的压力变化，不直接写结局。
- L5 macro director 只输出宏观压力候选，不作为可聊天 NPC。

不做：

- 不接 HeavenWill/Fate runtime。
- 不让 L5 决定玩家命运。
- 不暴露 hidden-adjacent 或方源证据。

## v4.6：高阶 lore / MiroFish intake 准入

目标：

- 决定 v5.0 第一刀是否需要真实原著锚点。
- 若需要，先做 RebornG request -> MiroFish export -> RebornG intake review -> rule/test/candidate material。

建议产物：

- 高阶战斗 topic-slice 需求评估。
- HeavenWill/Fate/era anchor topic-slice 需求评估。
- 是否需要方源/尊者/真实战役 evidence 的 blocking 决策。
- source-pointer / redaction / promotion chain 检查。

默认建议：

- 如果 v5.0 第一刀仍是 synthetic/generic 或 canon-safe shadow slice，MiroFish 可以保持 `not_needed` 或 `preferred`。
- 如果触及真实原著角色、真实战役、隐藏因果、尊者布局、方源证据或 formal lore conclusion，MiroFish 必须 `blocking`。

## v4.7：Auto-Theater runtime 准入包

目标：

- 给用户一个完整决策包：是否打开最小 Auto-Theater runtime first cut。
- 这一步必须把收益、风险、测试成本、live probe 需求、MiroFish 需求、save-format 需求和 UI/资产需求说清楚。

必须回答：

- 是否打开高阶战斗 runtime first cut。
- 是否允许 theater UI implementation。
- 是否允许素材生成或 runtime 资产晋升。
- 是否需要 save field / `SAVE_FORMAT_VERSION` bump。
- 是否需要 live DeepSeek。
- 是否需要 MiroFish blocking。
- 是否需要 backend/BFF 或外部 framework。
- 是否仍保持 WorldCore final authority。

默认建议：

- 不一次打开所有门。
- 首选 “最小可回滚、无 save 新字段、synthetic/generic、小范围、WorldCore 裁决”。
- 任何需要真实原著锚点的方案都必须先走 v4.6 intake。

## v4.8：最小 Auto-Theater first cut 候选

本阶段只有在 v4.7 获得用户明确批准时才做 runtime。

候选路线：

- 路线 A：report-only 继续硬化，适合证据不足时。
- 路线 B：non-runtime visual harness，仅用静态 mock data 验证 theater 可读性，仍不接游戏。
- 路线 C：最小 runtime first cut，在一个小范围 synthetic/generic 或 canon-safe shadow encounter 中接入 Auto-Theater 表达。

路线 C 的最小边界：

- 只允许一个 bounded scenario。
- 只允许 WorldCore / Combat Core 裁决结果。
- DeepSeek 只可表达和生成候选意图。
- 不写新 save field，除非用户另批 save-format gate。
- 不开放正式高阶地点、阵营、奖励、NPC 生死或仙蛊屋归属。

## v4.9：v5.0 前安全收束

目标：

- 像 v3.9 进入 v4.0 前一样，做 v5.0 go/no-go。
- 证明 v5.0 第一刀不是凭感觉冲刺，而是证据链足够。

必须复核：

- v4.0-v4.8 是否按路线推进，是否有文档写了但代码/测试没有兑现。
- `F-400` 或后续 future gates 是否有打开项，证据是否完整。
- old-save / no-save / rollback 是否稳定。
- Player Advocate 是否证明玩家能理解 Auto-Theater、Combat Ledger 和宏观压力。
- deterministic / live / visual / long drift gates 是否满足 v5.0 所需强度。
- MiroFish / knowledge / source-pointer / hidden boundary 是否干净。
- external framework 是否仍未越权。
- Git/CI/main merge/EdgeOne 边界是否清楚。

输出：

- v5.0 go/no-go。
- v5.0 前置授权包草案。
- v5.0 例外停机清单。
- v5.0 MiroFish need-level。
- v5.0 Player Advocate / live probe 档位建议。

## v5.0：高阶世界第一刀

v5.0 的建议定位：

`Auto-Theater Combat + Macro Pressure 的第一个受控可玩试点`

它应该做到：

- 玩家第一次在可玩流程里感受到高阶战斗不再只是平面棋盘。
- 玩家能看见准备、态势、杀招栈、自动结算、Combat Ledger 和宏观压力如何共同影响体验。
- WorldCore / Combat Core 仍拥有最终裁决权。
- DeepSeek 仍只做表达、候选意图和叙事润色。
- HeavenWill/Fate 仍只做 pressure / constraint / risk，不做结局裁决。
- 任何真实原著锚点都有 MiroFish intake、source pointer、visibility 和 promotion 状态。

它不能声称：

- 完整开放蛊仙。
- 完整开放五域两天。
- 方源、尊者或原著关键人物成为 live agent。
- 所有蛊虫、杀招、仙蛊屋可玩。
- 天道/宿命可直接决定玩家结局。
- DeepSeek 可裁决战斗、奖励、生死、地点或正史。

## 测试制度演进

v4.x 的测试不应只加脚本名，而要逐步覆盖以下门禁：

| 门禁 | 覆盖目标 |
|---|---|
| `V4X-CONTRACT` | Auto-Theater contract 字段完整、语义不漂 |
| `V4X-LITE` | 凡阶 Lite 映射不改变现有 runtime |
| `V4X-THEATER` | 空域/地面/地下/水域/洞天/阵法/领域/仙蛊屋/环境压力可区分 |
| `V4X-STACK` | 杀招栈成功、失败、反制、反噬和成本都需要证据 |
| `V4X-LEDGER` | Combat Ledger 区分 facts/candidates/rejected/needs_user_decision/pressure |
| `V4X-ART` | 视觉不把候选画成事实，不暗示奖励、生死、地点毁灭或正史改变 |
| `V4X-HEAVEN` | HeavenWill/Fate 只做压力，不做万能失败或结局裁决 |
| `V4X-MIROFISH` | 真实原著锚点触发 blocking，不绕过 intake |
| `V4X-DEEPSEEK` | DeepSeek 不裁决战斗结果、奖励、生死、地点或宿命 |
| `V4X-PA` | 玩家能理解准备、态势、事实、候选、拒绝、宏观压力和下一步 |

## 如何保证不偏

每个 v4.x 启动会必须做六件事：

1. 引用本总纲，说明当前小版本处于 v4.1-v4.9 的哪一段。
2. 写清楚本小版本是否仍是 report-only / design-gate / visual-harness / runtime first cut。
3. 重新列出打开了哪些 gate，没有打开哪些 gate。
4. 若触及真实原著、命名 NPC、hidden-adjacent、方源证据、尊者布局或 formal lore conclusion，先升级 MiroFish need level。
5. 把新增高风险样本纳入测试矩阵演进规则，不能只写 TODO。
6. 阶段完成时执行 Skill sync audit、dashboard/AGENTS/PROJECT-STATE/handoff 同步和 Git 证据记录。
