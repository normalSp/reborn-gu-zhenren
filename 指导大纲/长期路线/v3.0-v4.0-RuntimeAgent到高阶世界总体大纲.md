# v3.0-v4.0 Runtime Agent 到高阶世界总体大纲

日期：2026-05-24
状态：长期路线总纲；承接 `v2.0-v3.0-AgentLab到RuntimeAgent总体大纲.md` 与 `v2.0-v4.0-分层Agent与世界内核/`；不授权当前 runtime。

## 定位

这份总纲把 v3.0 完成后的路线拆成 v3.1-v3.9 与 v4.0 的可执行方向。

当前选择已经锁定：

- v3.1 先继续加固，不急着真正接 runtime agent。
- v3.x 按 v3.1-v3.9 逐小版本规划。
- v4.0 定位为“高阶战斗 + 天道/宿命”双预备，不承诺完整蛊仙开放。

核心判断：

`v3.0 -> v4.0` 不是从 report-only 直接跳到完整 agent society，也不是让所有 NPC 都 live DeepSeek。正确路线是先把 runtime agent 的门禁、回滚、旧档、Player Advocate、live probe 和长期漂移证据压实，再决定是否开启最小 L2/L3 runtime 第一刀；高阶战斗和天道/宿命只能作为 v4.0 架构预备。

- 不改 runtime/source/UI/store/prompt/save。
- 不实现 runtime agent implementation。

- 不引入 Hermes、LangGraph、Mastra、Dify、Flowise、AutoGPT、Agno、Browser-use、LlamaIndex 或其他外部 framework 的 PoC、dependency、vendored subset、subagent、patch artifact、项目读写、命令或 git 权限。

## v3.1：runtime agent 门禁二次加固

目标：

- 把 v3.0 的设计门禁从“能识别边界”推进到“能承受更像真实 runtime 的压力”。
- 复核 `AgentProposal -> WorldCore post-check -> approved expression / rejected / needs_user_decision`。
- 强化 rollback、old-save、Player Advocate、live metadata、drift checkpoint 的验收方式。

建议产物：

- v3.1 专家团启动会与前置授权包。
- v3.1-a1 `AgentProposal contract v2`，只扩展设计约束，不改 runtime。
- v3.1-a2 rollback / old-save / PA / live gate 二次加固。
- v3.1-b1 report-only checker 扩展，加入更长链路的 negative fixtures。
- v3.1-rc 质量收束，确认是否足以进入 v3.2 rehearsal。

不做：

- 不打开 F-300-001。
- 不实跑 live DeepSeek，除非用户单独批准 F-300-003 的样本、轮次、成本和接受标准。
- 不碰 save-format。

## v3.2：离线/报告型 runtime rehearsal

目标：

- 模拟一个“看起来像 runtime”的全链路，但保持 report-only/offline。
- 验证 agent 候选如何进入表达候选、如何被 WorldCore 拒绝或降级、如何被 Player Advocate 记录。

建议产物：

- synthetic world snapshot。
- L2/L3 proposal batch。
- WorldCore post-check rehearsal report。
- approved expression sample，不进入 UI。
- rejection / needs_user_decision / manual_review 趋势报告。

验收方向：

- P0/P1 falseNegative 必须为 0。
- 所有 formal authority、hidden leak、save write、DeepSeek authority、MiroFish blocking、external framework boundary 都必须能被识别。
- same-start difference 只能表现为候选差异，不得污染事实。

## v3.3：F-300 reopen 决策包

目标：

- 给用户一个可一次性决策的 F-300 reopen 包。
- 明确到底是否值得进入最小 runtime 第一刀。

必须回答：

- 是否打开 F-300-001：runtime agent implementation。
- 是否需要 F-300-002：save field / `SAVE_FORMAT_VERSION` / migration / `runFingerprint`。
- 是否打开 F-300-003：live DeepSeek 实跑、prompt/context/model/authority。
- 是否需要 F-300-005：MiroFish blocking topic-slice。
- 是否触发 F-300-004、F-300-007、F-300-008、F-300-009。

默认建议：

- v3.3 只做决策包，不默认实现。
- 若证据不足，继续 report-only hardening。
- 若证据足够，下一版本才进入最小 runtime 第一刀。

## v3.4-v3.6：最小 L2/L3 小区域试点候选

这段只有在未来用户明确批准 F-300-001 及相关 gate 后才进入实现。

候选切法：

- v3.4：一个 small-area、一个 L2 或 L3 proposal-only path。
- v3.5：加入 Player Advocate / live probe / deterministic replay 对照。
- v3.6：加入 rollback、old-save、same-start difference 和 drift hardening。

WorldCore 保持：

- 最终事实裁决。
- save 写入裁决。
- 奖励、地点、身份、NPC 生死、战斗结果、正式势力结果裁决。
- DeepSeek visible context 与 hidden/private boundary 裁决。

agent 只允许：

- 生成 NPC 意图候选。
- 生成表达候选。
- 生成记忆反思候选。
- 生成小势力压力提示。
- 提供 safe next step 建议。

## v3.7-v3.8：多 NPC、小势力和长期差异复核

目标：

- 评估最小试点能不能扩到多 NPC、小队成员、小势力。
- 证明同开局差异是“世界因果 + agent 候选”带来的有效变化，不是随机噪声。
- 检查长期记忆污染、目标遗失、重复空转、叙事漂移。

重点样本：

- 多 NPC 对同一公开事件形成不同利益判断。
- 小队成员记住玩家公开行为，但不把谣言写成事实。
- 小势力压力随公开行动变化，但不直接生成通缉、招揽、封锁。
- 同开局多次游玩出现不同对话和机会窗口，但不改变 WorldCore 事实。

硬停：

- 命名 NPC、真实势力、hidden-adjacent、方源公开证据、L4/L5 锚点进入 scope 时，MiroFish 必须升级为 blocking。
- formal relation、faction standing、reward、NPC life/death 仍需单独门禁。

## v3.9：v4.0 前安全收束

目标：

- 复盘 v3.x 是否已经真的满足进入高阶世界预备的条件。
- 统一 agent、DeepSeek、MiroFish、外部框架、save-format、Player Advocate 和 long drift 边界。

必须复核：

- F-300 是否有任何项被打开，打开证据是否完整。
- runtime agent 是否仍保持 WorldCore final authority。
- live probe 是否无 P0/P1，成本和 retry 是否可控。
- Player Advocate 是否证明玩家能理解 agent 带来的差异。
- old-save、rollback、same-start difference 是否稳定。
- 外部 framework 是否仍只停留在 architecture reference / adapter candidate。
- MiroFish topic-slice 是否只作为 candidate/rule/test material。

输出：

- v4.0 go/no-go。
- 高阶战斗预备准入清单。
- Fate/HeavenWill/L5 宏观系统预备准入清单。
- L4 原著关键人物仍默认不开放的复核记录。

## v4.0：高阶战斗与天道/宿命双预备

v4.0 的定位是预备，不是完整开放。

### 高阶战斗预备

候选方向：

- theater map：比低阶棋盘更能表达飞行、高低、领域、阵法、地形破坏。
- killer-move stack：杀招准备、反制、连锁、反噬、资源成本。
- formation / domain：阵法、领域、环境约束。
- Immortal Gu House state：仙蛊屋作为系统状态，而不是普通装备。
- environment damage：只显示 WorldCore / combat engine 已裁决的环境变化。

不做：

- 不开放所有仙蛊、杀招、仙蛊屋。
- 不让 DeepSeek 或表现层裁决命中、伤害、资源消耗、NPC 生死、奖励。
- 不把战斗旁白写成事实结算。

### 天道/宿命预备

候选方向：

- Fate state：沿用 `intact`、`fractured`、`destroyed` 的强边界。
- HeavenWill pressure：压力、修正、排斥、机会偏置，不是万能失败按钮。
- era anchor graph：时代、正史锚点、区域大势、尊者布局的 source-pointer 图。
- causality debt：玩家偏离正史后，世界压力如何升高。
- L5 macro director：只输出 pressure / constraint / risk，不作为可聊天 NPC。

不做：

- 不直接让天道/宿命裁决玩家结局。
- 不开放方源等原著关键人物自由 live agent。
- 不把隐藏因果链暴露给 UI 或 DeepSeek visible context。

## 外部框架位置

外部框架继续按 `Agent-Framework-Landscape-2026吸收矩阵.md` 处理。

默认结论：

- Hermes：P0 架构参考 / P1 隔离 PoC 候选，但当前不批准 PoC。
- LangGraph、Mastra：继续作为 orchestrator / workflow / memory / eval 的架构参考。
- Dify、Flowise、AutoGPT、Agno、Browser-use、LlamaIndex：继续按 WorldCore 服务能力吸收思想。

任何框架进入下一步前，必须单独审批：

- license / SBOM。
- 隔离 PoC 目录与删除/归档计划。
- 是否安装 dependency。
- 是否读取 RebornG 文件。
- 是否调用模型。
- 输出格式。
- 是否生成 patch artifact。

默认禁止：

- 外部 framework 直接写 RebornG runtime/canon/save/prompt/knowledge-index。
- 外部 framework 执行 RebornG 命令或操作 git。
- self-learning self-adjudicate。

## MiroFish 与知识库位置

v3.1-v3.3 若保持 synthetic/report-only 小区域样本，MiroFish 默认 `not_needed`。

升级为 blocking 的触发器：

- 命名 NPC。
- 真实原著势力。
- 方源公开证据。
- hidden-adjacent material。
- L4/L5 fact anchor。
- 真实地点/阵营/身份/奖励/NPC 生死。
- formal lore conclusion。
- knowledge-index body。
- DeepSeek visible lore/RAG。

吸收链仍是：

`RebornG request -> MiroFish export -> RebornG intake review -> rule/test/candidate material`

不得把 MiroFish 基础包、hidden body 或原著正文直接导入 runtime、UI、prompt 或 DeepSeek visible context。

## 测试与验收基线

v3.x 必须继续沿用：

- report-only checker。
- deterministic replay。
- long narrative drift checkpoint。
- Player Advocate。
- live probe metadata。
- old-save / rollback review。
- test matrix evolution。
- Skill sync audit。
- Git branch / push / CI 证据。

未来若进入 player-facing runtime：

- 小版本至少 10 轮 Player Advocate。
- rc 至少 50 轮 Player Advocate。
- runtime agent 高风险 rc 建议 80-120 轮。
- live probe 默认 `deepseek-v4-flash`，样本、轮次、成本、retry、失败率、报告路径必须记录。
- P0/P1 直接停机，不得靠措辞绕过。

## 当前结论

v3.0 已把 runtime agent 的“笼子”做好，但还没有把 agent 放进玩家 runtime。v3.1-v3.3 的正确任务是继续证明这套笼子足够可靠；只有当 F-300 reopen 包证据充分并获用户单独批准，v3.4 之后才考虑最小 L2/L3 小区域 runtime 第一刀。

v4.0 的正确定位不是“蛊仙全开放”，而是把高阶战斗表达和天道/宿命宏观压力拆成可测试、可回滚、可被 WorldCore 裁决的架构预备。
