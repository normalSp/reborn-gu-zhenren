# v4.0-v12.0 RebornG 终局形态总体大纲

日期：2026-05-26
状态：长期路线总纲；承接 `v4.0-v5.0-AutoTheater到高阶世界Runtime总体大纲.md`、`v3.0-v4.0-RuntimeAgent到高阶世界总体大纲.md`、`v2.0-v3.0-AgentLab到RuntimeAgent总体大纲.md` 与 `v2.0-v4.0-分层Agent与世界内核/`；不授权当前 runtime。

## 专家团结论

有必要做一份 `v4.0-v最终版本` 总纲。

原因很直接：`v4.0-v5.0` 只回答“Auto-Theater Combat 与 HeavenWill/Fate Pressure 如何走到第一个高阶世界可玩试点”，还没有回答“从第一个高阶试点怎么走到你想要的终局游戏”。如果没有这份总图，后续很容易在 v5 以后继续堆小功能，却忘记最终目标是一个可长期运行、可任意身份进入、能覆盖五域两天、NPC/势力/经济/战斗/宿命都有深度的活世界。

专家团建议把终局规划暂定为 `v4.0 -> v12.0 final-candidate`。这不是锁死最终版本号，而是一个工程规划地平线：按当前项目野心，离真正终局至少还差 `8 个左右大版本`。如果 v5-v7 进展异常顺利可以压缩；如果外部框架、后端、MiroFish、原著关键人物或高阶战斗打开后风险增大，可能会扩展到 v13+。

一句话判断：

> v5.0 可以做高阶世界第一刀；v8.0 才可能像一个多系统活世界；v10.0 才可能安全触碰原著关键时代与重要人物；v12.0 才有资格称为终局候选。

## 终局形态

RebornG 的终局形态不是 AI 酒馆，不是普通人生重开，也不是纯文字跑团。它应该是：

> 一个由 WorldCore 裁决、由分层 Agent 提供候选、由 DeepSeek 表达叙事、由 MiroFish/知识库守住原著边界的蛊真人活世界 RPG。

终局玩家体验应包含：

- 玩家可以选择不同时间点、地域、身份和立场进入世界，包括原创身份、低阶散修、小势力成员、商队人物、正道/魔道边缘身份，以及未来经过强门禁后的书中人物穿越模式。
- 同一开局可以因为 NPC 意图、势力压力、经济波动、公开行动和宏观时代压力产生不同体验，但底层事实不随机漂移。
- 每个选择、战斗、交易、小队关系、公开传闻和失败代价都能进入可解释后果链。
- NPC 有长期目标、情绪、记忆、误解、利益判断和公开反应，但不能越权写事实、奖励、生死、阵营、地点或 canon。
- 战斗不只是棋盘，也不是不可交互播片；凡阶用 `Auto-Theater Lite` 保持可读和可控，高阶用 `Auto-Theater Combat` 表达空域、地下、水域、洞天、阵法、领域、仙蛊屋、杀招栈和环境破坏。
- 天道/宿命不是结局按钮，而是 pressure / constraint / risk / era anchor 系统。它让玩家感受到时代大势和因果压力，但最终结论仍要经过 WorldCore、canon gate 和用户批准的版本门禁。
- DeepSeek 负责语言、表达、候选意图和叙事润色；最终事实、数值、位置、奖励、NPC 生死、正史锚点和存档写入仍由本地或未来服务端 WorldCore 裁决。

这些地基最宝贵的不是某个功能，而是一套工程节奏：

- `WorldCore final authority`
- `AgentProposal candidate-only`
- `MiroFish request -> export -> intake review -> rule/test/candidate`
- `Player Advocate + Long Narrative Drift`
- `frontloaded approval + exception stop`
- `Skill sync audit`
- `Git branch / push / CI / no auto-main-merge`
- `historical-index + dashboard + PROJECT-STATE`

这套制度是终局路线的核心资产。后续越自由，越不能削弱它。

## v4.x：Auto-Theater 与宏观压力地基

主线：`Auto-Theater Combat + HeavenWill/Fate Pressure`

v4.x 已有单独总纲：`v4.0-v5.0-AutoTheater到高阶世界Runtime总体大纲.md`。

它必须完成：

- Auto-Theater contract。
- 凡阶 `Auto-Theater Lite` 映射。
- 高阶 theater layer：空域、地面、地下、水域、洞天、阵法、领域、仙蛊屋、环境压力。
- killer-move stack 与 Combat Ledger。
- HeavenWill/Fate/L5 macro pressure。
- MiroFish/lore intake 准入。
- v5.0 go/no-go。

v4.x 不应该急着做“完整蛊仙玩法”。它应该把高阶玩法的语言、证据链、UI/视觉边界和测试门禁先打稳。

## v5.x：高阶世界第一刀

目标：

- 让玩家第一次在可玩流程里感受到高阶世界：不是纯文字描述，而是能看见准备、态势、杀招栈、自动结算、Combat Ledger 和宏观压力。
- WorldCore/Combat Core 仍裁决结果，DeepSeek 仍表达和提出候选。
- 不追求全量蛊仙，只做一个受控切片。

建议切分：

- v5.0：高阶世界 first cut go/no-go 与最小 playable slice。
- v5.1-v5.3：Auto-Theater runtime 最小闭环、视觉 harness、reduced-motion、移动端可读性。
- v5.4-v5.6：杀招栈、反制、反噬、资源消耗和 Combat Ledger 玩家理解。
- v5.7-v5.9：HeavenWill/Fate pressure 进入玩家可见但不裁决结局的体验链。

终局贡献：

- 证明高阶战斗不是只靠棋盘。
- 证明宏观压力能成为游戏系统，而不是剧情词。

## v6.x：多区域与多身份基础

目标：

- 从单区域活世界走向多区域、多身份、多路线。
- 支持玩家离开一个区域后，旧区域仍有状态、传闻、追击、供应、债务和记忆。
- 身份不只是开局文本，而是长期权限、风险、资源、目标和人际压力。

建议切分：

- v6.0：多区域 route/state contract。
- v6.1-v6.3：旅行、滞留、追击、供给、地图可见性。
- v6.4-v6.6：身份路线 contract：散修、商队、宗族边缘、小势力、临时伪装。
- v6.7-v6.9：多区域 old-save/rollback/long drift 与 Player Advocate。

参考：

- 沙盒 RPG 的区域压力可参考 `Kenshi`、`Mount & Blade`、`Crusader Kings` 的身份/势力/地缘思想。
- 实现上仍应走 RebornG-owned TypeScript core，不把外部游戏架构照搬为 runtime。

## v7.x：Agent Society 扩展

目标：

- 从 L2/L3 小范围 agent 候选扩展到多 NPC、小队、商队、小势力和城镇社会。
- NPC 有记忆、情绪、误解、利益判断和目标，但仍不拥有事实裁决权。
- 同开局差异来自公开事件、agent 候选、势力压力和经济状态，而不是无约束随机。

建议切分：

- v7.0：Agent Society contract v1。
- v7.1-v7.3：小队成员长期目标、事件后反思、信任/怀疑候选。
- v7.4-v7.6：商队、小势力、城镇传闻、公开压力。
- v7.7-v7.9：同开局差异度、记忆污染、空转、目标遗失和 hidden leak 长测。

参考：

- `Generative Agents`、`Concordia`、`AI Town`：记忆、反思、计划、社会模拟。
- `SOTOPIA`：社交智能评估。
- `CICERO`：战略推理和自然语言表达分离。
- `Hermes Agent`、`LangGraph`、`Mastra`：自学习、workflow、memory、eval 的架构思想。

边界：

- agent 输出仍是 `AgentProposal`。
- self-learning 只能生成 `candidate_memory_patch` / `candidate_skill_patch`。
- agent 不写 canon/runtime/save/knowledge-index/prompt。

## v8.x：经济、炼养用、宝黄天与资源生态

目标：

- 把经济从低阶生存扩展到高阶资源生态。
- 玩家能通过炼养用、商路、库存、价格、势力关系、风险和交易窗口理解世界运行。
- 宝黄天作为仙阶经济边界进入系统设计，但 mortal start 不可直接交易。

建议切分：

- v8.0：经济与资源生态总 contract。
- v8.1-v8.3：低阶/中阶市场、库存、价格、供需、反刷、商队。
- v8.4-v8.6：炼养用、蛊材/仙材、蛊方/残方、养蛊成本、失败代价。
- v8.7-v8.9：宝黄天、仙窍、仙材、仙元/仙元石边界预备。

参考：

- 管理和经济模拟可以参考 `RimWorld`、`Dwarf Fortress`、`Against the Storm` 的供需/风险/事件压力思想。
- 知识和材料仍以 `reverend-insanity-lore`、MiroFish intake、project canon 为准。

边界：

- DeepSeek 可以表达商贩态度、供需传闻和风险候选。
- 价格、库存、交易、奖励、掉落、套利封禁由 engine/canon 裁决。
- 不做全书 RAG，不让模型即兴创造可入库材料、蛊方、仙蛊或杀招。

## v9.x：高阶战斗与战争系统成熟

目标：

- 把 v5 的高阶第一刀扩展为可长期运行的高阶战斗系统。
- 能表达蛊仙上天入地、阵法、领域、洞天、仙蛊屋、环境破坏和大规模冲突。
- 玩家每一步准备、资源投入、杀招选择、反制窗口和失败原因都能被看见。

建议切分：

- v9.0：Auto-Theater Full contract v2。
- v9.1-v9.3：杀招栈、counter window、backlash、资源成本、证据链。
- v9.4-v9.6：仙蛊屋、阵法、领域、环境破坏、区域战场。
- v9.7-v9.9：战争/围剿/追杀/大势力冲突的 report-only 到 runtime 准入。

参考：

- 自走战斗可以吸收 `TFT`、`Mechabellum`、`Backpack Battles`、`Yi Xian` 的准备与自动结算优点。
- 开源侧可以参考 `boardgame.io` 的回合/状态思路、`OpenRA` 的 RTS 状态表达、`PixiJS`/`Phaser` 的 2D 渲染生态、`xyflow` 的图式可视化思想。
- RebornG 不应把这些变成直接依赖结论；先通过 license/SBOM/architecture fit，再考虑 isolated PoC。

边界：

- UI/动效不能创造命中、伤害、资源消耗、NPC 生死、环境毁灭、奖励或归属变化。
- DeepSeek 在战斗里只能表达和候选意图，不能裁决结果。

## v10.x：原著时代锚点与关键人物安全接触

目标：

- 让玩家有机会接触原著重要角色、关键时代、主线大事件和宏观因果压力。
- 接触不是自由聊天，也不是让关键人物 live agent 随便改写剧情。
- 每个关键人物、关键事件、隐藏因果都必须有 source pointer、visibility、promotion 状态和测试样本。

建议切分：

- v10.0：原著时代锚点准入总门禁。
- v10.1-v10.3：非关键人物/低风险公开锚点 first cut。
- v10.4-v10.6：重要人物 shadow-contact / rumor / indirect evidence。
- v10.7-v10.9：关键人物有限接触、正史压力、IF 偏离债务。

参考：

- 叙事选择和后果表达可参考 `Disco Elysium`、`Pentiment`、`Tyranny` 的对话后果和身份压力思想。
- 原著事实必须来自 MiroFish topic-slice intake 或项目 canon，不用模型记忆即兴补。

边界：

- 方源、尊者、天庭核心、幽魂等关键人物默认不能自由 live。
- hidden-adjacent、方源公开证据、尊者布局、真实战役、formal lore conclusion 必须 MiroFish blocking。

## v11.x：任意时代/身份与内容工厂

目标：

- 把“我想任意时间点魂穿任意人物或原创角色”从愿望变成可审计生产线。
- 不是每次人工堆内容，而是把 MiroFish、知识库、Agent Lab、eval farm、测试矩阵、Player Advocate 和版本门禁组成内容工厂。

建议切分：

- v11.0：era/start identity factory contract。
- v11.1-v11.3：身份模板、区域模板、资源模板、势力压力模板。
- v11.4-v11.6：topic-slice intake 自动化/半自动化、coverage report、redaction chain。
- v11.7-v11.9：内容晋升链：candidate -> rule/test -> limited runtime -> PA/live/drift -> canon subset。

参考：

- `TextWorld` 的文本环境评测思想。
- `LlamaIndex` / `Letta` 的知识和记忆模式，但只作为 visibility/retrieval/summary 思想，不做全书 DeepSeek RAG。
- `Hermes` 的 skills、cron、trajectory、self-learning 只能转译为候选补丁和 eval archive。

边界：

- 内容工厂不能自动 promotion。
- 生成内容、MiroFish 摘要、agent 记忆、DeepSeek 文本都不是 runtime truth。
- 用户仍要在正式地点、身份、阵营、奖励、NPC 生死、关键 lore 结论上拍板。

## v12.x：终局候选与世界级质量收束

目标：

- 把 RebornG 从“很多强系统”收束成一个稳定的终局候选。
- 长档、跨区域、多身份、高阶战斗、经济、NPC 社会、宏观大势、原著锚点都能长期运行。
- 玩家能在 1000+ 轮级别长线体验中保持目标清晰、世界一致、NPC 有反应、剧情不漂移、系统不无限刷、不泄露 hidden/private。

建议切分：

- v12.0：final-candidate go/no-go。
- v12.1-v12.3：1000+ round deterministic/live/mixed long soak。
- v12.4-v12.6：性能、存档、回滚、迁移、云存档/薄后端、成本观测和故障恢复。
- v12.7-v12.9：公开发布、法律/版权边界、主线合并、部署、文档、skill 和知识库长期维护。

终局候选必须证明：

- WorldCore final authority 没被 LLM、agent、UI、后端或外部框架稀释。
- DeepSeek 叙事质量在长线中没有大规模降级。
- MiroFish/knowledge/canon 边界没有泄露或污染。
- 高阶战斗和宏观压力能被玩家理解。
- 经济和奖励不能被稳定套利。
- NPC/势力长期反应不会空转、失忆或越权。
- 旧档、回滚、迁移、异常恢复和成本观测都能被工程制度支撑。

## 外部参考吸收原则

外部参考分三类：

| 类别 | 参考对象 | 用法 | 禁止 |
|---|---|---|---|
| 开源 agent/framework | Hermes、LangGraph、Mastra、OpenAI Agents SDK、Google ADK、AutoGen、PydanticAI、Letta、LlamaIndex | 架构、workflow、memory、eval、visibility、guardrail、trajectory 思想 | 未经 license/SBOM/PoC 审批不得引依赖、vendored subset、项目读写、命令或 git |
| 游戏设计参考 | Dwarf Fortress、RimWorld、Crusader Kings、Kenshi、Mount & Blade、Disco Elysium、TFT、Mechabellum、Yi Xian | 系统深度、身份压力、社会模拟、战斗准备、后果表达 | 不复制素材、文本、数值、玩法外壳；不让参考游戏替代蛊真人世界观 |
| 开源游戏/可视化技术 | boardgame.io、OpenRA、PixiJS、Phaser、xyflow、Playwright | 状态管理、战场表现、图式可视化、浏览器验收 | 不默认进入 runtime；先 architecture fit，再 isolated PoC |

吸收顺序永远是：

1. 先写 landscape / architecture fit。
2. 再做 license / SBOM。
3. 再申请 isolated PoC。
4. PoC 只能 report-only，不写 RebornG 项目。
5. 只有用户另批，才考虑 dependency、adapter、vendored subset 或 runtime 接入。

## 工程制度如何跟着终局进化

每个大版本必须保留以下制度：

1. 专家团启动会：给出主线、收益、风险、版本范围。
2. 前置授权包：一次性列 D 项和 F 项。
3. 例外停机清单：触发即停，不在 `/goal` 里硬冲。
4. 真相源索引：当前事实从哪里来，哪些来源禁止。
5. MiroFish need-level：not_needed / optional / preferred / blocking。
6. 测试矩阵：新增高风险样本必须 triage。
7. Player Advocate：玩家是否理解目标、风险、后果、下一步。
8. Long Narrative Drift：长线叙事、世界状态、lore、DeepSeek 权限是否漂移。
9. Skill sync audit：expert/game-dev/lore/combat/mirofish 必要时同步。
10. Dashboard / AGENTS / PROJECT-STATE / handoff：入口必须同步，不把制度扔在文档里没人管。
11. Git 计划：版本分支、提交、推送、CI、是否建议合 main，不能自动合并。
12. Historical index：旧版本压缩归档，不让旧阶段清单误导当前门禁。

v8 之后建议新增更强的制度：

- `world-scale soak gate`：1000+ round deterministic/live/mixed 长测。
- `cross-region save migration gate`：跨区域、跨身份、跨时代旧档迁移。
- `agent memory contamination gate`：NPC/势力/玩家/谣言/hidden/private 的记忆污染检测。
- `economy exploit farm gate`：资源、交易、宝黄天、奖励、炼养用套利检测。
- `canon contact gate`：原著关键人物和关键事件接触前的 source-pointer / visibility / MiroFish blocking 门禁。
- `visual truthfulness gate`：UI/动效/素材不得暗示未裁决事实。

## 当前路线对 v4.0 的影响

用户已经批准 `D-400-001` 至 `D-400-012`，并确认 `F-400-001` 至 `F-400-012` 全部继续 `future_gate_required`。

这意味着：

- v4.0 可以在授权包内完成设计门禁、report-only checker、PA/old-save/no-save/rollback 证据、文档收束、Skill sync、Git/CI。
- v4.0 仍不能实现高阶 runtime、凡阶 runtime 迁移、pure auto-battler runtime、theater UI、素材生成、live DeepSeek、MiroFish blocking、后端、外部框架 PoC、save field 或正式世界事实。
- 本 `v4.0-v12.0` 总纲只作为终局路线基线，不新增任何 D/F 授权。

## 总结

我们没有在造一个普通“AI 会聊天”的游戏。我们在造的是一个可裁决、可回滚、可长测、可扩展、可接原著锚点、可承载自由意图的活世界。

目前地基是稳的，但离终局还远。v2.x 和 v3.x 已经把 agent 和 WorldCore 的笼子搭出来；v4.x 和 v5.x 要把高阶战斗和宏观压力做成第一个高阶体验；v6-v9 要把区域、身份、社会、经济、战斗扩成真正的世界；v10-v12 才能安全触碰原著关键时代、关键人物、任意身份和终局质量。

终局路线的关键不是“让 DeepSeek 权力更大”，而是让世界更会判断、更会记忆、更会拒绝、更会反馈，也更会在长线里保持自己不崩。
