# v2.0-v3.0 Agent Lab 到 Runtime Agent 总体大纲

日期：2026-05-22
状态：长期路线总纲；承接 `v2.0-v4.0-分层Agent与世界内核/` 研究线；不授权当前 runtime。

## 定位

这份总纲把 v2.0 到 v3.0 的路线从愿景拆成可执行小版本。

核心判断：

`v2.0 -> v3.0` 不是“马上让所有 NPC live DeepSeek”，也不是“继续堆区域 UI”，而是从已验证的第一个区域活世界，逐步走到第一个受控 runtime agent 试点。

总路线：

1. v2.0：完成第一个区域活世界，形成 WorldCore、ledger、T3 长测基线。
2. v2.1-v2.3：完成 Agent Lab 和 eval farm 地基。
3. v2.4-v2.6：评估薄 BFF、private canon、job queue、replay archive 等基础设施边界。
4. v2.7-v2.9：完成 runtime agent 接入前的多 NPC 场景、准入门禁和安全收束。
5. v3.0：只在一个小区域接入有限 L2/L3 runtime agent。

## 硬边界

本文件是长期规划，不授权以下事项：

- 不改 runtime。
- 不新增后端/BFF。
- 不调用 live DeepSeek。
- 不新增 DeepSeek prompt/context/model/authority。
- 不新增 save 字段，不 bump `SAVE_FORMAT_VERSION`。
- 不启用 subagents。
- 不新增 MiroFish export/intake。
- 不开放正式地点、阵营、奖励、NPC 生死或 canon promotion。
- 不自动部署 EdgeOne。

任何长期目标进入当前版本前，必须另写当前版本启动审查、需求决策池、真相源索引、测试矩阵，并取得用户决策。

## 总体阶段

| 阶段 | 主线 | 目标 | 默认边界 |
|---|---|---|---|
| v2.0 | 第一个区域活世界 | 已完成，作为 T3 和 WorldCore 基线 | 不等于完整南疆或完整 agent 社会 |
| v2.1-v2.3 | Agent Lab | 离线证明 agent proposal、记忆、反思、势力压力和 L5 宏观导演可审计 | 不进玩家 runtime，不写正式存档 |
| v2.4-v2.6 | 薄 BFF / 私有事实 / 归档边界 | 评估 API proxy、private canon、job queue、cost/archive | 默认只评估，不实现后端 |
| v2.7-v2.9 | runtime agent 预备 | 扩大低阶多 NPC 样本，建立准入门禁和安全收束 | 不直接接入 L4/L5，不授权完整社会 |
| v3.0 | 有限 runtime agent 接入 | 只允许 L2/L3 在一个小区域提供候选和表达 | WorldCore 保持最终裁决权 |

## v2.0：第一个区域活世界

状态：已完成本地开发。

作用：

- 证明 `regionalEventLedger`、WorldCore、同开局 replay lane、旧档/回滚 review 和 T3 长测能守住第一个低阶区域活世界。
- 为 Agent Lab 提供真实失败样本和 hardening 输入。

关键证据：

- T3 deterministic/replay 160/160 accepted，P0/P1/P2=0/0/0。
- T3 live clean3 160/160 accepted，P0/P1/P2=0/0/0。
- mixed total 320/320 accepted。
- rc Player Advocate 100 轮通过。

不能外推：

- 不代表完整南疆。
- 不代表完整商家城。
- 不代表正式地点/阵营/奖励/NPC 生死开放。
- 不代表完整 agent 社会已验证。

## v2.1：Agent Lab 启动与架构门禁

主线：

- v2.0 T3 复盘。
- Claude Code 与开源 Agent 框架架构尽调。
- L0-L5 agent 分层。
- `AgentProposal` envelope。
- WorldCore post-check。
- 权限矩阵与 eval 指标。

默认产物：

- a0：T3 复盘与 Agent Lab 范围冻结。
- a1：Claude Code 与开源 Agent 框架架构尽调，只吸收官方/开源公开架构模式，不复制源码、不引入依赖。
- a2：Agent Lab 设计门禁草案。
- b1：用户已批准受限 report-only/offline runner 第一刀；仅 TypeScript-first、zero-dependency、dry-run only，报告可写入 `artifacts/v2.1.0/`。

不做：

- 不调用 live DeepSeek。
- 不启用子代理。
- 不实现 BFF/backend。
- 不引入第三方 agent SDK / framework 依赖。
- 不做外部 framework PoC。
- 不让外部 SDK/agent 访问项目文件、执行命令、生成补丁或操作 git。

后续决策：

- D-211 已批准 b1 受限 runner，不批准 b1 live DeepSeek。
- 外部 framework PoC 暂不批准；先让自有 AgentProposal 门禁跑通，再重议。
- 只读/分析型子代理暂不批准；后续需单独风险收益报告。
- 开源 SDK/agent 权限转入 D-212：未来可以评估 license、SBOM、隔离 PoC、vendored subset、只读扫描或 patch artifact，但不默认开放项目写入/命令/git。

## v2.2：Agent Lab offline runner 第一刀

默认目标：

- 在 v2.1-b1 受限 runner 基础上，扩展离线、report-only 的 Agent Lab runner。
- 输入 JSON 场景。
- 输出报告，不写正式存档。
- 默认跑 20 NPC、3 势力、1 个 L5 宏观导演。

runner 必须验证：

- agent 只能输出 `AgentProposal`。
- 所有 proposal 必须经过 WorldCore post-check。
- L2/L3 可以有记忆、反思、意图候选，但不能写事实。
- L5 只输出 pressure，不作为可聊天 NPC，不宣判结局。

禁止：

- 不接玩家 runtime。
- 不写 store/canon/save。
- 不把 hidden/private body 放进 DeepSeek visible context。
- 不让 agent 写奖励、正式地点、正式身份、NPC 生死。

需要用户拍板：

- 是否允许实现 runner。
- 是否允许 live DeepSeek 样本，或先只做 dry-run/replay。
- report 输出是否进入 `artifacts/v2.2.0/`。

## v2.3：agent eval farm 与失败分类

默认目标：

- 把 Agent Lab 从“能跑”推进到“能判定是否可靠”。
- 建立 agent failure taxonomy。
- 把 v2.0 T3 的 hardening 经验迁移到 agent 场景。

必须覆盖的失败类型：

- formal credential 越界。
- hidden leak / hidden echo。
- DeepSeek authority drift。
- agent 记忆污染。
- proposal 写事实。
- 重复空转。
- NPC 目标遗失。
- 势力压力不随公开行为变化。
- L5 宏观导演替玩家宣判结局。

默认产物：

- eval matrix。
- failure taxonomy。
- replay/rescore 报告。
- future_sample_pool。

禁止：

- 不把 eval 通过等同于 runtime agent 批准。
- 不把小样本通过写成完整南疆或完整社会验证。

## v2.4：薄 BFF 边界评估

默认目标：

- 只评估薄 BFF 是否必要，不默认实现。
- 明确后端能做什么、不能做什么。

候选能力：

- DeepSeek API proxy。
- private canon / hidden body 可见性服务。
- agent job queue。
- prompt hash、token、cache、cost、retry、failure 观测。
- replay/eval archive。
- 云存档预研。

硬边界：

- BFF 不能接管 WorldCore。
- BFF 不能批准奖励、NPC 生死、地点、身份或 canon。
- BFF 不能绕过 MiroFish intake。
- BFF 不能把 hidden/private body 直接喂给 DeepSeek。

需要用户拍板：

- 是否进入后端实现专项。
- 部署平台和密钥策略。
- 后端是否只服务开发/eval，还是未来进入公开 runtime。

## v2.5：private canon / knowledge visibility 试验设计

默认目标：

- 解决 agent 时代 hidden/private fact 如何安全使用。
- 明确 full-book knowledge、MiroFish 基础包、source pointer、knowledge index 和 runtime canon 的边界。

默认产物：

- visibility gate 设计。
- private summary ref 设计。
- source pointer policy。
- knowledge-index allowedUses / forbiddenUses 样板。

禁止：

- 不做全书 RAG。
- 不把全书 MiroFish 基础包导入 DeepSeek visible context。
- 不把 hidden/private body 放入 UI 或公开前端。
- 不让 L4 原著关键人物自由 live。

## v2.6：agent job queue / replay archive 工程预备

默认目标：

- 为批处理反思、成本观测、回放复评和长期 agent 实验准备基础设施设计。
- 仍以 report-only 或离线为默认。

候选能力：

- job schema。
- replay archive schema。
- cost/cache/retry 记录。
- prompt hash 与 evaluator version。
- failed sample 回流到测试矩阵。

禁止：

- 不默认接入生产 runtime。
- 不默认引入外部队列服务。
- 不让 job 直接写 save/canon/store。

## v2.7：低阶多 NPC / 小势力模拟样板扩展

默认目标：

- 扩大 Agent Lab 场景覆盖。
- 验证低阶区域中多 NPC、小队、小势力、商队和散修压力能长期保持一致。

候选场景：

- 商队短工被骗后续。
- 小队成员被救或被牺牲后的长期记忆。
- 散修聚落对玩家身份的怀疑。
- 地方小势力对反复投机的反应。
- 路人/商贩/L1 聚合反应与 L2 关键 NPC 反思的衔接。

禁止：

- 不开放正式商队身份。
- 不开放正式势力成员身份。
- 不写正式奖励。
- 不写 NPC 生死结论。
- 不接 L4 原著关键人物 live。

## v2.8：runtime agent 接入准入门禁

默认目标：

- 判断哪些 Agent Lab 能力有资格进入 v3.0 runtime。
- 明确每一层 agent 的准入标准。

准入候选：

- L2 事件后反思摘要。
- L2 小队成员长期目标候选。
- L2 小势力压力建议。
- L3 当前场景关键 NPC 语言与意图候选。

准入要求：

- 通过 offline runner。
- 通过 eval farm。
- 通过 hidden/private visibility gate。
- 通过 WorldCore post-check。
- 通过 Player Advocate 可理解性审查。
- 通过长线漂移检查。

默认拒绝：

- L4 原著关键人物 live。
- L5 天道/宿命 runtime 裁决。
- agent 写奖励、地点、身份、NPC 生死。
- DeepSeek visible lore/RAG。

## v2.9：v3.0 前安全收束

默认目标：

- 把 v3.0 的 runtime agent 试点收束成可执行门禁。
- 统一权限、prompt、eval、Player Advocate、MiroFish、skill sync 和 Git 边界。

必须完成：

- v3.0 design gate。
- runtime agent go/no-go。
- live probe 档位方案。
- Player Advocate 强度方案。
- rollback plan。
- old-save compatibility plan。
- MiroFish need classification。
- skill sync audit。

触发即停：

- 如果需要新持久字段，先做 save-format 设计门禁。
- 如果需要命名 NPC、hidden-adjacent、方源公开证据或正式 lore 结论，MiroFish 升级 blocking。
- 如果需要 BFF/backend，另开后端专项门禁。

## v3.0：有限 runtime agent 接入

默认目标：

- 只在一个小区域接入有限 L2/L3 runtime agent。
- agent 只参与候选和表达。
- WorldCore 仍拥有最终事实、数值、奖励、位置、NPC 生死、正史锚点和存档裁决权。

可做：

- 当前场景关键 NPC live 语言和意图候选。
- 事件后 L2 NPC 反思摘要。
- 小队成员目标候选。
- 小势力压力建议。
- WorldCore post-check 后的公开叙事表达。

不可做：

- L4 原著关键人物自由 live。
- L5 天道/宿命直接裁决玩家结局。
- agent 写奖励、地点、阵营、身份、NPC 生死或 canon。
- hidden/private 进入 DeepSeek visible context。
- 全书 RAG。
- 完整南疆/完整商家城/完整 agent 社会。

验收门：

- focused unit / schema / post-check tests。
- deterministic replay。
- live probe 记录模型、样本、轮次、成本、retry、失败率。
- Player Advocate 判断 NPC 是否有意图、世界是否有记忆、下一步是否清楚。
- 长线叙事漂移检查确认世界事实稳定。
- rollback / old-save / no-authority-expansion 复核。

## MiroFish 触发点

默认 v2.1-v2.3 的纯 Agent Lab 工程门禁不需要新 MiroFish export。

升级 blocking 的情况：

- 命名 NPC 或原著关键人物进入样本。
- hidden-adjacent 或方源公开证据进入样本。
- 正式地点、阵营、身份、奖励、NPC 生死或 canon 结论进入样本。
- L4/L5 事实锚点进入 runtime 候选。
- 知识库正文或 DeepSeek visible lore/RAG 被提议进入实现。

处理方式仍是：

`RebornG request -> MiroFish export -> RebornG intake review -> test matrix / rule draft`

## DeepSeek 与 live probe 策略

默认模型沿用项目现行 `deepseek-v4-flash`。

默认分层：

- v2.1：不调用 live DeepSeek。
- v2.2：优先 dry-run/replay；live 需用户批准。
- v2.3：可设计 live eval 档位，但执行前必须报样本、轮次、成本和接受标准。
- v2.4-v2.6：若涉及 BFF/API proxy，只评估，不默认调用。
- v2.7-v2.9：为 v3.0 runtime agent 准入准备 live/replay 对照。
- v3.0：runtime agent 接入前必须有 live probe 和 Player Advocate 双门禁。

## 成功标准

到 v3.0 前，必须能回答：

- 哪些 agent 能看见哪些事实。
- 哪些 agent 能输出哪些 proposal。
- 哪些 proposal 会被 WorldCore 拒绝。
- 哪些失败会阻塞 runtime 接入。
- hidden/private 如何不泄漏。
- 成本、retry、cache、失败率如何记录。
- 玩家是否能感到 NPC 有意图，而不是只看到后台热闹。

## 总结

v2.0 到 v3.0 的正确节奏是：

`一个稳的区域活世界 -> 一个稳的 Agent Lab -> 一个稳的基础设施边界 -> 一个严格准入的 runtime agent 试点`

RebornG 要追求世界级深度，关键不是让模型拥有更多权力，而是让每个 agent 的能力、记忆、上下文、候选和失败都被世界内核看见、审计和裁决。
