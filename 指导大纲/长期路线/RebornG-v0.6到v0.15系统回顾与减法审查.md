审查状态：已审阅；用户已批准第 2 至第 7 项减法方向，运行时代码尚未实施。
审查日期：2026-05-18。
审查范围：从历史 `v0.6.0` 到当前完成的 `v0.15.0`，只做设计、工程、玩家体验与减法评估，不改运行时代码。
MiroFish 状态：本轮是系统回顾与 UI/闭环审查，`not_needed`。后续如果把审查结论转成原著锚点、NPC/势力规则、路线事实或隐藏事实 runtime work，需要重新判定 MiroFish 包是否 `blocking` 或 `preferred`。
测试状态：本轮为文档审查草案，不触发 Player Advocate、e2e、build 或 save migration 检查。若进入小版本实施，必须按对应小版本补文档、测试、仪表盘、Git 状态。

## 1. 结论先行

RebornG 现在不是“功能没有做出来”，而是进入了另一个阶段：功能做得足够多，已经需要主动做减法、做权威归并、做玩家界面收束。

从 `v0.6.0` 到 `v0.8.0`，项目有明显的“大而全探索期”特征：升仙、宝黄天、势力、商会、小队、结局、传承、灾劫、战场、剧情锚点等系统都被打开过。它们提供了大量工程资产和测试资产，但也留下了两个风险：系统入口过多，部分高阶系统早于青茅山低阶体验出现；旧系统和新活世界系统之间存在权威重叠。

从 `v0.9.0` 开始，项目方向明显变稳。`线索 -> 出发 -> 本地结算 -> 行动账本 -> 回流文本` 成为核心协议；`v0.10.0` 做青茅山区域样板；`v0.11.0` 建立 `livingWorldState` 与自由意图闸门；`v0.12.0` 做原著事实/正史/IF 边界；`v0.13.0` 做 NPC/势力反应投影；`v0.14.0` 做路线承接；`v0.15.0` 做低阶经济、补给、炼养用边界。这条线是对的，而且比早期“大而全”更接近“活着的蛊真人世界”。

当前最大问题不是单点 bug，而是产品形态过重。玩家底栏按钮过多，`行动`、`自由目标`、`训练/道场`、`青茅/凡战/群像/大阵`、`蛊虫/杀招/炼蛊/蛊材`、`空窍/道痕`、`宿命/终局/传承` 等入口同时暴露，玩家会像在调试工具台里找功能，而不是在蛊真人世界里行动。

我的核心建议是：下一阶段不急着继续加大系统，而应先做一次“系统收束与 UI 减法”专项，目标不是删除功能，而是把功能按玩家语义合并，把高阶/调试/候选入口收起来，把核心闭环变成更少、更清楚、更像游戏的主界面。

## 3. 版本回顾

### v0.6.0：大系统压力暴露期

`v0.6.0` 的主题偏高阶系统与终局压力，例如升仙、宝黄天、商会、拍卖、蛊虫费用、NPC、阵营、数值闭环等。历史审查记录中曾出现 MerchantPanel update depth、天赋加成、叙事蛊费用、拍卖价格套利、重复仙蛊、Zod 阵营失败、NPC 性格不匹配等问题。

当前评价：

- 正面：它让项目很早暴露出经济、数值、AI 状态写入、阵营合法性、UI 无限更新等风险。这些经验后来进入 `.learnings/ERRORS.md`、Zod/validator、canary、经济测试和 DeepSeek 权限治理。
- 负面：它的内容层级太高，和“一转青茅山活世界”的玩家入口不完全匹配。宝黄天、升仙、仙蛊、终局相关入口如果过早暴露，会破坏低阶蛊师阶段的代入感。
- 当前定位：历史资产与风险教材，不应作为当前玩家主线入口。

### v0.7.0：系统广度扩张期

`v0.7.0` 打开了小队、势力、成就、扩展经济、战斗公式、动态 NPC 等方向。当前源码里仍能看到不少沉积字段，例如 `npcRelations`、`playerFaction`、`squadDispatchState`、`dynamicNPCs` 等。

当前评价：

- 正面：很多底层测试仍在，如 v070 combat、squad、achievement、data foundation，它们是长期系统可回归资产。
- 负面：旧势力/NPC/小队权威和 `v0.11+ livingWorldState` 的 `npcMemories/factionPressure/actionConsequences` 有潜在重叠。未来不能让两套系统同时决定同一件事。
- 当前定位：可保留为底层能力和历史系统，但需要在进入正式 NPC/势力账本前做“权威归并表”。

### v0.8.0：剧情锚点、IF 与战斗表现探索期

`v0.8.0` 探索了剧情锚点、IF、战场 UI、传承地、灾劫、结局框架、叙事蛊 affordance 等。它给后来的 Qingmao battle slice 和 story/IF 思路提供了资产。

当前评价：

- 正面：战场 UI、叙事结构、场景时间、传承/结局框架都有测试资产；后续青茅凡战能做起来，与这一阶段有关。
- 负面：早期 story anchor/IF 维度偏宏大，容易把宿命、尊者、仙界因果提前压到青茅山低阶阶段。后来 `v0.12.0` 把 Qingmao IF 收束到低阶轴线，这是正确纠偏。
- 当前定位：战斗表现资产可继续使用；宏大 IF 轴线只作为远期参考，不应直接进入当前青茅山低阶主线。

### v0.9.0：统一行动协议与青茅战斗竖切

`v0.9.0` 是转折点。核心协议收束为 `线索 -> 出发 -> 本地结算 -> 行动账本 -> 回流文本`，并完成训练场、传承/福地、灾劫、野外行动、旧战斗兼容、青茅凡战视觉竖切、DeepSeek telemetry、发布配置等。

当前评价：

- 正面：行动协议落地，是后来所有活世界系统能接上的主干。
- 正面：青茅凡战竖切让“战斗不是纯文本”这件事落到实处，并有 e2e/截图/资产扫描约束。
- 负面：青茅凡战仍偏“演示竖切/候选进入”，还不是所有剧情自然触发的完整战斗系统。
- 当前定位：核心协议可信；战斗表现可信；正式战斗生态仍需后续版本深化。

### v0.10.0：青茅山区域主线可玩化第一层

`v0.10.0` 建立 Qingmao 三寨区域板、区域行动、低阶蛊/战斗内容包、资源/喂养/炼蛊可读性、场景变体、生产预览 smoke 等。

当前评价：

- 正面：青茅山从“背景”变成了可行动区域，低阶蛊、资源缺口、战斗候选开始和玩家行为连接。
- 正面：EdgeOne 黑屏问题推动了 production-preview smoke，提升了发布稳定性。
- 负面：很多内容仍是 `candidate_review` 或只读 readiness，不是正式奖励表、掉落表、商店表或完整战斗平衡源。
- 当前定位：区域行动样板成立，但还不是完整青茅山主线。

### v0.11.0：活世界地基与自由意图闸门

`v0.11.0` 增加 `livingWorldState`，`SAVE_FORMAT_VERSION` 到 22，建立 world intent adjudication、free goal panel、事实卡 pilot、可见调查、白家接触窗口、逃离路线准备链、测试工程化与流程制度。

当前评价：

- 正面：这是“活世界”的真正地基。玩家自由输入不会直接改世界，而是先经过本地裁决、目标账本、候选后续、隐藏事实保护。
- 正面：DeepSeek 权限被稳定压在叙事/候选/线索层，正式奖励、地点、阵营、NPC 生死不交给模型。
- 负面：`行动` 和 `自由目标` 在 UI 上分离，但系统上已经互相咬合，玩家会不知道该去哪里做“我想做的事”。
- 当前定位：必须保留并继续加强，是活世界的入口权威。

### v0.12.0：青茅正史锚点与 IF 框架

`v0.12.0` 建立 Qingmao 原著事实卡、正史锚点、可见/隐藏/IF 分类、路线/补给/追索、轻量 NPC/势力反应桥、方源公开旁证、GitHub/CI 门禁。

当前评价：

- 正面：它解决了“DeepSeek 长上下文记不住原著”的关键风险，把原著事实变成本地结构化候选/规则/测试。
- 正面：玩家可见事实和隐藏事实分离，是方源、灵泉、回溯、关键因果不被 UI/AI 泄露的基础。
- 负面：Qingmao fact cards 只是青茅山第一层，离完整原著世界图谱还很远。
- 当前定位：canon/IF 的正确方向，后续必须继续按 MiroFish intake -> RebornG review -> local canon/tests 的流程扩展。

### v0.13.0：NPC 与势力反应投影

`v0.13.0` 做了社会记忆协议、NPC memory projection、faction stance projection、public event chronicle、social followups、Player Advocate 可读性 UI、远端 Actions/Git baseline。

当前评价：

- 正面：世界开始出现“别人怎么看你”的结构化影子。
- 正面：公开事件编年史避免完全依赖 DeepSeek 长上下文硬记。
- 负面：大多数仍是只读投影与候选后续，还不是正式 standing、通缉、招揽、任务来源或 NPC 长期记忆写入。
- 当前定位：社会反应样板成立，但还没形成真正长期 NPC/势力账本。

### v0.14.0：青茅后续路线承接

`v0.14.0` 做了路线条件预览、遮掩逃离痕迹正式前置行动、山路逃离路线候选、阵营/身份前置展示、FreeGoal priority summary、60 轮 Player Advocate rc。

当前评价：

- 正面：玩家说“逃离青茅山”“投靠白家”“去商家城”时，系统开始能给前置条件和风险，而不是只拒绝或胡编。
- 正面：正式地点、阵营、NPC 生死仍没有被过早打开，边界稳。
- 负面：路线承接还是候选/预览/前置，不是正式离开青茅山，也不是完整南疆路线。
- 当前定位：路线系统前置层完成，下一步必须谨慎决定何时开放正式 route state/location change。

### v0.15.0：低阶经济、补给、炼养用深循环

`v0.15.0` 完成低阶经济规则、补给/喂养、炼蛊边界、商队/市场窗口、灰色交易/委托边界、美术专项、rc 质量收束。

当前评价：

- 正面：低阶蛊师不再只有“战斗/剧情”，开始有补给、喂养、炼蛊、交易窗口、灰色边界这些日常压力。
- 正面：依然没有贸然引入正式 inventory consumption、price table、shop inventory、black-market、commission profit、identity wash 等高风险功能。
- 负面：玩家可能看到很多“可读性/边界/窗口/候选”，但真正能执行的经济链条仍有限，短期爽感不强。
- 当前定位：经济地基稳，但下一步需要决定是继续补深度，还是先做 UI/系统收束。

## 5. 系统闭环评价

### 已经闭合得比较好的链路

1. DeepSeek 输出治理闭环：模型输出 -> schema/canary/validator -> telemetry -> failure reason -> 本地日志/测试。
2. 统一行动基础闭环：线索 -> 出发 -> 本地引擎结算 -> local ledger -> 回流文本/面板提示。
3. 自由目标基础闭环：玩家意图 -> world intent adjudication -> preview -> confirm -> `livingWorldState.playerGoals`。
4. 隐藏事实保护闭环：fact cards -> visible/hidden 分离 -> hidden ref only -> 不进 DeepSeek visible context。
5. 工程门禁闭环：小版本文档 -> 测试/构建/e2e/Player Advocate -> 仪表盘/Git 状态 -> handoff/CI。

### 半闭合链路

1. 自由目标到正式行动：能给前置和候选，但许多目标还不能完成正式结果。
2. NPC/势力反应：能投影“怎么看你”，但长期关系账本和正式 standing 还没开放。
3. 路线承接：能准备、预览和解释路线风险，但还没有正式 route state/location change。
4. 低阶经济：能展示缺口和边界，但材料消耗、价格、市场库存、委托收益仍未成为完整循环。
5. 战斗后果：青茅战斗可展示、可进入候选，但战斗结果到 NPC/势力/经济/路线长期影响还不完整。

### 尚未闭合或不应过早闭合的链路

1. 完整南疆/商家城/多地域路线。
2. 正式阵营转移、通缉、招揽、任务系统。
3. 正式 NPC 生死、捕获、长期命运改写。
4. 宝黄天、仙蛊、尊者、宿命、永生等高阶世界规则玩家入口。
5. 100+ 蛊虫和几十个杀招的完整战斗/炼养用/图像资产覆盖。

这些不闭合不是失败，而是边界正确。真正的问题是：UI 不应该把尚未开放的高阶入口和调试入口都放在玩家底栏上。

## 6. 重复与权威冲突风险

### 6.1 UI 入口重复

当前 `src/components/game/GameScreen.tsx` 中 `TOOLBAR_BUTTONS_BASE` 从 `行动`、`自由目标` 开始，继续暴露传承、宿命、属性、蛊虫、杀招、炼蛊、蛊材、空窍、人物、小队、道痕、商会、成就、道场、事件等入口。底栏还额外有地图、青茅、凡战、群像、大阵等入口。

玩家视角问题：

- 玩家不知道“我要做事”应该点 `行动` 还是 `自由目标`。
- `青茅/凡战/群像/大阵` 更像开发/演示入口，不像正式游戏导航。
- `蛊虫/杀招/炼蛊/蛊材` 本质都属于蛊师资产与炼养用，但被拆成多个底栏主按钮。
- `宿命/终局/传承` 对一转青茅山开局来说过早，会削弱低阶代入感。
- `商会` 在没有正式市场/商店库存之前放在主按钮上，会制造过度承诺。

### 6.2 UI 状态权威重复

`GameScreen.tsx` 本地维护 `sidePanel`，同时 store 里还有 `uiSlice.activeTab`。这不一定是 bug，但说明 UI 导航经历过多代演化。后续减法时需要决定：主导航到底由本地 `sidePanel` 驱动，还是由全局 UI state 驱动，避免出现“按钮显示 A、实际面板 B、存档/热重载状态 C”的隐患。

### 6.3 NPC/势力权威重复

旧系统里存在 `npcRelations`、`dynamicNPCs`、`playerFaction`、`factionEvents`、`squadDispatchState` 等；新系统里存在 `livingWorldState.npcMemories`、`factionPressure`、`publicChronicle`、`actionConsequences` 等。

未来不能让旧 `npcRelations` 和新 `npcMemories` 同时决定 NPC 信任/怀疑，也不能让旧 `playerFaction` 和新 faction prerequisites 同时决定阵营身份。进入正式社会账本前，需要一份“社会权威归并表”。

### 6.4 经济权威重复

旧系统有 material bag、merchant、auction、economy service、shop/refine 相关能力；`v0.15` 又建立了低阶经济规则、补给/喂养/炼蛊/市场/灰色交易边界。下一步如果直接做正式经济，很容易出现同一材料、价格、交易窗口由多套系统解释。

建议在正式材料消耗、价格表、商店库存、委托利润开放前，先做“低阶经济权威表”：

- 哪个模块决定材料存在。
- 哪个模块决定材料来源。
- 哪个模块决定材料消耗。
- 哪个模块决定价格和交易合法性。
- 哪个模块决定 DeepSeek 能说什么。

### 6.5 高阶系统过早暴露

传承、福地、灾劫、终局、宿命、宝黄天、升仙等系统可以留在工程资产中，但不应在一转青茅山开局的主导航中强提示。否则玩家会误以为这些都是当前可触达玩法，和原著低阶蛊师阶段的压迫感冲突。

## 7. 游戏逻辑与原著世界观评价

### 做得好的地方

1. 凡人/低阶蛊师阶段的边界越来越稳：不随便给仙蛊、宝黄天交易、尊者传承、永生、十转等。
2. 方源、隐藏事实、正史压力、玩家可见事实已经有保护机制。
3. DeepSeek 不被允许直接写正式奖励、地点、阵营、NPC 生死，这是正确的。
4. 玩家极端意图会被降级成长期目标、传闻、前置条件或风险解释，而不是简单满足。
5. MiroFish 输出被定位为候选材料，不是 canon 真相和 runtime 权力来源，这能保护世界观稳定。

### 需要警惕的地方

1. 早期历史系统里有很多高阶概念，不能因为“代码里有”就让玩家早期看到。
2. `v0.15` 后低阶经济开始铺开，必须避免“玩家随便刷材料/交易/委托套利”破坏蛊真人世界的残酷成本。
3. NPC/势力反应正式化后，不能让 DeepSeek 临场决定“白家接纳你”“方源被你杀死”“古月一代暴露”等大因果。
4. IF 自由度必须有代价。玩家能选择偏离，但不能无成本改写正史硬因果。
5. 所有原著事实扩展都应走 MiroFish/人工审查/本地 canon/test，而不是靠记忆或 prompt。

## 8. 玩家体验评价

### 当前体验优点

1. 世界行动越来越有“前置、风险、后果”，不是单纯 AI 编故事。
2. 自由目标不会被粗暴拒绝，而是会被裁决、解释、降级或记录。
3. 青茅山开始有区域、资源、路线、社会压力、战斗视觉，不再只是纯文本背景。
4. Player Advocate gate 已经让“玩家是否看得懂下一步”进入工程流程，这是很大的优势。

### 当前体验痛点

1. 底栏按钮太多，玩家很难建立心智模型。
2. 很多面板的状态是“候选/只读/边界/未开放”，工程上正确，但玩家会觉得“看得到，做不了”。
3. `行动` 与 `自由目标` 分开，导致玩家不知道主动行为和长期目标的关系。
4. `商会/传承/宿命/终局/道痕` 等入口过早，可能让低阶青茅山体验失焦。
5. 调试/演示入口混在正式入口中，会削弱沉浸感。
6. UI 现在更像“系统全展开”，不是“按当前场景给你最该做的 3-5 件事”。

## 9. 减法建议

### 9.1 主导航目标

建议把正式玩家主导航控制在 5 个一级入口以内：

1. `行动`
2. `角色`
3. `蛊道`
4. `世界`
5. `记录`

地图可以是独立图标，但不应和所有系统按钮平铺竞争。调试入口必须移出正式底栏。

### 9.2 合并建议

| 当前入口 | 建议归并 | 理由 |
|---|---|---|
| `行动` + `自由目标` | 合并为 `行动` 工作台 | 玩家语义上都是“我接下来做什么”。可用 tabs：当前行动、自由目标、线索后续、路线/交易/社会机会。 |
| `道场/训练` | 放入 `行动` 或 `蛊道` | 它是行动的一种，不应独占一级按钮。 |
| `青茅/凡战/群像/大阵` | 移入开发/演示抽屉 | 这些是竖切入口和测试入口，不是正式玩家导航。 |
| `蛊虫` + `杀招` + `炼蛊` + `蛊材` | 合并为 `蛊道` | 蛊、杀招、炼养用、材料是同一套资产/成长心智。 |
| `空窍` + `道痕` + `aperture_management` | 合并为 `修行` 或放入 `角色` | 对玩家来说都是自身修行状态。低阶阶段可只展示凡窍/真元/资质。 |
| `人物` + `小队` + 未来势力关系 | 合并为 `人物与势力` | NPC、队伍、势力反应最终是一套社会关系界面。 |
| `商会` | 场景化到 `行动/交易` | 只有市场窗口、商队、交易场景可用时显示，不做常驻主入口。 |
| `传承` + `宿命` + `终局` | 延后到 `世界/命运` 高阶页 | 青茅低阶阶段只隐藏或弱提示，不作为一转初期主按钮。 |
| `事件` | 并入 `记录` | 事件日志、行动账本、公开摘要、测试回放都属于记录。 |

### 9.3 推荐的新界面结构

```text
底栏：
地图 | 行动 | 角色 | 蛊道 | 世界 | 记录

行动：
- 当前场景可行动作
- 自由目标输入与确认
- 线索/后续候选
- 路线、交易、社会机会

角色：
- 属性、状态、凡窍/空窍
- 修行进度
- 人物关系摘要

蛊道：
- 蛊虫
- 杀招
- 喂养
- 炼蛊
- 蛊材

世界：
- 青茅山局势
- 势力压力
- 路线前置
- 正史/IF 可见摘要

记录：
- 行动账本
- 公开事件
- DeepSeek/本地结算摘要
- 成就/长期记录
```

### 9.4 减法不是删除

这些建议不是删掉系统，而是换曝光方式：

- 当前场景无关的功能收起。
- 未开放功能不给一级入口。
- 高阶内容按境界/剧情/事实可见性逐步出现。
- 调试入口只在开发模式出现。
- 候选/只读内容必须清楚标注“这是前置/线索/预览，不是已发生事实”。

## 10. 后续可做的专项

### 方案 A：先做 v0.16.0-system-closure UI 减法与权威收束

目标：不扩大世界内容，先让现有系统更像一个游戏。

交付：

1. 主导航减法方案与用户审定。
2. `行动 + 自由目标` 合并设计。
3. 调试/演示入口迁出底栏。
4. 蛊虫/杀招/炼蛊/蛊材归并设计。
5. 旧 NPC/势力/经济系统与 `livingWorldState` 权威归并表。
6. Player Advocate 20-50 轮专项走查，重点问“玩家下一步是否清楚”。

优点：立刻解决玩家体验和系统重复问题。
风险：短期不增加新内容，开发观感像“整理”，但这是为 v1.0 必须做的整理。

### 方案 B：继续长期路线 v0.16 战斗/杀招/队伍深化，但穿插减法

目标：继续补游戏性，顺手清理相关 UI。

交付：

1. 低阶战斗系统深化。
2. 杀招/蛊虫/资源接入战斗后果。
3. 战斗结果回流 NPC/势力/经济。
4. 战斗入口与蛊道入口局部收束。

优点：内容推进更明显。
风险：在底栏已经拥挤的情况下继续加系统，后续减法成本更高。

### 我的建议

优先选择方案 A，做一个轻量但硬的 `v0.16.0-system-closure`。范围不要太大，不碰正式路线解锁、存档字段、DeepSeek 权限、NPC 生死、阵营转移。先把现有系统的入口、权威和闭环理顺，再进入更深战斗或多地域。

## 12. 本审查对小版本流程的要求

如果用户批准后续减法专项，每个小版本必须做到：

1. 更新对应版本文档和范围说明。
2. 更新或确认测试矩阵。
3. 若触及玩家 UI，执行 Player Advocate gate。
4. 若触及按钮/布局，增加 e2e 或视觉可读性断言，避免再次出现底栏/卡片遮挡类问题。
5. 更新项目仪表盘。
6. 写 handoff，记录测试、commit、push、CI 状态。
7. Git 只 stage 当前小版本显式文件，不使用 `git add -A`。

本轮审查本身是草案文档，不代表小版本完成，不更新 runtime，不提交发布承诺。

## 13. 最终判断

RebornG 的地基足以继续支撑“活着的蛊真人世界”，但现在已经到了必须收束的阶段。

项目真正的优势是：我们不是让 DeepSeek 胡编长篇剧情，而是在本地建立行动、事实、IF、路线、NPC/势力、经济和测试门禁。这个方向正确。

项目真正的风险是：系统越做越多，玩家界面和系统权威没有同步收束，最后变成“所有功能都存在，但玩家不知道该按哪个按钮，也不知道哪些东西真的发生了”。

因此，下一步最有价值的工作不是继续堆大功能，而是先让现有功能更少、更清楚、更闭环。尤其是底栏减法、行动/自由目标合并、调试入口迁出、蛊道资产归并、旧系统权威归并，这些应该进入近期正式议程。
