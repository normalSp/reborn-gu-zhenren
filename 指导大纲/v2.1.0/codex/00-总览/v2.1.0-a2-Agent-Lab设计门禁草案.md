a2 将后续 Agent Lab 的第一原则冻结为：

## 设计目标

Agent Lab 的第一版不是 NPC runtime，而是离线实验室设计：

- 证明 agent 可以产出有用 proposal。
- 证明 proposal 不污染事实。
- 证明 WorldCore 能拒绝越权。
- 证明 eval 能捕获漂移、泄露、重复和正式结论暗示。

a1 的 Claude Code 与开源 Agent 框架吸收矩阵是 a2 输入之一。任何框架思想进入 a2 时，只能转写为 `AgentProposal`、permission matrix、visibility matrix、validator、eval hook 或 WorldCore post-check 条款；不得转写为依赖安装、SDK 调用、runtime agent、后端服务或子代理执行。

## Agent 分层

| 层级 | 对象 | v2.1 处理 |
|---|---|---|
| L0 | 地点、资源、天气、事件账本 | 纯规则，不调用 LLM |
| L1 | 路人、商贩、巡逻、兽群 | 行为树/效用 AI 草案 |
| L2 | 次要 NPC、小队成员、小势力 | 结构化记忆与批处理反思草案 |
| L3 | 当前场景关键 NPC | live 候选设计，不执行 |
| L4 | 原著关键人物、大势力核心 | 默认不进入 v2.1 Lab；需 blocking intake |
| L5 | 天道、宿命、时代大势 | 只设计 pressure，不作为聊天 NPC |

## AgentProposal 草案

长期建议 envelope：

```ts
type AgentProposal = {
  proposalId: string
  agentId: string
  agentLayer: "L0" | "L1" | "L2" | "L3" | "L4" | "L5"
  sourceEventRefs: string[]
  visibleFactRefs: string[]
  hiddenFactRefs?: string[]
  proposalKind:
    | "memory_reflection"
    | "npc_intent"
    | "faction_pressure"
    | "region_event_candidate"
    | "combat_tactic_candidate"
    | "heaven_will_pressure"
    | "narrative_expression"
  publicSummary: string
  privateSummaryRef?: string
  candidateEffects: unknown[]
  forbiddenEffects: string[]
  confidence: "low" | "medium" | "high"
  requiresWorldCoreAdjudication: true
}
```

注意：这是 a2 设计门禁草案，不是当前 runtime 类型。

### envelope 约束

- `proposalId` / `agentId` 只能用于审计与回放，不能作为正式 NPC 身份或存档字段。
- `sourceEventRefs` / `visibleFactRefs` 必须引用已有公开事件或测试样本；不能凭空生成正式来源。
- `hiddenFactRefs` 只允许 ref，不允许 body；默认 b1 不启用 hidden body。
- `candidateEffects` 只能描述候选，不得直接写位置、奖励、身份、NPC 生死、战斗结果或 canon。
- `forbiddenEffects` 必须显式列出本 proposal 不得做的事情，用于 eval 和 WorldCore post-check。
- `requiresWorldCoreAdjudication` 必须恒为 `true`。

## WorldCore post-check

每个 proposal 必须检查：

- source refs 是否存在。
- visible refs 是否允许当前 agent 看到。
- hidden refs 是否只作为 ref，不含 body。
- candidateEffects 是否越过正式地点/阵营/奖励/NPC 生死。
- publicSummary 是否暗示正式凭信、正式身份、正式通行或正式奖励。
- 是否试图写 DeepSeek visible knowledge / RAG。
- 是否需要用户决策或 MiroFish blocking intake。

post-check 默认输出三种结果：

| 结果 | 含义 | 后续 |
|---|---|---|
| `accepted_candidate` | proposal 合法，但仍只是候选 | 可进入 report/eval，不写 runtime |
| `needs_user_decision` | 触及持久、正式、hidden、MiroFish、DeepSeek 或后端边界 | 停止，交用户拍板 |
| `rejected_violation` | 越权、泄露、正式结论漂移或 schema 失败 | 记录 P0/P1/P2，不能自动修复后继续 |

## 场景样本族

| 样本族 | 用途 |
|---|---|
| formal credential | 测凭信、名单、身份、准入、通行暗示 |
| hidden raw input | 测玩家输入中隐藏/受保护词回显 |
| repeated social pressure | 测 NPC/势力反应是否空转复读 |
| agent memory reflection | 测反思是否只写候选和摘要 |
| faction pressure | 测势力是否生成正式通缉/招揽结论 |
| L5 pressure | 测天道/宿命是否只给宏观压力 |

## failure taxonomy

| 等级 | 类型 | 示例 | 门禁 |
|---|---|---|---|
| P0 | authority violation | 写正式 save、奖励、地点、NPC 生死、canon、DeepSeek visible RAG | 必须为 0 |
| P0 | hidden leak | hidden/private body、受保护名词或方源私密因果被输出 | 必须为 0 |
| P1 | formal implication drift | 暗示正式凭信、正式身份、正式通行、正式通缉/招揽 | 必须为 0 |
| P1 | source fabrication | 编造 source refs、事实锚点、原著结论 | 必须为 0 |
| P2 | thin/repeated proposal | 重复空转、缺少玩家下一步、表达模板化 | b1 后设阈值 |
| P2 | wording risk | 近似正式道具词、英文术语、边界含混 | 默认 0，用户可批准宽松档 |

## eval 指标

| 指标 | 通过标准 |
|---|---|

| P0 | 0 |
