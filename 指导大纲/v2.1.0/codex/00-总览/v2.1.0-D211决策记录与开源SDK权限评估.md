## b1 准入边界

`v2.1.0-b1` 可以进入，但只能做：

- TypeScript-first。
- zero new dependency。
- dry-run only。
- report-only/offline runner 第一刀。
- 输出报告到 `artifacts/v2.1.0/`。
- 输入使用抽象/合成样本，不需要 MiroFish。
- agent 输出必须进入 `AgentProposal`。
- `AgentProposal` 必须经过 validator 与 WorldCore post-check。
- 最小通过标准：schema 100%、P0=0、P1=0、P2 阈值另定。

`v2.1.0-b1` 仍禁止：

- live DeepSeek。
- 第三方 SDK/framework 依赖。
- 外部 framework PoC。
- 外部 SDK/agent 访问项目文件、执行命令或产生补丁。
- 子代理。
- BFF/backend。
- MiroFish export。
- runtime agent。
- save/canon/store/DeepSeek context 写入。
- agent memory 正式持久化。

## D-211-007 专家团评估

结论：未来有可能引入、改造或部分并入开源 agent SDK，但不能把“开源”直接等同于“可以访问 RebornG 文件、执行命令、产生补丁或改 git”。开源只解决可审计与许可起点，不自动解决权限、安全、架构耦合、供应链、canon 污染和 WorldCore 绕行风险。

更稳的制度是把 D-211-007 拆成一条未来晋升链：

| 层级 | 名称 | 允许内容 | 禁止内容 | 是否当前批准 |
|---|---|---|---|---|
| L0 | 公开资料/源码阅读 | 阅读官方文档、公开源码、license、issue、架构图 | 运行其代码、装依赖、接入项目 | 已允许 |
| L1 | 架构转译 | 把有价值模式转写为 RebornG 自有 `AgentProposal`、validator、permission、eval 设计 | 复制大段源码、引入 SDK | 已允许 |
| L2 | 隔离 PoC | 在 scratch 或独立实验目录跑开源 SDK 对照，不接 RebornG repo/secrets/git | 访问真实项目文件、`.env`、DeepSeek key、git、runtime | 未批准，D-211-006 后续再议 |
| L3 | vendored subset / adapter | 经 license、SBOM、测试和用户批准后，把小型可控子集或适配器纳入项目 | 全量吞 SDK、绕过 WorldCore、写 save/canon | 未批准，未来 D-212 |
| L4 | 受限项目只读 | 只读扫描指定目录，输出 report/建议 | 写文件、执行 shell、改 patch、读 secrets | 未批准，未来 D-212 |
| L5 | patch artifact | 只在 `artifacts/` 生成补丁建议文本，由 Codex 人工审查后再决定是否手工采纳 | 直接 `apply_patch`、`git add`、`git commit`、运行命令 | 未批准，未来 D-212 |
| L6 | 项目写入/命令/git | 外部 SDK/agent 直接写文件、执行命令、产生补丁或操作 git | 默认禁止 | 不建议进入默认制度 |

## 可能收益

- 少造轮子：workflow、tool permission、guardrails、tracing、eval、memory、sandbox、human-in-loop 等问题已有成熟模式可借鉴。
- 提升实验速度：如果某个 SDK 的状态图、schema 校验、评估归档或工具沙箱足够贴合，可以缩短 v2.2-v2.4 的实验周期。
- 增强审计能力：成熟框架通常已有 tracing、run log、tool call 记录或 replay 概念，适合转译成 RebornG eval archive。
- 支持未来薄 BFF：部分框架的 job queue、server mode、deployment 思路可在 v2.4+ 评估 private canon / DeepSeek proxy / cost archive。
- 促进架构对照：自有 AgentProposal 跑通后，用外部框架做对照可以发现我们是否过度自造、过窄或遗漏成熟模式。

## 主要风险

- 权限越权：外部 agent 如果能读写项目、执行命令或生成补丁，很容易绕过 RebornG 的 WorldCore、MiroFish、Skill、Git 和用户决策门禁。
- hidden/private 泄露：SDK 的 memory、trace、prompt log、cache 或 telemetry 可能把 hidden facts、私有事实、`.env`、DeepSeek key 或未公开 canon 暴露出去。
- canon 污染：agent memory 或外部框架状态如果混入候选、谣言、幻觉，就可能被误当成事实。
- 供应链风险：npm/pip 依赖、transitive dependency、postinstall、网络调用、许可证边界和 enterprise 子目录都需要审计。
- 架构错配：多数 agent 框架服务的是通用任务或代码 agent，不天然适合“世界内核最终裁决”的游戏架构。
- 维护成本：SDK 升级、API 变化、license 变化和 runtime 兼容会长期消耗维护力。
- 错误激励：外部 agent 越强，越容易诱导开发者让它直接裁决世界事实；这和 RebornG 的核心原则相反。

## 推荐制度

D-211-007 不作为 b1 权限批准。它应升级为未来 `D-212 开源 Agent SDK/agent 引入评估门`：

| ID | 决策项 | 默认建议 | 最早触发 |
|---|---|---|---|
| D-212-001 | 是否启动某个开源 SDK/agent 的 license + SBOM + 架构适配评估 | 可批准 | 自有 AgentProposal b1 通过后 |
| D-212-002 | 是否允许隔离 PoC | 需逐项批准 | D-211-006 后续重议 |
| D-212-003 | 是否允许引入依赖或 vendored subset | 默认不批准，需架构评审 | v2.2+ |
| D-212-004 | 是否允许只读扫描 RebornG 指定目录 | 默认不批准，需风险收益报告 | v2.3+ |
| D-212-005 | 是否允许输出 patch artifact | 默认不批准，需人工复核链路 | v2.3+ |
| D-212-006 | 是否允许外部 SDK/agent 直接写文件、执行命令或操作 git | 默认禁止 | 不建议常规开放 |

2026-05-23 用户进一步决策：

- D-212-001 已批准，但只允许 license、SBOM、架构适配评估，不安装、不运行、不接项目。
- D-212-002 需先解释“隔离 PoC”，之后逐项申请；当前未批准。
- D-212-003 若后续需要，先做架构评审，判断收益和实际必要性。
- D-212-004 需要时先做风险收益报告。
- D-212-005 需先解释 patch artifact，不因本表而批准。
- D-212-006 禁止。

任何 D-212 晋升都必须满足：

- 官方来源或可信开源仓库。
- license 与 RebornG 使用方式兼容。
- 明确排除 enterprise、closed-source、反编译、泄露或未授权来源。
- 无默认网络 exfiltration。
- 不读取 `.env`、密钥、SQLite/log/hibernation、plugin cache 或未授权私有文件。
- 不写 runtime、canon、save、DeepSeek context、MiroFish intake、知识库或 git。
- 输出先变成 report 或 `AgentProposal`，再由 Codex/WorldCore/用户门禁处理。
