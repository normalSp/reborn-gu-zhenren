# v3.0.0 总体开发大纲

日期：2026-05-24
状态：completed

## 主线

`有限 L2/L3 runtime agent 试点设计门禁`

## 目标

v3.0 的目标不是完整 agent 社会，而是证明一个最小 runtime agent 试点可以被 WorldCore、测试、回滚和 Player Advocate 守住。

默认目标：

1. 一个小区域。
2. 仅 L2/L3。
3. agent 输出候选和表达。
4. WorldCore 保持最终裁决。
5. 不让 agent 写正式事实、存档、奖励、地点、身份、NPC 生死或 canon。

## 非目标

- 不做完整南疆。
- 不做完整商家城。
- 不做所有 NPC live。
- 不做 L4 原著关键人物 runtime。
- 不做 L5 天道/宿命 runtime 裁决。
- 不做 backend/BFF/service。
- 不做全书 RAG。
- 不引入外部 agent framework 依赖或 PoC。
- 不让 self-learning 直接改 canon/runtime/save/prompt/knowledge-index。

## 核心设计问题

v3.0 必须回答：

- L2/L3 agent 能看到哪些 visible context。
- AgentProposal schema 是什么。
- WorldCore post-check 拒绝哪些候选。
- 被接受的候选如何转成玩家可见表达。
- 候选是否持久化。
- 若持久化，是否需要 save-format bump。
- live DeepSeek 如何测试。
- Player Advocate 如何判断 agent 是否提升体验。
- rollback 和 old-save 如何保证不破坏旧档。

## 阶段路线

- startup：专家团启动会和 D-300/F-300 审批包。
- a0：v2.9 复盘与 runtime agent 试点范围冻结。
- a1：AgentProposal contract 与 WorldCore post-check 设计门禁。
- a2：live probe / Player Advocate / drift / rollback / old-save 门禁。
- b1：是否允许最小 runtime 第一刀的 go/no-go。
- b2：若 b1 获批，接入最小 L2/L3 proposal-only runtime；否则继续 report-only hardening。
- b3：同开局差异、NPC 意图、势力压力、玩家可理解性复核。
- process-1：前置审批制度第六轮复核。
- process-2：长线叙事漂移与知识边界复核。
- rc：质量收束、live/PA/rollback/old-save 证据。

## 成功标准

最小成功：

- agent 只输出 proposal。
- WorldCore 拒绝越权 proposal。
- UI/叙事只显示 post-check 后表达。
- 旧档可加载。
- 不新增未批准 save field。
- 不扩 DeepSeek authority。
- live probe 和 Player Advocate 证明体验有增益且无 P0/P1。

## 失败标准

以下任一出现，必须停机：

- agent 直接写存档或正式事实。
- prompt/context 扩权未获批准。
- hidden/private 泄漏。
- MiroFish blocking 被触发但未完成。
- 需要 backend/service/external dependency。
- 需要 L4/L5 runtime。
- P0/P1 false negative 或 live probe P0/P1。
