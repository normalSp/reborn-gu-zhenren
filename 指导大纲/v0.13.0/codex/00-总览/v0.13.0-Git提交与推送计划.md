# v0.13.0 Git 提交与推送计划

日期：2026-05-16
状态：已批准，纳入 v0.13 阶段门禁
用途：把 v0.13 的 commit / push 节奏、回滚点、分支边界和脏项隔离写成可执行制度。

## 结论

需要制度化到每个大版本。

RebornG 已经不是短期 demo，后续 NPC、势力、正史锚点、IF、DeepSeek、MiroFish 和测试门禁会越来越多。每个大版本都必须有独立的 Git 计划，否则几天工作堆在工作区里，会导致：

- 无法按小版本回滚。
- 难以定位某个 bug 是哪个阶段引入的。
- 跨 Codex / CodeBuddy / GitHub / EdgeOne 流程时上下文容易断。
- 阶段文档、源码、测试、MiroFish intake 容易混在一个不可审查的提交里。

## 职责

不新增一个会写代码的独立专家。

新增轻量职责：`Git / 回滚守门人`。

该职责由以下角色共同承担：

- `QA / Release Guardian`：确认验证结果、rc 证据、CI 状态、发布边界。
- `Codex Safety Officer / Dev Environment Steward`：确认分支、stage 范围、脏项隔离、回滚点、push 状态。

职责口径：

- 每个小版本完成后，确认是否需要 commit / push。
- 每个大版本开始前，建立本版本 Git 计划。
- 每次提交前，确认只 stage 当前阶段相关文件。
- 每次 push 前，确认测试证据和文档交接已经写清。
- 禁止把密钥、`.env`、插件缓存、SQLite/log、无关压缩包、无关素材、外部参考脏项混入提交。

## 何时必须 commit

- 小版本阶段完成且验证通过。
- MiroFish 包完成 intake review。
- 新增或修改 GitHub/CI/PR/issue 模板。
- 新增持久化字段、迁移、DeepSeek 权限、canon/IF gate、社会记忆写入规则。
- 修复关键 bug 并补回归测试或 `.learnings/ERRORS.md`。
- 长任务上下文风险升高，需要跨窗口继续。

## 何时必须 push

- 小版本阶段完成并能作为稳定回滚点。
- 大阶段收束。
- rc / 发布候选前。
- GitHub Actions / PR 模板 / CI 配置变化。
- 跨工具、跨窗口、CodeBuddy / Codex 协作前。

可以先 commit 暂不 push 的情况：

- 正在拆分历史脏项。
- 只做本地 checkpoint，验证尚未完整。
- 当前提交是中间态，不能代表阶段完成。

这种情况必须在交接文件写明“不 push 的原因”和“下一次 push 的范围”。

## stage 规则

允许 stage：

- 当前阶段源码、测试、canon、文档、上下文交接。
- `.github/` 下已批准的 CI / PR / issue 模板。
- 项目镜像状态文件，例如 `.codex/skills/reborn-expert-council/references/PROJECT-STATE.md`，但必须确认不是 `.codex` runtime state。

禁止 stage：

- `deepseek密钥.txt`
- `.env`
- `.codex` runtime state、插件缓存、SQLite、log
- `RebornG_codebuddy.zip`
- 未确认归属的 `bgm/` 或大体积素材
- 外部参考目录中未转化为项目-owned 的文件
- 无关删除、无关旧文档迁移、无关测试存档重排

默认命令原则：

- 使用 `git status --short` 和明确路径 stage。
- 不使用 `git add -A`。
- 不使用 destructive checkout / reset。

## 需要用户决策的 Git 事项

以下事项不得自动做：

- branch protection。
- 自动部署 EdgeOne。
- auto-merge。
- 强制所有 PR 默认跑完整 Playwright / long e2e。
- 迁移默认分支或重写远端历史。
- 删除或重写历史提交。

用户已经明确：自动部署 EdgeOne 不由本 Codex 负责。
