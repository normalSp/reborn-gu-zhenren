# v4.2.0 专家团启动会纪要

状态：startup_prepared；等待用户审批 D-420 / F-420。
日期：2026-05-26。
主线建议：`Auto-Theater Lite / 凡阶战斗映射设计门禁`。

## 背景

`v4.0.0` 已把 Auto-Theater Combat、Auto-Theater Lite、battle theater、killer-move stack、Combat Ledger、HeavenWill/Fate/L5 pressure 建成高阶世界表达模型。`v4.1.0` 已把这些概念收束为 contract、schema、ExpressionAuthority 和 WorldCoreEvidenceChain，并用 `v410` report-only checker 验证 negative fixtures。

长期路线把 `v4.2` 定位为 `Auto-Theater Lite 凡阶映射`。这里的关键不是“把凡阶战斗立刻改成自走棋”，而是先回答：

- 现有棋盘 / 行动卡 / 战斗 trace 能不能映射到 Auto-Theater Lite。
- 玩家能不能通过 Combat Ledger 看懂准备、行动、失败原因、支援、反制、风险和结果。
- 未来若迁移凡阶 runtime，哪些合同、样本和验收标准必须提前冻结。

## 专家意见

Project Lead：
`v4.2` 应当是从“高阶概念”回到“现有凡阶战斗”的校准版本。凡阶战斗不能因为 Auto-Theater 愿景而被仓促推倒重做。当前版本应让后续改造有证据、有映射、有边界。

Systems Architect：
建议把现有凡阶战斗拆成 `source action`、`preparation cue`、`resolution fact`、`ledger lane`、`expression candidate` 五段。`v4.2` 不新增 runtime 类型或 save 字段，只在文档与 report-only checker 草案中定义映射合同。

Gameplay/Numerics Architect：
凡阶阶段的爽点来自可理解的选择、低阶资源压力、蛊虫特性和失败代价。Auto-Theater Lite 应增强这些解释，而不是增加自动结算黑箱。棋盘和行动卡可继续保留，Lite 负责把“为什么这一招有效 / 无效 / 被防御 / 只是准备”讲清楚。

Combat/Motion Lead：
凡阶战斗可以采用 `Lite theater strip`，不是完整高阶 theater。移动端和 reduced-motion 要优先：不追求大演出，先追求明确的 turn transcript、ledger lane、目标关系、风险色和原因标签。任何表现层不得把候选、准备或压力画成已发生事实。

Lore & World Designer：
`v4.2` 可以使用既有项目中的泛用凡阶例子，如月光蛊、白玉防御、酒虫支援、低阶兽战和失败原因，但必须保持 synthetic/generic 或已存在 runtime-safe 边界。若要使用真实原著战斗、命名 NPC、真实势力、隐藏因果或正式 lore conclusion，MiroFish 升级为 blocking。

AI Pipeline Architect：
DeepSeek 将来可以参与战斗表达和候选意图，但 `v4.2` 不应调用 live DeepSeek。设计门禁必须写清：DeepSeek 只能表达 WorldCore / Combat Core 已确定的事实，或生成带拒绝边界的候选，不得裁决命中、伤害、消耗、奖励、NPC 生死或地点变化。

QA/Test Engineering Guardian：
`v4.2` 的首要测试应是 mapping matrix 与 negative fixtures：同一凡阶 action 映射成 Lite ledger 时，不能改变结果、不能新增数值、不能泄露 hidden body、不能把支援写成攻击、不能把失败原因写成奖励。建议后续 b1 做 `v420` report-only checker。

Player Advocate：
玩家感知目标应是“我还是在做熟悉的凡阶战斗选择，但现在战斗为什么这样发展更清楚了”。如果玩家以为 v4.2 已经开放新战斗模式、自动胜负、战斗奖励或高阶 theater，文案就失败了。

Git/Rollback Steward：
v4.2 已切到 `codex/v420-startup-auto-theater-lite-mortal-mapping`。启动包只提交文档、入口和 skill sync，不自动合并 main。

Skill Sync Steward：
v4.2 触发 `reborn-expert-council`、`game-dev-text`、`reverend-insanity-lore`、`reborn-combat-motion`。`mirofish-reborng-export` 当前 `no_update_needed`，除非后续变为真实原著 topic-slice。

## 路线比较

| 路线 | 内容 | 收益 | 风险 | 专家建议 |
|---|---|---|---|---|
| A | Auto-Theater Lite 凡阶映射设计门禁 | 最稳，承接 v4.1 contract，给未来迁移准备证据 | 短期不改变玩家可玩操作 | 推荐 |
| B | 直接迁移凡阶 runtime 到 Auto-Theater Lite | 玩家短期能看到新模式 | 容易破坏既有战斗、测试和可读性 | 暂不推荐 |
| C | 做完整 theater UI 原型 | 视觉吸引力强 | 打开 UI/素材/runtime 多重 future gate | 不推荐 |
| D | 先做高阶战斗 Full Auto-Theater | 符合远景野心 | v4.2 还没证明凡阶映射，风险过高 | 禁止，留到后续 gate |

## 启动会结论

专家团建议 v4.2 采用路线 A。先做 Auto-Theater Lite / 凡阶战斗映射设计门禁，并准备后续 report-only checker 的样本与验收标准；不开放 runtime 迁移、theater UI、素材、DeepSeek、MiroFish、save、后端或外部框架权限。
