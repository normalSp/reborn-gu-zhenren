# v3.1.0 专家团启动会纪要

日期：2026-05-24
状态：startup completed；等待用户审批 D-310
主线：runtime agent 门禁二次加固

## 会议结论

专家团一致建议：v3.1 不应急着打开 F-300 runtime implementation，而应先做门禁二次加固。

v3.0 已证明我们能用 report-only checker 把 runtime agent implementation、save write、hidden leak、formal authority drift、backend、external framework 等高风险项拦下来。但 v3.0 仍只是设计门禁和合成证据，不足以直接进入 runtime。v3.1 的价值是把证据链从“能识别边界”推进到“能承受更像真实运行链路的压力”。

## 专家意见

### Project Lead

v3.1 应当继续遵循长期路线：v3.1 加固、v3.2 rehearsal、v3.3 再给 F-300 reopen 决策包。当前不把 v3.1 伪装成 runtime 第一刀。

### Systems Architect

核心要补的是 WorldCore post-check 的二次证据：AgentProposal 只能提供候选，不能写事实、存档、奖励、地点、关系、NPC 生死或战斗结算。v3.1 应冻结更严格的 rejected / needs_user_decision / manual_review 语义。

### AI Pipeline Architect

live DeepSeek 仍不默认实跑。v3.1 只把 live metadata schema、成本字段、样本/轮次/报告路径、P0/P1 停机规则写清楚。未来如果用户批准 live，必须能追踪模型、样本、轮次、成本和报告路径。

### Lore Guard

MiroFish 当前保持 `not_needed`。只要触及命名 NPC、真实势力、hidden-adjacent、方源公开证据、L4/L5 原著锚点、知识库正文或正式 lore 结论，立即升级 blocking，不得在 v3.1 里顺手吸收。

### QA / Test Engineering

v3.1 需要把 rollback、old-save、same-start difference、manual review、negative fixture、rescore stability 和 Player Advocate metadata 纳入测试矩阵。即使仍是 report-only，也要保留可追溯报告路径。

### Git / Rollback Steward

本版本应使用独立语义分支；不使用 `git add -A`；阶段结束必须显式列出 staged files、提交、推送、CI 结果。
