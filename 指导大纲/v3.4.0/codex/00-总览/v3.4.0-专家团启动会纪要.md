# v3.4.0 专家团启动会纪要

日期：2026-05-24
状态：startup；pending user route decision
主题：继续 report-only hardening，还是重开最小 transient proposal-only runtime first cut

## 启动前证据

- v3.0 完成有限 L2/L3 runtime agent 试点设计门禁，`runtimeImplementationApproved=false`。
- v3.1 完成门禁二次加固，`v32RehearsalAdmissible=true`，仍不进 runtime。
- v3.2 完成 offline / report-only runtime rehearsal，`v33DecisionPackageReady=true`。
- v3.3 完成 F-300 reopen 决策包，结论是 `f300ReopenDecisionPackageComplete=true` 且 `runtimeImplementationApproved=false`。
- v3.3 给出三条路线：继续 hardening、单独打开最小 transient proposal-only 第一刀、继续 defer。

## 既定路线怎么设定

长期路线不是“永远 report-only”。既定路线是：

- v3.1-v3.3 先把笼子造稳。
- v3.4-v3.6 若用户明确批准，做最小 L2/L3 小区域试点。
- 如果用户不批准，v3.4-v3.6 继续 report-only hardening。
- v4.0 之前不碰完整 agent society、高阶战斗 runtime、L4/L5 runtime、天道/宿命 runtime 裁决。

所以 v3.4 是第一个真正适合问“要不要开最小窄口”的版本。

## 路线 1：继续 report-only hardening

### 收益

- 零 runtime 风险。
- 不触发 F-330/F-340。
- 可以继续扩展 negative fixtures、manual review、same-start variance、rehearsal report。
- 更适合用户希望在 agent 进入游戏前再压一次安全边界。
- 不会引入玩家可见误解，不会影响旧档，不需要 Player Advocate runtime 走查。

### 风险与代价

- 边际收益开始下降：v3.0-v3.3 已连续证明门禁、rehearsal、reopen package。
- 没有玩家可见体验提升，NPC 意图和同开局差异仍停在报告里。
- 可能对 synthetic fixtures 过拟合，不能验证 UI / store / player comprehension 的真实摩擦。
- 会把长期路线的 v3.4-v3.6 runtime pilot 往后推。

## 路线 2：单独重开最小 transient proposal-only runtime first cut

### 最小定义

只允许：

- 一个小区域。
- L2/L3 非原著关键 NPC 或抽象小队/小势力压力对象。
- transient AgentProposal。
- WorldCore post-check 最终裁决。
- expression-only：只把通过 post-check 的候选转成表达或机会窗口。
- rejected proposal 不改变状态。
- 不写正式事实、不写玩家存档、不新增字段。
- 不调用 live DeepSeek。
- 不做 MiroFish、backend、external framework、subagent。

### 收益

- 第一次验证 agent 候选在真实游戏链路中的价值，而不是只在报告里漂亮。
- 玩家能开始感到 NPC 意图、机会窗口、同开局差异更鲜活。
- 可以验证 UI 是否会把“候选表达”误读成事实。
- 可以验证 WorldCore post-check 在真实入口前是否足够硬。
- 不新增 save field，不碰 migration，回滚成本最低。
- 走通后，v3.5/v3.6 可以自然承接 Player Advocate、deterministic replay、rollback、old-save、same-start difference hardening。

### 风险与代价

- 这是 runtime 代码，必须单独打开 F-330-001 / F-340-001。
- 即使不写存档，也可能出现“表达被玩家理解为事实”的 UI/文案风险。
- 需要更严格的测试：unit/schema、post-check rejection、old-save open、rollback、10 轮 Player Advocate、30 轮 deterministic。
- 如果第一刀设计太宽，容易滑向 save field、live DeepSeek、MiroFish、正式地点/阵营/奖励/NPC 生死。
- 不调用 live DeepSeek 的第一刀会比较克制，鲜活度有限；但这是安全换来的。

## 路线 3：继续 defer 到 v3.5+

### 收益

- 最大限度避免 player-facing agent 风险。
- 可以转向高阶战斗、天道/宿命或外部框架研究。

### 风险与代价

- 不建议作为默认路线。
- 会让 v3.0-v3.3 的证据链失去推进势能。
- 高阶战斗和天道/宿命仍需要 runtime agent / WorldCore / proposal chain 经验，过早转向会把架构问题推迟。

## 专家团意见

### Project Lead

建议 v3.4 选择路线 2，但必须把授权口径写得像手术刀一样窄。批准 route 2 不等于打开 save、live、MiroFish、backend、external framework 或正式世界事实。

### Systems Architect

路线 2 是技术上最合理的下一步，因为它能把 report-only 证据变成真实 runtime 摩擦证据。底线是 transient-only，不进存档；任何 `runFingerprint`、agent memory save、same-start persistent seed 都必须另批。

### AI Pipeline Architect

第一刀不建议 live DeepSeek。先用本地 deterministic proposal generator 或已有 synthetic proposal fixture 验证链路，避免把“模型不稳定”和“runtime 接入不稳”两个变量叠在一起。

### Lore Guard

路线 2 只能用 synthetic / RebornG-owned 泛化样本，不碰命名 NPC、真实势力、hidden-adjacent、方源证据和正式 lore。否则 MiroFish 立刻从 `not_needed` 升级为 `blocking`。

### QA / Player Advocate

路线 2 的关键不是“代码能跑”，而是玩家是否能理解：NPC 表达是候选与机会窗口，不是已经写入世界事实。必须把 Player Advocate 作为 implementation 小版本硬门。

### Git / Rollback Steward

如果路线 2 被批准，建议新开 implementation 分支，不混在 startup 决策提交里。每个阶段显式 stage，不用 `git add -A`。

## 专家团推荐

推荐路线 2：单独重开最小 `transient proposal-only` runtime first cut 授权包。

不推荐直接路线 1 作为完整 v3.4 主线，除非用户当前明确追求零 runtime 风险。

不推荐路线 3，除非用户决定把 runtime agent 推迟到 v3.5+。
