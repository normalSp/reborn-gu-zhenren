# v0.11.0 测试工程化总纲

日期：2026-05-15
状态：v0.11 长期门禁；第一版已建立。
适用范围：`v0.11.0-a3` 起，所有自由意图、活世界状态、原著事实卡、DeepSeek 回流、青茅活世界闭环、UI 接入和发布收束。

## 为什么要单独建测试总纲

RebornG 的优势不是“让 AI 随便写故事”，而是把 AI 带来的自由感压回可测试、可复现、可保存的游戏系统。

随着自由意图裁决、活世界状态、NPC 记忆、势力压力、原著事实卡和 DeepSeek 回流接入，测试一定会越来越难。

真正的风险不是测试数量变多，而是：

- 只有 e2e，没有纯引擎样本，定位慢。
- 只有 happy path，没有极端输入，模型越权时挡不住。
- 只有功能测试，没有世界观边界测试。
- 只有当前版本测试，没有旧档迁移和长期回归。
- 只在用户提醒后补测试，没有阶段性复盘机制。

本总纲把测试当成版本资产维护，而不是功能完成后的附属品。

## QA/测试工程负责人

专家团中固定设立 `QA/测试工程负责人`，由 `QA/发布稳定专家` 升级承担。

职责：

- 每个小版本维护测试矩阵。
- 每个新系统落地前定义硬指标。
- 每个 bug 修复后补 `.learnings/ERRORS.md` 或回归样本。
- 阻止“只做功能不补测试”的开发。
- 阻止 DeepSeek、UI、store 绕过本地裁决。
- 每 2 个小版本做轻量复盘，每个大阶段或 rc 前做完整复盘。

协作角色：

- 世界观审校：维护 lore 边界样本。
- 系统架构师：维护状态写入、存档、经济/奖励越权样本。
- AI Pipeline 专家：维护 DeepSeek 越权、JSON、缓存、token 指标。
- 前端/玩家体验专家：维护 UI 可读性、移动端、阻断文案样本。
- Codex 安全官：维护上下文交接、长任务稳定、测试命令清单。

## 测试分层

### L0 类型与 Schema

目标：数据不坏。

覆盖：

- TypeScript 类型。
- canon JSON 结构。
- Zod 或手写 normalize 函数。
- 存档默认值。

典型命令：

```powershell
npx tsc --noEmit --pretty false
npm test -- src/store/defaultLivingWorldState.test.ts
```

硬指标：

- 类型检查通过。
- 新增持久化字段必须有默认值、归一化、旧档迁移测试。

### L1 Canon 规则测试

目标：规则不漂。

覆盖：

- `src/canon/*` 静态规则。
- 世界观硬边界。
- 青茅身份边界。
- 意图类型、裁决分类、DeepSeek 契约。

硬指标：

- 未注册蛊、蛊方、仙蛊、仙材不得成为正式奖励。
- 凡人宝黄天交易、九转蛊、春秋蝉、永生、宿命蛊归属等硬边界样本通过率 `100%`。

### L2 纯引擎裁决测试

目标：自由输入可复现。

覆盖：

- `WorldIntentEngine`
- `applyLivingWorldPatch`
- Qingmao region/resource/combat engine
- story/canon/IF engine 后续切口

硬指标：

- 同一输入同一上下文必须得到同一 ruling。
- DeepSeek candidate 只能被本地 ruling 覆盖，不能反过来覆盖本地引擎。
- 非法状态写入次数 `0`。

### L3 Store 与存档测试

目标：状态不污染，旧档不坏。

覆盖：

- `livingWorldState.playerGoals`
- `knownFacts`
- `hiddenFactRefs`
- `npcMemories`
- `factionPressure`
- `actionConsequences`
- `ifDeviations`
- `SAVE_FORMAT_VERSION`

硬指标：

- 旧档迁移失败 `0`。
- UI/DeepSeek/叙事正文直接写活世界状态 `0`。
- 玩家确认长期目标后只写允许字段。
- 重复目标要去重或更新，不得无限堆积。

### L4 UI 与 E2E

目标：玩家能看懂，界面不遮挡，按钮不越权。

覆盖：

- 自由目标面板。
- 阻断理由、前置条件、风险提示、替代路径。
- 确认/取消/转为长期目标。
- 桌面、移动端、reduced motion。

硬指标：

- 关键面板文本遮挡/溢出 `0`。
- 关键按钮可点击。
- UI 不显示隐藏事实正文。
- UI 不展示未经 engine 认可的奖励、地点、NPC 生死。

### L5 DeepSeek Eval

目标：模型可用但不能越权。

覆盖：

- DeepSeek 候选生成。
- JSON/schema 合格率。
- 越权候选覆盖。
- cache hit/miss token 观测。
- retry cost 观测。

硬指标：

- DeepSeek 越权覆盖率 `100%`。
- 隐藏事实泄露次数 `0`。
- 非法奖励/地点/NPC 死亡写入次数 `0`。
- prompt hash 和 cache tokens 可观测。

### L6 长测与发布烟测

目标：上线不黑屏，不坏档。

覆盖：

- `npm test`
- `npx tsc --noEmit --pretty false`
- `npm run build`
- runtime/Qingmao asset scan
- long e2e
- production preview smoke

硬指标：

- 大阶段收束和 rc 前必须全跑。
- 生产预览黑屏为阻断级问题。
- 新 500KB+ chunk warning 需评估，不能默认忽略。

| 隐藏事实泄露 | 0 | 0 | 0 | 0 |
| 非法状态写入 | 0 | 0 | 0 | 0 |
| 存档迁移失败 | 0 | 0 | 0 | 0 |

- v0.11 rc 前，样本矩阵必须覆盖所有接入 UI/store 的路径。

## 样本库演进规则

样本库分为五类：

1. `golden`：必须稳定通过的核心样本。
2. `boundary`：世界观、奖励、存档、隐藏事实边界样本。
3. `identity`：不同出身、势力、境界下裁决不同的样本。
4. `fuzzy`：同义句、错别字、口语、模糊目标。
5. `deepseek_override`：模型越权候选必须被本地覆盖的样本。

每接入一个新系统，必须补样本：

- 接 UI：补显示、交互、移动端、遮挡样本。
- 接 `playerGoals`：补写入、去重、更新、读档样本。
- 接青茅活世界：补 NPC 记忆、势力压力、区域变化样本。
- 接原著事实卡：补可见事实、隐藏事实、IF 偏离样本。
- 接 DeepSeek：补 JSON、schema、越权、缓存/token 样本。

## 阶段性自我进化机制

### 轻量复盘

频率：每完成 2 个小版本，或连续 3 个功能切口后。

产出：

- 当前测试缺口。
- 新增高风险样本。
- 是否需要更新专家团角色。
- 是否需要新增门禁。
- 是否需要更新 skill。

建议文件：

- 当前版本 `复盘-YYYYMMDD.md`。
- 或追加到对应阶段验收文档。

### 完整复盘

频率：

- 每个大阶段结束。
- 进入 rc 前。
- 出现一次严重回归或生产黑屏后。

必须检查：

- 设计是否跑偏。
- 文档是否互相冲突。
- 测试矩阵是否覆盖新系统。
- `.learnings/ERRORS.md` 是否已沉淀新 bug。
- `.learnings/LEARNINGS.md` 是否记录新原则。
- 专家团 skill 和项目 `AGENTS.md` 是否需要更新。
- 是否需要新增或废弃门禁。

### 复盘结论类型

每条复盘结论必须归为：

- `新增测试`
- `新增门禁`
- `更新 skill`
- `更新文档`
- `架构债`
- `延期需求`
- `用户决策`

这样后续不会只有泛泛总结，而能变成下一轮行动。

## 本轮必要性和难度评估

必要性：高。

理由：

- 自由意图和活世界会显著增加组合复杂度。
- 玩家输入不可穷举，必须靠分层样本和硬边界测试兜底。
- 项目目标是 v1.0 之后再公布，现在正适合用慢一点的节奏打地基。
- 测试工程化是 RebornG 和一般 AI 游戏拉开差距的核心能力。

难度：中等偏高，但可控。

难点：

- 样本库需要持续维护，不是一版写完。
- UI/e2e、DeepSeek eval、存档长测会增加运行成本。
- 每个新增系统都要补测试，短期开发速度会变慢。
- 原著事实卡需要人工审校，不能只靠模型。

降低难度的方法：

- 先维护固定矩阵，不急着做大规模随机测试。
- 引擎测试优先，e2e 只覆盖关键路径。
- DeepSeek live eval 只在关键阶段跑，日常用 replay/dry-run。
- 每 2 个小版本轻量复盘，rc 前完整复盘。
