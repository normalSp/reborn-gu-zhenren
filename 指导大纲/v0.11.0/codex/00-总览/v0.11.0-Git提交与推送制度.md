# v0.11.0 Git 提交与推送制度

日期：2026-05-15
状态：第一版已建立，纳入阶段完成门禁。
用途：把 commit/push 从口头约定变成 RebornG 的工程流程。

## 目标

RebornG 不能把很多天的工作堆在一个未提交工作区里，也不能把每个小文件都碎片化提交。

本制度采用：小阶段、可验证、可追溯、少混杂。

## 当前仓库状态

- remote：`origin git@github.com:normalSp/reborn-gu-zhenren.git`
- 当前本地分支名：`codex/v090-b1-world-action-protocol`
- 当前分支名已落后于实际阶段。
- 工作区存在大量历史改动和未跟踪文件。

因此，后续提交前必须先做范围筛选，不能把无关历史脏项混入当前阶段提交。

## 提交频率

默认规则：

1. 每个小阶段完成并通过对应验证后，必须 commit。
2. 每个可推送阶段完成后，必须 push。
3. 长任务超过一天或上下文风险升高时，做 checkpoint commit。
4. 重大迁移、发布配置、DeepSeek 权限、存档字段变化前后，各做一次 checkpoint。
5. bug 修复必须和回归测试或 `.learnings/ERRORS.md` 更新一起提交。

不推荐：

- 每个小文件都提交。
- 多个小版本堆到一次提交。
- 把 unrelated dirty files 混进阶段提交。
- 在测试失败时推送普通阶段提交。

## 何时必须 push

以下情况必须 push：

- 小版本阶段完成，例如 `v0.11.0-b2-1`。
- 大阶段完成，例如 `v0.11.0-b2` 收束。
- rc / 发布候选前。
- 重要 bug 修复完成且验证通过。
- 需要跨工具或跨窗口继续工作前。

以下情况可以先 commit 暂不 push：

- 正在拆分历史脏项。
- 本地 checkpoint 只用于防上下文丢失。
- 验证还没完整通过，但需要保存中间态。

## 分支规则

建议后续建立当前阶段分支：

- `codex/v011-living-world`
- 或更细：`codex/v011-b2-visible-investigation`

当前旧分支 `codex/v090-b1-world-action-protocol` 不适合长期继续承载 v0.11 工作。

切换或创建分支前必须确认：

- 当前工作区中哪些是本轮改动。
- 哪些是历史脏项。
- 是否需要先创建 checkpoint。
- 是否有用户明确要求提交/推送。

## 提交信息规则

提交信息使用中文，格式建议：

```text
类型: 范围 + 结果
```

例子：

- `docs: 建立v0.11项目仪表盘与Git推送制度`
- `feat: 接入v0.11-b2可见范围调查闭环`
- `test: 补齐v0.11-b2调查回流样本`
- `fix: 修复青茅战场行动卡遮挡回归`
- `chore: 同步v0.11阶段交接与专家团状态`

## 提交前检查

每次 commit 前必须确认：

1. `git status --short` 已检查。
2. 只 stage 本阶段相关文件。
3. 未误 stage 密钥、`.env`、`.codex` runtime state、插件缓存、SQLite/log、无关素材包。
4. 已运行与阶段风险匹配的测试。
5. 已更新必要文档和上下文交接。
6. 项目仪表盘已更新。

## 推送前检查

每次 push 前必须确认：

1. 最近一次 commit 内容边界清楚。
2. 当前分支名与阶段匹配或有明确说明。
3. focused tests 通过。
4. 若是阶段完成，至少 `tsc` 或 build 之一通过；高风险阶段需全量。
5. `PROJECT-STATE.md` 和仪表盘不与源码事实冲突。

## 暂不做

本阶段暂不强制：

- GitHub PR 流程。
- branch protection。
- GitHub Actions。
- GitHub Projects 看板。

这些进入 `v0.12.0` 候选专项池。

## 阶段完成门禁

每个阶段完成时必须在交接文件中写明：

- 是否 commit。
- 是否 push。
- 如果没有 commit/push，原因是什么。
- 下一次提交建议范围。
- 当前分支和 remote 状态。
