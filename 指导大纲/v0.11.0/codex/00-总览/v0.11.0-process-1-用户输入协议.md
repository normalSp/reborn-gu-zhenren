# v0.11.0-process-1 用户输入协议

日期：2026-05-15
状态：第一版已建立。
用途：把用户从“临时想到什么说什么”转为“关键节点输入高价值判断”。

## 基本原则

用户负责：

- 方向。
- 边界。
- 口感。
- 取舍。
- 发布承诺。

Codex 负责：

- 拆解。
- 实现。
- 测试。
- 文档。
- 复盘。
- 主动提案。

专家团负责：

- 提出候选需求。
- 暴露风险。
- 审查边界。
- 维护门禁。
- 在需要用户拍板时停下来。

## 用户输入类型

### 1. 方向输入

触发时机：

- 新版本开始。
- 新专项开始。
- 长期路线需要调整。

用户需要回答：

- 本阶段玩家应该感受到什么？
- 这个阶段不是为了做什么？
- 哪些远期愿景只进入长期路线，不进入本期？

示例：

- “v0.11 先铺自由意图和活世界地基，不追求爽快。”
- “v1.0 前不公开，先追求质量。”

### 2. 边界输入

触发时机：

- 原著事实、IF 偏离、自由意图、奖励、地点、NPC 生死、势力归属出现风险。

用户需要回答：

- 这个偏离能不能接受？
- 哪些原著事实不能被玩家当前阶段改变？
- 哪些只允许成为长期目标、传闻或不可达目标？

示例：

- “一转玩家想拿九转蛊，只能作为长期野心或骗局风险。”
- “讨论里提到的狼潮/灵泉枯竭不是已冻结锚点，必须先查原著。”

### 3. 优先级输入

触发时机：

- 专家团给 3-5 条候选需求。
- 多个技术债/体验债/内容债同时存在。

用户只需要做：

- 批准。
- 驳回。
- 延期。
- 合并。
- 调整顺序。

用户不需要做：

- 拆文件。
- 设计测试。
- 管实现细节。

### 4. 口感输入

触发时机：

- 出现可玩样板。
- 出现 UI/视觉/叙事样板。
- 出现自由意图或活世界交互样板。

用户需要回答：

- 像不像蛊真人？
- 玩家是否感觉自己真的在世界里行动？
- 是否太机械、太绕、太不爽或太轻浮？
- 有没有破坏原著氛围？

这是用户最重要的输入之一，不能被测试替代。

### 5. 取舍输入

触发时机：

- 质量、速度、范围、稳定性、自由度冲突。

用户需要回答：

- 本阶段优先保质量还是速度？
- 要先做自由度还是先做门禁？
- 要先做可玩闭环还是先做视觉表现？
- 是否接受第一版不爽快，换取长期稳定？

### 6. 发布输入

触发时机：

- 对外公告。
- 截图/短录屏。
- EdgeOne 公开预览。
- 版本命名。
- 对外承诺。

用户需要回答：

- 哪些内容可以公开？
- 哪些只是内部样板？
- 哪些不能承诺最终效果？

## Codex 提问格式

后续 Codex 需要用户输入时，应明确标注类型：

- `方向决策`：决定阶段目标。
- `边界决策`：决定世界观、IF、奖励、地点、自由度边界。
- `优先级决策`：批准/驳回/延期候选需求。
- `口感验收`：确认体验是否符合目标。
- `取舍决策`：在冲突中选择保哪一项。
- `发布决策`：决定是否可以对外说。

避免使用模糊问题：

- 不问“你怎么看？”
- 不问“要不要优化一下？”
- 不把技术细节包装成用户必须理解的问题。

## 必须停下来问用户的情况

- 改变版本目标。
- 改变原著/IF 大边界。
- 开放新的自由意图结果类型。
- 新增持久化字段或提升 `SAVE_FORMAT_VERSION`。
- 让 DeepSeek 拥有新的写入权。
- 引入外部运行时依赖。
- 启用可写子代理。
- 对外发布或承诺。
- 需要牺牲一个重要目标换另一个目标。

## 可以自动推进的情况

- 已批准范围内的实现。
- 已批准范围内的测试补齐。
- 文档事实同步。
- 交接记录更新。
- 小型 bug 修复和回归测试。
- 不改变用户承诺的小重构。
- 专家团候选需求整理。

## 用户输入记录要求

每次用户做出关键决策后，至少同步到以下之一：

- `v0.11.0-需求决策池.md`
- 当前阶段文档。
- `PROJECT-STATE.md`
- 最新 `codex上下文信息/`

重大技术/路线选择还应进入 ADR-lite。
