# v0.11.0-process-1 专家团阶段门禁

日期：2026-05-15
状态：第一版已建立。
用途：让专家团在每个阶段稳定发挥作用，而不是只在用户点名时临时出现。

## 阶段入口门禁

进入任一小版本或专项前，必须检查：

1. 当前版本、阶段、完成状态和 `SAVE_FORMAT_VERSION` 是否一致。
2. `PROJECT-STATE.md`、当前版本 README、路线图、真相源索引、最新交接是否冲突。
3. `v0.11.0-项目仪表盘.md` 是否反映当前阶段、下一步、阻塞、最近验证和 Git 状态。
4. 本阶段是否触碰：
   - 活世界状态。
   - 自由意图。
   - DeepSeek 回流。
   - 原著/IF 边界。
   - 存档迁移。
   - 战斗/奖励/经济。
   - UI 可读性。
   - 对外发布。
5. 是否需要用户输入，输入类型是什么。
6. 是否已有测试矩阵或需要先补矩阵。

## 实现前门禁

动运行时代码前必须确认：

- 是否有 canon/数据真相源。
- 是否需要 schema/type。
- 是否需要纯 engine 先行。
- store 是否只能通过 action 或受控 patch 写入。
- UI 是否只消费 engine/store 事实。
- DeepSeek 是否只写候选/叙事/解释。
- 是否已有测试样本。
- 是否需要旧档迁移。
- 是否需要用户决策。

默认实现顺序仍为：

`canon -> types/schema -> engine -> store -> UI -> tests/docs`

## 用户输入门禁

以下任一情况必须停下来：

- 范围扩大。
- 原著事实可能被改写。
- 玩家自由意图产生新结果类型。
- DeepSeek 可能获得新权限。
- 持久化字段变化。
- 运行时依赖变化。
- 可写子代理或多代理流程。
- 对外发布承诺。

如果不确定是否需要用户输入，按需要输入处理。

## 测试门禁

任何触碰自由意图、活世界、DeepSeek、原著/IF、UI/store 的改动，必须先确认：

- 是否要更新 `v0.11.0-测试工程化总纲.md`。
- 是否要更新 `v0.11.0-a3-裁决样本矩阵.md` 或对应阶段测试矩阵。
- 是否有新极端玩家意图、bug、走查、MiroFish 或 DeepSeek eval 样本需要按 `指导大纲/流程制度/测试矩阵演进规则.md` 三分流为 `current_matrix`、`future_sample_pool` 或 `discarded`。
- 是否需要新增 L0-L6 某层测试。
- 是否能用旧档、长测或 e2e 复现关键风险。

## 完成门禁

阶段完成前必须检查：

1. 相关测试是否运行。
2. 文档事实是否同步。
3. 最新上下文交接是否写入。
4. `.learnings/ERRORS.md` 是否需要新增 bug 经验。
5. `.learnings/LEARNINGS.md` 是否需要新增设计经验。
6. `reborn-expert-council`、`game-dev-text`、`reverend-insanity-lore`、`reborn-combat-motion` 是否需要更新。
7. 是否存在需要用户复盘或验收的口感问题。
8. `v0.11.0-项目仪表盘.md` 是否已更新。
9. 交接文件是否记录 commit/push 状态；若未 commit/push，是否写明原因和下一次提交建议范围。

## 复盘节奏

- 每 2 个小版本或 3 个功能切口：轻量复盘。
- 每个大阶段、rc、严重回归或线上事故后：完整复盘。

复盘结论必须归类为：

- 新增测试。
- 新增门禁。
- 更新 skill。
- 更新文档。
- 架构债。
- 延期需求。
- 用户决策。
