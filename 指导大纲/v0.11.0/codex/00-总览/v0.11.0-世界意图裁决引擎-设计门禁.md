# v0.11.0 世界意图裁决引擎设计门禁

日期：2026-05-15
状态：硬门禁；未通过前不得进入 `v0.11.0-a2/a3/b2` 的运行时代码开发。

## 门禁目的

世界意图裁决引擎是后续“活世界”的入口裁判。它不能做成一个绕开现有系统的新大脑，也不能让 DeepSeek 变成事实写入者。

本门禁解决三个问题：

1. 现有子系统是否会和自由意图裁决冲突。
2. 外部项目和论文的设计思想如何吸收，而不是盲目引入依赖。
3. 后续每次进入活世界、自由输入、NPC 记忆、正史/IF、区域持久化开发时，如何强制回到同一套契约。

## 触发条件

出现以下任一情况，必须先执行本门禁：

- 新增或修改 `WorldIntentEngine`、`IntentCandidate`、`IntentRuling`、自由输入解析、玩家长期目标。
- 新增活世界持久化字段：区域状态、NPC 记忆、势力压力、已知/隐藏事实、IF 偏离、行动后果。
- DeepSeek 参与生成行动候选、NPC 反应、正史/IF 解释、失败原因或替代路径。
- 任何系统试图根据玩家自然语言直接写入背包、奖励、地点、战斗结果、NPC 生死、正史锚点、结局。
- 青茅山活世界最小闭环进入运行时。

## 核心原则

### 意图裁决不是活世界全部

世界意图裁决引擎只负责：

- 理解玩家目标。
- 判断当前能不能做。
- 解释为什么不能做。
- 给可执行、需前置条件、传闻、长期目标、危险注意、重大 IF 偏离等裁决。
- 把可执行意图路由到现有行动协议和本地引擎。

它不负责：

- 直接结算战斗。
- 直接发奖励。
- 直接开放地点。
- 直接改 NPC 生死。
- 直接改正史结果。
- 直接写完整世界剧情。

### 活世界至少需要五个内核

1. 世界意图裁决引擎：自由目标入口和本地裁决。
2. 活世界状态协议：世界时间、区域状态、已知/隐藏事实、NPC 记忆、势力压力、行动后果。
3. NPC/势力反应引擎：信任、警惕、追查、招揽、报复、封锁。
4. 正史/IF 锚点引擎：保护原著硬事实，记录有代价的偏离。
5. 本地结算引擎群：战斗、修行、炼蛊、资源、交易、路线解锁。

v0.11 只做地基，不追求一次性完成完整大世界。

## 现有系统集成契约

| 子系统 | 允许读取 | 允许写入 | 禁止事项 | 门禁要求 |
| --- | --- | --- | --- | --- |
| DeepSeek / response pipeline | 玩家输入、可见事实、裁决结果 | 叙事、候选、解释、传闻 | 直接写奖励、地点、NPC 生死、正史、结局 | 所有 DeepSeek 输出必须经过本地 ruling 覆盖 |
| 统一行动协议 | `IntentRuling` 中可执行候选 | 行动账本、回流上下文 | 接受未裁决自由文本直接结算 | 所有自由输入必须先转成裁决结果 |
| Qingmao region engine | 身份、线索、行动槽、区域边界 | 当前已有 action candidate / blocked reason | 绕过持久化区域状态开放三寨委托 | 新区域状态字段冻结前只读或临时派生 |
| combat candidate / battlefield | 裁决后的战斗候选、入口校验 | 现有战斗结果回流 | 自由输入直接杀死 NPC、直接掉落 Gu/材料 | 战斗只接收本地验证候选 |
| resource / refine / feeding | 裁决后的资源行动、材料缺口 | 既有材料包、食料提示、残方进度 | DeepSeek 或意图裁决直接发完整蛊方、仙蛊、稳定白玉材料源 | 奖励和材料必须由资源/炼蛊引擎决定 |
| story/canon/IF | 正史事实、隐藏事实、IF 边界 | IF 压力/候选账本 | 直接改原著核心结果或泄露隐藏事实 | 正史/IF 写入必须有来源和偏离等级 |
| store/save | 已批准字段、默认值、迁移 | 玩家长期目标、NPC 记忆、区域压力等批准字段 | 未冻结字段直接持久化 | 新字段必须列出 `SAVE_FORMAT_VERSION`、默认值、迁移测试 |
| UI | 裁决结果、原因、替代路径 | 玩家选择、确认、取消 | UI 私算成功率、私开奖励、隐藏阻断原因 | UI 只解释，不拥有事实裁决权 |

## 必交付物

执行本门禁时必须产出：

1. `IntentCandidate` / `IntentRuling` 字段草案。
2. 现有系统读写权矩阵。
3. DeepSeek 输入输出契约。
4. 存档字段表：字段名、默认值、迁移、是否提升 `SAVE_FORMAT_VERSION`。
5. 正史/IF 隐藏事实处理规则。
6. UI 展示规则：阻断原因、替代路径、风险提示。
7. 测试矩阵：极端样本、越界 DeepSeek、旧档迁移、无副作用、长测。

缺任一项，不进入运行时代码。

## 外部设计思想融入

### Concordia：GM 裁决模式

参考来源：

- GitHub: https://github.com/google-deepmind/concordia
- Paper: https://arxiv.org/abs/2312.03664

可吸收思想：

- 使用类似桌游 GM 的实体来解释环境与行动结果。
- 自然语言行动必须经过环境合理性检查。
- 物理、社交、数字空间中的行动都要被 grounded。

RebornG 融入方式：

- `WorldIntentEngine` 是“蛊世界 GM”，但它是本地规则引擎，不是纯 LLM。
- 每次玩家自由输入都走：意图候选 -> 本地合理性裁决 -> 可执行候选/阻断/传闻/长期目标。
- 不引入 Concordia 依赖；只借鉴 GM 分层和 grounded adjudication。

### Generative Agents：记忆、反思、计划

参考来源：

- Paper: https://arxiv.org/abs/2304.03442
- GitHub: https://github.com/joonspk-research/generative_agents

可吸收思想：

- Agent 记录经历。
- 周期性总结成高层反思。
- 根据相关记忆和计划生成可信行为。

RebornG 融入方式：

- NPC 不直接拥有无限自然语言记忆，而是写入结构化 `npcMemoryLedger`。
- 重要行动写成短事件：谁、何时、在哪、对谁、做了什么、影响什么。
- 反思只能变成结构化倾向：信任、警惕、恐惧、利用价值、仇怨、招揽意图。
- DeepSeek 可把结构化记忆写成文本，但不能把文本反思当成事实源。

### AI Town：共享状态、事务、模拟循环

参考来源：

- GitHub: https://github.com/a16z-infra/ai-town

可吸收思想：

- 共享全局状态。
- 事务式更新。
- 模拟引擎和前端展示分离。
- JS/TS 生态可做可扩展 AI 世界。

RebornG 融入方式：

- store action 应像事务：输入裁决、输出补丁、日志和副作用说明。
- 活世界状态写入集中到明确 slice，不让 UI 或 DeepSeek 私自散写。
- 不引入 Convex 或 AI Town 架构；保留 React + Zustand + 本地引擎。

### TextWorld：文本动作空间与可测试性

参考来源：

- GitHub: https://github.com/microsoft/TextWorld
- Paper: https://arxiv.org/abs/1806.11532

可吸收思想：

- 文本世界需要显式动作空间。
- 自然语言自由感必须落回有限、可测试、可解释的动作。
- 环境状态和动作结果可重复验证。

RebornG 融入方式：

- v0.11 首批只支持 `obtain_item`、`join_faction`、`investigate`、`travel`、`long_term_goal`。
- 每个意图类型必须有合法目标、非法目标、降级路径和测试样本。
- 测试中要验证原状态不被非法候选污染。

### ReAct / Reflexion / Tree of Thoughts：候选生成与失败学习

参考来源：

- ReAct: https://arxiv.org/abs/2210.03629
- Reflexion: https://arxiv.org/abs/2303.11366
- Tree of Thoughts: https://arxiv.org/abs/2305.10601

可吸收思想：

- ReAct：推理和行动交替，有助于生成更贴近环境的候选。
- Reflexion：失败后形成语言反馈，提高后续表现。
- Tree of Thoughts：多候选探索和比较，而不是单一路径。

RebornG 融入方式：

- DeepSeek 可以生成 3-5 个候选，但候选必须被本地评分和过滤。
- 失败反馈写入测试/门禁/`.learnings`，不让模型自由改规则。
- 多候选只用于“给玩家有意义选择”，不用于绕过本地裁决。

## 不吸收的做法

以下做法暂不进入 RebornG：

- 让 AI agent 全天候自主模拟整个世界并持续写存档。
- 把向量记忆当作事实源。
- 让 NPC 文本反思直接改变数值关系。
- 让 DeepSeek 自己维护正史锚点。
- 引入新的后端数据库、Convex、Python simulation runtime 或大型 agent 框架。
- 为了“活世界”牺牲可测试性、可复现性和存档稳定。

## 首批门禁测试样本

必须覆盖：

1. 我要拿九转蛊。
2. 我要获得春秋蝉。
3. 我要去宝黄天交易。
4. 我要投靠白家。
5. 我要跟踪方源。
6. 我要逃离青茅山。
7. 我要杀死白凝冰。

每个样本至少验证：

- 裁决类型正确。
- 阻断或降级原因正确。
- 不写入非法奖励、地点、NPC 生死、正史结果。
- DeepSeek 越界候选会被覆盖。
- UI 能显示当前可做什么。

## 通过标准

本门禁通过时必须满足：

- 冲突矩阵已覆盖现有相关子系统。
- 新增字段已冻结，或明确本阶段不新增持久化字段。
- `SAVE_FORMAT_VERSION` 决策明确。
- 至少一个青茅低阶样本有从玩家目标到本地裁决再到 UI 展示的完整路径设计。
- 测试计划能防止 DeepSeek 越权和 UI 私算。
- 文档已同步到 v0.11 README、需求决策池、真相源索引、专家团 skill 和上下文交接。

## 失败处理

如果门禁发现现有子系统冲突，优先处理顺序：

1. 先收窄 v0.11 范围。
2. 再修改契约。
3. 再补测试。
4. 最后才改运行时代码。

不能用“先写出来再说”绕过门禁。
