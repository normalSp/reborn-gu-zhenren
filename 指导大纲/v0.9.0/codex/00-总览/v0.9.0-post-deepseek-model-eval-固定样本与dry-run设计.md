## 样本结构

每条样本建议包含：

```json
{
  "id": "v090-action-001",
  "category": "world_action_return",
  "mode": "narrative",
  "systemPromptProfile": "v090-runtime-json",
  "stateFixture": "summary only, no original novel text",
  "userPrompt": "runtime-style user message",
  "mustContain": ["json", "choices"],
  "mustNotContainFacts": ["Immortal Gu reward", "rank ten", "mortal Treasure Yellow Heaven trade"],
  "expectedStateUpdateDomains": ["world_action_candidates"],
  "loreRisk": "medium"
}
```

原则：

- 样本只放项目状态转述，不复制原著长段文本。
- `mustNotContainFacts` 用于自动标记明显越界。
- `expectedStateUpdateDomains` 只允许候选、线索、传闻、压力和叙事承接。
- 所有正式奖励、战斗结果、地点、福地/洞天归属、仙蛊、完整蛊方必须由本地 canon/engine 决定。

## Dry-run 模式

dry-run 不调用 API，只做三件事：

1. 读取样本，确认每条都有 `id/category/mode/systemPromptProfile/stateFixture/userPrompt/mustNotContainFacts`。
2. 生成将要发送的 system/user prompt 摘要和 `promptPrefixHash`，检查同一 `systemPromptProfile` 的 hash 是否稳定。
3. 输出预估 token 大小和风险标签，但明确标记为 estimate，不写缓存命中率。

dry-run 通过标准：

- 样本 JSONL 可解析。
- 每类样本数量达到目标。
- 同一 `systemPromptProfile` 的 `promptPrefixHash` 稳定。
- 样本不含原著长段文本。
- 所有高风险样本都有 `mustNotContainFacts`。

## Live 模式

live 模式是真实调用 DeepSeek API 的评测模式，会消耗 API token。用户已在 2026-05-13 表示 token 消耗无所谓；runner 已实现真实 API 调用、报告文件、失败重试统计和成本估算。

运行前必须记录：

- 评估日期。
- 官方价格页截图或链接时间戳。
- 模型列表。
- 每个模型的 temperature、maxRetries、timeoutMs。
- 样本总数和预计最大调用次数。

运行后报告必须包含：

- 每个模型的 JSON 通过率、Zod 通过率、L3/L4 违规率。
- total tokens、cached tokens、retry tokens、elapsed ms。
- prompt prefix hash 稳定性。
- 单位有效叙事成本。
- 人工抽样备注。
- 是否建议进入正式切换决策。
