# v0.16.0-a1 信息架构与权威表设计门禁

日期：2026-05-18
状态：第一版完成
阶段性质：docs/design gate only
Player Advocate：豁免 runtime 走查，原因是本阶段不改玩家可见运行时代码
MiroFish：blocking `not_needed`

## 结论

`v0.16.0-a1` 冻结的不是最终 UI 视觉稿，而是后续 UI 减法必须遵守的信息架构和权威边界。

正式玩家主入口收束为 6 个一级心智分区：

`地图 / 行动 / 角色 / 蛊道 / 世界 / 记录`

运行时文案可以保留更有味道的称呼，例如当前域地图显示为 `青茅舆图`。但系统设计上必须按上面 6 个一级分区归并，不再继续把每个子系统都暴露成底栏按钮。

## 当前问题

当前 `GameScreen` 底栏暴露了：

- `地图`。
- `青茅 / 凡战 / 群像 / 大阵` 演武入口。
- `行动 / 自由目标 / 传承 / 宿命 / 终局 / 蛊师属性 / 蛊虫 / 杀招列表 / 炼蛊 / 蛊材 / 空窍状态 / 人物图鉴 / 小队 / 流派道痕 / 商会 / 成就 / 道场 / 事件日志`。

这让玩家看起来像在操作开发工具台，而不是在一个活着的蛊真人世界里行动。问题不是功能多，而是一级入口权重失控。

## a1 冻结规则

1. `行动` 是玩家做事的总入口。
2. `自由目标` 不再作为一级入口独立存在，归入 `行动`。
3. `蛊虫 / 杀招 / 炼蛊 / 蛊材` 归入 `蛊道`。
4. `青茅 / 凡战 / 群像 / 大阵` 从正式底栏移入开发/演示抽屉，正式战斗必须从剧情、行动候选或战斗入口进入。
5. `传承 / 宿命 / 终局 / 商会` 不作为一转青茅开局常驻一级入口，按境界、剧情、场景或开发条件显示。
6. `角色` 负责玩家自身、空窍、人物、道痕、成就等角色信息。
7. `世界` 负责世界态势、剧情锚点、传承/终局/商会等高阶或场景化内容的受控入口。
8. `记录` 负责事件日志、行动账本、可见事实回顾、系统回流记录。

## 运行时代码边界

a1 不改 runtime。后续 b 阶段改 UI 时必须遵守：

- 不新增持久字段。
- 不 bump `SAVE_FORMAT_VERSION`。
- 不新增 DeepSeek 权限。
- 不开放正式奖励、地点、阵营、NPC 生死。
- 不把候选、只读、演示、开发入口包装成正式世界事实。
- 不用 UI 减法名义重写 engine/store 目录。

## 后续小版本承接

| 小版本 | 承接 a1 的方式 | 硬边界 |
|---|---|---|
| `a2` | 同步仪表盘、handoff、AGENTS、PROJECT-STATE 和 Git 计划入口 | 不改 runtime |
| `b1` | 把 `行动 + 自由目标` 合并为一个玩家操作工作台 | 不扩大 ActionPanel/FreeGoalPanel 写权限 |

| `rc` | 全量质量门和 60 轮 Player Advocate | 不以 rc 名义扩范围 |

## 必停决策

以下事项即使在 v0.16 内出现，也必须停下来让用户决策：

1. 需要保存 UI 偏好或新增任何存档字段。
2. 需要让 DeepSeek 直接决定奖励、地点、阵营、NPC 生死或正史事实。
3. 需要打开正式市场、黑市、委托利润、正式路线成功或地点切换。
4. 需要新增原著事实、隐藏事实、NPC/势力规则事实或 IF 规则。
5. 需要 MiroFish blocking 包。
6. 需要对外发布承诺、EdgeOne 自动部署、branch protection 或 auto merge。
