## 一、设计哲学与核心原则

### 1.1 根本矛盾

本书覆盖面临一个根本性矛盾：AI驱动的动态叙事追求自由度和涌现性，而原著IP追求还原度和确定性。如果每条故事线都是完全自由生成的文本，名场面的"名场面感"就会消失；如果将名场面写死为固定脚本，又丧失了AI叙事的意义。

解决这一矛盾的核心设计原则是：**约束注入，而非固定脚本**。名场面不是一段预设文本，而是一组注入AI的叙事约束——规定了"必须发生什么"和"不能发生什么"，但允许AI在约束框内自由组织语言。这类似于电影导演给演员一个场景目标和情绪基调，而不是逐字逐句的台词。

### 1.2 五条核心设计原则

**原则一：名场面是约束，不是脚本。** 四十多个名场面不是四十多段固定文本，而是四十多组约束规则。每个名场面对应一条`scene_constraint`对象，包含事件核心要素（人物、地点、后果）、必须出现的标志性元素（如开窍大典的测灵石）、禁止出现的偏差（如开窍大典不能写成低调内敛的私下测试）。AI在这些约束框内自由发挥语言的文学性。

**原则二：全局事件是涟漪，不是强制剧本。** 全局事件（如三王山、宿命大战）不应强制所有域玩家以相同方式参与。设计为"涟漪系统"——事件在源发域以完整形态呈现，在相邻域以"波及叙事"呈现（如远方天际的震动、流言的传来、势力态度的微妙变化），在远离域可能仅以"传闻"形式出现。玩家可以选择是否介入。这保证了每个域的叙事独立性，同时让世界感觉是一个联动的整体。

**原则三：章节是状态区间，不是叙事容器。** 章节不应被理解为"这一段故事的内容"，而应被理解为"这个区间内世界的规则和约束"。章节定义了出场NPC池、可用地点、势力关系、关键事件条件和名场面约束。不同的域共享相同的全局章节区间，但在每个域内的具体展开完全不同。

**原则四：flag驱动一切，不硬编码路径。** 游戏状态的所有推进——故事进展、名场面触发、NPC态度变化、跨域影响——都应通过flag系统驱动，不依赖代码中的if-else硬编码。这确保系统可无限扩展，新增一条故事线只需新增一组flag定义，不需修改引擎代码。

**原则五：先验证架构，再扩展内容。** P1阶段仅做南疆三章不是"偷懒"，而是必要的架构验证。用一个域跑通完整的章节状态机、转章流程、上下文注入、名场面约束的所有环节后，再复制到其他四域。这避免了"做五个域才发现架构有致命缺陷需要推翻重来"的灾难。

## 二、原著内容全景盘点

### 2.1 篇章与名场面矩阵

蛊真人原著（截至天庭终战）可划分为8个主要篇章，每篇有标志性的核心名场面：

**第一卷·青茅山篇**（凡人-三转，原著约200章）
核心名场面：开窍大典（测灵石、空窍开启）、月下炼蛊（月光蛊初成）、狼潮攻城（群狼围寨、血战求生）、山寨覆灭（古月山寨焚毁、族人四散）、血颅蛊屠族（白凝冰失控、血案发生）。这一卷的特点是"从凡人到蛊师的蜕变"和"第一次面对蛊界残酷法则"。

**第二卷·商队南疆篇**（三转-五转，原著约300章）
核心名场面：投奔商队（山寨覆灭后的流亡选择）、商心慈初遇（商队据点、商人世家的第一印象）、商路劫匪（第一次商队护送战斗）、南疆商会入会（进入南疆最大的商业网络）、势力网初探（了解到南疆正道联盟、魔道散修、家族势力三足鼎立格局）、第一次跨势力博弈（利用商会信息周旋于各方之间）。

**第三卷·三王山篇**（五转-蛊仙，原著约250章）
核心名场面：三王传承开启（王庭福地前的三王试炼）、定仙游炼制（炼制跨域传送仙蛊）、白凝冰背刺（昔日伙伴的反目）、首次接触仙级力量（感受到蛊仙与凡蛊的天壤之别）、升仙抉择（选择三气升仙的路线）。

**第四卷·北原·王庭福地篇**（蛊仙-七转，原著约350章）
核心名场面：王庭福地开启（巨阳血脉的试炼场）、狐仙福地争夺（多位蛊仙混战的资源战）、巨阳传承现世（巨阳仙尊的遗留意志和传承）、黄金血脉觉醒（北原特有的血统力量体系）、长生天现踪（北原长生天的势力介入）。

**第五卷·义天山篇**（七转-八转，原著约300章）
核心名场面：义天山巅峰之战（多方势力在义天山的决战）、春秋蝉丢失（方源的底牌被破）、天庭第一次正面介入（天庭势力的展示）、多方混战（正魔散三方的全面对抗）、天道反噬初现（逆天行为的第一次天道回应）。

**第六卷·逆流河篇**（八转，原著约250章）
核心名场面：逆流河上溯（对抗时光逆流的极限挑战）、宿命蛊线索浮现（追寻宿命蛊的踪迹）、天庭全面介入（天庭倾力阻止宿命蛊被破坏）、时间线交织（春秋蝉与逆流河的时间冲突）、逆流而上突破（在不可能中创造可能）。

**第七卷·宿命大战篇**（八转-九转，原著约300章）
核心名场面：宿命蛊摧毁（宿命蛊的最终破坏，命运枷锁断裂）、命运枷锁断裂（整个蛊界的因果链重写）、天道反噬降临（天道对宿命破坏者的终极惩罚）、大爱仙尊崛起（方源从这一刻起进入新的层次）、全世界势力的反应（正道恐慌、魔道狂欢、散修观望）。

**第八卷·天庭终战篇**（九转，原著约200章计）
核心名场面：天庭覆灭（天道最后堡垒的崩塌）、炼天魔尊出世（方源以魔尊身份君临天下）、尊者之战（多位仙尊级别的对抗）、蛊界格局重塑（旧的秩序崩溃，新的秩序建立）。

以上合计约40-45个核心名场面，覆盖原著约2150章的内容跨度。

### 2.2 五域差异化潜力评估

五域的核心差异不仅在于地名和出场NPC不同，而在于整个叙事范式、成长路径和价值取向的根本差异：

**南疆**（山寨/家族·丛林法则）：叙事范式是"家族内部的等级斗争和外部势力的渗透"。成长依赖于山寨内的资源分配和人际关系操作。名场面参与方式最接近原著（方源线），是三王山、义天山的近距离参与者。

**北原**（部落/游牧·血统至上）：叙事范式是"部落间的征服与部落内的血脉竞争"。成长依赖于战功和血脉觉醒。王庭福地和巨阳传承是其核心叙事驱动力。全局事件中更容易站在"争夺传承者"的立场。

**东海**（海岛/散修·自由探索）：叙事范式是"在不同岛屿间航行探索和散修间的松散合作"。成长依赖于探索发现和交易积累。较少受大陆政治影响，更适合喜欢"避开主线自由发展"的玩家。全局事件中往往是"旁观者"或"后期介入者"。

**西漠**（沙漠/绿洲·生存优先）：叙事范式是"在资源匮乏的环境中争夺生存权"。成长依赖于绿洲控制权和古老遗迹的发现。人祖传说和蓝海秘境是其独特内容。全局事件中可能以"人祖传承守护者"的身份参与。

**中洲**（宗门/正统·秩序维护）：叙事范式是"在正道秩序内爬升地位并应对道德困境"。成长依赖于宗门贡献和师承关系。天庭线是其终极叙事驱动。全局事件中常常站在"秩序维护者"或"变革者"的十字路口。

**关键设计判断**：五域不是五个皮肤，而是五套不同的叙事范式。如果用同一套模板套五个域，玩家会在第二遍就发现"换了个地名和NPC名字而已"。因此五域的核心投入不在于增加NPC数据库条目和工作量，而在于为每域设计独立的叙事范式、开局情感基调、以及该域独有的"不可替代体验"——例如北原的部落征服感、中洲的宗门晋升感、东海的探索发现感。

## 三、系统架构设计

### 3.1 章节路由架构：从线性到条件图

当前P1设计为线性三章：青茅山→商队→南疆初探。这足以验证章节系统的可行性，但不足以承载五域+全书覆盖的需求。

**P1即需预留多域扩展接口。** 由于决策19已确认要做五域开局，即使P1实际只实现南疆三章，chapterSlice和chapters.json的数据结构也必须从现在起就支持多域。这不是"先做三章以后再改"——而是"P1设计数据结构时就为五域预留全部槽位，但P1只填充南疆三章的数据"。

扩展到全书覆盖需要升级为**条件路由图**（Conditional Route Graph），核心数据结构如下：

```
ChapterRoute = {
  chapterId: string;
  domain: '南疆' | '北原' | '东海' | '西漠' | '中洲' | 'global';
  prerequisites: {
    requiredFlags: string[];     // 触发条件：必须全部为true
    requiredRealm: number;
    requiredDomain?: string;     // 仅适用于特定域
    requiredChapter?: string;    // 前置章节
  };
  alternatives: string[];         // 如果此章被跳过，可选替代章节
  isExclusive: boolean;           // 是否为域专属章节（global章的false）
  domainOpeningChapter: boolean;  // [P1预留] 是否为该域的开局章节（五域开局用到）
}
```

**P1阶段必须完成的接口预留（0.5天，嵌入P1工期）**：

1. `chapters.json`顶层结构预留五域命名空间。即使P1只填充`南疆`域的三章，JSON结构也必须预留`北原`、`东海`、`西漠`、`中洲`四个空数组槽位。做法是在chapters.json中定义：
   ```json
   {
     "domains": {
       "南疆": [ /* P1填充3章 */ ],
       "北原": [],
       "东海": [],
       "西漠": [],
       "中洲": []
     },
     "global": [ /* 全局章节，P2填充 */ ]
   }
   ```
2. `chapterSlice`中的状态结构预留`currentDomain: string`字段。P1阶段固定赋值为`"南疆"`，P2阶段由开局选择赋值。所有涉及域的判断（如NPC过滤、场景描述风格）均使用`state.currentDomain`而非硬编码`"南疆"`。
3. `checkChapterProgression()`接口设计为接收`(state, domain)`两个参数，而非仅接收`state`。P1阶段domain固定传入`"南疆"`，P2阶段传入实际域值。
4. `chapters.json`类型定义（`ChapterDefinition`接口）包含所有字段，P1章节允许`domainOpeningChapter`为false，P2域专属开局章节设置为true。

**P1不需要做的**（留给P2）：域选择UI、多域实际数据填充、域间路由逻辑、跨域NPC过滤。P1只需要让数据结构和接口**支持**这些东西，而非**实现**它们。

关键设计点：
1. 章节分为"域专属章节"和"全局章节"两类。域专属章节仅在特定域可触发（如北原的"黄金试炼"章），全局章节对所有域开放但参与方式不同（如"三王山"章在南疆可近距离参与，在北原需通过王庭势力进入）。
2. 条件路由不做固定顺序——玩家可能通过不同路径到达同一章节。例如南疆玩家可能先经历"南疆风云"再进入"义天山"，而北原玩家可能通过"巨阳传承"直接跳入"义天山"。
3. `alternatives`字段解决"跳过推进"的后续路径问题——如果某个必要章节被跳过，系统自动指向替代路径而非死胡同。

### 3.2 全局事件涟漪系统

全局事件是全书覆盖中最复杂也最关键的设计。该系统的核心挑战是：如何让一个发生在南疆的全局事件，让远在北原的玩家也能感受到而不会被"强制传送"。

设计为四层涟漪模型：

**L0 — 核心事件**（事件发生域）：完整的事件体验。玩家身处事件发生域，获得完整的叙事线、战斗场景、选择分支。例如三王山事件中，位于南疆的玩家可以参与三王试炼、进入传承地、获得传承。

**L1 — 直接影响**（相邻域）：事件通过势力网络传导。位于北原的玩家可能收到王庭的紧急召集（"三王山有变，速去探察"），位于中洲的玩家可能收到天庭的监视任务。玩家可以选择介入或观望，介入后进入简化版事件（比L0的叙事线更短但核心结果相同）。

**L2 — 间接涟漪**（远隔域）：事件以"世界动态"形式呈现。位于西漠的玩家可能在下一轮叙事中看到："远方的天际隐约传来一阵震动，像是某种古老的存在被惊醒了。商路上的旅人们纷纷议论，说南疆三王山那边出了大事。"这种涟漪叙事不强制玩家任何行动，但让世界感觉是活的。

**L3 — 长尾影响**（全域网）：事件发生后的持久世界状态变化。无论玩家在哪个域，宿命蛊被摧毁后，全世界所有人都会感受到——境界突破变得更容易、天意变得模糊、某些古老的禁制失效。这些通过"全局flag"实现——宿命大战结束后设置`global_flag.fate_destroyed = true`，所有域后续叙事都会读取此flag调整叙事方向。

涟漪系统的实现方法：
- 每轮RESOLVED后，检测所有全局事件池：如果某事件的触发条件满足且尚未触发，则按玩家所在域计算涟漪层级
- L0参与：注入完整scene_constraint + NPC列表 + 势力动态
- L1参与：注入"介入选项"（特殊Choice）+ 简化版scene_constraint
- L2波及：注入一条"涟漪叙事文本"（不附带选择，纯叙事背景）
- L3长尾：持续存在的全局flag，影响后续所有篇章的叙事基调

### 3.3 Context-Builder扩展：泛域上下文注入

当前context-builder在每次AI调用时构建system prompt，章节上下文注入约500-1000 tokens。全书覆盖后需要新增的注入层：

**章节路由注入**（~200 tokens）：当前章节的id、名称、阶段、可用NPC池上限。替代当前硬编码的`flags.current_chapter || '青茅山期'`。

**名场面约束注入**（~300 tokens per scene）：仅注入"当前可能触发的"名场面约束，不是全部40条。由proximity检测器判断：如果玩家的境界、位置、已完成flag与某名场面的触发条件接近（距离≤2个前置flag），则注入该名场面的约束文本。

**全局事件涟漪注入**（~200 tokens）：当前所有处于"活跃状态"的全局事件（已触发但未完成）对当前域的影响描述。

**域势力动态注入**（~150 tokens）：当前域的关键势力关系状态——谁在崛起、谁在衰退、哪些势力之间有摩擦。

**跨域NPC状态同步**（~200 tokens）：曾经在本域出现过但已离开的NPC的"去向摘要"。例如："古月方源已于3轮前离开南疆，据传前往北原。"

总新增约1000-1200 tokens，在DeepSeek V4的128K窗口内完全可控。关键设计是**按需注入**而非全量注入——不同于把所有信息堆进prompt，而是通过proximity检测器只注入"当前相关的"规则。

## 四、五域出生点系统设计

### 4.1 开局体验差异化

五域开局不是简单地选择"where you start"——它是选择"what kind of story you want to experience"。每域的开局需要让玩家在前5轮内感受到该域的核心范式：

**南疆开局**：你从山寨/家族中醒来，首先面临的是家族内的地位竞争（"你是旁系子弟还是嫡系？"），然后是获得第一只蛊的流程。整体氛围是"等级森严但可以攀爬"。

**北原开局**：你从部落营帐中醒来，首先面临的是部落的生存竞争（"今年冬天部族是否有足够食物？"），然后是血脉觉醒的第一次暗示。整体氛围是"强者生存，血统至上"。

**东海开局**：你从渔村或散修聚居地醒来，首先面临的是"跟随哪个师父"的选择（师徒关系是东海的核心纽带），然后是第一次出海探索。整体氛围是"自由但孤独"。

**西漠开局**：你从绿洲村落中醒来，首先面临的是水资源分配（"你的家族控制水井吗？"），然后是寻找人祖遗迹的第一条线索。整体氛围是"每一滴水都很珍贵"。

**中洲开局**：你从宗门入门试炼中醒来，首先面临的是宗门选拔（"你被分配到哪个堂？"），然后是正道规则的第一次道德制约。整体氛围是"规则就是规则"。

### 4.2 开局数据层设计

开局选择后的初始状态设定：
- `flags.origin_domain`: 出生域ID
- `flags.origin_background`: 开局背景（山寨子弟/部落族人/散修学徒/绿洲居民/宗门弟子）
- 初始NPC关系网差异：南疆开局与古月方正初始关系为"兄弟"，北原开局与部落族长之子初始关系为"竞争者"
- 初始位置和地图认知：每域有不同的已知地图节点
- 初始蛊虫偏向：南疆开局更容易获得常见蛊，北原开局有概率获得血脉相关蛊

这些差异全部通过`playerSlice.flags`和`mapSlice`管理，不改变store架构。

### 4.3 域间迁移设计

在全书覆盖中，玩家可能会跨域迁移（如从南疆前往北原参加王庭福地）。跨域迁移的流程设计：
1. 触发条件：全局事件在玩家所在域外优先级触发，并提供"前往参与"的选项
2. 迁移叙事：AI生成跨域旅程的过渡叙事（3-5轮），消耗AP和资源
3. 迁移代价：离开原域后，原域NPC态度会因"久不联系"而衰减；在原域的部分势力声望"冻结"
4. 新域适应：到达新域后，所有NPC初始态度为中立，需要重新建立关系网

## 六、对现有系统的冲击评估

### 6.1 无冲击层（无需改动）

以下系统无需任何修改即可承载全书覆盖：
- **StateUpdate应用器**：纯数据驱动，flag扩展无上限
- **PIPE管线架构**：BUILDING_CONTEXT → PARSING → VALIDATING → RESOLVED的流程与章节数无关
- **Zustand store架构**：切片式设计，新增chapterSlice不影响其他12个切片
- **存档/持久化**：partialize白名单模式，新增flag自动进入持久化
- **UI组件层**：StatusBar/ChoicePanel/GameScreen均通过store驱动，无需重构

### 6.2 轻冲击层（需小幅扩展）

**NPC数据库**：当前120+人，全书覆盖约需200+人。但这不是线性增长——五域NPC天然分域，每域40-60人，通过域过滤机制避免全部注入prompt。dynamicTitles已预留跨章节结构，只需补充新章节条目。工作量约2-3天。

**Canary Assertions**：从13条扩展到25条。每条规则独立，新增不影响已有规则。工作量约1-2天。

**PlayerSlice.flags**：当前扁平KV存储。全书覆盖后flag数量可能达到200+，建议在P2阶段引入flag命名空间（如`chapter.xxx`、`global.xxx`、`relation.xxx`）以利于管理。这不是架构重构，只是命名规范升级。工作量约1天。

### 6.3 中冲击层（需重构）

**Chapter系统**：P1的线性3章设计需在P2重构为条件路由图。核心改动是`checkChapterProgression()`从"确认下一章条件"变为"扫描可达章节列表"。不影响已实现的其他模块（转章UI、目标管理、eventStatus），因为这些模块与路由逻辑解耦。工作量5-7天。

**Context-Builder**：当前注入总量约2500 tokens，需新增章节上下文、全局事件涟漪、跨域NPC同步约1000-1200 tokens。不是架构问题而是token预算控制问题——需要proximity detector来决定"注入什么"而非"注入全部"。

### 6.4 无冲击总结

全书覆盖对现有系统的最大冲击在于"内容量"而非"架构"。现有架构的核心优势——AI驱动、flag驱动、PIPE管线、切片式Store——正是支撑大规模叙事扩展的理想基础。不存在需要推翻重来的架构缺陷。

## 八、风险分析与缓解策略

### 8.1 AI叙事一致性风险（高风险）

**风险描述**：DeepSeek V4虽然有128K上下文窗口，但不代表它能完美记住50轮前的叙事细节。随着章节推进和全局事件累积，AI可能"遗忘"早期事件的关键信息，导致叙事前后矛盾。

**缓解策略**：
- 滚动摘要已实现（`rollingSummary`），每10轮压缩一次历史。在全书覆盖后，建议改为每5轮压缩一次，并新增"跨域摘要"——专门记录全局事件的发生和影响。
- 章节隔离原则：不是让AI同时记住所有章节的约束，而是每轮仅注入"当前章节的约束+当前激活的全局事件涟漪"。
- 金丝雀规则兜底：C14-C22覆盖最常见的跨域一致性违规。

**残留风险**：AI偶尔会出现"引用未发生事件"或"忽略已发生事件"的错误。这是AI驱动模式的固有限制，只能通过后续持续优化提示词和金丝雀规则来降低频率，无法完全消除。

### 8.2 内容膨胀风险（中风险）

**风险描述**：五域×8篇章=40条潜在叙事路径，每条路径都需要独特的内容。如果全部由人工定义scene_constraint，工作量极其庞大且难以维护。

**缓解策略**：
- 场景约束复用：环境描写、NPC群体行为、天气/季节/氛围描述等非核心约束可以跨章节复用（如"山寨大典的紧张氛围"和"部落大会的紧张氛围"可以是同一套情感约束模板）。
- 涟漪叙事模板化：L2/L3涟漪叙事的文本结构可以模板化——"远处的天际传来一阵__声，像是__被惊醒了。__的路上__们纷纷议论，说__那边出了大事。"
- 优先保核心：不是40个名场面都需要同等级别的精细scene_constraint。8个全局事件（L0-L3四层）需要完整定义，其他域内名场面（如"海岛散修篇的第一次出海"）可以用更轻量的约束。

### 8.3 开发进度风险（中风险）

**风险描述**：3-5个月的单人开发周期，在缺乏外部激励和监督的情况下容易陷入拖延。

**缓解策略**：
- P1先行验证：先用3-4天跑通南疆3章，验证架构可行性，建立信心。
- P2分批交付：不要一次性做五个域。先做南疆+北原（1-2周），发布一个"双域版"内部测试，获得反馈后再决定是否继续做东海+西漠+中洲。
- 模块化交付：每个域独立交付，每个全局事件独立交付——随时可以有"当前做到哪算哪"的功能完整版本，而非必须全部做完才有可玩内容。

### 8.4 玩家体验碎片化风险（中风险）

**风险描述**：五域开局+40+名场面可能导致某些域的叙事线内容密度不足——玩家可能在某个域玩到中期发现"无事可做"，因为该域的内容定义不够充分。

**缓解策略**：
- 内容密度检测器：在每域内容完成后运行模拟测试，检测是否存在"事件干旱区间"（连续多轮没有可选名场面或关键事件）。
- 支线事件模板填充：对内容密度较低的域和区间，使用模板化支线事件自动填充空白——不需要每个支线事件都是原著名场面，AI生成的"蛊材采集"、"NPC闲聊"、"随机遭遇"可以填充常规轮次。
- 轻量域可接受：有些域（如东海、西漠）在原著中的篇幅确实较少，玩家选择这些域本身就预期了"更松散的故事线"，不需要强行把每个域的内容密度拉到和南疆一样。

## 九、专业设计建议

### 9.1 必要性确认（基于已确认决策）

三项决策已经确认，以下是对每项决策的实现策略分析：

**五域开局（决策19→B，已确认做）**：这是增加游戏重玩价值最有效的手段。AI驱动游戏的核心价值在于"每次体验不同"，而单域单线几乎无法提供真正的重玩价值。五域开局的差异化叙事范式让每次新游戏真正不同。实现策略：P1预留接口→P2-A制作五域章节组。每域4-5章，按"南疆+北原"先做、其余三域后做的顺序分批推进。关键不在数量而在差异度——玩家需要感受到南疆的"家族丛林法则"与北原的"部落血统至上"是本质不同的体验，而非同一模板换地名。

**全部40+名场面波及（决策20→C，已确认做）**：全部名场面都做波及系统，但采用分层策略。8个全局核心事件做完整L0-L3四层涟漪定义（这是世界联动感的骨架），其余32+域内事件做"域内完整L0 + 跨域一句L2提及"的简化波及（这是血肉但不需要每条血肉都和骨架一样粗）。这样做既满足了"全部波及"的完整性要求，又避免了每个事件都定义四层涟漪的边际成本爆炸。关键在8个核心事件的质量——如果这8个事件的涟漪能让玩家感受到"世界是联动的"，其余事件的"一句提及"就足够锦上添花。

### 9.2 已确认的执行路线

基于三项决策的确认结果，执行路线如下：

**P1（当前，4-5天）**：按原计划做南疆三章 + 多域接口预留 + scene_constraint结构预留。核心目标是验证章节系统的基础架构是否可行，同时确保P1完成时数据结构已为五域和40+名场面打开。

**P1→P2闸门（关键决策点，P1退出时）**：P1七条退出标准全部满足后，审视以下问题再决定P2启动节奏：
- 章节状态机在实际运行中是否稳定（有无状态跳转bug、死循环、flag未正确更新）？
- scene_constraint对AI叙事的约束效果是否可接受（名场面核心不偏，偏差容忍度参数合理）？
- 多域接口预留是否合理（P2填充北原数据时会不会发现结构缺陷）？
- AI上下文注入总量是否在token预算内（单次调用≤6000 input tokens）？
如果以上四项全部通过，**确认启动P2全部四大模块**。如果某项有疑虑，先修P1架构再启动P2。

**P2-1（闸门通过后，15-20天）**：做五域开局章节组。先做南疆扩展+北原（最核心的两条原著线），两条线跑通验证后，并行推进东海+西漠+中洲。五域全部做完后，已有完整的"开局选择→域内发展→跨域触发"叙事闭环。

**P2-2（五域章节完成后，18-25天）**：做全部40+名场面波及系统。先做8个核心事件的完整L0-L3四层涟漪（10天），再做其余32+域内事件的简化波及（10-15天）。全部名场面做完后，世界联动感完整，每个域在任何章节区间都有可感知的世界动态。

**P2-3（涟漪系统完成后，8-12天）**：做章节路由重构和扩展NPC交互。将P2-1和P2-2积累的数据接入条件路由图，实现完整的跨域叙事驱动。

**P3（时间待定，视玩家反馈）**：补齐后四篇章（蛊仙阶段的章节和名场面）和金丝雀扩展，以及原P3内容。这取决于游戏是否发展出了稳定的玩家群体，以及当前章节内容是否需要高境界内容的支撑。

### 9.3 关键设计陷阱警示

**陷阱一：过度约束。** 名场面的scene_constraint如果写得过细（"方源应该说这句话"、"白凝冰应该用这个表情"），会扼杀AI的创造性，让叙事读起来像填词游戏。约束应该定义"事件的核心结果和不变量"，而非"具体的对话和细节"。

**陷阱二：以为玩家会玩完250轮。** 大部分玩家不会体验完整的6-8篇章。游戏设计的重心应该放在前50轮的体验密度——让玩家在前几章就感受到蛊界法则的残酷、蛊虫系统的深度、名场面的震撼感。后四卷的内容深度可以比前四卷低30%。

**陷阱三：为了覆盖而覆盖。** 不是原著有的内容就一定要做进游戏。有些名场面在原著中很精彩是因为文学描写而非游戏交互——单纯把"方源炼制定仙游"作为一个章节目标并不能给玩家带来等价的体验。选择名场面时应该问自己："这个场面变成玩家的游戏交互后，它还好玩吗？"

**陷阱四：flag爆炸。** 全书覆盖后flag可能达到300+，如果不加命名规范和管理工具，维护会成为噩梦。强烈建议在P2启动时引入flag命名空间（`chapter.xxx` / `global.xxx` / `scene.xxx` / `relation.xxx`）和flag依赖图工具。

## 十、总结

从专业游戏开发设计角度评估，蛊真人模拟器全书覆盖在技术上是完全可行的。核心原因在于该游戏的AI驱动架构天然适合大规模叙事扩展——名场面不是手写脚本而是约束注入，全局事件不是固定分支而是涟漪系统，章节不是线性容器而是状态区间。这些设计模式让"40+名场面×5域"的复杂度被分散为"每个AI调用只关注当前相关的一小部分规则"。

三项关键决策已确认，对项目的指导意义如下：

**决策19（五域开局 — 是）** 意味着：P1不能仅关起门来做好南疆三章——必须在P1阶段就为五域预留数据结构和接口槽位（+0.5天），否则P2扩展时会发现架构需要"凿墙"来添加新域，代价远超P1阶段的0.5天预留成本。好在这0.5天的预留工作只是"数据结构定义"，不是"功能实现"——P1依然只跑南疆三章，但chapters.json的结构已经为北原、东海、西漠、中洲敞开了大门。

**决策20（全部40+名场面波及 — 是）** 意味着：P2-B的工作量从8-12天增至18-25天，但采用分层策略控制复杂度——8个全局核心事件做完整L0-L3四层涟漪，其余域内事件做"域内完整+跨域一句提及"的简化波及。这既满足了"全部名场面都有波及"的完整性要求，又避免了每个事件都做四层涟漪的边际成本爆炸。P1阶段也需要为每条名场面定义预留`rippleLayers`和`globalEventId`字段（空对象/null），确保P2填充时数据结构已就绪。

**决策21（先P1验证再扩展 — 是）** 意味着：不在一开始就承诺全部3-5个月的工期。P1完成后的闸门审查是关键的决策点——如果P1验证发现架构有致命缺陷（如scene_constraint对AI的约束效果远低于预期），可以及时止损或调整方向，而非在错误架构上堆积几十天的内容数据。这不是拖延，而是对开发资源的理性管理。

从必要性角度，五域开局是必不可少的（没有它就没有重玩价值），全局事件涟漪系统是强烈推荐的（没有它世界就不像活的），全部名场面波及通过分层策略可控制在合理成本内。后四篇章可以推迟到蛊仙系统完善后再补充。

从执行角度，已确认路线为：P1南疆三章+多域接口预留（4-5天）→ 闸门审查 → P2-1五域开局章节组（15-20天）→ P2-2全部40+名场面波及系统（18-25天）→ P2-3路由重构+NPC扩展（8-12天）→ P3后四卷+原P3内容（25-43天）。总工时约69-105天（3.5-5个月单人全职）。

全书覆盖不是一个"能不能做"的问题——而是"用多长时间、投入多少资源、做到什么粒度"的问题。答案是：技术可行，架构就绪，P1已为五域和全书覆盖预留接口，时间可控（3.5-5个月单人全职），但需要分阶段交付、持续验证、在P1闸门处做理性的go/no-go决策。

## 附录B：关键术语定义

- **章节（Chapter）**：一个状态区间，定义了境界范围、可用NPC池、势力关系、名场面约束和推进条件。不是一段固定叙事。
- **名场面（Canon Scene）**：原著中的标志性事件。在游戏中被建模为一组scene_constraint（AI叙事约束），而非一段预设文本。
- **全局事件（Global Event）**：影响多个域甚至整个蛊界的重大名场面。有L0-L3四层涟漪定义。
- **域（Domain）**：玩家出生和主要活动的蛊界区域。五域各自有独立的叙事范式、势力网络和开局体验。
- **涟漪（Ripple）**：全局事件向非源发域传播影响的方式。L0=完整参与，L1=简化参与，L2=叙事提及，L3=长尾世界变化。
- **条件路由图（Conditional Route Graph）**：章节间的非线性连接结构。每章定义前置条件和替代路径，玩家可通过不同路径到达同一章。
- **scene_constraint**：注入AI的叙事约束文本。规定事件必须包含的元素和禁止出现的偏差，但不规定具体对话和描写。
- **deviationTolerance**：名场面的偏差容忍度。strict=核心事件绝对不能跳过和改变，moderate=允许细节偏差但主体不变，loose=允许AI较大幅度的创造性展开。
