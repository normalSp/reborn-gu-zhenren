## 一、总体策略

### 1.1 执行哲学

当前代码库存在一个结构性矛盾：大量系统被完整实现（类型定义+业务逻辑）但从未接入生产流程——"成品未接线"。在此之上直接堆叠新功能会导致接线更复杂、Bug更隐蔽。

**策略决定**：先修复基础设施断裂（阶段A），确保现有系统完整可用；再在此坚实基础上实施大纲新功能（阶段B）。两条线不能并行，因为阶段B的多项新功能直接依赖阶段A修复的引擎和Store Action。


## 二、关键系统重构方案（阶段A前置决策）

在执行任何修复或新功能之前，需先完成两项类型/架构重构。这两项是基础设施，影响后续所有工作。

### 2.1 refine-engine 扩展方案

**决策：扩展（Extend），不重写**

**理由**：
- 当前引擎 169 行，核心 `refineGu()` 结构正确（Input→校验→成本→成功率→产出）
- 大纲要求的反噬/合炼/拆炼/仙蛊升炼是正交功能，适合作为独立函数共享工具层
- 重写风险：现有调用方（多处UI组件）依赖当前接口签名，重写会连锁影响
- TS 模块天然支持逐步扩展：新导出函数不影响已有调用方

**扩展后的模块结构**（预估 400~500 行）：

```
refine-engine.ts
├── refineGu(input, state) → RefineResult          [扩展现有: 加入蛊材分级/反噬/耗时]
├── combineGu(input, state) → CombineResult        [新增: 合炼]
├── disassembleGu(input, state) → DisassembleResult [新增: 拆炼]
├── ascendImmortalGu(input, state) → AscendResult  [新增: 仙蛊升炼]
│
├── 共享工具层:
│   ├── resolveMaterialGrade(materialName) → MaterialGrade  [新增: 蛊材分级解析]
│   ├── calculateMaterialCost(tier, grade) → number         [新增: 统一成本公式]
│   ├── calculateBacklash(input, state) → BacklashResult    [新增: 反噬计算]
│   ├── calculateRefineTime(tier, hasSpeedGu) → number      [新增: 耗时计算]
│   └── calculateSuccessRate(...) → number                  [重构: 加入全部加成源]
│
├── 类型定义:
│   ├── RefineInput (扩展现有, 加入 materialGrades)
│   ├── CombineInput / DisassembleInput / AscendInput
│   ├── BacklashResult / BacklashSeverity
│   └── MaterialGrade enum
```

**成本公式统一**（以 economy.json 为唯一事实源）：

| 转数 | economy.json 设计 | 当前代码 | 统一后 |
|------|-------------------|----------|--------|
| 1 转 | 5-10 + 普通×2 | tier×5=5 | **8 + 普通蛊材×2** |
| 2 转 | 15-30 + 普通×3 | tier×5=10 | **20 + 普通蛊材×3** |
| 3 转 | 40-80 + 精品×3 | tier×5=15 | **60 + 精品蛊材×3** |
| 4 转 | 100-200 + 精品×4 | tier×5=20 | **150 + 精品蛊材×4** |
| 5 转 | 300-500 + 稀有×5 | tier×5=25 | **400 + 稀有蛊材×5** |
| 6+ 升炼 | 2000 + 稀有×5+仙材×1 | 拦截不执行 | **2000 + 稀有蛊材×5 + 仙材×1** |

取 economy.json 范围中值，确保数值一致性。

### 2.2 KillMove 类型扩展方案

**决策：新增可选字段，完全向后兼容**

**理由**：
- 38条已有杀招数据不能破坏
- 新功能（熟练度/进化/自创）需要额外字段
- TypeScript 可选字段 (`?`) 提供零破坏性扩展

**扩展后的 KillMove 类型**（`src/types/index.ts`）：

```typescript
export interface KillMove {
  // ─── 现有字段（保持不变）───
  id: string;
  name: string;
  path: PathType;
  level: number;
  baseCost: number;
  multiplier: number;
  cooldown: number;
  description: string;

  // ─── 新增字段（全部可选，向后兼容）───
  /** 熟练度等级 (0=入门 ~ 4=宗师)，undefined=无熟练度追踪 */
  proficiency?: KillMoveProficiency;
  /** 累计使用次数，用于熟练度升级判定 */
  usageCount?: number;
  /** 核心蛊虫ID列表（自创杀招必需） */
  coreGu?: string[];
  /** 辅助蛊虫ID列表（自创杀招必需） */
  supportGu?: string[];
  /** 进化阶段 (0=原始, 1~N=进化次数) */
  evolutionStage?: number;
  /** 获取来源 */
  source?: 'innate' | 'taught' | 'discovered' | 'created' | 'event';
  /** 原创者（自创=玩家名，传授=NPC名） */
  creator?: string;
  /** 是否可传授他人（宗师级解锁） */
  canTeach?: boolean;
  /** 额外效果标签（用于炼蛊加速等联动） */
  effectTags?: KillMoveEffectTag[];
}

export type KillMoveProficiency = 0 | 1 | 2 | 3 | 4;
// 0=入门(-10%倍率)  1=熟练(标准)  2=精通(+10%/-1冷却)  3=大师(+20%/特效解锁)  4=宗师(+30%/可传授)

export type KillMoveEffectTag = 'refine_speed' | 'refine_success' | 'refine_double' | 'refine_legendary';
```

**38条已有数据兼容保证**：
- `proficiency` 默认 `undefined` → UI 不显示熟练度条
- `usageCount` 默认 `undefined` → 不追踪
- `coreGu`/`supportGu` 默认 `undefined` → 不参与自创检测
- `source` 默认 `undefined` → 不影响现有获取路径
- 加载时通过 `normalizeKillMove(raw)` 函数填充默认值

**KillMove ↔ DuelMove 桥接**（新增 `src/engine/killmove-bridge.ts`）：

```typescript
// 将已学习的KillMove转换为战斗用的DuelMove
export function convertKillMoveToDuelMove(
  killMove: KillMove,
  playerState: PlayerState
): DuelMove {
  return {
    name: killMove.name,
    damageMultiplier: killMove.multiplier * getProficiencyMultiplier(killMove.proficiency),
    pathBonus: getPathProficiencyBonus(playerState, killMove.path),
    description: killMove.description,
    killerMoveId: killMove.id,
    requiredCoreGu: killMove.coreGu,
  };
}
```

此桥接让战斗系统能正确使用玩家的已学杀招（含熟练度加成和流派加成）。

## 十、关键风险

1. **economy.json 是数值唯一真理源**：阶段A修改 `refine-engine.ts` 和 `shop-engine.ts` 的成本公式后，所有数值必须与 `economy.json` 对齐。任何地方的硬编码成本都是 bug。

2. **38 条杀招数据的兼容性**：B2.0 KillMove 类型扩展后，必须运行自动化加载测试验证所有 38 条数据能正确加载、所有新字段有合理默认值。

3. **advanceTurn 膨胀**：随着 P0+P1 修复越来越多 logic 接入 `advanceTurn()`，需注意性能。每回合执行的计算量可控（非实时渲染），但需保持代码清晰——按子系统分组调用，不要全摊在一层。

4. **refine-engine 复杂度**：从 169 行扩展到 450+ 行后，函数间可能产生隐含耦合。策略是保持纯函数风格（尽量不直接操作 store，通过参数传入所需状态），降低测试难度。

5. **旧存档加载**：旧存档缺少所有新增字段（KillMove.proficiency、materialGrades 等）。加载器需为所有新字段提供安全默认值，确保旧存档不崩溃。

6. **LLM 叙事一致性**：阶段A P4（遭遇系统去 LLM 化）未包含在本计划中。在 B1.2 材料获取路径实现前，LLM 仍然"声称"发放奖励但可能不一致。这是一个已知的降级风险。

## 十一、用户已确认的决策（2026-05-04）

1. **B1.2 仙窍产出公式**：遵循 economy.json，修正代码加入 `areaMu` 面积参数。面积是原著重要设定，不可移除。

2. **B2.3 自创杀招命名**：自动生成名称（流派+效果关键词）+ 允许玩家手动编辑。在自动生成后弹出编辑框，默认填充自动名称，玩家可修改。

4. **NPC 传授杀招的专属保护**：`isExclusive` 的杀招不可通过 NPC 传授（仅事件/探索获取）。自创系统的"改良配方"规则可作为规避手段——若玩家独立自创出相似配方，不受专属保护限制（先创后得原则）。
