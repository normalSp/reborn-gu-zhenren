## 三、Phase 1 详细设计：空窍/福地类型分离（3天）

### 3.1 核心设计原则

在蛊真人原著中，1-5转蛊师拥有**空窍（Mortal Aperture）**，其概念要素为：
- 元海（Primeval Sea）：青铜→赤铁→白银→黄金→紫晶，对应一转→五转
- 窍壁（Aperture Wall）：希望蛊光膜层，初阶厚实→巅峰透明
- 蛊虫容量（Capacity）：按转数递增（一转3→二转5→三转8→四转12→五转15）
- 无面积、无时间流速、无资源节点、无道痕密度、无天劫

6转以上蛊仙拥有**仙窍（Heavenly Land）**，其概念要素为：
- 福地/洞天类型，面积(亩)、时间流速(1:N)
- 资源节点、道痕密度
- 天劫系统（地灾→天劫→浩劫→万劫）
- 地灵（Earth Spirit）

代码中已有正确的 `HeavenlyLand` 接口（位于 `types/index.ts`），应将其作为唯一仙窍模型，废弃 `ImmortalAperture`。

### 3.3 境界晋升联动逻辑

```typescript
// 在 applyStateUpdate 的 realm 更新块中追加：

function advanceApertureWithRealm(
  currentAperture: MortalAperture | HeavenlyLand | null,
  newGrand: number,
  newSub: SubRealm
): MortalAperture | HeavenlyLand | null {
  // case 1: 无空窍 → 不操作
  if (!currentAperture) return null;

  // case 2: 已是仙窍 → 不操作（仙窍数据由 HeavenlyLandEngine 管理）
  if (currentAperture.type !== 'mortal') return currentAperture;

  const ma = currentAperture as MortalAperture;

  // case 3: 六转升仙 → 替换为 HeavenlyLand（由 state-update-applier 处理）
  if (newGrand >= 6) return ma; // 标记等待替换

  // case 4: 同转数内升小境界 → 仅更新窍壁状态
  if (newGrand === ma.rank) {
    const wallState = getWallStateForSubRealm(newSub);
    return {
      ...ma,
      subRank: newSub,
      apertureWall: wallState,
    };
  }

  // case 5: 跨转数晋升 → 更新全部字段
  const newColorName = REALM_TO_COLOR[newGrand] || '深黑';
  const colorConfig = APERTURE_COLOR_MAP[newColorName];
  const capacityMap: Record<number, number> = { 1: 3, 2: 5, 3: 8, 4: 12, 5: 15 };
  return {
    ...ma,
    rank: newGrand,
    subRank: newSub,
    primevalSea: {
      color: colorConfig.primary,
      colorName: newColorName,
      fillPercent: 50,
    },
    apertureWall: getWallStateForSubRealm(newSub),
    capacity: capacityMap[newGrand] || 20,
    carriedGu: Math.min(ma.carriedGu, capacityMap[newGrand] || 20),
  };
}
```

// v7→v8: 将旧版 ImmortalAperture 迁移至 MortalAperture
if (persistedVersion < 8) {
  const oldAperture = (persistedState as any).aperture;
  if (oldAperture && 'area_mu' in oldAperture) {
    // 旧存档中 aperture 是 ImmortalAperture 类型
    // 如果 realm.grand <= 5，转换为 MortalAperture
    const realm = (persistedState as any).playerSlice?.profile?.realm;
    const grand = realm?.grand || 1;
    if (grand <= 5) {
      const realmColor = REALM_TO_COLOR[grand] || '深黑';
      const colorConfig = APERTURE_COLOR_MAP[realmColor];
      persistedState.aperture = {
        type: 'mortal',
        rank: grand,
        subRank: realm?.sub || '初阶',
        primevalSea: {
          color: colorConfig?.primary || '#1a1a24',
          colorName: realmColor,
          fillPercent: 50,
        },
        apertureWall: {
          state: '完整',
          opacity: 1,
          description: '窍壁恢复如初。',
        },
        capacity: { 1: 3, 2: 5, 3: 8, 4: 12, 5: 15 }[grand] || 20,
        carriedGu: 0,
      };
    }
    // grand >= 6 的存档：aperture 数据在六转升仙时已被 HeavenlyLand 覆盖，
    // 只需将 aperture 置为 null（HeavenlyLand 独立存在于 playerSlice 中）
    else {
      persistedState.aperture = null;
    }
  }
}

### 🔴 Critical: 破坏原著核心设定

#### C1. 蛊虫流派（path）字段存在9处功能性类别冒充流派

**文件**: `src/canon/gu-database.json`
**严重度**: 🔴 Critical

原著中蛊虫的"流派"（path）必须是48种天地大道之一：炎道、水道、金道、木道、土道、风道、雷道、冰道、光道、暗道、力道、智道、魂道、奴道、变化道、律道、宙道、宇道、食道、毒道、骨道、血道、运道、梦道、刀道、剑道、人道、炼道、等等。

当前数据库中存在9处用**功能性类别**冒充流派的条目：

| 蛊虫名 | 当前path | 问题 | 应改为 |
|--------|----------|------|--------|
| 铜皮蛊 | `"防御"` | 防御是功能不是流派 | `"金道"` (金属防御) |
| 天元宝莲蛊 | `"辅助"` | 辅助是功能不是流派 | `"木道"` (莲属木) |
| 月影蛊 | `"隐匿"` | 隐匿是功能不是流派 | `"暗道"` (暗系隐匿) |
| 电眼蛊 | `"侦查"` | 侦查是功能不是流派 | `"雷道"` (电眼=雷系侦查) |
| 狼吞蛊 | `"后勤"` | 后勤是功能不是流派 | `"奴道"` 或 `"食道"` (狼属奴道) |
| 横冲直撞蛊 | `"移动"` | 移动是功能不是流派 | `"风道"` 或 `"力道"` (冲锋=力道) |
| 金缕衣蛊 | `"防御"` | 防御是功能不是流派 | `"金道"` (金缕=金系防御) |
| 金罡蛊 | `"防御"` | 防御是功能不是流派 | `"金道"` (金刚罡气=金系防御) |
| 铁皮蛊 | `"防御"` | 防御是功能不是流派 | `"金道"` (铁=金道) |
| 正气蛊 | `"心类"` | 原著无"心类"流派 | `"人道"` (正义=人道范畴) |

**修改影响**: 这9处修改会使流派统计从"混合了不存在的流派"变为"全部属于48大道"，同时影响流派互斥计算和前后端流派展示。

#### C2. 蛊虫喂养系统完全无视食物类型匹配

**文件**: `src/store/slices/guSlice.ts` L135-157 `feedGuHunger()`
**对照**: `world-rules.json` L28-32 明确声明 "每种蛊虫有特定食物，不可乱喂"
**严重度**: 🔴 Critical

当前代码中 `feedGuHunger(guId)` 方法仅做计数器减法操作，**不检查玩家是否拥有该蛊虫对应的食物资源**。gu-database.json 中定义了85种不同的 `feedRequirement.type`（月光、美酒、石粉、生肉、文字、晨露...），但代码完全不区分。

原著核心设定：喂错食物→蛊虫可能反噬蛊师。此规则在代码层面完全未被遵守。

**修复方向**: feedGuHunger 需增加参数 `foodType: string`，与 `gu-database.json` 中的 `feedRequirement.type` 做匹配检查。不匹配则触发反噬（扣减生命/真元）。

#### C3. 流派互斥在3层实现中不一致

**文件**: `world-rules.json` L34-44 vs `combat-config.json` pathMatrix vs `DaoMarkPanel.tsx` PATH_MUTUAL_EXCLUSION
**严重度**: 🔴 Critical

| 互斥对 | world-rules | combat-config | DaoMarkPanel |
|--------|-------------|---------------|--------------|
| 炎道↔冰道 | **互相削弱** | 炎→冰=1.3(克) / 冰→炎=0.7(被克) | **缺失** |
| 雷道↔风道 | 互相削弱 | 存在 | 存在 |
| 其余8对 | 一致 | 一致 | 一致 |

**关键冲突**: world-rules.json 声明"炎道↔冰道: 互相削弱"（双向互克），但 combat-config.json 的 pathMatrix 中 `炎道→冰道=1.3`（炎克冰，单向克制），`冰道→炎道=0.7`（冰被炎克，单向被克）。这在原著中有争议——原著中炎道通常克冰道（火焰融化冰雪），并非"互相削弱"。此外 DaoMarkPanel 的前端展示缺少"炎道↔冰道"这对。

**修复方向**: 统一为 combat-config.json 的矩阵定义（以原著为准，炎克冰而非互克），将 world-rules.json 修正，DaoMarkPanel 补齐缺失对。

#### C4. 道痕（dao_marks）在战斗引擎中完全未被使用

**文件**: `src/types/index.ts` L60-61 (dao_marks定义) vs `src/engine/combat-engine.ts` (无dao_mark引用)
**对照**: `world-rules.json` L16-21 "道痕战力模型: 战力由道痕数量决定而非单纯境界"
**严重度**: 🔴 Critical

原著核心设定："道痕战力模型"——"战力由道痕数量决定而非单纯境界。无极魔尊刚升仙有八转巅峰级道痕(30万)可抗衡普通八转"。当前 combat-engine.ts 的伤害公式为 `(ATK - DEF×0.5) × pathMult × realmMult × moveMult × crit × variance`，**完全没有道痕因子**。

这意味着一个道痕极低的蛊师和一个道痕极高的蛊师，在同境界同装备下战斗力完全相同——**直接违背原著最核心的战力设定**。

**修复方向**: 在伤害公式中加入 `daoMarkMult` 因子，从 player.pathBuild.dao_marks 读取攻击/防御流派的道痕值进行计算。

### 🟠 High: 明显设定冲突或系统缺失

#### H1. 杀招系统数据与战斗引擎完全脱节

**文件**: `src/canon/killer-moves.json` vs `src/engine/combat-engine.ts`
**严重度**: 🟠 High

2. 检查 activationContext（夜间/水源/室内等）

**对照**: `world-rules.json` L208 "空窍容量: 一转3只、二转5只、三转8只、四转12只、五转15只"
**严重度**: 🟠 High

**修复方向**: 战斗引擎中，当玩家使用蛊虫技能时，检查真元是否足够、环境是否匹配（如水域使用火鸦蛊），不满足则触发 usageFailure 效果。

#### H6. 蛊虫死后销毁/战利品规则未强制执行

**严重度**: 🟠 High

#### H7. 气运数值默认可见，违反原著设定

**严重度**: 🟠 High

world-rules.json 明确声明："蛊师/蛊仙默认看不见自己和他人气运"、"若玩家无运道相关蛊虫，气运应显示'???'而非具体数值"。当前 initialState 中气运有具体数值，且前端无隐藏逻辑。
