| 级别 | 数量 | 含义 |
|------|------|------|

### 2.5 幽灵计数器导致起源/成就永久锁定（7 项）

以下计数器在 store 类型定义和成就/起源条件中均被引用，但在代码中**永不递增**：

| 计数器字段 | 锁定目标 | 影响范围 |

| `maxRealmReached` | 天才起源 / "登峰造极" 成就 | 1 个起源 + 1 个成就永久锁定 |
| `totalCurrencyEarned` | 富商起源 / "富甲一方"、"日进斗金" 成就 | 1 个起源 + 2 个成就永久锁定 |
| `renZuLegendsHeard` | 学者起源 / "人祖传说" 成就 | 1 个起源 + 1 个成就永久锁定 |
| `totalBattlesFought` | "百战不殆" 成就 | 1 个成就永久锁定 |
| `refinedGuCount` | "初窥炼蛊"、"炼蛊大师" 成就 | 已修复 (v1.7 M1.3) |

**修复方向**：每个计数器需在对应的系统事件中递增：
- `domainsVisited` — 角色首次移动到新区域时递增
- `maxRealmReached` — 境界突破时比较并更新
- `totalCurrencyEarned` — 所有元石获取路径累加
- `renZuLegendsHeard` — 触发人祖传说事件时递增
- `totalBattlesFought` — 每场战斗结束时递增

## 六、额外发现

### 6.1 遭遇 rewards 完全依赖 LLM

`consumeEncounter()` 函数零调用。遭遇奖励（物品、元石、蛊虫）完全由 LLM 在叙事文本中"声称"给予，`state-update-applier.ts` 解析 LLM 的 JSON 输出后执行。这意味着：
- 奖励不一致（取决于 LLM 的即兴发挥）
- 无程序化保底（LLM 可能忘了给奖励）
- 无法做数值平衡

### 6.2 商店定价硬编码

`shop-engine.ts` 的定价逻辑硬编码在 TS 源码中，未读取 `economy.json` 或 `shop-items.json`。数据文件中精心设计的定价/稀有度数据完全白费。

### 6.3 测试覆盖盲区

vitest 的 158 个测试集中在数据处理函数（parser/validator/serializer），但上述所有断裂的 engine 和 store action 调用链均无集成测试覆盖。

## 十、风险与注意事项

1. **测试回退风险**: 当前 158 测试通过，大量修改 store action 调用链可能导致测试失败。每批次修改后需 re-run vitest。
2. **存档兼容**: P0.2 引入新计数器字段，需确保旧存档 migrate 路径兼容（新字段 default 为 0）。
3. **性能风险**: P1.1 拍卖系统涉及定期事件轮询，需确认不会在每回合触发大量 NPC 竞价计算。
4. **LLM 联动风险**: P4 改造遭遇奖励系统时，需确保 LLM 叙事中已提及的奖励描述与程序化奖励一致（不一致会造成玩家困惑）。
5. **优先级可调整**: 以上优先级基于当前审计发现的严重程度排列。如设计目标有变，可重新排序。
