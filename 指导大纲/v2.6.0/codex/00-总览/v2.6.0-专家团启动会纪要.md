## 专家团意见

### Project Lead

v2.6 不应该直接实现后端，也不应该继续只写抽象规划。最合理的下一步是把 v2.5 的 visibility / redaction / promotion chain 转成最小可检查工程门：schema、fixture、dry-run checker、报告和 rc 复核。

### Systems Architect

建议 v2.6 采用“工程预备，不上服务”的路线：

- a1 冻结 job / replay / archive boundary model。
- a2 冻结 private canon classifier interface。
- b1 在用户批准后实现最小自有 schema checker。
- b2 增加 negative synthetic fixtures 和 manual review queue 草案。
- b3 给 v2.7-v2.9 Agent Lab 扩展和 v3.0 runtime agent 准入提供 checklist。

这套设计服务 WorldCore，不替代 WorldCore。

### AI Pipeline Architect

DeepSeek 仍保持 `deepseek-v4-flash`，但 v2.6 默认不调用 live DeepSeek，不改 prompt/context/model/authority。

v2.6 可以设计 prompt hash、evaluator version、retry summary、cost/cache/replay metadata 字段，但不能保存 prompt body、DeepSeek raw hidden-adjacent body 或 DeepSeek visible lore/RAG。

### Canon / Knowledge Curator

v2.6 仍不需要 MiroFish export。所有 fixture 必须 synthetic-only，所有 source pointer 只能是 class 或 fake id，不得引入真实原著事实、命名 NPC、hidden-adjacent、方源公开证据或正式 lore 结论。

如果后续任一阶段需要真实 lore body、知识库正文、runtime canon、正式结论或 MiroFish topic-slice，必须升级为 blocking 并停机。

### Lore Guard

private canon 在 v2.6 仍是“可见性与审计边界”，不是“把全书喂给模型”的借口。

自学习、Hermes memory、LlamaIndex retrieval、Letta memory 等都只能作为候选机制参考。任何 candidate memory / candidate skill patch 都不得自裁决，不得写 canon/runtime/save/prompt/knowledge-index body。

### QA / Eval Guardian

v2.6 的最小 checker 若获批，必须验证：

- redacted envelope 不含 hidden/private body、original text、prompt body。
- source-pointer-only 不得直达 DeepSeek visible、runtime canon 或 save。
- AgentProposal 不得写正式地点、阵营、身份、奖励、NPC 生死。
- report boundary assertions 必须显式证明无 runtime/save/DeepSeek/backend/MiroFish/external framework 越权。

Player Advocate 和 live narrative probe 默认豁免，因为没有玩家可见 runtime/UI，也没有 DeepSeek 叙事变更。

### Dev Environment / Git Steward

v2.6 继续沿用前置批量审批制度。启动会给全版本授权包，用户批准后 `/goal` 可连续完成 a1/a2/b1/b2/b3/process-1/rc。

仍然显式 stage，不使用 `git add -A`，推送后查 GitHub Actions，不自动部署 EdgeOne。
