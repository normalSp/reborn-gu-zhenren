#### 1.2.2 势力创建条件（双轨制）

势力创建分两个档次，数值上做出明显区分——5转山寨和蛊仙仙门的发展天花板不可同日而语：

| 条件 | 5转蛊师级（山寨/帮派） | 6转+蛊仙级（仙门/洞天势力） |
|------|----------------------|---------------------------|
| **最低境界** | 5转蛊师巅峰 | 6转蛊仙 |
| **消耗** | **8000元石**（纯元石，v0.7.0审查修正：取消仙元石要求——5转玩家无仙元石获取途径） | 3000仙元石 |
| **领地要求** | 当前域声望≥500（在该域有一定名声） | 拥有至少一片福地/洞天 |
| **初始成员槽** | 2个 | 3个 |
| **势力等级上限** | 5级 | 10级 |
| **最大成员数** | 6人（等级5时） | 12人（等级10时） |
| **可解锁功能** | 基础招募、成员分配、势力任务 | 全部功能（含势力外交、高级事件） |
| **升级消耗系数** | ×1.2（仙元石+声望） | ×1.0（基准） |
| **AI叙事定位** | 地方豪强 | 一方霸主 |

设计理由：
- 5转蛊师创建山寨/帮派符合原著世界观——方源5转时已在南疆建立血翼魔教雏形，黑楼兰5转时掌控北原黑家部分势力。5转是蛊师巅峰，具备"拉一支队伍"的资格，但没有福地洞天做根基，只能是小规模的地面势力
- 蛊仙才能创建真正的"仙门"，拥有福地洞天作为大本营，这是原著设定的硬分界线
- 5转山寨的6人上限让玩家在低转阶段也能体验势力管理，但不会让山寨发展到"与蛊仙势力平起平坐"的程度——要突破6人上限必须升6转

势力创建后获得势力旗帜、初始招募槽。

#### 1.3.2 NPC性格系统

| 性格 | 战斗行为 | 忠诚变化 | 特殊效果 |
|------|---------|---------|---------|
| **loyal(忠诚)** | 优先保护玩家 | 不易下降 | 濒死时替玩家挡刀 |
| **cunning(狡诈)** | 优先偷袭 | 势力弱时快速下降 | 侦察/情报成功率+20% |
| **reckless(莽撞)** | 优先强攻 | 胜利+忠诚, 失败-忠诚 | 暴击率+10% |

**性格来源**：
- 原著NPC：从 `npcs.json` 中读取预设性格（如太白云生=无私、黑楼兰=狡诈）
- 随机招募NPC：性格随机，但受势力类型影响（魔道→更多狡诈/莽撞, 正派→更多忠诚/谨慎）
- 性格可在长期相处中微调（每50回合有1次性格转变事件）

#### 1.3.3 NPC死亡处理

| 战斗类型 | HP归零结果 | 恢复方式 |
|---------|-----------|---------|
| **训练**（道场对决） | 重伤（冷却恢复[游戏回合冷却]，10回合不可出战） | 自动恢复 |
| **实战**（剧情/势力战/白天黑天） | **阵亡**（永久不可用） | 不可恢复 |

> 设计理由：实战阵亡增加战术深度和情感重量，训练重伤降低挫败感。

#### 1.3.3 忠诚度系统

```
忠诚度 0-100

影响：
  100-80：全力战斗，不使用逃跑，有概率触发"舍身保护"
  79-60：正常战斗，偶尔防御
  59-40：战斗中概率防御/逃跑（30%）
  39-20：战斗中高概率逃跑（60%），可能拒绝执行危险任务
  19-0： 背叛——离开势力，可能带走装备/蛊虫

忠诚变化事件（仅由势力层面行为触发——与个人层面的信任系统职责分离）：
  +忠诚：每10回合+1（基础）、战斗胜利（势力战）+2、救治疗伤+3（势力管理层面）、授予职位+5~10
  -忠诚：每10回合未出战-2、战斗失败（势力战）-3、势力声望下降-5、长期未分配资源-1/回合
  ⚠ 注意：给予蛊虫/资源、分战利品等个人层面奖励统一走信任度(adventureTrust)系统，不再触发忠诚变化
```

#### 1.3.4 信任-利益双轴组队意愿模型

NPC是否愿意跟你出门冒险，不能只用"忠诚度"单一数值——忠诚是对势力的归属感，但出门冒险是"把自己的命交到你手上"。引入信任-利益双轴模型解决这个问题：

```
                         利益驱动 →
                 低                        高
         ┌─────────────────┬─────────────────┐
    高   │ 主动要求入队     │ 愿意入队          │
         │ 不索要报酬       │ 但可能拒绝危险任务  │
    ↑    │ 例：太白云生(后期)│ 例：商心慈         │
    信   ├─────────────────┼─────────────────┤
    任   │ 拒绝组队         │ 可以入队          │
    │    │ ("我不想跟你去") │ 但每次索要报酬     │
    低   │ 例：敌对NPC      │ 例：黑楼兰(前期)   │
         └─────────────────┴─────────────────┘
```

| 信任度 | 利益驱动 | AI叙事标签 | NPC在对话中的典型表现 |
|--------|---------|-----------|-------------------|
| 高 | 高 | `willing_eager` | "正好我也想去那边采药，一起走吧" |
| 高 | 低 | `willing_cautious` | "这次恐怕...但我信你，走" |
| 低 | 高 | `mercenary` | "可以，但我要这次探索所得的三成" |
| 低 | 低 | `unwilling` | "以后再说吧"（对话选项不出现邀请） |

**信任度**（代码字段`adventureTrust`，区别于`CharacterRelation.trust`）不同于忠诚度——信任是"我信你这个人的判断力"，忠诚是"我对这个势力的归属感"：
- 信任度来源：战斗中救过NPC(+15)、NPC被你治疗(+10)、分战利品给NPC(+物品价值/100)、长期组队不受伤亡(+2/次)
- 信任度消耗：战斗中抛弃NPC(-20)、带NPC进入远超其境界的战斗(-10)
- 信任度独立于忠诚度存储，但两者共同影响NPC行为。注意：`adventureTrust`(0-100)完全不同于`CharacterRelation.trust`(NPC间社交信任)——后者是NPC之间的好感网络，前者是NPC对玩家带队能力的信赖

**利益驱动**来自NPC当前的资源需求：
- NPC缺少某类蛊虫时，对该类蛊虫作为报酬的接受度+30
- NPC境界即将突破时，对修炼资源的接受度+50
- NPC个人任务未完成时，如果玩家的目的地与任务相关，利益驱动+40

> **大模型驱动架构适配**：信任度字段命名为`adventureTrust`（区别于`CharacterRelation.trust`——后者是NPC之间的社交关系信任，前者是NPC对玩家的组队信任）。`adventureTrust`和`interestDrive`作为数值状态存在于Zustand store中，context-builder将这两个值注入AI叙事上下文。AI在生成NPC对话时读取`willingnessTag`决定语气和条件。不需要AI"判断"NPC意愿——意愿由数值公式预先计算，AI只负责用自然语言表达这个结果。

#### 1.3.5 组队条件四层递进

```
玩家想带NPC出门
  │
  ├─ 第一层 基本条件（硬门槛，任一不满足→不可组队）
  │    ├─ 玩家境界 ≥ NPC境界-1（差2转以上NPC认为跟你混没前途）
  │    │   例外：信任度≥80时无视境界差
  │    ├─ NPC当前不处于"重伤/闭关/执行势力任务"状态
  │    └─ 队伍当前人数 < 4
  │
  ├─ 第二层 关系条件（软门槛，决定NPC是否愿意，由信任-利益双轴计算）
  │    ├─ willing_eager → 必定同意，可能主动要求入队
  │    ├─ willing_cautious → 同意但对话中表达顾虑，拒绝极高风险任务
  │    ├─ mercenary → 弹出报酬条件（索要资源/蛊虫/仙元石），玩家接受后入队
  │    └─ unwilling → 不出现邀请选项
  │
  ├─ 第三层 情境条件（什么时候可以组队）
  │    ├─ 锚点窗口：在当前城镇/山寨"酒馆/议事厅"场景中，菜单面板可用
  │    ├─ 剧情触发：关键剧情节点自动触发组队/离队（AI叙事引擎判定）
  │    └─ 战斗前准备：进入副本/据点/白天黑天前，队伍编成面板自动弹出
  │
  └─ 第四层 阵营约束
       ├─ 同势力NPC：信任门槛-20%（自己人好说话）
       ├─ 中立NPC：正常门槛
       └─ 敌对势力NPC：信任门槛+40%（需先策反或利益极大才可能）
```

> **大模型驱动架构适配**：四层递进中的"对话语气生成"和"报酬条件协商"由AI叙事引擎完成。但**是否出现邀请选项、报酬条件的具体数值**由游戏逻辑预先计算。AI只生成对话包装，不做数值决策。

### 1.4 小队编成与战斗

#### 1.4.1 势力与队伍的定位区分

设计意图上，势力是"资产管理系统"（战略层），队伍是"战术搭配系统"（战术层）。二者的数据关系为：**势力成员池 ⊇ 队伍成员**。NPC在势力中时"经营势力"（产出资源、触发事件），编入队伍后"跟你冒险"（参与战斗、积累信任）。

从势力调NPC到队伍有摩擦成本——这不是"打开菜单随便拖拽"：
- 消耗1AP（代表"去找他商量"的时间）
- 如果NPC刚从战斗中退出（重伤/疲劳），该NPC在势力中对应岗位的产出暂停3回合
- NPC在队伍中时，其在势力中的岗位产出暂停（不能同时两处工作）
- **战术姿态**（替代阵型——原著战斗为"杀招流"非"阵型流"，参见原著考证）:

| 战术姿态 | 效果 | 原著依据 |
|---------|------|---------|
| **合击** (Combined Strike) | 两人攻击同一目标+15%伤害 | 铁家四老合击、三兄弟合击 |
| **牵制** (Pinning) | 一人牵制敌方最强单位，其闪避-20% | 全书最高频战术(128处) |

##### 1.4.4.1 信任养成（事件驱动）

信任度的变化不由"挂机时长"决定，而是由具体事件驱动——这与大模型叙事引擎天然契合。

**战斗中**：
| 事件 | 信任变化 | 触发条件 |
|------|---------|---------|
| 玩家帮NPC挡致命一击 | +15 | 战斗中玩家使用"保护"指令，NPC本回合HP本应归零 |
| NPC被你治疗 | +10 | 你（或你的蛊虫）对NPC使用治疗 |
| 战斗胜利 | +3/场 | 全员存活 |
| 你在战斗中逃跑而NPC还在战斗 | -20 | 玩家使用逃跑指令 |
| NPC在战斗中重伤而你未施救 | -30 | NPC HP<20%且该场战斗中玩家从未对其使用治疗 |

**非战斗中**：
| 事件 | 信任变化 | 触发条件 |
|------|---------|---------|
| 分给NPC战利品 | +物品价值/100 | 战斗后分配战利品界面 |
| NPC个人任务完成 | +20 | 1.4.4.4个人任务模板完成 |
| 分配NPC势力职位 | +5~10 | 任命NPC为势力干事后 |
| 长期未联系 | -1/20回合 | 存在于势力但从未被编入队伍 |

##### 1.4.4.2 战术配合度（合击系统）

两个NPC共同战斗次数越多，配合度越高，解锁更高级战术效果。配合度以`Map<"charA_charB", number>`存储（SortedKey确保A_B与B_A一致），最大12人共66个pair——远低于性能瓶颈。

| 配合等级 | 共同战斗次数 | 效果 |
|---------|-------------|------|
| Lv1 初识 | 3次 | 无额外效果 |
| Lv2 默契 | 10次 | 合击时伤害+15% |
| Lv3 心有灵犀 | 25次 | 解锁合击杀招（预设组合） |
| Lv4 生死与共 | 40次 | 一方濒死(HP<20%)时另一方自动保护/治疗 |

**预设合击组合**（仅原著有依据的组合）：
- 方源 + 太白云生 → 合击"春秋如故"：方源春秋蝉×太白江山如故——一次战斗中复活一名阵亡队友（限1次/战斗）
- 白凝冰 + 黑楼兰 → 合击"冰力无双"：冰系控制×力道爆发——对单体300%伤害并冻结1回合

> 配合度数据仅在Zustand中存储，不进入AI叙事上下文（节省token）。合击触发由战斗引擎纯数值判断，AI仅生成合击名称的美化描述。

##### 1.4.4.3 队伍共享资源

- **共享经验池**：非出战NPC每回合自动获得出战NPC平均经验值的30%
- **队伍羁绊等级**：羁绊经验=全员共同战斗次数总和
  - Lv5 "生死相依"：全员HP<30%时，伤害+25%
  - Lv10 "心有灵犀"：战术姿态切换不再消耗回合
- **队伍蛊虫库**（v0.8.0+）：放入库中的蛊虫，所有成员可在非战斗间隙临时装备（使用后自动归还）

##### 1.4.4.4 NPC个人成长（任务模板+AI叙事填充）

每个重要NPC有一个"任务模板"——定义核心欲望和完成条件，AI叙事引擎在上下文匹配时动态生成具体任务。模板是Zod-validated的JSON配置，AI负责叙事填充。

```typescript
interface NPCPersonalQuestTemplate {
  npcId: string;
  desire: string;           // AI围绕此主题创作任务
  unlockCondition: string;  // adventureTrust阈值或剧情flag
  completionHint: string;   // 完成方向（不指定步骤，AI动态铺路）
  domainHint?: string[];    // 🆕 该任务与哪些域相关。AI在玩家进入此域时优先注入任务线索
  stallTimeout?: number;    // 🆕 解锁后N回合未推进，触发一次"偶遇"事件来推进（默认30回合）
  reward: { trustBoost: number; passiveName?: string; passiveDesc?: string };
}
```

**v0.7.0首批NPC个人任务模板**：

| NPC | 核心欲望 | 解锁条件 | AI叙事方向 | 域提示 | 停滞时限 | 完成奖励 |
|-----|---------|---------|-----------|--------|---------|---------|
| **太白云生** | 完善"人如故"仙蛊的完美形态 | adventureTrust>70 | 寻找稀有仙材/宙道遗迹探索 | 北原/西漠 | 30回合 | 人如故复活不消耗太白自身寿命 |

> **大模型驱动架构适配**：任务模板的`desire`和`completionHint`注入AI叙事上下文，AI在正常叙事中自然嵌入个人任务线索。例如玩家探索冰道遗迹时，AI可能同时推进白凝冰的个人任务线——不需要专门写"白凝冰任务关卡"。这是大模型驱动游戏相对于传统脚本游戏的最大优势：任务与主线叙事可以无缝融合。

### 5.2 高压迫槽位机制

**设计依据**：原著中十绝体空窍被十成真元"充斥得全满"，窍壁承受巨大压迫力，随时有崩塌风险。但原著从未说蛊虫无法存放——白凝冰正常使用冰刃蛊、霜妖蛊等战斗。因此采用"高压迫槽位"而非"无空腔"方案。

- 十绝体空窍 capacity = **3**（普通蛊师 capacity = 资质值，甲等9→9槽位，十绝体仅3槽位）
- 每次将蛊虫**放入/取出**空窍时，扣当前HP的5%（窍壁摩擦伤害）
- **HP不足5%时阻止操作**并提示"窍壁压力过大，当前状态无法承受"（v0.7.0审查决策：防止玩家因换蛊操作意外死亡）
- 3个槽位足够携带核心战斗蛊虫，但玩家必须做出取舍，不能"全都要"

**第二空窍蛊**作为解决方案：宝黄天稀有拍卖品，定价8000-12000仙元石，扩容+3槽位。仅在蛊仙阶段可用。

### 5.3 类型选择与体质觉醒叙事

**选择方式**：资质roll到10时，系统随机分配一种十绝体类型（不可手动挑选）。提供1次免费重掷（重掷后随机新类型，不可回退）。重掷次数可在角色创建界面的十绝体选择UI中看到。

**体质觉醒叙事**：roll到十绝体类型后，AI生成一段该类型的简短觉醒叙事（50-80字），增强沉浸感。示例：

- 北冥冰魄体："一股彻骨寒意自丹田深处涌出。你的指尖触及之处，水汽凝结成霜，呼出的气息化作白雾。空窍之中，十成真元如极北冰海般沉静而危险。"
- 炎煌雷泽体："体内仿佛有什么东西炸开了。炽热的真元裹挟着电弧在你的经脉中奔涌，皮肤表面隐隐浮现赤金色的裂纹。你甚至能闻到空气中臭氧的焦味。"
- 宇宙大衍体："空窍中的真元不再静止，而是开始自发地流转、变形、分裂、重组。万物在你眼中都变得模糊不定——你感觉自己可以成为任何东西。"

### 5.4 道痕禁制完整实现

在引擎层面实现亲和=0禁止使用、亲和为负使用反噬的逻辑。

**UI标记规则**：
- 亲和=0的流派蛊虫：在炼蛊/宝黄天/蛊虫面板中**灰色**显示，标注"流派禁制·不可使用"
- 亲和为负的流派蛊虫：**红色**显示，标注"流派相克·使用反噬：-X HP/回合"

**引擎实现**（`src/engine/combat-formulas.ts` 追加 `checkAffinityBlock`）：
```typescript
function checkAffinityBlock(guAffinity: number, playerAffinity: number): { blocked: boolean; selfDamage: number } {
  if (playerAffinity === 0) return { blocked: true, selfDamage: 0 };
  if (playerAffinity < 0) return { blocked: false, selfDamage: Math.abs(playerAffinity) * 2 };
  return { blocked: false, selfDamage: 0 };
}
```

## 七、BGM音效配音系统

> 设计原则：四通道独立控制、BGM三层自适应、SFX从合成音切换到真实素材、木成配音离线集成、名场面优先2-3个核心场景。遵循状态驱动设计——音频状态完全由soundSlice托管，UI仅读取状态渲染；遵循数据视图分离——audio-manager作为纯数据层，不感知React生命周期。

### 7.1 当前音频系统差距分析

v0.6.0现状（`src/store/slices/soundSlice.ts` + `src/utils/audio.ts`）：

| 维度 | 现状 | 差距 |
|------|------|------|
| 通道数 | 2通道（master→bgmGain + master→sfxGain） | 缺voice独立通道，配音与SFX共用通道导致音量不可独立调节 |
| Ducking | 无 | 配音播放时BGM盖过语音，需voice播放时bgmGain自动-6dB |
| SFX音源 | 全量OscillatorNode合成音（纯波形） | 13个MP3文件（combat×5/env×3/event×3/ui×2）已存在于public/audio/sfx/但代码从未调用 |
| BGM覆盖 | 仅东海/西漠有938KB有效音频；南疆/北原/中洲为0字节空文件 | 五域BGM缺口60%（3/5域无声） |
| SPECIAL_BGM | 常量已定义但无实际音频文件 | 逆流河/定仙游/定场诗等名场面无音频 |
| 配音 | 零 | 木成授权111文件未集成 |
| 音量控制 | sfxVolume/bgmVolume可调 | 无voice独立音量、无全局ducking开关 |
| 域切换 | 硬切（立即替换src） | 缺少crossFade平滑过渡 |

### 7.2 四通道音频架构设计

```

                    │         AudioContext (全局单例)        │
                    │         sampleRate: 44100Hz           │
                    └──────────────┬──────────────────────┘
                                   │
                    ┌──────────────▼──────────────────────┐

                           │       │       │
              ┌────────────▼─┐ ┌──▼────┐ ┌▼──────────┐ ┌──────────────┐

         │  ┌─────────────────┐ │   │   │ voice播放→    │       │
         │  │ L1 域氛围层      │ │   │   │ bgmGain       │       │

         │  │    (intro片段)   │ │   │   │ voice停止→    │       │
         │  │ L3 剧情情绪层    │ │   │   │ bgmGain       │       │
         │  │    (瞬时切入)    │ │   │   │ .linearRamp   │       │
         │  └─────────────────┘ │   │   │ (0dB,0.5s)    │       │
         └──────────────────────┘   │   └───────────────┘       │
                                    │                           │
                              ┌─────▼──────┐            ┌───────▼──────┐
                              │ MP3文件     │            │ OscillatorNode│
                              │ 播放器      │            │ (按钮/UI音效) │
                              │ (combat/env │            │ 保留合成音    │
                              │  /event)    │            │ 轻量无延迟    │
                              └────────────┘            └──────────────┘
```

**通道职责**：
- **bgmChannel**：承载BGM三层混音器输出，受ducking控制。域切换时crossFade 2秒，角色曲/情绪层直接mix。
- **sfxChannel**：承载所有游戏音效（战斗/环境/事件），从OscillatorNode切换到`public/audio/sfx/`下的MP3文件。保留UI按钮音效为合成音（延迟敏感）。
- **voiceChannel**：承载木成111配音文件。独立音量控制。播放时触发ducking压低bgmChannel -6dB。
- **uiChannel**：保留OscillatorNode合成音，用于按钮/UI交互（需零延迟响应）。

**Ducking实现**（`audio-manager.ts`核心逻辑）：
```typescript
// voice播放时压低BGM
voiceSource.onstart = () => {
  bgmGainNode.gain.linearRampToValueAtTime(
    bgmVolume * 0.5,  // -6dB = 振幅×0.5
    audioCtx.currentTime + 0.2
  );
};
// voice停止后恢复BGM
voiceSource.onended = () => {
  bgmGainNode.gain.linearRampToValueAtTime(
    bgmVolume,
    audioCtx.currentTime + 0.5
  );
};
```

### 7.3 BGM素材三层方案

BGM采用AAA游戏自适应音乐系统的**Horizontal Re-Sequencing**模式——三层独立音轨按情境叠加/切换，而非单一长音频文件。

#### 7.3.1 Layer 1：域氛围层（Domain Ambient）

|----|--------|-----------|---------|---------|
| 东海 | `east-sea.mp3` | ✅ 已有（938KB） | 保留现有 | >60s 可循环 |
| 西漠 | `west-desert.mp3` | ✅ 已有（938KB） | 保留现有 | >60s 可循环 |

| 北原 | `north-plain.mp3` | 🆕 v0.7.0补齐 | 社区翻唱版/AI纯音乐变奏 优先 | >60s 可循环 |
| 中洲 | `central-continent.mp3` | 🆕 v0.7.0补齐 | 社区翻唱版/AI纯音乐变奏 优先 | >60s 可循环 |

域切换crossFade逻辑：当前域BGM线性fadeOut 2秒，目标域BGM线性fadeIn 2秒，中间交叉区同时播放避免静音间隙。

#### 7.3.2 Layer 2：角色主题层（Character Theme）

角色主题层采用\"事件触发→淡入→高光播放→淡出\"模式，**不做持续混音**。角色主题曲的触发**不受角色与玩家的阵营关系限制**——无论该角色是队友、NPC中立角色还是敌方强敌，只要其出现在叙事的\"高光时刻\"（关键剧情节点、情感爆发、绝境逆转、名场面等），均可触发其角色曲。例如：龙公作为友方守护人族时触发春泥，作为天庭敌人对玩家出手时同样可触发春泥——敌方强敌的出场往往更有压迫感，角色曲的烘托效果更佳。每次触发播放12-20秒后淡出，恢复纯域氛围层。触发算法设计详见7.3.2.5。

##### 7.3.2.1 角色曲选曲策略：现有流行歌曲匹配角色

角色主题层的选曲策略与B站/抖音蛊真人社区文化一致——**不创作新曲，而是为每个角色匹配一首气质契合的已有知名歌曲**。这种\"角色×歌曲\"匹配模式在社区有深厚基础：方源×年轮、龙公×春泥是最成功的两个案例，玩家一听到前奏就产生角色条件反射。

选曲标准：
- **旋律气质匹配**：歌曲的节奏/调性/情绪氛围与该角色的核心气质一致（如方源的沧桑轮回→年轮的叙事感）
- **歌词可映射**：歌词主题与该角色的命运或性格有天然共鸣（如龙公的\"化作春泥更护花\"→守护人族的龙人之祖）
- **传唱度高**：必须是B站/抖音/华语乐坛广泛传播的歌曲，避免太小众导致玩家无共鸣
- **版权风险可控**：非商业项目引用已有歌曲仅用于角色BGM，属于合理使用范畴；另备AI纯音乐变奏版作为纯乐器版替代方案

##### 7.3.2.3 v0.7.0实装方案

v0.7.0角色主题层仅实装**方源×年轮**一个角色曲组合。选择年轮的三点理由：
- **传播度最高**：B站/抖音\"仙尊进行曲\"标签下年轮MAD播放量远超其他曲目，玩家一听到前奏就产生\"方源\"的条件反射。2023年起社区自发将年轮配蛊真人后迅速扩散为全网现象级二创
- **情绪匹配精准**：年轮的\"数着一圈圈年轮/我认真将心事都封存\"天然对应方源五百年重生轮回的核心叙事。副歌\"世间最毒的仇恨是有缘却无分\"暗合方源与天意的宿命对抗
- **版权风险低**：非商业项目引用已有歌曲作为角色BGM属于合理使用范畴；同时准备社区翻唱版和AI纯音乐变奏版作为纯乐器替代方案

v0.7.0角色主题层设计要点：
- 年轮不持续循环——仅在方源触发高光时刻时播放（播放intro→verse→chorus约15秒后淡出），触发算法见7.3.2.5
- 播放时域氛围层自动降低至30%（相当于ducking），年轮占70%，通过各自的GainNode独立控制
- 播放完毕后1.5秒crossFade回纯域氛围层

##### 7.3.2.5 角色主题曲触发算法设计

主题曲播放由\"高光时刻检测→优先级仲裁→播放控制\"三阶段算法驱动。整体设计遵循状态驱动原则——触发判断由`context-builder.ts`和游戏状态切片共同完成，`audio-manager.ts`仅负责播放执行。

**A. 高光时刻分类与权重**

每种高光时刻有明确的触发条件和权重值，权重越高越优先播放：

| 权重 | 分类 | 触发条件 | 示例 | 播放策略 |
|------|------|---------|------|---------|
| 100 | **绝境逆转** | 角色HP<20%时击败强敌 / 濒死时触发逆转杀招 | 方源逆流河27步逆转、龙公三气归来反杀、黑楼兰残血解封秒杀 | 完整播放intro→verse→chorus(15-20s)，BGM音量从0骤起至70%（冲击感） |
| 95 | **觉醒/突破** | 十绝体觉醒 / 境界突破 / 隐藏能力解锁 | 黑楼兰解开大力真武体封印、白凝冰阴阳转身蛊转性后觉醒、方源炼道成尊 | 播放intro→verse(12-15s)，BGM从域氛围层crossFade 1.5秒过渡 |
| 90 | **杀招释放** | 角色使用标志性杀招且造成高额效果 | 方源春秋蝉自爆、太白云生人如故复活队友、龙公龙御上宾 | 播放verse→chorus(10-12s)，BGM 0.8秒快速淡入 |
| 85 | **关键抉择** | 玩家/角色做出改变叙事走向的重大选择 | 商心慈十里相送、红莲打碎宿命蛊、方源选择炼道 | 播放intro+短verse(8-10s)，弱起→渐强 |
| 80 | **情感爆发** | 角色忠诚度剧变(±30) / 重要关系揭示 / 牺牲 | 商心慈重逢、太白牺牲暗示、龙公最终守护 | 播放chorus段落(8-10s)，0.5秒快速淡入 |

**B. 高光时刻检测器（HighlightDetector）**

集成在`src/engine/context-builder.ts`的叙事上下文中，每回合结束时执行：

```typescript
interface HighlightDetector {
  // 每回合结束后扫描所有活跃角色的状态变化
  scanTick(state: GameState, prevState: GameState, context: NarrativeContext): HighlightEvent[];
}

interface HighlightEvent {
  characterId: string;
  category: HighlightCategory;  // 上述8种分类
  weight: number;
  context: {
    actionTag: string;           // 'breakthrough' | 'killerMove' | 'decision' | 'sacrifice' | ...
    narrativeKey: string;        // 用于匹配具体的高光场景
    involvedCharacters: string[];// 涉及的其他角色（用于判断是否\"名场面\"）
  };
}
```

检测逻辑伪代码：
```
FUNCTION scanTick(state, prevState, context):
  events = []
  
  // 扫描范围：所有活跃角色（含队友、中立NPC、敌方角色）
  // 角色曲触发不受阵营限制——敌方角色的高光时刻同样播角色曲
  allNarrativeCharacters = state.partyMembers 
    + state.sceneNPCs 
    + state.combatLog.enemies  // 敌方角色也纳入扫描
  
  FOR EACH character IN allNarrativeCharacters:
    // 检测1：绝境逆转 — HP<20% 且 本回合造成了>200伤害
    // 敌方角色濒死反杀尤为精彩，权重更高
    baseWeight_1 = character.isEnemy ? 100 : 100
    IF character.hp / character.maxHp < 0.2 
       AND state.combatLog.thisTurn[character.id]?.totalDamageDealt > 200:
      events.push({ character, category: '绝境逆转', weight: baseWeight_1 })
    
    // 检测2：觉醒/突破 — 发生了境界变化（敌我通用）
    IF character.realm != prevState.getCharacter(character.id).realm:
      events.push({ character, category: '觉醒/突破', weight: 95 })
    
    // 检测3：杀招释放 — 使用了标记为'killerMove'的动作（敌我通用）
    IF state.combatLog.thisTurn[character.id]?.actions.some(a => a.tag === 'killerMove'):
      events.push({ character, category: '杀招释放', weight: 90 })
    
    // 检测4：关键抉择 — 叙事引擎判定为'decision'节点（敌我通用）
    IF context.narrativeNode.type === 'decision' AND context.narrativeNode.actorId === character.id:
      events.push({ character, category: '关键抉择', weight: 85 })
    
    // 检测5：情感爆发 — 队友忠诚度变化>30 或 敌方与玩家的\"仇恨值\"激增
    IF character.isAlly:
      IF abs(character.loyalty - prevState.getCharacter(character.id).loyalty) > 30:
        events.push({ character, category: '情感爆发', weight: 80 })
    ELSE IF character.isEnemy:
      // 敌方角色：检测是否首次使用全力/揭露真相/与玩家有深仇大恨
      IF character.narrativeFlags.powerRevealed OR character.narrativeFlags.truthRevealed:
        events.push({ character, category: '情感爆发', weight: 80 })
    
    // 检测6：首次登场 — 该角色在当前存档中首次出现（敌我通用）
    IF character.firstAppearedThisSession:
      events.push({ character, category: '首次登场', weight: 70 })
    
    // 检测7：名台词 — 叙事文本匹配定场诗正则（敌我通用）
    IF MATCH(context.narrativeText, KNOWN_SIGNATURE_LINES[character.id]):
      events.push({ character, category: '名台词', weight: 60 })
  
  RETURN events
```

**C. 优先级仲裁器（ThemeConflictResolver）**

同一回合可能有多个角色同时触发高光时刻，需仲裁选择：

```
FUNCTION resolveThemeConflict(events: HighlightEvent[]): HighlightEvent | null:
  IF events.length == 0: RETURN null
  
  // 步骤1：过滤冷却中的角色（同一角色60秒内不重复触发）
  activeEvents = events.filter(e => !isOnCooldown(e.characterId))
  IF activeEvents.length == 0: RETURN null
  // 冷却期间若有更高权重事件，可破例（需权重高10以上）
  higherEvents = events.filter(e => isOnCooldown(e.characterId) AND e.weight > lastTriggerWeight[e.characterId] + 10)
  activeEvents = [...activeEvents, ...higherEvents]
  
  // 步骤2：过滤正在播放名场面音效的角色（7.6节名场面优先级更高）
  activeEvents = activeEvents.filter(e => !isSceneAudioPlaying(e))
  
  // 步骤3：按权重降序排列
  activeEvents.sort((a, b) => b.weight - a.weight)
  
  // 步骤4：权重差距>15，直接选最高
  IF activeEvents[0].weight - activeEvents[1].weight > 15:
    RETURN activeEvents[0]
  
  // 步骤5：权重接近时，按\"叙事关联强度\"排序
  // 叙事关联强度 = 角色与玩家当前剧情的紧密程度，阵营无关
  // 敌方强敌的出场往往比队友的行动更有叙事冲击力
  scored = activeEvents.map(e => {
    narrativeAffinity = getNarrativeAffinityScore(e.characterId, context)
    // narrativeAffinity 考虑：角色在剧情中的戏份权重 + 最近出场频率 + 与当前叙事节点的关联度
    // 敌方角色若为当前主线BOSS，其narrativeAffinity可以与队友同等甚至更高
    return {
      event: e,
      score: narrativeAffinity * 0.4 + e.weight * 0.6
    }
  })
  RETURN maxBy(scored, 'score').event
```

**D. 播放控制器（ThemePlaybackController）**

集成在`src/utils/audio-manager.ts`中：

```
FUNCTION playCharacterTheme(event: HighlightEvent):
  config = CHARACTER_THEME_CONFIG[event.characterId]  // 歌曲文件路径、intro/verse/chorus分段信息
  IF config == null: RETURN  // 该角色无主题曲配置
  
  // 确定播放长度（根据事件权重）
  playDuration = 
    weight >= 95 ? config.fullDuration :      // 15-20s 完整播放
    weight >= 85 ? config.mediumDuration :    // 12-15s
    weight >= 70 ? config.shortDuration :     // 8-12s
    5  // 仅intro片段
  
  // 执行播放序列
  audioManager.fadeOutBgm(domainGain → 0.3, 500ms)          // 域BGM降至30%
  await sleep(500ms)
  audioManager.playTheme(config.filePath,                    // 角色曲
    fadeInMs: weight >= 90 ? 800 : 1500,                     // 绝境/觉醒快速淡入，登场/名台词慢淡入
    volume: 0.7,
    duration: playDuration
  )
  await sleep(playDuration * 1000)
  audioManager.crossFade(themeGain → 0, domainGain → 1.0, 1500ms)  // 交叉淡出回域BGM
  
  // 记录冷却
  updateCooldown(event.characterId, event.weight)
```

**E. 冷却与破例机制**

| 条件 | 冷却时间 | 破例规则 |
|------|---------|---------|
| 权重60-70（名台词/首次登场） | 90秒 [实时秒冷却，非游戏回合] | 权重≥100的事件可无视冷却 |
| 权重80-90（情感/杀招/抉择） | 60秒 [实时秒冷却，非游戏回合] | 权重≥100的事件可无视冷却 |
| 权重95-100（突破/绝境逆转） | 45秒 [实时秒冷却，非游戏回合] | 同角色第二高权重事件需权重高于上次30以上才破例 |

> ⚠ 冷却类型区分：角色曲触发冷却使用**实时秒**（防止音频频繁切换），重伤冷却和势力任务冷却使用**游戏回合**。两种冷却互不干扰。冷却记录存储在`audio-manager.ts`内部Map中（非持久化，刷新页面重置）。

**F. 与名场面音效的层级关系**

角色主题曲触发与名场面音效（7.6节）的层级关系：

```
优先级（高→低）：
  L3 剧情情绪层 (名场面BGM) — 名场面期间独占bgmChannel
  L2 角色主题曲          — 普通剧情段落中事件触发
  L1 域氛围层            — 始终作为底层的默认BGM

冲突仲裁：
  - 名场面播放期间 → 角色主题曲触发被抑制，等待名场面结束
  - 角色主题曲播放期间 → 名场面触发时，角色曲立即fadeOut 0.5s，名场面BGM接管
  - 两个角色同时触发 → 见优先级仲裁器（C节）
```

#### 7.5.2 配音管理器（voice-manager.ts）核心设计

遵循"事件溯源"原则——配音播放是纯副作用，不产生新状态，仅读取当前游戏上下文决定播放内容：

```typescript
// 配音管理器接口
interface VoiceManager {
  // 预加载队列（前5个即将触发的配音）
  preload(voiceIds: string[]): Promise<void>;
  // 播放指定配音（触发ducking）
  play(voiceId: string, options?: { volume?: number; interrupt?: boolean }): Promise<void>;
  // 停止当前配音（淡出0.3s）
  stop(fadeOutMs?: number): void;
  // 驱逐缓存（超过20条未使用的解码buffer）
  gc(): void;
  // 加载manifest索引
  loadManifest(): Promise<VoiceManifest>;
}
```

**性能策略**：111个文件总量预估约30-50MB（单文件~300-450KB，MP3 128kbps CBR，每条约15-30秒）。不全部预加载——基于游戏上下文预加载接下来5个可能触发的配音，已播放超过20条的buffer被GC驱逐。Web Audio API的decodeAudioData在后台线程执行，不阻塞主线程。

### 7.6 名场面音效（v0.7.0优先做2-3个核心场景）

名场面是"BGM+SFX+Voice"三通道协同播放的复合音频事件，是四通道架构的集成验证场景。

#### 7.6.1 名场面优先级

| 优先级 | 名场面 | 触发条件 | 音频组合 |
|--------|--------|---------|---------|
| 1（最高） | **逆流河** | 方源遇到逆流河时（AI叙事判定） | 水流SFX(env层)+坚持意志BGM(L3情绪层)+木成"坚持仙蛊"配音(voice层，ducking BGM -6dB) |
| 2 | **定仙游** | 方源施展定仙游时（战斗/探索事件） | 空间穿越SFX(event层)+仙道氛围BGM(L3情绪层)+木成"定仙游"配音(voice层) |
| 3 | **定场诗** | 游戏开始/升仙/章节切换 | 木成朗诵(voice层)+低沉暗衬BGM(L3情绪层，-12dB低音量) |

#### 7.6.2 名场面音频数据流

```
名场面触发（context-builder检测）
  │
  ├─→ 1. audio-manager.prepareScene(sceneId)       // 预解码所需voice+sfx
  ├─→ 2. audio-manager.startScene(sceneId)          // 同时启动各通道
  │      ├─ bgmChannel: 切换到情绪层BGM（fadeIn 1s）
  │      ├─ sfxChannel: 播放环境/事件SFX（可loop）
  │      └─ voiceChannel: 播放配音（触发ducking）
  │
  ├─→ 3. 名场面结束（叙事完成/玩家跳过）
  │      ├─ voiceChannel: 停止配音（fadeOut 0.3s）
  │      ├─ bgmChannel: 恢复L1+L2层（ducking release）  
  │      └─ sfxChannel: 停止场景SFX
  └─→ 4. audio-manager.endScene()  // 释放资源
```

#### 7.6.3 名场面跳过与中断处理

- 玩家点击"跳过"→所有通道渐弱0.5秒后停止，恢复域BGM
- 另一个名场面触发→当前场景渐弱后切换，不硬切
- 战斗插入名场面中间→名场面立即终止，切换战斗BGM+SFX

## 九、诊断日志埋点系统

> 设计原则：遵循"最小侵入、最大化诊断价值"——日志调用仅1行代码，零性能影响（console.debug在production构建中自动剔除），localStorage持久化支持"崩溃后回溯最后500ms发生了什么"。日志是开发工具而非游戏功能——玩家永远不可见。

### 9.1 设计目标与原则

| 原则 | 说明 |
|------|------|
| **零生产开销** | 仅使用`console.debug`级别，生产构建（`NODE_ENV=production`）时被打包工具自动tree-shake/剔除 |
| **持久化审计** | 所有日志同时写入localStorage key `gu-zhenren-diag-log`，支持崩溃后导出分析 |
| **结构化格式** | 统一`[版本][模块] 操作: {JSON payload}`格式，支持grep/正则快速过滤 |
| **循环覆盖** | 最多保存1000条日志，超出后覆盖最早条目（环形缓冲区），避免localStorage膨胀 |
| **非阻塞** | 写入localStorage使用`requestIdleCallback`或microtask，永远不阻塞游戏主循环 |
| **开发期默认开** | 开发模式下默认启用；打包发布时通过环境变量`VITE_DIAG_LOG_ENABLED`关闭 |

### 12.3 v0.6.0经验教训 — 开发过程防错机制

**教训1: Zustand `||{}`反模式**
- 问题：59+处 `|| {}` 创建不稳定对象引用→无限重渲染→MerchantPanel白屏崩溃
- 根因：开发者不熟悉React订阅机制，以为 `useStore(s => s.obj || {})` 是安全的
- 对策：所有对象/数组selector必须用 `useShallow`，已修6个关键组件
- 🔧 **已机制化至Skill**: Zustand反模式检测规则

**教训2: state_update字段映射 = 项目与DeepSeek API的唯一数值通信途径**
- 问题：AI文本说"消耗了30真元"但未写 `player.essence` → UI显示满真元→玩家困惑
- 根因：LAYER2中的字段映射规则被视为"可压缩的冗余格式指导"而不是"通信协议"
- 对策：**LAYER2字段映射规则完全保留不删**，仅压缩世界观风格重复
- 🔧 **已机制化至Skill**: state_update校验闭环

**教训3: 开发顺序跳跃导致数据缺口**
- 问题：achievements.json缺reward/progressMax、gu-database缺combatStatus、npcs缺adventureTrust/interestDrive
- 根因：跳过数据阶段直接写引擎→引擎编码时无数据可用
- 对策：强制开发顺序: **数据填充 → 类型定义 → 引擎逻辑 → Store → UI**
- 🔧 **已机制化至Skill**: 开发流程强制检查清单

**教训4: 术语不同步导致设计-代码脱节**
- 问题：style-guide自身"中州"与代码中"中洲"矛盾，economy将仙元(能量)当作仙元石(货币)
- 根因：规范文件和代码双向修改后无交叉验证
- 对策：`terminology.json` 作为唯一真相源，所有文件术语引用需对照
- 🔧 **已机制化至Skill**: 术语宪章机制

**教训5: IP还原度需要专门审查维度**
- 问题：太白云生两条目personality完全矛盾（"善良"vs"亦正亦邪"）、白凝冰alias填体质名
- 根因：代码审查无原著对照维度
- 对策：专家E（蛊真人IP）作为固定审查维度，知识库固化
- 🔧 **已机制化至Skill**: E专家IP知识库
