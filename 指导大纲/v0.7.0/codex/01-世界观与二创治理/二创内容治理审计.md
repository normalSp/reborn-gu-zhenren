# 二创内容治理审计

更新时间：2026-05-07
治理决策：保留二创内容，但每条必须标注 `canon | derived | original | unknown`，并通过数值审计。
版权边界：本审计只记录摘要、来源等级和开发定位，不复制原著全文。

## 1. 治理原则

项目允许存在二创角色、仙蛊、杀招、训练场和事件，但不得把二创内容混称为原著事实。进入 v0.7.0 的数据必须能回答三件事：

| 问题 | 必填答案 |
| --- | --- |
| 它来自哪里 | `canon` 原著直接存在，`derived` 原著设定推演，`original` 项目原创，`unknown` 暂未判定。 |
| 它在玩法里做什么 | 战斗、经济、叙事、势力、训练、交易、成就或测试锚点。 |
| 它有没有数值风险 | 是否影响价格、掉落、伤害、成长速度、招募强度、资源产出。 |

`unknown` 数据不得进入 v0.7.0 可交付清单，只能留在开发候选池。

## 2. 当前数据扫描

本轮扫描对象为 canon 数据，不包含运行时生成对象。

| 数据域 | 条目数 | formal provenance 缺失 | 当前结论 |
| --- | ---: | ---: | --- |
| 一至五转蛊虫 | 128 | 128 | 数据中存在大量原著与推演混合条目，但没有显式来源字段。 |
| 仙蛊 | 41 | 41 | 含原著仙蛊与项目扩展仙蛊，v0.7.0 宝黄天会直接消费这些数据，必须先标注。 |
| 杀招 | 47 | 47 | 原著杀招、推演杀招、项目自创杀招混在同一表中。 |
| NPC | 202 | 202 | `_meta.source` 声称原著，但单条没有来源等级，且项目资产目录已有 `original` 角色。 |
| 训练场 | 29 | 29 | `_meta.totalGrounds` 写 31，但实际 `grounds` 为 29，存在元数据不一致。 |
| 随机遭遇 | 55 | 55 | 事件模板多为玩法推演，应多半归入 `derived/original`，不能统一视作原著。 |

证据：`src/canon/gu-database.json`、`src/canon/immortal-gu.json`、`src/canon/killer-moves.json`、`src/canon/npcs.json`、`src/canon/training-grounds.json`、`src/canon/encounters.json`。

## 3. 分类规范

| 字段 | 允许值 | 说明 |
| --- | --- | --- |
| `provenance` | `canon` | 原著中明确存在，名称与核心设定可回指。 |
| `provenance` | `derived` | 原著世界观可推出，但不是原著明确条目，例如某域通用训练场。 |
| `provenance` | `original` | 项目原创或同人扩展，例如原创 NPC、原创仙材、原创任务。 |
| `provenance` | `unknown` | 暂无判断，禁止进入 v0.7.0 正式交付。 |
| `balanceTier` | `low/mid/high/critical` | 影响数值闭环的强度，critical 必须有单测或模拟。 |
| `designRole` | 自由短语 | 它服务哪个玩法闭环，例如宝黄天交易、势力招募、小队战、十绝体反噬。 |

## 4. 分域审计结论

### 4.1 角色与 NPC

- `src/canon/npcs.json` 的 `_meta.source` 不能替代单条 provenance。
- 资产目录已显式区分 `public/rebrng/characters/canon` 与 `public/rebrng/characters/original`，但 canon 数据层没有同步这个边界。
- v0.7.0 小队势力系统会把 NPC 变成可招募、可入队、可参战对象；因此 NPC provenance 不只是文案问题，也是战斗和经济问题。

进入要求：

1. 核心原著人物标为 `canon`，并保持性格、立场、关系不反原著。
2. 原创山寨成员、商人、账房、药农、情报贩子等标为 `original`。
3. 原著势力下的无名成员可标为 `derived`，但文案要写成“某势力修士/弟子”，不要伪装成原著重要人物。

### 4.2 蛊虫、仙蛊、杀招

- 一至五转蛊虫、仙蛊、杀招均缺单条来源字段。
- 宝黄天现在会从 `immortal-gu.json` 和 `killer-moves.json` 生成交易池，未标注来源的原创内容可能直接被玩家购买，造成世界观和数值双重污染。
- `Spring Autumn Cicada/春秋蝉` 等唯一性、主线强约束物应继续被 auction 排除或强约束。

进入要求：

1. 主线唯一仙蛊标为 `canon` 且 `marketAllowed=false` 或进入排除表。
2. 推演类仙材、残方、杀招传承标为 `derived/original`，并注明是“宝黄天交易推演”。
3. 完整传承与残方必须有去重、合成和学习规则，不能只靠名称排重。

### 4.3 训练场与遭遇

- `training-grounds.json` `_meta.totalGrounds=31`，实际 `grounds` 为 29，元数据需要修复。
- 白天/黑天狩猎是可接受的原著设定推演，但具体荒兽、掉落率、成本多为项目数值设计，应标 `derived`。
- 随机遭遇 55 条均缺 provenance，绝大多数应是玩法模板，而不是原著事件。

进入要求：

1. 原著著名场景可以标 `canon_anchor` 或在 `sourceNote` 中标注章节锚点。
2. 通用训练场和模板事件标 `derived/original`。
3. 高收益训练场必须进入数值闭环报告，尤其黑天、白天、七转门槛、仙蛊掉落。

## 5. v0.7.0 交付闸门

| 闸门 | 通过条件 |
| --- | --- |
| 数据完整 | 角色、蛊虫、仙蛊、杀招、训练场、遭遇至少 95% 有 provenance，剩余必须有阻塞清单。 |
| 世界观安全 | `unknown` 不进入宝黄天、小队招募、成就奖励、训练场掉落。 |
| 数值安全 | `original` 和 `derived` 的 critical 条目必须有数值测试或模拟。 |
| 文案安全 | UI 与叙事中不把原创内容写成原著既定事实。 |

## 6. 推荐实现顺序

1. 在 canon 数据 schema 中增加 `provenance`、`sourceNote`、`designRole`、`balanceTier`。
2. 先处理会进入运行时池子的表：`immortal-gu.json`、`killer-moves.json`、`training-grounds.json`、`encounters.json`。
3. 再处理 NPC 与一至五转蛊虫。
4. 最后把 `unknown` 数据从宝黄天、训练场、成就奖励、小队招募候选中过滤出去。
