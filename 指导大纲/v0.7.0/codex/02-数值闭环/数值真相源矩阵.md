# 数值真相源矩阵

更新时间：2026-05-07
目的：避免同一数值在 canon、engine、store、UI、文档中各自定义，导致后续持续崩坏。

## 1. 单源原则

| 数值域 | 应成为真相源的文件 | 当前冲突源 | 处理原则 |
| --- | --- | --- | --- |
| 术语 | `src/canon/terminology.json` | UI 文案、旧文档、技能旧注释 | 术语以 canon 为准，`仙元` 和 `仙元石` 不得混用。 |
| 凡蛊价格 | `src/canon/economy.json` + `src/engine/shop-engine.ts` | `shop-items.json` 的 basePrice 是蛊材/消耗品局部价格 | engine 用 economy 的境界基数，items 用自身价格。 |
| 仙元石换算 | 待新增配置 | `MerchantPanel.tsx:119` 的 `price/10000` | 禁止 UI 私自定义换算。 |
| 宝黄天价格 | `src/engine/auction-engine.ts`，后续应迁移到 canon 配置 | 旧数值报告、交叉验证报告、设计大纲目标值 | 先以源码为当前事实，再用模拟决定目标值。 |
| 势力收入/维护 | `src/store/slices/factionSlice.ts`，后续应迁移公式参数到 canon | 旧数值报告 | 当前已有 tick，问题是通胀。 |
| 仙窍产出 | `src/canon/economy.json` 与 `src/store/slices/immortalSlice.ts` 尚未统一 | 月产公式、逐日 tick 公式 | 选一个时间单位后合并。 |
| 资源点成本 | `src/canon/economy.json.RESOURCE_NODE_BUILD_COST` | `immortalSlice.ts` 内联 `100 * gradeMultiplier` | engine/store 应读取 canon 配置。 |
| 训练场成本/收益 | `src/canon/training-grounds.json` | 旧数值报告 | 修 `_meta.totalGrounds`，然后用模拟审计收益。 |
| 战斗基础属性 | `src/engine/combat-stats.ts` | 旧报告中的黑狱龙手算 | 以 engine 为准，训练场只写设计注释。 |
| 小队战 | `src/engine/squad-combat-engine.ts` | 设计大纲 | 需要加 TTK 测试作为真相验证。 |
| 成就奖励 | `src/canon/achievements.json` + `achievementSlice.ts` | 旧跟踪文档 | 字段已补，奖励强度仍需模拟。 |
| 存档版本 | `src/store/initialState.ts` | 旧技能说明、旧文档 | 当前事实为 `SAVE_FORMAT_VERSION=12`，已覆盖目标蛊效果等 pre 新字段默认迁移。 |

## 2. 当前最危险的重复定义

### 2.1 仙元石换算

- `economy.json`：仙元石是蛊仙通用货币，与元石经济脱钩。
- `MerchantPanel.tsx:119`：隐式 `1仙元石=10000元石`。
- `factionSlice.ts:68`、`:81`：5转和6转用不同币种，不做换算。

判定：必须新增统一配置，例如 `currencyExchangePolicy`，并让 UI、store、engine 共同读取。

### 2.2 仙窍产出

- `economy.json` 写月产：面积、时间流速、节点数。
- `immortalSlice.ts:143-183` 写逐日 tick：节点 output_rate、quality、道痕密度、总道痕。

判定：这两者可以并存，但必须声明一个是设计参考，一个是运行时公式，并能在典型福地上对齐到同一量级。

### 2.3 宝黄天定价

- 旧报告写 `{6:4,7:12,8:35,9:100}`。
- 当前源码写 `{6:60,7:240,8:960,9:3800}`。
- 交叉验证报告提到另一个目标 `{6:3600,7:12000,8:40000,9:150000}`。

判定：三个版本不能同时存在。当前开发文档一律以源码值为事实，以模拟结果决定是否调价。

## 3. 统一后的数据流

推荐 v0.7.0 前修复方向：

1. canon 数据定义参数：
   - 货币换算策略。
   - 宝黄天 tier base price。
   - 资源点建造/升级成本。
   - 势力维护/收入参数。
2. types 定义读取结构。
3. engine 只做计算，不内联常量。
4. store 调用 engine，不重复公式。
5. UI 只展示计算结果，不自己换算货币。

## 4. 审计规则

每次新增或修改数值，必须记录：

| 字段 | 示例 |
| --- | --- |
| `sourceOfTruth` | `src/canon/economy.json.currencyExchangePolicy` |
| `runtimeConsumer` | `MerchantPanel`, `auctionSlice`, `factionSlice` |
| `unit` | turn/day/month, 元石/仙元石/仙元 |
| `expectedRange` | 6转小福地 20 turn 可买低价仙材，但不能稳定买 6转仙蛊 |
| `test` | `auction-economy-loop.test.ts` |

没有这些字段的数值，不得作为 v0.7.0 平衡基线。
