# v3.9.0 专家团启动会纪要

状态：startup completed；D-390 后续已批准并执行。
日期：2026-05-25。
主线建议：`v4.0 前安全收束`。

## 背景

`v3.4` 打开最小 transient proposal-only L2/L3 first cut；`v3.5` 做 lifecycle hardening；`v3.6` 做 synthetic L2/L3 微扩；`v3.7` 做 multi-NPC / small-faction proposal graph；`v3.8` 做 long-horizon stability 与 multi-pressure 复核。

到 v3.8 为止，项目已经有一个很窄但真实玩家可见的 `意图` tab transient AgentProposal 链路。它仍然没有持久 agent state、没有 save 写入、没有 live DeepSeek、没有 MiroFish、没有后端、没有外部框架、没有正式世界事实裁决权。WorldCore 仍然是最终权威。

长期路线把 `v3.9` 定位为 `v4.0 前安全收束`。专家团一致认为，v3.9 的价值不应是继续加功能，而是判断 v3.x 证据链是否足以支撑 v4.0 做高阶战斗与天道/宿命双预备。

## 专家意见

Project Lead：
v3.9 是 v3.x 的结账版本。它要统一回答：F-300/F-380 类 future gate 是否有任何被误开、v3.4-v3.8 的 transient runtime 是否真的可控、v4.0 能不能进入预备，而不是继续把 `意图` tab 做得更花。

Lore & World Designer：
v4.0 会靠近高阶战斗、天道、宿命、L5 宏观压力和原著关键人物边界。v3.9 必须先把 MiroFish blocking 条件写得更硬：命名 NPC、真实势力、真实地点、hidden-adjacent、方源公开证据、L4/L5 fact anchor、formal lore conclusion 一律不能靠 synthetic 样本绕过去。

Systems Architect：
必须复核 save-format、rollback、old-save、no-save、runtime fact authority、formal credential、formal standing、warrant/recruitment/blockade、reward、NPC life/death。v3.9 可以设计 report-only go/no-go checker，但不能把 checker 变成新世界状态。

AI Pipeline Architect：
v3.9 默认不跑 live DeepSeek。真正要跑 live probe，必须单独批准样本、轮次、模型、成本和 P0/P1 停机标准。当前更重要的是确认 DeepSeek 没有因 `意图` tab 获得新的 prompt/context/model/authority。

Player Advocate：
玩家体验重点是“候选差异可理解，候选不是事实”。v3.9 应复盘 v3.4-v3.8 Player Advocate 记录，判断玩家是否能理解 candidate/rejected/expired/needs_user_decision、多 NPC 压力、同开局差异和 WorldCore 裁决边界。

QA/Test Engineering Guardian：
建议 v3.9 做 v3.x 测试体系收束审计：v300-v380 checkers、deterministic probes、PA gate、old-save/no-save/rollback、player-visible-copy scan、stale-entrypoints、full e2e、long e2e 是否覆盖了 v4.0 前需要的风险。如果后续开发获批，建议新增 `v390` report-only go/no-go checker 和 180 轮 deterministic pre-v4 stability。

Combat/Motion Lead：
v4.0 会讨论 high-rank theater combat、killer-move stack、domain/formation、Immortal Gu House state，但 v3.9 只应定义准入标准，不改战斗表现。表现层仍不能创造命中、伤害、环境破坏、NPC 生死或奖励结论。

Git/rollback Steward：
v3.9 已切到 `codex/v390-startup-v4-safety-closure`。由于 v3.8 未获批合并 `main`，v3.9 从 v3.8 完成态继续，必须在 Git 计划中写明基线和不自动合并主线。v3.9 完成时必须再次给出是否建议合并 `main` 的结论。

Skill Sync Steward：
本轮触发 `reborn-expert-council`、`game-dev-text`、`reverend-insanity-lore`。`reborn-combat-motion` 因 v4.0 高阶战斗预备进入审计，但当前建议 `no_update_needed`，除非 v3.9 或 v4.0 真的改战斗表现。`mirofish-reborng-export` 当前 `no_update_needed`，除非 MiroFish need level 升级为 blocking。

## 路线比较

| 路线 | 内容 | 收益 | 风险 | 专家建议 |
|---|---|---|---|---|
| A | v4.0 前安全收束与准入框架 | 把 v3.x 证据链收拢，避免 v4.0 带债起步 | 产出偏治理与工具，不如新功能显眼 | 推荐 |
| B | 继续扩 `意图` tab，多加小势力/关系/文案 | 短期可见变化更多 | 可能绕过 v4.0 前安全收束，测试债继续上升 | 不推荐 |
| C | 提前做高阶战斗或天道/宿命 runtime | 方向令人兴奋 | 直接触发 L4/L5、MiroFish、lore、save、combat authority 大量 future gate | 禁止默认进入 |
| D | v3.9 直接合并 main / 发布 | 可以整理主线 | 当前未获用户主线合并批准，且 v4.0 前仍需启动包和证据 | 需另批 |

## 结论

v3.9 建议采用路线 A。请用户审批 `D-390-001` 至 `D-390-012`，并确认 `F-390-001` 至 `F-390-012` 全部继续 `future_gate_required`。审批后，Codex 可在该授权包内完成 v3.9；触发例外停机清单时必须立即停止自动推进并报告用户。
