# v3.9.0-a2 v4.0 Readiness 设计门禁

状态：complete。
授权：D-390-004、D-390-005。

## 目标

给 v4.0 提供两个预备方向的准入门禁：

1. 高阶战斗：theater / killer-move stack / domain / formation / terrain / Immortal Gu House 的架构预备。
2. HeavenWill / Fate：fate state / HeavenWill pressure / era anchor / causality debt / L5 macro pressure 的架构预备。

## Go 条件

| 方向 | Go 条件 |
|---|---|
| 高阶战斗预备 | 只做架构模型、状态边界、表现消费 WorldCore 事实，不裁决伤害/奖励/生死/地点破坏 |
| HeavenWill/Fate 预备 | 只做压力、约束、风险、时代锚点图；不做结局裁决，不暴露隐藏因果 |
| Agent 协作 | L2/L3 仍只能给候选；L4/L5 仍必须 future gate |
| Lore | 真实原著锚点、命名 NPC、hidden-adjacent、方源证据必须 MiroFish blocking + intake |
| Save | 任何持久字段、migration、runFingerprint 都必须另批 |
| DeepSeek | live probe 或 prompt/context/model/authority 扩大必须另批 |

## No-Go 条件

- 高阶战斗 UI/engine 直接进入 runtime。
- HeavenWill/Fate 直接决定玩家结局或事件成败。
- DeepSeek 把战斗旁白、天道压力或命运判断直接写成事实。
- Agent 绕过 WorldCore post-check。
- 玩家把 candidate / pressure / rumor 误读为已发生事实。
- 外部框架进入项目依赖、PoC、read-only scan、patch artifact 或子代理。

## v4.0 启动会建议问题

v3.9 完成后，v4.0 专家团应先让用户决定：

- v4.0 优先高阶战斗预备，还是 HeavenWill/Fate 预备，还是双线设计门禁。
- 是否需要 MiroFish blocking topic-slice。
- 是否继续 report-only/design-gate，还是打开最小 prototype。
- 是否先做 main 稳定分支合并，再进入 v4.0。
