# v1.9.0-process-2 只读分析型子代理试点评估

状态：complete locally；评估-only
日期：2026-05-22
性质：只读/分析型子代理风险收益评估；本阶段不启用子代理。

## 结论

专家团建议：

`v1.9 不启用子代理；v2.0 或 v2.1 可让用户单独批准只读/分析型子代理试点。`

理由：

- v1.9 当前主线是 readiness/report，不需要并行写作或实现。
- 子代理若参与 MiroFish、runtime、canon、git，容易打破责任链。
- 只读/分析型子代理对 T3 长测、coverage gap、测试矩阵覆盖有潜在收益，但必须被严格限制。

## 可接受试点范围

未来如获用户批准，只读/分析型子代理只能做：

- 覆盖缺口扫描。
- 测试矩阵与 source pointer 交叉检查。
- T3 replay finding 聚类。
- 术语/P2 风险归类。
- 文档入口过期检查。

## 禁止范围

子代理不得：

- 写 `src/**` runtime。
- 写 `src/canon/**`。
- 写 store、DeepSeek context、prompt 或 eval 判定权。
- 写 MiroFish export/intake 正文。
- 写知识库正文。
- stage/commit/push。
- 替用户作决策。
- 宣布正式地点/阵营/奖励/NPC 生死/canon promotion。

## 风险收益表

| 维度 | 收益 | 风险 | 控制 |
|---|---|---|---|
| coverage | 多人并行找缺口 | 误把建议当事实 | 只输出 findings，不改文件 |
| T3 长测 | 快速聚类 300+ 轮问题 | 标签不一致 | 主线程统一裁决 |
| MiroFish | 快速比对 source pointer | 误导晋升 canon | 只读，不碰 intake decision |
| Git | 无直接收益 | 极高风险 | 禁止子代理 stage/commit/push |
| 成本 | 节省主线程时间 | 上下文碎片化 | 使用固定 checklist 和报告模板 |
