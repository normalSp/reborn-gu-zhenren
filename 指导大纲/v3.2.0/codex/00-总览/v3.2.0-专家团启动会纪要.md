# v3.2.0 专家团启动会纪要

日期：2026-05-24
状态：completed；D-320 approved/executed

## 会议背景

v3.1 已完成 runtime agent 门禁二次加固。其 checker 结果：

- `v32RehearsalAdmissible=true`。
- `runtimeImplementationApproved=false`。
- P0/P1 falseNegative：0/0。

专家团一致判断：可以开 v3.2，但 v3.2 只能做 offline / report-only runtime rehearsal，不应越过 F-300/F-310 去实现 runtime agent。

## 专家团共识

项目负责人：

v3.2 的职责是把“候选能不能进入表达链”跑出完整证据。它不是功能版本，不给玩家看到新 agent UI，也不写存档。

系统架构：

应把 rehearsal 拆成四段：synthetic world snapshot、L2/L3 proposal batch、WorldCore post-check rehearsal、expression/rejection/manual-review report。所有阶段都必须可重放、可审计、可拒绝。

游戏设计：

同开局差异在 v3.2 只能表现为候选层差异，例如不同 NPC 意图候选、不同表达候选或不同机会窗口候选。不能把这种差异写成事实、奖励、地点、阵营、NPC 生死或正式关系。

Lore guard：

MiroFish 当前为 `not_needed`。只要进入命名 NPC、真实势力、hidden-adjacent、方源公开证据、L4/L5 锚点、知识库正文、正式 lore 结论，立即升级 blocking。

AI pipeline：

不调用 live DeepSeek。v3.2 可以继续要求报告字段写清楚“是否调用 live DeepSeek：否”。如果未来要 live probe，必须另开决策包。

QA / release：

v3.2 最小可验收证据是 report-only checker 或文档自检；不运行 runtime e2e 作为必须项，因为本版本不改 runtime。若后续 D-320 获批并进入 b1，checker 必须覆盖 false negative、result mismatch、boundary assertions、rescore stability、manual review 和 future gate 分类。

Skill sync steward：

v3.2 startup 会更新 expert / game-dev / lore 当前 override。combat-motion 与 mirofish-export 当前不需要更新，因为本版本不进入战斗表现或 MiroFish export。
