# v3.7.0 分层 Agent 与 WorldCore 测试体系审计

日期：2026-05-25
状态：startup proposed；pending user approval

## 结论先行

RebornG 的分层 Agent 与 WorldCore 测试已经制度化、工程化，并且随开发持续进化。当前不足不是没有测试，而是 v3.7 开始测试对象从“单 AgentProposal”变成“多主体候选网络”，因此需要新增 proposal graph 级别的测试门禁。

## 现有工程层

Agent / WorldCore 相关测试从 v2.1 起逐层进化：

| 阶段 | 测试工程化成果 | 意义 |
|---|---|---|

## v3.7 新增测试对象

v3.7 的测试对象从单个候选变为 proposal graph：

- 多个 synthetic/generic NPC candidate。
- 一个 generic small-faction pressure source。
- 同一公开事件输入。
- 不同 NPC 的利益判断差异。
- 小势力压力候选。
- WorldCore post-check 对越权候选的降级、拒绝或 future gate。

## v3.7 必须新增的测试维度

| 维度 | 必测问题 | 失败信号 |
|---|---|---|
| 多主体差异 | 多 NPC 是否有不同动机/风险/下一步 | 全部 NPC 输出同质空话 |
| rumor-vs-fact | 传闻、怀疑、观察是否不会变成事实 | UI 或 report 把 rumor 写成 known fact |
| 小势力压力 | 压力是否只是候选，不是 formal standing | 出现正式通缉、招揽、封锁、阵营结论 |
| WorldCore post-check | 越权候选是否被 rejected / needs_user_decision | 越权候选进入 approved lane |
| no-save boundary | 是否没有 store/save/prompt/canon 写入 | 出现新 save field、migration、runFingerprint |
| same-start variation | 同开局差异是否可解释、可复现 | 纯随机噪声或无法复盘 |
| old-save/rollback | 旧档和回滚是否不受影响 | 旧档缺 v370 字段时报错 |
| Player Advocate | 玩家是否理解“候选不是事实” | confused rounds 超阈值 |

## v3.7 测试强度建议

- Focused unit：覆盖 proposal graph、WorldCore post-check、negative families。
- Focused e2e：`意图` tab 显示多 NPC / 小势力 candidate lane，文案不误导。
- Deterministic drift：120 轮 same-start / multi-NPC variation，不使用 `runFingerprint`。
- Player Advocate：40 轮，必须记录 live DeepSeek：否。
- Old-save/no-save/rollback：证明旧档不需要 v370 字段。
- Full verification：若进入开发，执行 full unit、typecheck、build、asset/copy scans、full e2e、long e2e、production-preview。

## v3.7 不需要的测试

- 不跑 live DeepSeek，除非用户另批模型、样本、轮次、成本和接受标准。
- 不跑 MiroFish intake，因为 v3.7 默认 synthetic/generic；一旦触及真实 lore，立即停机升级 blocking。
- 不测试 external framework adapter，因为 v3.7 不引入 framework PoC/dependency。
- 不测试 persistent memory migration，因为 v3.7 不新增持久 agent state。

## 结论

现有测试体系是制度化和工程化的；v3.7 需要做的是把测试粒度从 `single AgentProposal` 升到 `multi-agent proposal graph`。这一步如果写好，会为 v3.8 长期差异复核和 v3.9 v4.0 前安全收束打下非常硬的测试地基。
