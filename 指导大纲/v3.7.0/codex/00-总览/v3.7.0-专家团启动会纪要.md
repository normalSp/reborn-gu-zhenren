## 专家团分工意见

项目负责人：

v3.7 应按长期路线进入 `v3.7-v3.8 多 NPC / 小势力扩展复核`。但范围必须是 transient / synthetic / proposal-only，不要把 v3.7 做成完整 NPC 社会。

游戏系统架构：

v3.7 的关键不是多加几个 NPC 文案，而是验证多主体候选之间的约束关系：同一公开事件下，不同 synthetic NPC 有不同利益判断，小势力压力只形成候选，不直接写正式 standing、warrant、recruitment 或 blockade。

WorldCore / QA：

测试必须从单 proposal gate 升级为 proposal graph gate。验收重点是 no-save-write、no-formal-authority、rumor-vs-fact、rejected/future gate、same-start stability、old-save/rollback。

Lore guard：

v3.7 可使用 synthetic/generic NPC 和小势力压力源。只要触及真实命名 NPC、真实势力、hidden-adjacent、方源公开证据、L4/L5 原著锚点或正式 lore 结论，MiroFish 必须升级为 `blocking` 并停机。

Player Advocate：

玩家需要看懂三件事：这些 NPC 是候选表达，不是事实；小势力压力是风险提示，不是正式通缉/招揽/封锁；同开局差异来自世界因果与候选视角，不是随机噪声。

## 路线 A：多 NPC / 小势力 transient 复核（推荐）

在 v3.6 的 multi-lane 基础上，加入一个 synthetic proposal graph：

- 3-5 个 generic L2/L3 NPC candidate。
- 1 个 generic small-faction pressure source。
- 一个公开事件输入，多主体产生不同意图候选。
- WorldCore post-check 将越权候选降级为 rejected / needs_user_decision / future_gate_required。

收益：

- 正好承接长期路线 v3.7-v3.8。
- 比 v3.6 更接近未来“小队成员、小势力、区域社会”的真实压力。
- 仍不开放 persistent agent、live DeepSeek、MiroFish、backend 或外部框架。

风险：

- 候选图会更复杂，UI/copy 容易让玩家误以为关系已经正式成立。
- 多主体样本容易产生谣言变事实、重复空转、目标遗失。
- 如果设计不严，后续很容易滑向正式 faction standing 或 NPC memory。

## 路线 B：继续 v3.6 原地硬化

继续增加 v360 negative cases、copy guard 和 deterministic 轮次，不进入多主体。

收益是风险最低；缺点是 v3.x 的社会模拟路线推进太慢，无法回答“多个 NPC 是否能围绕同一事件形成不同压力”。

## 路线 C：live probe 或 MiroFish topic slice

直接引入 live DeepSeek 或真实 lore topic slice。

当前不推荐。v3.7 的核心问题可以先用 synthetic/generic 测清，不需要引入 live 成本、真实 lore 边界或 hidden 风险。

## 专家团建议

选择路线 A。v3.7 做 `transient multi-NPC / small-faction AgentProposal 复核`，保持 synthetic/generic、WorldCore final authority、no-save/no-runFingerprint/no-live/no-MiroFish。测试重点从“单候选是否安全”升级为“候选网络是否仍被 WorldCore 管住”。
