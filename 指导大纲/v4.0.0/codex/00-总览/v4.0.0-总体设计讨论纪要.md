# v4.0.0 总体设计讨论纪要

状态：design consensus recorded；用于修订 D-400 授权包。
日期：2026-05-25。

## 背景

用户在 v4.0 启动后提出：高阶战斗很可能不适合继续沿用棋盘战斗路线，并询问是否可以把战斗做成自走棋。随后用户明确赞成 `Auto-Theater Combat`，并进一步询问凡阶战斗是否也需要迁移到同一范式。

专家团复核后形成共识：传统棋盘和传统自走棋都不能单独承载蛊真人高阶战斗。v4.0 应把战斗路线从“高阶棋盘 / 纯 theater”修正为 `Auto-Theater Combat`：

`自走棋式准备 -> WorldCore 自动结算 -> Battle Theater 态势表达 -> Killer-Move Stack -> Combat Ledger 复盘 -> DeepSeek 表达/候选意图`

## 已确认用户决策

| 项目 | 决策 |
|---|---|
| 天道/宿命 | 只做 `pressure / constraint / risk`，不做玩家结局裁决 |
| v4.0 范围 | 不做高阶 runtime，只做总设计、schema/envelope、report-only checker |
| 文档顺序 | 先补本纪要，再重写 D-400 授权包，防止后续 `/goal` 跑偏 |
| 战斗主路线 | 赞成 `Auto-Theater Combat`；不采用纯棋盘或纯自走棋作为高阶主容器 |
| 凡阶战斗 | 纳入同一底层范式，但采用 `Auto-Theater Lite`，不立刻重做凡阶 UI/runtime |

## 专家团共识

### 不采用纯自走棋

纯自走棋适合表达：

- 准备阶段。
- 阵容 / 站位 / build。
- 自动结算。
- 复盘日志。

但纯自走棋不足以表达：

- 空域 / 地面 / 地下 / 水域 / 洞天福地。
- 阵法嵌套、领域压制、仙蛊屋状态。
- 高阶杀招的准备、压栈、反制、打断、反噬。
- 环境破坏证据链。
- 天道/宿命宏观压力。
- 战后势力关注、因果债和传闻扩散。

因此 v4.0 不应把所有战斗改成“摆棋子然后自动打”。传统棋盘继续保留为低阶/中阶/局部镜头，不能作为高阶战斗主容器。

### 采用 Auto-Theater Combat

`Auto-Theater Combat` 的核心是：

- 准备阶段像自走棋：玩家配置蛊虫、杀招、队友、资源、站位/态势、底牌暴露策略、撤退阈值。
- 结算阶段由 WorldCore 自动跑：命中、伤害、资源消耗、反噬、环境变化、NPC 生死候选和奖励候选都必须经过本地内核。
- 表达阶段使用 Battle Theater：展示空域、地面、地下、水域、洞天、阵法、领域、仙蛊屋、环境压力。
- 博弈阶段使用 Killer-Move Stack：展示准备、蓄势、锁定、释放、反制、打断、反噬和失败降级。
- 复盘阶段使用 Combat Ledger：记录每一步的事实、候选、拒绝原因、代价和后果。

### 凡阶采用 Auto-Theater Lite

凡阶战斗也应纳入同一底层范式，但不能立刻表现硬改。

推荐分层：

- 一转到二转：保留棋盘和行动卡，加入局部 `Auto-Theater Lite`，强调可读教学、真元消耗、距离、目标和失败原因。
- 三转到五转：保留小队/棋盘战术，同时加入自动结算片段、战术倾向、撤退阈值、队友职责和 Combat Ledger。
- 蛊仙及以上：进入完整 `Auto-Theater Combat`，棋盘降级为局部战术镜头，主表达转为 theater / stack / ledger。

结论：`Auto-Theater Combat` 是全阶段战斗架构基线；凡阶采用 Lite 表现，高阶采用 Full 表现。v4.0 只设计跨阶段架构，不实施凡阶 runtime 迁移。

### 美术与交互总纲补充

用户进一步确认看好 Auto-Theater Combat 后，专家团补充判断：Auto-Theater Combat 应成为 RebornG 后续战斗表现的核心招牌。它的美术方向不是大粒子堆叠，也不是普通自走棋皮肤，而是：

`水墨战争沙盘 -> 道痕规则可视化 -> 杀招栈舞台 -> Combat Ledger 复盘 -> DeepSeek 表达`

因此 v4.0 需要新增专门的美术与交互总纲，冻结 theater 布局、色彩语言、杀招栈、Combat Ledger 信息架构、GSAP/Motion 分工、素材分层、外部参考矩阵和 Playwright 截图验收计划。该总纲只授权设计语法，不授权 theater UI implementation、素材生成、运行时接入或战斗结算。

### DeepSeek 参与边界

DeepSeek 可以参与：

- 敌方战斗意图候选。
- 战斗旁白。
- 小队成员反应。
- 战后复盘文本。
- 天道/宿命压力的可读表达。

DeepSeek 不可以参与：

- 命中、伤害、资源消耗、反噬结算。
- NPC 生死、地点毁灭、正式奖励、阵营变化。
- 仙蛊、杀招、传承生成。
- 天道/宿命结局裁决。
- hidden-adjacent / Fang Yuan evidence / L4/L5 原著锚点正文。

|---|---|---|
| Teamfight Tactics | 准备、部署、自动结算、阵容/build节奏 | 平面棋盘作为高阶主容器 |
| Mechabellum | 部署后自动战斗、大规模态势变化、减少微操 | 纯兵种/单位互殴 |
| Backpack Battles | build 组合、物品位置联动、异步自动战斗 | 把蛊虫系统简化成普通道具栏 |
| Yi Xian | 修行题材的构筑与自动出招 | 纯卡牌循环替代世界内核 |
| boardgame.io | phase、turn order、log、replay/time-travel 思想 | 直接引入依赖或替代 Combat Core |
| OpenRA | actor/trait/mod 组件化思想，阵法/领域/仙蛊屋可参考 | RTS 运行时或单位寻路逻辑 |

当前仅吸收设计模式，不引入外部 dependency、PoC、vendored subset、read-only scan、patch artifact 或 subagents。

v4.0 主线应从：

`高阶战斗 theater 与 HeavenWill/Fate 宏观压力双预备`

修订为：

`Auto-Theater Combat 与 HeavenWill/Fate Pressure 高阶世界表达模型`

v4.0 可以完成：

- `AutoTheaterCombatEnvelope` 设计。
- `AutoTheaterLiteEnvelope` 设计。
- `BattleTheaterEnvelope` 设计。
- `KillerMoveStackEnvelope` 设计。

- theater UI implementation。

- 杀招栈数值结算。

## 需要进入 D-400 的原则

1. 高阶战斗主路线采用 `Auto-Theater Combat`，不是纯棋盘或纯自走棋。
2. 棋盘保留为凡阶/中阶/局部战术镜头，高阶主表达为 theater / stack / ledger。
3. 凡阶战斗底层按 `Auto-Theater Lite` 预备，但 v4.0 不实施 runtime 迁移。
4. Auto-Theater Combat 美术与交互总纲必须进入 a1 上游，且只冻结视觉语法和验收计划。
5. DeepSeek 只做表达和候选意图，不能裁决战斗、奖励、生死、环境或宿命。
6. 天道/宿命只输出 `pressure / constraint / risk`，不裁决玩家结局。
7. v4.0 默认只做设计门禁和 report-only checker，不做 runtime。
8. 任何真实原著锚点、命名 NPC、hidden-adjacent、方源证据、L4/L5 或 formal lore 结论都升级 MiroFish blocking。
9. v4.0 到 v5.0 的长期推进按 `指导大纲/长期路线/v4.0-v5.0-AutoTheater到高阶世界Runtime总体大纲.md` 执行：先 contract、Lite、高阶 theater、杀招栈、宏观压力、MiroFish/lore 准入和 v5.0 go/no-go，再考虑高阶世界第一刀。
