# v4.0.0 专家团启动会纪要

状态：startup pending user decision。
日期：2026-05-25。
主线建议：`高阶战斗 theater 与 HeavenWill/Fate 宏观压力双预备`。

## 背景

`v3.9.0` 已完成 v4.0 前安全收束：v390 report-only checker、180 轮 deterministic、60 轮 Player Advocate、old-save/no-save/rollback、MiroFish / DeepSeek / external framework / save-format / WorldCore 边界均已复核。v3.9 结论允许进入 v4.0 专家团与设计门禁，但不授权高阶 runtime。

长期路线把 v4.0 定位为“高阶战斗 + 天道/宿命双预备”。专家团一致认为，v4.0 的任务不是一口气开放蛊仙世界，而是把两个最危险、最核心、最影响未来上限的系统拆成可测、可回滚、可由 WorldCore 裁决的架构草案。

## 专家意见

Project Lead：
v4.0 应是高阶世界的“设计地基版”。它要把战斗表现、宏观时代压力、AgentProposal、WorldCore final authority 和 lore source-pointer 边界统一起来。不要从 v3.9 直接跳到完整蛊仙、五域两天或方源 live agent。

Lore & World Designer：
天道、宿命、尊者布局、方源、原著关键人物、隐藏因果链都属于 L4/L5 级风险。v4.0 可以设计 `FateState`、`HeavenWillPressure`、`EraAnchorGraph` 和 `CausalityDebt` 的概念接口，但真实原著锚点、hidden-adjacent material、formal lore conclusion 必须走 MiroFish blocking topic-slice。

Systems Architect：
高阶战斗必须先区分表达层和结算层。theater map 可以比低阶棋盘更有表现力，但命中、伤害、资源成本、环境破坏、NPC 生死、奖励、地点变化仍必须由 WorldCore / combat engine 裁决。v4.0 第一轮只做 schema / envelope / checker，不写 save。

Gameplay/Numerics Architect：
蛊仙级战斗不适合简单扩 5x3 棋盘。建议设计 theater + stack：theater 负责空间、地形、领域、阵法、距离、环境压力；killer-move stack 负责杀招准备、打断、反制、连锁、反噬、资源成本。数值不能进入 UI 私有公式。

Combat/Motion Lead：
高阶战斗表现可以研究 full-screen theater、层级时间轴、杀招栈、环境破坏回放、仙蛊屋状态面板，但 v4.0 启动包只写准入和表现边界。表现层不得暗示已获得仙蛊、已开放完整杀招传承、NPC 已死或地点已毁。

AI Pipeline Architect：
DeepSeek 在 v4.0 默认仍不扩大权限。即使未来用于高阶战斗旁白或天道压力文本，也只能表达 WorldCore 已裁决或 report-only 候选，不得生成正式事实、正史锚点、隐藏因果或结局裁决。

Player Advocate：
v4.0 的玩家理解风险非常高。必须反复检查玩家是否能区分：候选、压力、传闻、宏观趋势、已裁决事实。设计文案要避免把 HeavenWill pressure 写成“系统强杀”，也避免把杀招栈预备写成正式战斗结算。

QA/Test Engineering Guardian：
建议 v4.0 继续 report-only / design-gate-first。第一批测试矩阵应覆盖高阶战斗越权、天道万能失败、hidden leak、L4/L5 原著关键人物越权、formal location/reward/NPC-life 越权、DeepSeek authority drift。

Git/Rollback Steward：
v4.0 已从 v3.9 完成态切分支 `codex/v400-startup-high-world-prep`。本启动包完成后应提交推送并等 CI；版本完成时仍需单独记录是否建议合并 `main`，但不自动合并。

Skill Sync Steward：
本轮触发 `reborn-expert-council`、`game-dev-text`、`reverend-insanity-lore`、`reborn-combat-motion`。`mirofish-reborng-export` 当前 `no_update_needed`，除非后续 MiroFish need level 升级为 blocking。

## 路线比较

| 路线 | 内容 | 收益 | 风险 | 专家建议 |
|---|---|---|---|---|
| A | 双预备：高阶战斗 + HeavenWill/Fate | 符合长期路线，能同时打通表现和宏观压力地基 | 设计复杂，需要严格禁止 runtime 越权 | 推荐 |
| B | 只做高阶战斗预备 | 产出更聚焦，战斗表现可见性强 | 宿命/时代压力继续缺位，后续接原著大事件会吃力 | 可选 |
| C | 只做 HeavenWill/Fate 预备 | 世界观深度强，适合长期叙事 | 玩家可见短期收益弱，战斗表现问题延后 | 可选 |
| D | 直接做高阶 runtime prototype | 短期刺激 | 触发 save、lore、MiroFish、combat authority、DeepSeek 多重 future gate | 不推荐 |
| E | 先合并 main / 暂停 v4.0 | 主线更整洁 | 会打断长期路线节奏，仍需单独审批 | 需另批 |

## 结论

专家团建议采用路线 A：v4.0 做双预备启动版，先设计、审计、建 checker，不直接开高阶 runtime。请用户审批 `D-400-001` 至 `D-400-012`，并确认 `F-400-001` 至 `F-400-012` 全部继续 `future_gate_required`。
