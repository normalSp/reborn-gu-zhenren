# v4.0.0-process-1 开发流程与系统连续性审计

日期：2026-05-26
状态：completed；建立项目级 `系统连续性与孤儿系统审计制度.md`

## 触发原因

用户要求严谨评估 RebornG 的开发流程和工程化制度是否科学、可靠，是否存在每个大版本各干各的、没有接上、出现孤儿代码或孤儿系统的问题。该问题触发项目级治理审计。

本审计只检查流程、路线、文档、脚本、测试和当前代码接续关系；不改 runtime、不新增 save field、不调用 live DeepSeek、不做 MiroFish intake、不引入后端或外部依赖。

## 总结论

当前开发制度整体可靠，适合 RebornG 这种长线 AI RPG：

- 前置批量审批和例外停机解决了用户反复小审批的问题。
- Skill 同步审计解决了专家团、工程 skill、lore skill、combat skill 口径滞后的问题。
- Git 分支/推送制度和主线合并补丁解决了“每个版本开分支但没有主线归宿”的问题。
- Player Advocate、长线叙事漂移、测试矩阵演进、MiroFish 双仓、知识库治理共同覆盖了玩家口感、长线漂移、原著边界和候选材料晋升链。

但审计发现一个制度缺口：上述制度分别很强，却缺少一个专门检查“上一阶段产物是否被下一阶段承接、替代、归档或明确废弃”的横向门禁。因此本次新增项目级制度：

- `指导大纲/流程制度/系统连续性与孤儿系统审计制度.md`

它把系统产物分类为 `active_runtime`、`active_gate`、`report_archive`、`planning_baseline`、`historical_reference`、`superseded_with_pointer`、`remove_candidate`、`blocking_orphan`，要求每次大版本启动、runtime first cut 前和 rc 收束时检查路线、代码、脚本、测试、制度、Skill 的接续关系。

## 对现有制度的评价

| 制度 | 科学性 / 可靠性 | 当前建议 |
|---|---|---|
| 前置批量审批与例外停机 | 高。能让用户只批方向和硬边界，后续由 Codex 机械执行 | 保持，新增系统连续性审计作为授权包/rc 项 |
| Skill 同步审计 | 高。已经避免 lore skill、expert skill 口径滞后 | 保持；制度变化时继续强制健康门 |
| Git 分支与主线合并 | 高。语义分支适合当前节奏，main 合并需用户批准 | 保持；v4.0 rc 继续给是否建议 merge main 的结论 |
| Player Advocate | 高。补自动测试看不到的玩家理解和口感 | v4.0 b3 按 D-400 执行 60 轮，live DeepSeek 默认否 |
| 长线叙事漂移 | 高。分 T0/T1/T2/T3 很适合防 AI 跑团退化 | v4.0 文档/设计阶段豁免 live，但 b1/b2 checker 要覆盖越权边界 |
| MiroFish / 知识库治理 | 高。三层晋升链保护 hidden/private 和 canon 边界 | v4.0 维持 not_needed；触及真实锚点立即 blocking |
| 系统连续性审计 | 新增。补齐跨版本“有没有接上”的横向门禁 | 从 v4.0-process-1 起执行 |

1. a0 必须引用本审计，确认 v3.9 -> v4.0 的上游证据链。
