### 执行策略：五波次并行推进

P2的94-131天对于一个单人开发者项目来说是最长也最关键的阶段。核心策略是——**M5叙事深度期作为串行骨架先行推进，M6/M7/M8在M5推进期间穿插并行，M9作为最后的决战集成期**。

**第一波次（M5先行，38-54天）** — 这是P2最长、最硬、最不可并行的部分：

```
P2-3 路由重构 (5-7d)
  └→ P2-1a 南疆扩展 (3-4d)
      └→ P2-1b 北原章节 (3-4d)
          └→ P2-1c 东/西/中三域 (6-8d)
              └→ P2-2a 8核心涟漪 (8-10d)
                  └→ P2-2b 32+域内涟漪 (10-15d)
                      └→ P2-审计C 叙事数据审计 (3-5d) — 穿插在涟漪编写间隙
```

**关键判断**：M5的串行依赖链无法打破（涟漪需要章节定义完，章节需要路由重构完）。但M5期间的"编写工作"（chapters.json数据填充、涟漪文本撰写）是纯内容工作，与后续编码工作不同质。建议在M5期间：

- 上午做大块数据编写（章节定义/涟漪约束文本）
- 下午穿插M6/M7/M8中的编码任务（作为"换脑休息"）
- 每完成一个域的章节组，立即跑一次该域的开局叙事测试（3-5轮），确保AI叙事能正常适应新域约束

**第二波次（M6并行于M5，28-37天有效工时）**：

```
P2-4a 决斗引擎 (5-7d) ────── 依赖P1战斗接口
  └→ P2-4b 叙事战斗框架 (4-6d) ── 依赖P2-4a
      └→ P2-4c 战斗UI (5d) ────── 依赖P2-4b

P2-5 NPC对话 (6-8d) ──────── 完全独立
P2-6 经济闭环 (4-6d) ──────── 依赖P1 yuanStoneSlice
P2-11 引导填充 (2-3d) ─────── 依赖P1 tutorialSlice
```

**第二波次的并行原则**：战斗系统（P2-4a→b→c）是其内部的串行链，但整体可以与NPC/经济/引导并行推进。建议按系统复杂度分配主副时间——战斗系统占60%精力，其余3个系统合计占40%。P2-9随机遭遇与P2-10起源解锁两项在交叉审计中发现完全不存在，需从零新建。

**第三波次（M7前端动效，14-21天，与M5/M6/M8高度并行）**：

```
P2-F0 Token落地 (1-2d)
  ├→ P2-F1 组件动效 (4-6d)
  │   └→ P2-F2 场景动画 (5-7d) — 可与F3部分并行
  └→ P2-F3 材质氛围 (3-4d)
      └→ P2-F4 调试优化 (1-2d) — 依赖F1+F2+F3全完成
```

前端动效重构是**独立实施轨道**——不依赖任何后端系统变更，也不需要战斗/经济/NPC完成。Phase 0-4的全部工作都是CSS/组件视觉层的改动。因此M7在整个P2期间随时可以启动和推进。建议放在M5中期（章节数据编写感到疲劳时）作为"视觉激励"启动——做出一两个组件的动效后看到实际成品效果会显著提振士气。

**第四波次（M8并行于M5/M6/M7，14-20天有效工时）**：

```
P2-7 音效系统 (3-4d) ──── 完全独立
P2-8 成就UI (3-4d) ────── 依赖P2-4完成（需战斗事件触发成就检测）
P2-12 空窍表现层 (4d) ─── 依赖P1空窍AI修正
P2-13 动态补完 (3d) ───── 依赖P1 guSlice/factionSlice
P2-审计D 剩余审计 (6-8d) ─ 完全独立，纯数据
P2-流派 本命蛊 (3.5d) ─── 依赖P1流派接口
```

所有M8子系统互相独立，可任意穿插推进。音效和审计是纯资源/数据工作，适合在"脑子累了不想写逻辑"的日子做。空窍SVG渲染和成就UI是可视化组件工作，适合在动效重构期（M7）穿插。

**第五波次（M9集成，8-13天）** — 前置条件：M5+M6+M7+M8全部完成：

```
P2-3b NPC扩展+跨域过滤 (3-5d)
  └→ P2-QA 全局联调+17项退出标准全验收 (5-8d)
```

这是P2的最后决战阶段。所有子系统就绪后，需要做的是：把17项退出标准逐一跑通，把南疆+北原两条完整故事线从头到尾打一遍，把五个域的出生开局各验证3轮，把8个核心事件涟漪的四层触发路径全部覆盖。M9不应该有"写新代码"——只有"调已经写好的代码"。

## 第二部分：章节弧光系统

本章节是对「全书覆盖可行性评估与开发指导文档」的精简整合。完整的设计哲学、原著内容全景、风险分析请参考原文档。

### 2.1 核心设计原则

**原则一：名场面是约束，不是脚本。** 四十多个名场面不是四十多段固定文本，而是四十多组约束规则(scene_constraint)，包含事件核心要素、标志性元素、禁止偏差。AI在约束框内自由发挥语言文学性。

**原则二：全局事件是涟漪，不是强制剧本。** 设计为四层涟漪模型——L0完整参与(源发域) → L1简化参与(相邻域) → L2叙事提及(远隔域) → L3长尾影响(全局flag)。玩家可选择是否介入。

**原则三：章节是状态区间，不是叙事容器。** 章节定义出场NPC池、可用地点、势力关系、关键事件条件和名场面约束。不同域共享全局章节区间但展开完全不同。

**原则四：flag驱动一切，不硬编码路径。** 所有状态推进通过flag系统驱动，新增故事线只需新增flag定义，不需修改引擎代码。

**原则五：先验证架构，再扩展内容。** P1仅做南疆三章验证完整环节后，再复制到其他四域。

## 第三部分：战斗系统设计

### 3.1 可行性评估

**现有基础**: killMoveSlice.ts（杀招管理+冷却系统）+ killer-moves.json（8.54KB杀招数据）+ playerSlice（HP/真元/境界/AP）+ 语义验证层R01/R05/C01（境界压制检测）。无实际战斗引擎。

**风险评估**: 中等。核心风险不在于"AI驱动vs TS代码"的张力——回合制对于文字游戏不是可选问题，而是唯一可行的时间框架（文字需要阅读时间，实时战斗在文字媒介中无法成立）。真正的风险在于**1v1回合制框架与群像战斗的粒度不匹配**：原著中狼潮攻城、义天山混战、宿命大战、天庭终战都是多方大规模战斗，如果让玩家在"寨子快被攻破了"的叙事之后进入两个血条的战斗界面，会产生严重的认知断裂。结论：采用**分层战斗系统**——群像战斗走AI动态生成的叙事选择链（玩家做出战略决策而非每回合选择技能），1v1决斗走确定性TS回合制引擎（有策略深度和操作快感），两者之间通过COMBAT_ROUTER在运行时自动判定切换。

**工时估算**: P2阶段14-18天（含决斗引擎+叙事战斗框架+COMBAT_ROUTER+StatBridge+南疆三章战斗场景数据+UI组件）。

### 3.2 战斗类型判定

分层战斗系统的第一层逻辑是**在触发战斗时自动判定战斗类型**，由AI场景分析引擎完成，不依赖人工预标记。

```typescript
type CombatType = 'duel' | 'narrative' | 'group';

interface CombatTrigger {
  combatType: CombatType;
  autoDetect: boolean;  // true=AI自动判断战斗类型, false=数据中手动标注
}
```

**AI自动判断规则**（嵌在state_update生成逻辑中）：

AI在生成state_update时检查以下信号，综合输出combat_type决策：

| 信号 | 倾向duel | 倾向narrative |
|------|---------|--------------|
| enemy_count | 1 | ≥3（多方混战默认为narrative） |
| enemyRealm | 任意 | 任意（高境界单敌仍可为duel） |
| 场景标签 | — | force_battle_scale: 'war'/faction_mix: ≥3 |
| 名场面临近 | 非核心名场面 | 宿命大战/天庭终战等L0核心名场面 |
| 玩家是否孤立 | 是（无盟友） | 否（有盟友参战） |

**手动标注**：chapters.json和encounters.json中的关键战斗场景可显式指定combat_type，覆盖AI自动判断。这用于确保关键战斗体验的一致性（如"白凝冰决斗"必须走duel引擎，不能因AI状态偏差误判为narrative）。手动标注的优先级高于autoDetect。

### 3.3 COMBAT_ROUTER 路由层

在PIPE管线的VALIDATING→RESOLVED之间插入COMBAT_ROUTER分流逻辑。这不是新PIPE阶段，而是VALIDATING完成后的路由判断——根据state_update.combat_type将流程导向不同的执行路径。

```
VALIDATING 完成
  → COMBAT_ROUTER 检查 state_update.combat_type
    → 'duel' → COMBAT_DUEL阶段（确定性回合制引擎，详见3.4节）
    → 'narrative' → COMBAT_NARRATIVE阶段（AI动态生成战略选择链，详见3.5节）
    → 'group' → P2返回兜底文本（AI叙事描述团战过程），P3走团战引擎（详见3.6节）
    → null → 直接进入 RESOLVED（无战斗触发）
```

路由决策在response-pipeline.ts中实现，约40行逻辑。路由层本身不做任何战斗计算，只做类型判定和流程分发。

### 3.4 决斗引擎（Duel Combat）

决斗引擎适用于玩家vs单个敌方单位的战斗（与敌对蛊师的决斗、野兽/荒兽遭遇战、关键NPC的切磋/复仇战）。这是唯一使用完整HP/真元/杀招数值计算的战斗类型，其余战斗类型不进入此引擎。

**核心数据模型**:


**回合制战斗循环**:

1. **回合开始** — AP重置为1，检查持续效果(buff/debuff/dot)到期和触发
2. **玩家行动选择** — 消耗1AP，可选行动：
   - 普通攻击（0真元消耗，基础伤害=境界基础值×武器蛊加成）
   - 使用杀招（消耗真元，伤害=杀招baseDmg×multiplier×境界修正，杀招进入冷却）
   - 使用蛊虫辅助（消耗真元，施加buff/debuff/治疗）
   - 尝试逃跑（消耗AP，按逃跑判定公式计算成功率）
3. **敌方AI行动** — 根据attackPattern选择行动：
   - 攻击型：70%概率攻击/20%技能/10%防御
   - 防御型：50%防御/30%攻击/20%技能
   - 狂暴型（HP<30%时）：80%技能/20%攻击
4. **执行阶段** — 双方行动同时结算。伤害计算：`finalDamage = baseDmg × (1 + realmDiff × 0.3) × (1 - defenseBuff) × random(0.85, 1.15)`
5. **判定阶段** — 检查HP是否归零（玩家→死亡，敌方→胜利）。检查逃跑是否成功。
6. **日志输出** — 生成本回合的叙事文本（"你的月牙蛊化作一道寒光刺向敌人..."），由AI填充语言但数值由引擎提供。

| 四转 | 50 | 55 | 60 | 65 |
| 五转 | 70 | 76 | 82 | 88 |

注：每提升一个小境界基础伤害+3（一转区间累进），每跨越一个大境界额外+3。以上为基础普通攻击值（不消耗真元、不使用杀招时），杀招伤害 = 杀招基础值 × 境界修正系数 × multiplier。

**杀招自然语言到数值转换表**（用于将killer-moves.json中的描述性成本转换为战斗引擎可用的数值）：

| 自然语言描述 | 真元消耗 | 伤害类型 | 建议baseDmg | 冷却回合 |
|------------|---------|---------|------------|---------|
| 轻消耗 | 5 | 轻攻击 | 基础值×1.2 | 0 |
| 中消耗 | 10 | 标准攻击 | 基础值×1.5 | 1 |
| 重消耗 | 20 | 重攻击 | 基础值×2.0 | 2 |
| 极重消耗 | 35 | 终结技 | 基础值×3.0 | 4 |
| 轻连锁 | 8 | 多段攻击 | 基础值×0.6(×3段) | 1 |
| 中范围 | 15 | 范围攻击 | 基础值×1.3(波及全体) | 2 |

**条件逃跑机制**:

逃跑判定公式：`逃跑成功率 = baseChance + realmBonus × (playerRealm - enemyRealm) + guBonusPerGu × assistGuCount + fieldBonus`

- baseChance: 20%（基础逃跑概率。从30%下调至20%，玩家初始HP=100，野兽级敌人约一转初阶水平，逃跑不应过于容易——更低的逃跑率迫使玩家认真评估每次战斗决策，也符合蛊真人世界"每一步都可能致命"的基调）
- realmBonus: 15%（每高一个境界+15%）
- guBonusPerGu: 10%（每只辅助蛊+10%，如月光蛊/飞翔蛊/隐匿蛊）
- fieldBonus: 10%（玩家方有环境优势时）

逃跑成功后：触发逃跑叙事（AI生成），应用escapePenalty（从敌方state_update中读取），COMBAT_DUEL阶段结束。
逃跑失败后：本回合无法再次逃跑，直接进入敌方回合。

**设计原则**: 逃跑不应无代价——否则玩家会养成"见敌就跑"的习惯。逃跑惩罚应与战斗收益成比例：一场可能掉落稀有蛊材的战斗，逃跑代价应大于一场随机遭遇。

### 3.5 叙事战斗框架（Narrative Combat）

叙事战斗适用于多方大规模群像战斗——狼潮攻城、义天山混战、宿命大战、天庭终战等场景。在这些战斗中，玩家的个人杀招选择对大局影响微乎其微，真正有意义的是**战略层面的决策**（守哪个门、救谁、跟谁结盟、何时撤退）。因此叙事战斗不走回合制引擎，而是由DeepSeek根据combat_constraint动态生成战略选择链——玩家在关键节点做出战略选择，AI生成衔接叙事文本。

#### 3.5.1 combat_constraint 数据模型

借鉴scene_constraint的设计模式（见评估文档第三章），combat_constraint定义叙事战斗的约束边界：

```typescript
interface CombatConstraint {
  sceneId: string;
  combatType: 'narrative';
  scale: 'skirmish' | 'battle' | 'war';

  // 约束边界（借鉴scene_constraint模式）
  mustHappen: string[];       // 必须发生：如"狼潮突破木墙防线"
  mustNotHappen: string[];    // 禁止发生：如"玩家单枪匹马击退全部狼群"
  keyFactions: string[];      // 参战势力
  keyNPCs: string[];          // 关键NPC出场名单

  // AI生成指令
  strategicChoiceCount: number;  // 每轮生成2-4个战略选择
  narrativeStyle: string;        // 叙事风格基调

  // StatBridge配置
  statBridge: {
    enabled: boolean;
    realmWeight: number;       // 境界对概率的权重（默认0.1）
    guTagInfluence: { tag: string; bonus: number }[]; // 蛊虫标签加成
  };
}
```

**combat_constraint与scene_constraint的关系**：

- scene_constraint用于名场面的叙事约束
- combat_constraint是scene_constraint的子集——名场面中的战斗场景同时定义scene_constraint和combat_constraint
- 数据存储：combat_constraint作为scene_constraint的嵌套字段存在（`scene_constraint.combat`），不新建独立数据文件
- 注入时机：proximity detector检测到战斗型名场面临近时，同时注入scene_constraint和combat_constraint

#### 3.5.2 StatBridge 轻量数值桥接

叙事战斗中玩家的境界和蛊虫标签影响战略选择的成功概率，但不消耗HP/真元。StatBridge是一个独立的概率修正模块，在玩家做出战略选择后计算最终成功率。

**概率修正公式**：

```
finalProbability = baseChance × (1 + realmDiff × realmWeight) × (1 + sum(guTagBonus))
```

- baseChance: 每个选择的基础概率（50-80%），AI生成时设定，受金丝雀规则约束（不得低于50%，不得高于80%）
- realmDiff: (playerRealm - challengeRealm)，范围[-3, +3]
- realmWeight: 默认0.1（每个境界差影响10%概率）
- guTagBonus: 玩家携带蛊虫的标签与选择匹配时，每只+5~15%（如选择"守住防线"且玩家携带防御类蛊虫，tagBonus+10%）

**StatBridge不介入的场景**：

- 纯叙事选择（如"与谁对话"），概率固定为100%
- 有combat_constraint.mustHappen标记的选择（无论概率计算结果如何，故事走向必须发生）
- 叙事战斗中的决斗分支触发（此时切换至决斗引擎，走完整数值系统）

#### 3.5.3 AI动态选择链生成流程

叙事战斗的完整执行流程：

1. **context-builder.ts 注入 combat_constraint**（~300 tokens），包含mustHappen/mustNotHappen/keyFactions/keyNPCs/strategicChoiceCount/narrativeStyle
2. **DeepSeek 根据约束生成**：场景描述（150-300字）+ 2-4个战略选择 + 每个选择的baseChance（50-80%）
3. **PIPE VALIDATING阶段用金丝雀规则校验AI输出**——不能生成mustNotHappen中禁止的选项内容，baseChance必须在50-80%范围内，选项数量必须等于strategicChoiceCount
4. **UI展示**：NarrativePanel显示场景描述 + ChoicePanel以特殊样式显示战略选项（与普通行动选项在视觉上区分——带概率标注和风险等级颜色）
5. **玩家选择后**：StatBridge计算最终概率 → 判定成功/失败 → AI生成结果叙事（成功叙事和失败叙事走向不同，但都需在mustHappen约束内）
6. **结果叙事生成state_update**（flag变更/affinity变化/元石变动/道心变化），不生成HP/真元变化

**设计原则**：叙事战斗中的失败不是"你死了"，而是"战略目标未达成但故事继续"——这与决斗引擎的二元胜负有本质区别。叙事战斗失败可能导致：失去战略位置、NPC受伤或死亡、势力声望损失、错过关键道具或情报，但玩家本人不会因此死亡（除非后续叙事收束到一场必死的决斗）。

#### 3.5.4 叙事战斗中的道心联动规则

叙事战斗中的道心变化与决斗引擎不同——没有"处决敌人"或"使用杀招"这类动作触发的道心变化，而是由**战略选择类型**驱动：

| 战略选择类型 | 道心变化 | 说明 |
|------------|---------|------|
| 选择挺身而出/正面迎敌 | ambition +3~6 | 主动面对危险 |
| 选择守护他人/断后掩护 | mercy +3~6, ambition +2~4 | 牺牲精神与勇气并存 |
| 选择趁乱牟利/浑水摸鱼 | scheme +4~8 | 乱中取利 |
| 选择屠杀/赶尽杀绝 | kill +5~8, mercy -3~5 | 残暴选择 |
| 选择撤退/明哲保身 | scheme +2~4 | 审时度势（非懦弱） |
| 选择背刺盟友/出卖队友 | scheme +8, mercy -10, kill +3 | 极端背叛 |

#### 3.5.5 叙事战斗中的蛊虫饥饿

- 叙事战斗不推进蛊虫饥饿计时（无回合循环，不存在tickGuHunger调用点）
- 但叙事战斗前的场景仍消耗叙述AP，叙述AP消耗过程中正常推进饥饿
- 如果叙事战斗的某个选择触发决斗分支（如"你选择拦截这个七转蛊师，双方对峙——战斗开始"），进入决斗引擎后恢复饥饿计时（每回合正常调用tickGuHunger）

#### 3.5.6 叙事战斗示例：狼潮攻城

以狼潮攻城（南疆青茅山篇）为例展示叙事战斗的完整选择链：

```
第1轮场景描述（AI根据combat_constraint生成）：
"狼群的嚎叫声从四面八方传来，古月山寨的木墙上点燃了火把。白凝冰站在你左侧，眼神冷漠。古月方正握着刀，手心出汗。木墙在第一波冲击下发出了可怕的呻吟——东门的防御是整个山寨最薄弱的点。"

战略选择（AI生成，各带baseChance）：
A. 守住东门——那里是防御最薄弱的点（baseChance 70%）
   成功：东门守住，山寨士气+10，古月方正存活flag=true
   失败：东门被突破，山寨HP-30%，部分族人死亡
B. 去救被困在寨中央的族人（baseChance 60%）
   成功：救出三族老（获得稀有蛊材奖励），亲和+5
   失败：救援中被狼群围攻，触发决斗分支（玩家 vs 3只灰狼，进入duel引擎）
C. 寻找机会趁机搜刮山寨仓库（baseChance 55%）
   成功：元石+50，flag: looted_warehouse
   失败：被白凝冰发现，affinity-15，flag: caught_looting

StatBridge修正（假设玩家一转中阶，携带防御蛊和隐匿蛊）：
- realmDiff: 0（狼潮无特定挑战境界，视为同阶）
- guTagBonus: 选A（防御蛊+10%）→ final 70% × 1.10 = 77%
- guTagBonus: 选C（隐匿蛊+10%）→ final 55% × 1.10 = 60.5%
```

### 3.6 小团战接口预留（Group Combat）

小团战（2-10人规模混战）在P2阶段不实现完整逻辑，但预定义数据结构以确保P3扩展时不需要"凿墙"。

```typescript
interface GroupCombatState {
  combatType: 'group';
  allies: {
    npcId: string;
    hp: number;
    maxHp: number;
    realm: number;
    role: 'tank' | 'dps' | 'support';  // NPC盟友战场角色
    autoDecision: 'aggressive' | 'defensive' | 'supportive'; // NPC自动决策风格
  }[];
  enemies: {
    id: string;
    hp: number;
    maxHp: number;
    realm: number;
    attackPattern: AttackPattern;
  }[];
  formation: 'frontline' | 'flanking' | 'defensive'; // 站位（影响伤害分配）
}
```

P3实现计划：2v2/3v3简化回合制——使用共享行动池（每方1-2AP/回合），NPC盟友根据autoDecision风格自动决策，玩家手动控制自己角色。站位（formation）影响伤害分配和目标选择。

P2兜底方案：当combat_type判定为'group'时，COMBAT_ROUTER跳过任何引擎逻辑，直接由AI生成一段团战叙事描述作为兜底文本，确保体验不中断。

### 3.7 道心-战斗-NPC三角联动规则

现有playerSlice已包含daoHeart四维（kill/mercy/scheme/ambition，范围0-100），state_update管道也已处理道心增量。道心不是孤立的数值系统，而是贯穿战斗选择和NPC交互的深层驱动机制。

**道心四维定义**：

kill（杀心）：战斗中处决敌人、选择致命杀招、默许暴力解决方案时增长。高杀心（>60）使战斗叙事更残暴——AI在生成战斗文本时会加入更暴力的描写，NPC（尤其是正直型）会对此产生态度变化。低杀心（<20）则使玩家在处决场景中感受到叙事阻力（AI会描述内心的犹豫）。

mercy（仁心）：战斗中饶恕敌人、选择非致命手段、在NPC交互中选择理解和宽恕时增长。高仁心（>50）解锁"饶恕"战斗选项——对已被击败的敌人可选择放走而非处决。被饶恕的NPC可能在未来章节中以援助者身份出现（affinity+15，概率触发回报事件）。低仁心（<15）则使NPC（尤其是懦弱型）对玩家产生恐惧（affinity自动-10）。

scheme（谋心）：使用计谋杀招（如陷阱类、伪装类杀招）、在交易中设套、在委托中耍手段时增长。高谋心（>40）解锁"设伏"战斗选项——战斗开始前可消耗1AP设置陷阱，使首回合敌方命中率-20%。高谋心（>70）在情报获取场景中效率+30%（NPC提供的线索更具体）。低谋心（<15）则使阴狠型NPC更可能欺骗玩家。

ambition（野心）：主动介入名场面、在势力晋升中积极表现、在关键决策中选择高风险高回报路径时增长。高野心（>50）解锁"强势介入"名场面选项——不满足于旁观而主动参与核心事件。高野心（>80）使势力晋升速度+50%，但代价是暴露增长+20%（更多人注意到你）。低野心（<20）则使势力内部对玩家的评价为"不思进取"。

**决斗中的道心变化**：

| 战斗行为 | 道心变化 | 触发条件 |
|---------|---------|---------|
| 使用致命杀招击杀敌人 | kill +2~5 | 杀招类型为direct_damage且最终伤害>敌方HP |
| 处决已投降的敌人 | kill +8, mercy -5 | 敌方HP<20%且敌方选择投降 |
| 饶恕被击败的敌人 | mercy +5~10 | 选择饶恕选项 |
| 使用陷阱/欺骗型杀招 | scheme +3~6 | 杀招标签含"trap"/"deception" |
| 在明显劣势下选择继续战斗 | ambition +4~8 | 境界差≥2且选择"继续战斗" |
| 逃跑（非必要） | ambition -3, scheme +2 | 逃跑时敌方境界≤玩家 |
| 逃跑（必要） | 无惩罚 | 逃跑时敌方境界>玩家+2 |

**叙事战斗中的道心变化**（详见3.5.4节）：

| 战略选择类型 | 道心变化 |
|------------|---------|
| 选择挺身而出/正面迎敌 | ambition +3~6 |
| 选择守护他人/断后掩护 | mercy +3~6, ambition +2~4 |
| 选择趁乱牟利/浑水摸鱼 | scheme +4~8 |
| 选择屠杀/赶尽杀绝 | kill +5~8, mercy -3~5 |
| 选择撤退/明哲保身 | scheme +2~4 |
| 选择背刺盟友/出卖队友 | scheme +8, mercy -10, kill +3 |

**NPC交互中的道心变化**：

| 交互行为 | 道心变化 | 触发条件 |
|---------|---------|---------|
| 挑衅NPC后将其击败 | kill +3~5, mercy -3 | 先挑衅后进入战斗并获胜 |
| 深交话题中分享过去 | mercy +3~7（正直型NPC）/ scheme +3~7（阴狠型NPC） | affinity≥40且选择deepen_bond |
| 交易中狠压价格 | scheme +2~5 | 商队后人出身或affinity<20 |
| 接受高难度委托 | ambition +3~6 | 委托难度标记为"hard" |

**道心与势力/NPC反应**：

- 杀心>60且暴露>50：正道势力发出警告（affinity-20），部分正道NPC拒绝交互
- 仁心>70且暴露<30：部分散修NPC主动接近（概率触发援助事件）
- 谋心>60：阴狠型NPC affinity自动+5（视为"同类"），正直型NPC affinity-5（视为"不可信"）
- 野心>80且势力声望>60：势力晋升速度+50%，但势力内竞争对手（同rank NPC）affinity自动-10

**设计原则**：道心不设上限惩罚（高道心≠坏结局），而是开启不同的叙事路径。一个高杀心的玩家可以通过暴力路线达成结局，一个高仁心的玩家可以通过人脉网络达成结局——两种路线都有风险与回报，由玩家风格决定而非系统强制。

### 3.8 蛊虫饥饿与战斗联动规则

现有蛊虫饥饿系统：每回合蛊虫有概率进入饥饿状态，概率按转数递增：一转蛊5%/回合、二转10%/回合、三转15%/回合、四转20%/回合、五转25%/回合。饥饿分为fed→hungry→starving→dying四个状态，每个状态影响蛊虫效果。

**决斗中的饥饿推进**：

战斗回合也是回合——蛊虫在战斗中持续计算饥饿推进。一场10回合的战斗中，一只三转蛊有约80%概率至少推进一级饥饿（1 - 0.85^10 ≈ 80%）。这会产生策略深度——战斗前需要喂养蛊虫。

**关键联动规则**：

1. **杀招核心蛊饥饿则杀招不可用**：如果一个杀招需要的核心蛊（core_gu_requirements中定义的蛊虫）处于starving或dying状态，该杀招在战斗面板中变灰，提示"你的[蛊虫名]过于饥饿，无法驱动[杀招名]"

2. **战斗中达到starving的蛊虫**：在当前回合的杀招/蛊虫辅助使用后立即结算饥饿。如果使用的杀招核心蛊在结算后变为starving，该杀招本回合仍生效，但下回合不可用

3. **战斗结束后统一同步**：战斗中的饥饿变化在COMBAT_DUEL→RESOLVED阶段一并通过DeltaApplier写入guSlice，触发StatusBar的饥饿状态更新

4. **战斗前检测**：COMBAT_DUEL阶段初始化时，检测玩家所有携带蛊虫的饥饿状态，对starving/dying蛊虫给出警告提示："你的[蛊虫名]状态极差！在战斗中可能无法使用相关杀招。"

**叙事战斗中的饥饿**（详见3.5.5节）：

- 叙事战斗不推进蛊虫饥饿计时（无回合循环）
- 叙事战斗前的场景消耗叙述AP，正常推进饥饿
- 叙事战斗触发决斗分支时，进入决斗引擎后恢复饥饿计时

**策略含义**：

这条规则创造了三个有意义的玩家决策点：
- **战前喂养**：进入预期战斗前需要消耗元石购买蛊材喂养核心蛊虫（产生元石消耗）
- **杀招选择**：如果多个杀招依赖同一核心蛊，该蛊饥饿后所有杀招同时失效，促使玩家培养多元化的杀招组合
- **战斗节奏**：长时间战斗对高转蛊虫不利（饥饿概率更高），在持久战中低转杀招可能比高转杀招更可靠

这条规则无需新增代码架构——饥饿推进逻辑已在现有guSlice中（通过advanceTurn→tickGuHunger），只需在COMBAT_DUEL阶段的每回合结算中调用相同的tickGuHunger流程并在杀招选择时加入饥饿检查。

## 第四部分：NPC交互系统设计

### 4.1 可行性评估

**现有基础**: npcs.json（120+NPC完整数据库，含alias/faction/rank/role/relationship/personality/tier/dynamicTitles）+ injectNPCContext()三层过滤系统（按位置/章节/势力过滤→评分排序→取前12注入prompt）+ factionSlice（势力声望+角色关系）+ CharacterRelation类型（affinity -100到100，8种relation_type）。无独立对话系统。

**风险评估**: 低。现有数据层非常完整，交互系统的核心挑战不是"能不能做"而是"怎么做才不破坏现有AI叙事流的连贯性"。NPC交互应作为AI叙事的增强层而非替代层——玩家选择话题后，AI基于话题类型和NPC性格生成回应，而非预设对话树。

**工时估算**: P2阶段6-8天。

### 4.2 话题选项制设计

核心思想：当玩家处于"可与NPC交互"的场景时（同地点、非战斗、NPC可访问），StatusBar或场景叙事中显示可交互NPC列表。选择一个NPC后，展示6类话题选项。

**6类话题定义**:

| 话题类型 | 说明 | affinity影响 | 信息产出 | 限制条件 |
|---------|------|-------------|---------|---------|
| 闲聊 (small_talk) | 天气、近况、八卦 | ±0~3 | 低（可能获得传闻线索） | 无 |
| 请教 (seek_advice) | 蛊道修习、战斗技巧 | +2~5 | 中（可能获得知识/技能） | NPC境界≥玩家境界 |
| 交易 (trade) | 蛊材交换、情报买卖 | ±0~2 | 高（获得蛊材/情报） | NPC有交易意愿 |
| 委托 (request_help) | 请求NPC协助 | -3~5（看结果） | 高（获得战斗/任务协助） | affinity≥20 |
| 挑衅 (provoke) | 激怒NPC | -5~15 | 低（可能触发战斗） | NPC性格非懦弱 |
| 深交 (deepen_bond) | 谈论过去/理想/秘密 | +5~15或-10~20 | 极高（解锁角色故事/隐藏flag） | affinity≥40 |

**NPC性格×话题响应矩阵**（决定AI生成回应的风格基调）:

| NPC性格 | 闲聊回应风格 | 请教回应风格 | 挑衅后果 |
|---------|------------|------------|---------|
| 狂傲 | 不屑一顾，敷衍 | 藐视问题简单 | 极高概率触发战斗 |
| 阴狠 | 话里有话，暗藏机锋 | 给出半真半假的建议 | 表面不动怒，暗中记仇(affinity-20+后续背刺flag) |
| 正直 | 和善回应，关心近况 | 认真指导，倾囊相授 | 严肃批评，但不轻易动手 |
| 贪婪 | 话题总绕到利益 | 收费指导(affinity-1+元石条件) | 权衡利弊，不一定还手 |
| 懦弱 | 小心翼翼，不敢多言 | 谦虚推辞 | 被挑衅后逃跑或求饶 |

### 4.3 轻度感情线设计

感情线不是独立系统（避免GalGame化），而是嵌入在话题交互中的"双轨affinity系统"：

**双轨设计**:
- 普通affinity轨：-100到100，影响NPC对玩家的态度（友善/中立/敌意），所有交互类型都可能变化
- 感情flag轨：独立的三元状态（none/interest/attachment），仅通过deepen_bond话题推进。当affinity达到40+且选择deepen_bond话题时，AI叙事中可能出现感情暗示。当affinity达到70+且感情flag为attachment时，NPC可能主动表达感情

**关键阈值事件**:
- affinity ≥ 20: NPC在随机叙事中出现频率提升
- affinity ≥ 40: NPC可能主动发起闲聊/请教话题
- affinity ≥ 60: NPC可能在战斗中主动协助
- affinity ≤ -30: NPC可能在叙事中成为阻碍
- affinity ≤ -60: NPC可能成为主动敌对者

**设计约束**:
- 感情线不强制、不主线化——没有"攻略NPC"的成就或奖励，纯粹是叙事深度的自然延伸
- 每局最多1-2个NPC能发展到attachment状态（通过叙事自然限制，非硬编码）
- 感情进展速度受章节推进控制——早期章节affinity变化幅度更小

### 4.4 混合任务刷新机制

任务不是"打开任务面板接任务"的传统模式，而是嵌入在NPC交互和势力声望变化中的**混合刷新**：

**三种任务来源**:

| 来源 | 刷新条件 | 任务类型 | 频率 | 奖励 |
|------|---------|---------|------|------|
| NPC委托 | 与该NPC affinity≥10 | 收集蛊材/传递消息/护送/清除威胁 | 每章每NPC最多1次 | 元石+蛊材+affinity+声望 |
| 势力公告 | 该势力声望≥20 | 势力建设/情报收集/外交/讨伐 | 每5-10轮刷新1个 | 元石+势力声望+稀有蛊材 |
| 随机遭遇 | 内容密度检测器触发 | 各种小型遭遇(详见第九部分) | 每3-8轮概率触发 | 微量资源+叙事内容 |

**任务数据模型**:


## 第五部分：经济系统设计

### 5.1 可行性评估

**现有基础**: playerSlice含元石/仙元石字段 + canon/economy.json存在（经济规则定义）+ advanceTurn()每回合推进。

**风险评估**: 低。经济系统是文字RPG中最容易做到"够用"的系统——不需要复杂的供需曲线，但必须有闭环防止通胀。现有代码的DeltaApplier模式已为元石变动提供了成熟的注入点。

**工时估算**: P1数据模型1天 + P2闭环实现4-6天，合计5-7天。

### 5.2 三大闭环设计

**闭环一：元石周转**

元石收入来源：
- 战斗胜利掉落（敌方境界×10-50元石）
- 任务完成奖励（NPC委托30-100元石，势力任务50-200元石）
- 蛊材出售（在商店/宝黄天出售多余蛊材，价格=基础价×稀有度系数×章节通胀系数）
- 成就首次解锁（微量，5-20元石）

元石消费出口：
- 蛊材购买（商店/宝黄天）
- 炼蛊费用（炼蛊消耗蛊材+元石）
- 情报购买（从NPC获取关键信息）
- 贿赂/人情（影响NPC态度或势力声望）
- 治疗费用（重伤后加速恢复）

**闭环二：章节通胀**

```typescript
// economy.json中的章节定价系数

  "青茅山期": 1.0,      // 基准
  "商队期": 1.2,
  "南疆初探": 1.5,

  "王庭福地期": 2.5,

```

所有商品基础价格乘以当前章节的系数。这确保早期章节获得的元石不会在后期变成废纸，同时后期章节的元石收入也相应提高（高境界敌人掉落更多），形成一个自然的通胀-收入匹配。

**闭环三：稀有资源隔离**

某些蛊材/蛊虫不进入通用商店，仅通过特定行为获取：
- 击败特定敌人（如月光蛊仅从月下狼王掉落）
- 完成势力任务（如血战令仅从北原王庭任务获得）
- 名场面参与（如三王传承中获得定仙游残片）
- 隐藏发现（如探访人祖遗迹获得特殊蛊材）

这些稀有资源在economy.json中以`sourceRestriction`字段标记，商店过滤逻辑会排除这些条目。

### 6.1 音效系统设计

#### 6.1.1 可行性评估

**现有基础**: package.json已引入howler@2.2.4（Web Audio API封装库），但无独立音频管理器文件。

**免费音效来源**: Freesound.org（CC0/CC BY许可）+ Pixabay Sound Effects（免费商用）+ Mixkit（免费音效库）。覆盖需求：
- 环境音：山林风声、溪流水声、市集嘈杂、洞窟回音、雷雨声（10-15个音频文件，总大小约5-8MB）
- 战斗音：蛊虫嗡鸣、骨骼碎裂、剑气破空、术法爆炸、逃跑脚步声（8-12个音频文件，总大小约3-5MB）
- UI音：按钮点击、成就解锁、警告提示、转章音效、新消息提示（5-8个音频文件，总大小约1-2MB）
- 背景音乐：lo-fi/古风/氛围音乐（3-5首，通过ccMixter获取CC BY许可曲目，总大小约15-25MB）

总体积约25-40MB。对于web应用不会造成加载问题（分批懒加载）。

**风险评估**: 低。Howler.js已是成熟库，三通道音量控制是标准功能。主要工作量在音频文件的筛选和授权确认。

**工时估算**: P2阶段3-4天。

#### 6.1.2 架构设计

**三通道独立音量**:

```typescript
interface SoundSlice {
  masterVolume: number;          // 主音量 0-1
  bgmVolume: number;             // 背景音乐音量 0-1
  sfxVolume: number;             // 音效音量 0-1
  uiVolume: number;              // UI音效音量 0-1
  bgmMuted: boolean;             // 背景音乐静音
  sfxMuted: boolean;             // 音效静音

```

#### 6.2.2 隐藏成就机制设计（20%比例）

**成就分级**:

|------|------|--------|------|

**成就数据模型**:

```typescript
interface Achievement {
  id: string;
  name: string;
  description: string;            // 解锁后显示
  hint: string;                   // 隐藏成就的提示（如"在某个月夜...")，常规成就为解锁条件描述

  icon: string;                   // 成就图标名称(使用Lucide图标)
}

  type: 'flag' | 'count' | 'threshold' | 'combo';

  value: number | string;
}
```

**20%隐藏成就的判定管理**:
1. achievements.json中定义所有成就（包括隐藏成就），hidden字段控制可见性
2. 成就面板显示时：hidden=true且未解锁→显示"??? + hint提示文本"
3. 隐藏成就解锁时：正常弹出成就通知（名称+描述），之后在面板中正常显示
4. 检测逻辑与常规成就完全相同——隐藏只影响UI展示，不影响条件检测

**微量元石奖励设计**:

- 隐藏成就：20-50元石（稍高，作为探索奖励）

- 成就元石仅首次解锁时发放一次（通过achievementSlice的unlocked列表去重）

**成就画廊（局外跨存档）**:
- 已解锁成就ID列表写入localStorage（独立于存档的Zustand persist）
- 新游戏开始时读取localStorage中的已解锁成就列表，在成就面板中用"跨存档已达成"标记展示
- 成就解锁流程：检测到条件满足→解锁→写入achievementSlice.unlocked→同步写入localStorage
- 这是Roguelite死亡循环的第一层（详见第八部分）

### 7.2 六步FSM引导设计

```
  step_4: 蛊虫初识 — "蛊虫是蛊师最核心的力量..."
```

## 第八部分：Roguelite死亡循环

### 8.1 可行性评估

**评估结论**: 做前两层（成就画廊+起源解锁），不做第三层（天赋点数）。核心原因已在缺口分析中阐明——天赋点数的"死得越多越强"与蛊真人世界的残酷基调根本矛盾。成就画廊和起源解锁是横向扩展（解锁更多起点选项）而非纵向堆叠（数值加成），既保留了死亡的重量，又给了玩家"这局没白玩"的正向反馈。

**风险评估**: 低。成就画廊是纯数据存储（localStorage读写），起源解锁是条件判断+UI选项扩充。技术实现极简。

**工时估算**: P2阶段2-3天。

### 8.2 第一层：成就画廊

**实现方案**:
1. localStorage键：`gu-zhenren-achievement-gallery`，存储格式：`string[]`（已解锁成就ID列表）
2. achievementSlice中每次unlockAchievement时同步写入localStorage
3. 成就面板（AchievementPanel）在显示时读取localStorage，对所有成就标注"局外已达成"（跨存档标记为银色边框+勾号图标，局内达成为金色边框+勾号图标）
4. 新游戏开始时，achievementSlice初始化时读取localStorage，但unlocked数组清空（局内重新解锁），仅galleryUnlocked数组保留localStorage数据
5. **不提供任何局外数值奖励**——成就画廊纯粹是展示和回忆，不影响任何一局游戏的数值

```typescript
interface AchievementSlice {
  unlocked: string[];              // 当前局内已解锁
  progress: Record<string, number>;
  galleryUnlocked: string[];       // 局外跨存档已解锁（从localStorage读取）

  unlockAchievement: (id: string) => void;  // 同时写入unlocked和localStorage
  getGalleryAchievements: () => string[];   // 读取localStorage
}
```

### 8.3 第二层：起源解锁

**设计哲学**: 起源解锁不是"增强"，而是"横向扩展"。解锁的起源选项提供不同的起点体验（不同的初始资源/初始蛊虫/初始NPC关系），但不提供数值优势。所有起源选项在开局时元石总量一致（只是分配方式不同），确保公平。

**可解锁起源示例**:

| 起源名称 | 解锁条件（任意一局达成） | 初始资源差异 | 初始蛊虫差异 | 初始关系差异 |
|---------|----------------------|------------|------------|------------|
| 蛊师世家 | 炼出任意三转蛊 | 元石-20 | 多一只随机一转蛊 | 家族声望+10 |
| 散修浪人 | 跨域旅行过一次 | 元石+10 | 无初始蛊 | 无初始关系网 |
| 商队后人 | 在南疆商队篇完成3个交易 | 元石+30 | 一只运输蛊 | 商心慈初始affinity+10 |
| 战场遗孤 | 在任何战斗中失败过3次 | HP上限+5 | 一只防御蛊 | 北原某部落初始敌意 |
| 人祖信徒 | 在西漠探访过人祖遗迹 | 元石-30 | 一只特殊蛊（随机） | 西漠绿洲声望+15 |

**实现方案**:

```typescript
interface OriginUnlockSlice {
  unlockedOrigins: string[];       // 已解锁的起源ID列表（localStorage持久化）

  checkUnlockCondition: (condition: OriginUnlockCondition, gameState: GameState) => boolean;
  unlockOrigin: (originId: string) => void;
  getAvailableOrigins: () => OriginOption[]; // 基础起源+已解锁起源
}

interface OriginOption {
  id: string;
  name: string;
  description: string;
  unlocked: boolean;               // 是否已解锁
  unlockCondition: string;         // 解锁条件描述（如"在任意一局中炼出三转蛊"）
  initialResources: { yuanStone: number; items?: string[] };
  initialGu?: string[];            // 初始携带蛊虫ID列表
  initialRelations?: { npcId: string; affinity: number }[];
}
```

**起源解锁流程**:
1. 游戏过程中，在关键检测点（炼蛊成功/跨域完成/交易达成/战斗失败）调用checkUnlockCondition
2. 条件满足时弹出通知："你已解锁新起源：蛊师世家"
3. 写入localStorage（独立于存档）
4. 下一局新游戏时，origin_select界面中该起源显示为可用（带"新解锁"标记）
5. 已解锁起源永久可用，不受存档删除影响

**P1预留**: originUnlockSlice接口定义 + localStorage键`gu-zhenren-unlocked-origins`约定 + 5个初始起源条件定义在canon/origins.json中

### 8.4 为什么不做第三层（天赋点数）

专业评估理由：天赋点数的核心问题是"死得越多越强"与IP基调的根本冲突。蛊真人世界的核心体验之一就是"生命脆弱、每一步都可能致命"——如果玩家知道"反正死了有天赋点"，死亡的冲击力就消失了，游戏的核心张力也就崩溃了。成就画廊和起源解锁不提供数值优势，只提供"这一次我换个方式试试"的横向扩展，这与IP的残酷基调完全兼容。

### 8.5 分阶段实施清单

**P1（不单独计工时，嵌入P1-3成就Slice预留）**: originUnlockSlice接口定义 + origins.json初始数据 + localStorage键约定

**P2（2-3天）**: 
- 第1天: 成就画廊localStorage读写实现 + AchievementPanel局外标记展示
- 第2天: originUnlockSlice实现（条件检测+localStorage持久化）+ origin_select界面解锁标记
- 第3天: 5个起源选项联调 + 死亡后起源解锁检测（在game_over触发时运行checkUnlockCondition）

**P3（不单独计工时）**: 根据P2玩家反馈增加新的可解锁起源

## 第九部分：随机遭遇/内容密度填充系统

### 9.1 可行性评估

**评估结论**: 必须做。现有评估文档第八节已识别"事件干旱区间"风险，随机遭遇是解决这一问题的系统方案。采用模板化而非AI动态生成的策略——每条遭遇5秒内完成拼装，不调用AI。

**风险评估**: 低。系统本质是一个带权重的随机模板选择器+场景文本拼装器。不引入新的复杂度。

**工时估算**: P2阶段3-4天。

### 9.2 架构设计

**核心原则**: 随机遭遇不抢戏——它是名场面之间的填充物，不是另一种名场面。每条遭遇应控制在2-4句文本（约100-200字），附带0-1个简单选择。

**模板分层**:

| 层次 | 内容 | 数量 | 示例 |
|------|------|------|------|
| 场景模板 | 地点+氛围+时间 | 30-40条 | "月夜竹林，竹叶沙沙作响..." |
| 遭遇类型 | 5类 | — | 蛊材发现/野外战斗/NPC偶遇/天气事件/传闻获取 |
| 稀有度 | 3层 | — | 70%常见/20%罕见/10%稀有 |

**五类遭遇详解**:

**1. 蛊材发现（~30%触发率）**
常见："路边的石缝中闪烁着微光——你发现了一株常见的蛊材。"（获得普通蛊材1-2个）
罕见："悬崖边缘，一株散发幽香的百年灵芝引起了你的注意。"（获得稀有蛊材1个）
稀有："古老石碑下埋藏着一截枯木，上面寄生着肉眼难见的虫卵——这是一窝野生的蛊虫胚胎。"（获得野生蛊虫1只）

**2. 野外战斗（~25%触发率）**
常见："一只饥饿的野狼从灌木丛中蹿出，龇牙咧嘴地朝你扑来。"（低境界野兽战斗）
罕见："密林深处，一只三转境界的毒蟒盘踞在古树上，它似乎把你当成了猎物。"（中等境界蛮兽战斗）
稀有："前方的山洞中传来蛊虫的嗡鸣——一只野生蛊虫正在脱壳转生，此刻是捕获它的最佳时机。"（蛊虫捕获机会，不一定是战斗）

**3. NPC偶遇（~20%触发率）**
常见："商道上，一位行色匆匆的商贩向你打听前方的路况。"（随机NPC交互，可能触发小交易）
罕见："一位同境界的蛊师在山路上与你擦肩而过，他的目光在你身上停留了片刻。"（可能触发切磋/情报交换）
稀有："一个浑身是血的蛊师跌跌撞撞地从前方跑来，他的背后似乎有什么可怕的东西在追赶。"（可能触发连锁事件/隐藏任务）

**4. 天气事件（~15%触发率）**
常见："天空突然阴沉下来，一场暴雨即将来临。你需要在雨势变大前找到避雨之处。"（消耗额外AP用于寻找避雨处，或承担淋雨造成的小额HP损失）
罕见："远方的天际传来低沉的雷鸣，空气中弥漫着不安的气息——这不是普通的雷雨，而是蛊虫群的迁徙前兆。"（可能触发蛊虫潮事件）
稀有："月全食发生了。月光蛊和所有月属性蛊虫在这一刻变得异常活跃。"（月属性蛊虫效果临时翻倍，持续5轮）

**5. 传闻获取（~10%触发率）**
常见："酒馆里，几个蛊师正在议论南疆最近的大事。"（获得一条域内传闻，可能关联到名场面线索）
罕见："一个神秘的老者低声告诉你，北方的王庭最近有异动。"（获得跨域传闻，标记为待验证情报）
稀有："一封匿名信出现在你的行囊中，上面写着某个古老遗迹的坐标。"（解锁隐藏地点/特殊机缘线索）

### 9.3 触发机制

```typescript
interface EncounterSystem {
  // 在advanceTurn后、AI叙事生成前调用
  checkEncounterTrigger: (state: GameState) => Encounter | null;

  // 触发概率计算
  calculateTriggerChance: (state: GameState) => number;
  // 基准40%，但有激活名场面时降为0%（不抢戏），连续5轮无遭遇+20%
}

// 触发概率动态调整
function calculateTriggerChance(state: GameState): number {
  let chance = 40;  // 基准40%
  // 如果有激活的名场面→不触发随机遭遇
  if (state.chapter.activeEvents.length > 0) return 0;
  // 如果本章已有名场面即将触发(proximity≤1)→降概率
  if (hasNearbyCanonScene(state)) chance -= 20;
  // 如果连续多轮无遭遇→概率递增(+10% per 2 rounds without encounter)
  chance += Math.floor(state.roundsWithoutEncounter / 2) * 10;
  return Math.min(chance, 80);  // 上限80%
}
```

```typescript
interface DensityCheckResult {
```

### 10.1 设计原则声明

本文档只做**架构级设计参考**，不做像素级UI规范。核心产出三张图：屏幕状态转换图、面板组件树、信息优先级分层。P1可维持当前UI，P2新增系统时按本文档的信息架构原则放置新面板。

### 10.4 信息优先级分层

**第一优先级（始终可见）**: HP、真元、AP、当前章节名+域、元石数量。这些信息在StatusBar第一行展示，字号14px，颜色使用主对比色。

**第二优先级（常驻但收敛）**: 暴露、债务、伤势。在StatusBar第二行展示，字号12px，仅在数值>0时显示对应图标+数值，数值为0时隐藏以减少信息噪音。

**第三优先级（按需展开）**: 成就进度、任务列表、NPC关系摘要、蛊虫状态。不进入StatusBar，在各自独立面板中间隔展示。玩家通过StatusBar的按钮或快捷键呼出这些面板。

**第四优先级（事件触发）**: 战斗浮层、NPC对话面板、商店、转章弹窗、成就通知。这些面板临时覆盖在GameScreen之上，交互结束后自动消失。

**信息密度控制原则**:
- StatusBar总占用不超过屏幕高度15%
- 第一优先级不超过5个数据点
- 第二优先级不超过3个数据点（且0值隐藏）
- 同一时刻最多1个第四优先级面板覆盖

### 13.1 存档架构概述

### 13.6 存档质量门禁

每个迭代结束时必须验证：

1. **新游戏无污染**：从 Title 点击开始 → 角色创建 → 游戏中，store 中无任何旧存档残留数据
2. **向后兼容**：上一个版本的存档文件可正常导入读取（migrate 函数正确填充默认值）
3. **无损往返**：存档 → 导出文件 → 导入文件 → 状态完全一致（排除 non-serializable 字段）
4. **死亡→重置**：GameOver → 重入轮回 → store 全量清零，persist 写入干净状态
5. **persist partialize**：新增 slice 状态确认在 partialize 白名单中，不会丢失

## 第十二部分：十绝体类型差异化评估与设计

### 12.1 现有实现评估

**当前状态**：项目仅将"十绝体"视为一个二元类别——资质=10，其余属性保底≥8。在terminology.json中仅定义为"十成真元，空窍完全充满元海"，系统提示词中描述为通用模板。白凝冰（北冥冰魄体）和狂蛮魔尊（大力真武体）的十绝体类型仅在NPC的canonicalNotes中以文本注释形式存在，CharacterCreate.tsx中任何十绝体玩家角色均无类型区分。

**问题严重性**：高。原著中十绝体不仅仅是"资质更好"，每种十绝体都有独特的道痕亲和、修行倾向、体质特征和致命缺陷。不区分类型意味着：(a) 十绝体玩家的游戏体验千篇一律，与原著"每种十绝体都是独特的存在"严重矛盾；(b) NPC白凝冰的北冥冰魄体沦为空洞标签，叙事AI无法利用其冰属性特质生成有差异的互动文本；(c) 道痕体系的核心概念（不同流派依赖不同道痕亲和）在十绝体身上无法体现。

### 12.3 游戏机制差异化方案

**方案一：道痕亲和矩阵（推荐，P2实施）**

在CharacterCreate.tsx的十绝体分支中，根据roll到的十绝体类型，修改talentSlice中的道痕亲和（DaoMarkAffinity）初始值：


**方案二：特有灾劫类型（P2实施）**

在升仙灾劫系统（三气升仙逻辑）中，每种十绝体触发特有灾劫类型而非通用灾劫。例如北冥冰魄体升仙时触发"冰魄反噬劫"（极寒侵袭自身空窍），大力真武体触发"力溃劫"（肉身力量失控自毁），纯梦求真体触发"梦魇劫"（被自身梦境吞噬）。灾劫类型决定AI叙事中的升仙场景描写和成功条件。

**方案三：剧情影响权重（P2-P3实施）**

每种十绝体对全局叙事的影响体现在：(a) NPC反应——拥有同流派亲和NPC在初见时affinity+10（如紫山真君对太古星光体持有者），拥有相克流派NPC可能有敌意；(b) 名场面介入方式——北冥冰魄体在冰原类场景中有独特选项，大力真武体在蛮力挑战中有独特解法；(c) 势力偏好——中洲古板派对纯梦求真体持有者极端排斥（视为"不稳定因素"）。

### 12.4 角色创建界面的十绝体选择

**P2实施**：在CharacterCreate.tsx中，当roll到资质=10时，不直接进入通用十绝体描述，而是展示10种十绝体选项供玩家选择（或不选择——随机十绝体）。每种十绝体展示：名称、核心流派、体质特征简述、相应缺陷提示（但不暴露具体数值）、剧情风格预告（如"北冥冰魄体：极寒之路，冷静而危险"）。

**十绝体对天赋点的消耗**：十绝体消耗10点资质点，仅剩0-2点可用于选择额外天赋（当前CharacterCreate已有限制"天赋余裕所剩无几——仅能选基础天赋"），此限制保持合理。

### 13.3 入库策略

**策略一：保持现有数据文件格式，不做结构重构**

项目现有canon文件使用JSON格式，NPC数据文件结构清晰可扩展。所有新入库数据沿用现有JSON格式放入对应文件（如新增凡蛊直接追加到gu-database.json），不需要建新文件或改变数据结构。这样避免P1-P2过渡期的数据迁移风险。

**策略二：先补足"战斗可见"数据，再补"叙事可见"数据**

战斗系统在P2是核心交付物。因此优先入库战斗直接使用的数据：杀招（killer-moves.json battleProperties填充）、荒兽（wild-beasts.json具体条目）、凡蛊（gu-database.json扩展）。战斗可运行后再补叙事相关数据（阵道/仙蛊屋）。

**策略三：利用NPC数据入库的经验**——统一字段规范

NPC数据的成功在于统一字段（alias/faction/rank/role/relationship/personality/tier）和严格原著考证。凡蛊、杀招、荒兽的入库也应遵循相同模式：

```
凡蛊入库规范：{ id, name, rank_band[1-5], path_tags[], role_tags[], rarity, 
  domain_origin, description, feeding_cost, refines_to[], appears_in_chapters[] }
  
杀招入库规范：{ id, name, required_realm, core_gu[], optional_gu[], battle_type, 
  accuracy, baseDmg, dmgMultiplier, costType, costValue, cooldown, 
  path_tags[], appears_in_chapters[] }
  
荒兽入库规范：{ id, name, rank, type[普通/荒兽/上古荒兽/太古荒兽], 
  domain, habitat, abilities[], dropTable, appears_in_chapters[] }
```

**策略四：不追求"一万条数据"而是"每条数据都有用"**

这与NPC入库时的策略一致——120个NPC而非500个，但每个都经过原著验证且有游戏内交互。凡蛊扩展至35-40种、杀招扩展至30个、荒兽扩展至15-20条，每条都有使用场景和战斗交互，不堆无用条目。

## 第十四部分：空窍系统设定修正与表现层重构

### 14.1 现状评估

**已有的正确设定**：

空窍容量映射 `{1:3, 2:5, 3:8, 4:12, 5:15}` 已正确实现在 `playerSlice.ts`、`world-rules.json`、`economy.json`、`context-builder.ts` 等多处，甲等-丁等资质分级正确限定于空窍真元容量评价。terminology.json 中"空窍：蛊师体内的空间，容纳蛊虫和真元"的定义是正确的。

**存在的三处问题**：

问题一（AI叙事质量）：系统提示词中未明确约束"蛊虫存放于空窍内"，导致 DeepSeek 有时生成"腰间挂着竹制蛊虫笼"等与原著世界设定冲突的物理携带描述。蛊界蛊师可将蛊虫收入空窍——空窍是蛊师体内的独立空间，蛊虫在本命蛊之外通过空窍进出，不存在腰间显式挂载蛊虫的设定。只有凡人（未开窍）或极特殊情况才会用外部容器携带蛊虫。

问题二（UI功能错误）：`AperturePanel.tsx` 当前仅处理 `ImmortalAperture`（六转以上的仙窍=福地/洞天）。当 `aperture === null` 时，面板显示"尚未开辟空窍"。但角色创建后玩家即为"一转蛊师"，空窍已经开辟（开窍大典已通过）。目前没有"蛊师阶段空窍"的独立类型——`types/index.ts` 中只有 `ImmortalAperture` 接口，`immortalSlice.ts` 中 `aperture` 初始值为 `null`。玩家从一转开始游戏，空窍状态面板却始终显示"尚未开辟"，直到六转升仙才可见——严重破坏沉浸感。

问题三（表现层缺失）：原著中不同转数的空窍元海有明确的颜色区分，同一转数内不同小境界的窍壁也有不同的物理状态，这些视觉反馈在游戏中完全缺失。

### 14.2 原著空窍设定（核心参考）

**空窍基本结构**（原著约第1-200章）：

空窍呈球形，位于蛊师体内胸腔位置。开窍时希望蛊炸开，在空窍表面留下一层光膜支撑窍壁不塌陷。空窍内部下方为元海（真元液体聚集处），元海之上为空腔区域，蛊虫可在空窍内活动。蛊师晋升时元海扩大、光膜变薄——初阶时光膜厚实，巅峰时几近透明，突破后旧窍壁碎裂重组新窍壁。

| 转数 | 真元名称 | 元海颜色 | 色值参考 | 初阶-巅峰色泽渐变 |
|------|---------|---------|---------|----------------|

| 五转 | 紫晶真元 | 紫色系（淡紫→深紫） | #a6d → #624 | 初阶浅淡紫，巅峰深邃紫晶 |

### 15.1 审计概述

本节基于对 RebornG 项目 v0.5.1 源代码的全面扫描，结合原著（蛊真人/ Reverend Insanity）设定和公开 wiki 信息，识别出以下五类需要在大纲中补充的缺口。

### 15.2 五域文化差异与势力体系（大纲缺失）

当前大纲的第二章仅以列表形式区分五域叙事风格（南疆家族阴谋→北原部落生存→东海资源争夺→西漠遗迹探索→中洲秩序维护），但缺乏对各域蛊师文化、势力结构和特色流派的系统性定义。

**原著设定**（需纳入设计）：

**南疆**：十万大山，家族制度。一转后勤、二转中坚、三转家老、四转/五转族长、六转+为太上长老。人口密度五域最高，蛊师数量庞大。以家族血脉传承蛊虫。曾出过乐土仙尊（土/音/阵/天道）、幽魂魔尊（魂道开创者）两位尊者。武家为南疆第一超级势力（风道）。文化特点：重血缘纽带、蛊虫多为虫/兽/木系、凡蛊基数最大。

**北原**：草原王庭，部落制度。黄金血脉为核心（巨阳仙尊后裔），以实力争霸。资源五域最贫瘠，蛊仙财力垫底，但战力五域第一、最好斗悍勇。曾出过狂蛮魔尊（力道开创者）、巨阳仙尊（血/阳道）两位尊者。长生天为北原圣地。文化特点：弱肉强食、逐水草而居、战斗蛊虫发达。

**东海**：散修天堂，海岛文化。无统一政权，多散修群体。资源以水/气/光道为主，海兽/荒兽资源丰富。蛊仙数量五域最多（无统一势力压制）。曾出过海神（水道开创者）一位尊者。文化特点：自由散漫、航海探索、水生蛊虫。

**西漠**：绿洲城邦，商路文化。以绿洲为中心分布城邦国家，商贸网络发达。资源集中在绿洲和遗迹中。萧家为西漠超级势力（三十年河东三十年河西）。曾出过元莲仙尊（木/生道开创者）。文化特点：沙漠生存法则、遗迹猎宝、驼队贸易。

**中洲**：中央正统，门派制度。十大古派（仙鹤门等）为正道核心。天庭为最高圣地（历代尊者的道统汇聚处），底蕴最厚，综合实力第一。灵缘斋为首的古派垄断正道话语权。曾出过元始仙尊（天道开创者）、星宿仙尊（智道开创者）两位尊者。文化特点：正道规则森严、升仙考核、魔道诛杀令。

**五域界壁**：各域之间有天然界壁阻隔——南疆瘴气界壁、北原甘草界壁、东海风暴界壁、西漠狂沙界壁、中洲圣贤界壁。凡蛊级蛊师无法跨域，只有蛊仙级才能跨越界壁。跨域远征是顶级势力的战略行为。

**需在新章节「五域势力与蛊师文化体系设计」中补充**。

### 15.4 真元质量/耐用比系统（完全缺失）

**原著设定**：小境界间真元耐用比约 1:2（如初阶与巅峰），大境界间约 1:3（如一转巅峰与二转初阶）。高转真元驱动低转蛊虫消耗极低——二转蛊师使用一转杀招时真元消耗仅为同转杀招的 1/3。

**代码现状**：`types/index.ts` 中 `essence: { current: number; max: number }` 仅有数值无质量维度。真元上限对所有转数统一为 100（参见 `realmGrowthTable`），无法体现"二转真元远超一转"的原著设定。

**所需补充**：在 `vitals.essence` 中增加 `quality: number` 字段（1-5映射转数），在蛊虫催动/杀招使用时乘以 `quality` 修正系数计算实际消耗。在 `economy.json` 中表未实现的元石充能递减规则（一转+100、二转+70、三转+40、四转+20、五转+10）。

### 15.5 修行/突破系统补完

**现有基础**：`BreakthroughAnimation.tsx` 组件存在但从未被触发，`advanceTurn` 方法不包含修行进度。

**严重缺失**：

1. **无修行(cultivate)动作** — 玩家无法主动消耗 AP 积累修行进度
2. **境界晋升仅靠 AI 叙事 set** — `state_update.realm` 是唯一触发路径，无游戏内机制
3. **六转及以上境界零覆盖** — `realmMap` 只到五转巅峰
4. **无瓶颈机制** — 初→中→高→巅峰→跨转应有瓶颈判定
5. **无升仙机制** — 五转巅峰→六转需要"三气升仙"（world-rules.json 有描述但零实现）
6. **道痕不参与突破** — 原著中道痕是突破关键因素，代码中零关联

**设计方案**（需新增在修行系统设计章节）：引入 `cultivationProgress` 进度条（每轮修行消耗 1AP + 消耗真元，根据资质修正效率）。瓶颈判定公式：`breakthroughDifficulty = baseDifficulty + daoMarkPenalty`。升仙需同时满足三气条件：天气（天劫淬体）、地气（福地/洞天雏形）、人气（道心/势力/机缘）。

### 15.6 蛊虫反噬机制（零实现）

`gu-database.json` 中每个蛊虫的 `usageFailure` 字段详细描述了反噬后果，但**代码中完全没有实现任何反噬逻辑**：

| 缺失项 | 说明 |

| 使用失败反噬 | `guSlice.ts` 无 backlash 方法 |

| 本命蛊死亡反噬 | 最严重的反噬——本命蛊死亡会导致蛊师重伤甚至死亡 |

## 第十六部分：存档恢复机制——方案A（本地自动恢复）

### 16.1 设计目标

玩家刷新页面或关闭浏览器后重新打开时，应能一键返回上次离开时的游戏状态，而非强制从标题画面重新开始。方案A利用现有 Zustand persist 在 localStorage 中已持久化的完整游戏数据（turn/profile/messages/vitals/inventory/flags 等），在标题画面增加"继续冒险"入口。

### 16.2 用户流程

```
浏览器刷新 / 重新打开
  → Zustand persist 从 localStorage 自动恢复游戏数据（turn=12, profile.name='方源', messages=[...], vitals=...）
  → screenState 默认为 'title'（从 persist 中排除，确保不跳过标题画面）
  
标题画面:
  ┌──────────────────────┐
  │    蛊 真 人 模 拟 器   │
  │                      │
  │  [ 开始新冒险 ]       │  ← 始终可见
  │  [ 继续冒险 ]         │  ← 仅当 turn > 1 且 profile.name 非空时显示
  │                      │
  │  继续冒险提示：        │
  │  第12回合 · 一转初阶   │
  │  南疆 · 古月山寨       │
  └──────────────────────┘

点击 [ 开始新冒险 ] → 走现有流程（resetStore → mode_select → origin_select → character_create → game_play）
点击 [ 继续冒险 ]   → 直接 goTo('game_play')，startGame() 传入 isOpening=false
```

### 16.3 技术实现

**TitleScreen.tsx 改动**（约30行）：

```typescript

const hasSavedGame = useStore(s => s.turn > 1 && (s.profile?.name?.length ?? 0) > 0);
const saveTurn = useStore(s => s.turn);

  // 不调用 resetStore——保留已恢复的游戏状态

};
```

**useGamePipeline.ts 改动**（约25行）：

`startGame` 增加 `isResume` 参数：

```typescript

  pipeline.reset();
  syncState('BUILDING_CONTEXT');
  const result = await pipeline.process(null, !isResume); 
  // isOpening = !isResume: 续档时 isOpening=false，使用标准系统提示词而非SYSTEM_PROMPT_OPENING

```

### 16.4 续档时的AI行为差异

| 维度 | isOpening=true（开局） | isOpening=false（续档） |
|------|----------------------|------------------------|

续档时 AI 看到完整历史上下文 + 当前状态，会自然生成"继续"叙事——角色可能正在休息后醒来、从一个地点走向另一个、或者正在进行上次未完成的对话。

### 16.5 首屏体验处理

### 16.6 安全考虑

## 第十七部分：v1.4补完方案的游戏性/沉浸感平衡评估

### 17.1 评估原则

蛊真人模拟器的核心产品形态是 **AI驱动文字叙事RPG**。玩家核心体验来自"在蛊界中做出选择→AI生成后果→世界响应"的叙事循环。任何数值系统的引入必须满足两个标准：

1. **增强叙事权重** — 数值应服务于叙事选择，而非取代叙事选择。如果玩家开始"计算最优解"而非"代入角色做出选择"，则数值系统越界了。
2. **不打断叙事流** — 数值操作（修行、喂养、采集）应作为叙事轮次中自然发生的动作，而非独立的"菜单→操作→确认"子流程。

以下对 v1.4 审计识别的 10 类缺口逐一评估是否可能影响沉浸感与游戏性的平衡。

### 17.2 逐项评估

#### 项目1：蛊虫数据库去重（纯数据清洗）— 无影响

修复8处重复/矛盾条目。不影响任何游戏机制或叙事生成——只是确保AI和UI引用的数据一致。**建议直接执行，无平衡问题。**

#### 项目2：真元质量/耐用比系统 — ⚠️ 需谨慎设计

**风险**：引入 `essence.quality` 和消耗修正系数后，玩家可能需要在每个回合计算"这只蛊虫消耗多少真元、我还有多少真元可用"——这将叙事游戏拉向资源管理模拟。

**平衡方案**：
- **表现层隐藏数值**：真元消耗修正作为AI上下文中的隐性规则，AI在生成选择时自动考虑玩家的真元余量。如果真元不足，AI在中风险/高风险选项的 `risk_note` 中提示"真元可能不够"。不暴露具体消耗数字给玩家。
- **StatusBar 轻量提示**：不添加真元数值条，而是用文字描述："真元充沛"（>70%）、"真元尚可"（30-70%）、"真元见底"（<30%）。此设计已在 StatusBar 信息优先级体系中预留。
- **不新增独立UI面板**：真元质量/消耗计算完全在后端完成，前端仅感知HP和AP变化。

**不确定点**：真元质量的数值层（`quality × baseCost`）是否需要暴露给玩家？如果完全不暴露，玩家可能不理解为什么同样动作在不同转数时消耗不同。**建议：先实现全隐藏版本，P2后期根据玩家反馈决定是否增加轻量展示。**

#### 项目3：修行/突破系统 — ⚠️⚠️ 最高风险

**风险**：这是本次所有补完方案中**最容易破坏沉浸感的一项**。如果玩家需要"点修行按钮→看进度条→等回合一到→再点一次"，叙事游戏立刻变成挂机游戏。文字游戏的核心是"选择与后果"，而非"重复劳作与数值积累"。

**平衡方案**：
- **修行融入叙事选项**：不新增独立的"修行"按钮。而是在每轮叙事选项中自然融入修行类选择（如"你决定在石室中静坐修行一个时辰"），选中后 AI 根据资质和真元计算修行进度，在后续叙事中反馈结果（"你感到空窍微微震动，元海潮汐似乎更活跃了几分——距离中阶又近了一步"）。
- **被动修行背景进度**：在 `advanceTurn` 中自动推进微量修行进度（受资质修正），让不刻意选择的玩家也能随时间自然成长。进度积累到阈值时触发 `BreakthroughAnimation`。
- **瓶颈融合叙事冲突**：突破瓶颈时不是"进度条卡住"，而是 AI 在叙事中描述瓶颈现象（"元海狂暴，窍壁欲碎不碎"），玩家需要通过特定选择（寻找机缘、炼制辅助蛊虫、向NPC长老请教）来突破——将瓶颈转化为叙事驱动力。
- **不新增进度条UI**：修行进度仅作为内部状态存在，外部表现为叙事文本而非数值显示。

**不确定点**：修行进度是否需要可视化？我个人倾向"完全不显示"——让境界突破成为突如其来的叙事事件而非可预期的进度条事件，符合蛊界"突破皆在生死之间"的基调。**但这可能让部分玩家感到"不知道自己在进步"——需要你确认方向。**

#### 项目4：反噬机制 — ⚠️ 中等风险

**风险**：过度惩罚会导致玩家不敢使用蛊虫，选择保守路线——这与蛊界"冒险才有收获"的基调冲突。

**平衡方案**：
- **分层反噬**：轻反噬（HP-5~10，叙事中描述"虎口微麻"）→无UI提示；中反噬（HP-20~30，"空窍剧震"）→NarrativePanel 底部轻量警示；重反噬（本命蛊死亡、"空窍裂痕"）→触发特殊叙事事件，不直接致人死亡
- **反噬融入叙事链**：不弹出"反噬：-15HP"对话框，而是 AI 在生成叙事时自然描述反噬后果——"蛊虫反噬的剧痛让你单膝跪地，空窍中的元海掀起惊涛骇浪"——HP减少在 state_update 中静默处理
- **反噬概率与蛊虫饥饿挂钩**：饥饿状态越差的蛊虫，使用时反噬概率越高。这让喂养系统有了战略意义

**不确定点**：重反噬（本命蛊死亡）是否应作为可能的结局路径？原著中确实存在蛊师因本命蛊死亡而丧命的情节。**建议：P2阶段仅实现轻/中反噬，重反噬和本命蛊死亡留P3——这需要与死亡检测系统联动，不应草率上线。**

#### 项目5：蛊材数据库 — 低风险，纯数据补完

不影响游戏机制。`canon/materials.json` 定义为结构化数据后，AI在炼蛊相关叙事中能引用真实蛊材名称（而非"通用蛊材"占位符），提升叙事真实感。**建议直接执行。**

#### 项目6：timeline.json 名场面扩展 — 无风险

纯数据/设计文档补完，不涉及代码逻辑变更。**建议直接执行。**

#### 项目7：五域文化差异注入AI提示词 — 正面增益

在AI系统提示词中注入各域文化特色（南疆家族制/北原部落制/东海散修/西漠商贸/中洲门派）只会让AI生成更真实、多样的叙事，不增加任何操作负担。**建议直接执行。**

#### 项目8：NPC数据补全 — 低风险

`personality`字段补全和缺失NPC入库不影响游戏机制，仅丰富AI叙事素材。**建议直接执行。**

#### 项目9：凡蛊补完 — 低风险

新增10种凡蛊条目不影响游戏机制——它们需要通过机缘/炼蛊/交易获得，AI可以引用它们丰富叙事。**建议直接执行。**

### 17.3 最终决策（已确认）

| 项目 | 决策 | 方案细节 |
|------|------|---------|
| 真元展示 | **5段电池图标（模糊）** | StatusBar 显示 5 个空/实方块（█░░░░），无数字标注。悬停时显示文字等级（"充沛/尚可/不足/见底"），不显示百分比。5段粒度足以支撑策略判断但无法反推精确数值 |
| 修行展示 | **零UI + AI叙事暗示** | 不给任何修行进度UI。AI根据内部进度值在叙事中自然提及修行状态——"元海平静如镜"→"窍壁水珠凝结"→"元海翻涌，突破契机在即"。突破作为突发的叙事事件，不可预期不可计算 |
| 修行机制 | **被动背景 + 叙事融入** | `advanceTurn` 中自动推进微量进度（受资质修正）。叙事选项中自然出现修行类选择（"你盘膝修行一个时辰"），AI描述其效果。不设独立的"修行"按钮 |
| P2反噬范围 | **仅轻中反噬** | 轻（HP-5~10，叙事描述）→中（HP-20~30，轻量警示）→无反噬UI弹窗。重反噬和本命蛊死亡留P3，需与死亡检测系统联动 |

### 17.4 总体评估结论

v1.4 审计识别的 10 类缺口中，**5 类可安全推进**（蛊虫去重/蛊材数据库/timeline扩展/五域文化/NPC补完），**4 类按上述决策推进**（真元质量：5段电池图/修行：AI叙事暗示/反噬：仅轻中/凡蛊补完），**0 类应当放弃**。核心设计原则：所有数值系统作为叙事引擎的"物理规则"存在——真元"是否够用"是合理策略信息、修行"是否进展"通过阅读感知——文本游戏中"感觉"比"数字"更重。

## 第十八部分：流派/道痕/本命蛊补充设计

### 18.1 流派系统（Path System）

**原著设定**：蛊师修何种流派（力道/炎道/水道/魂道/变化道/……）由空窍中积累的道痕类型决定。道痕数量决定杀招威力、蛊虫亲和度和互斥程度。流派之间有互斥关系（如冰火互斥、光暗互斥）。

**当前代码状态**：`playerSlice.pathBuild` 定义了 primary/secondary/path_levels/dao_marks 结构；`pathSlice` 另有 primaryPath/secondaryPaths/pathLevels/daoMarks（两套平行结构）；`world-rules.json` 有"流派互斥"条目。游戏内无流派选择流程——玩家开局后流派由 AI 叙事自然决定。

**设计决策**：流派不走硬性选择界面。AI 根据玩家叙事选择（炼什么蛊、修什么道）在 `state_update` 中自动写入 `daoMarks`。互斥规则作为 AI 上下文的隐性约束——如果玩家在炎道积累了大量道痕但尝试炼冰道蛊虫，AI在叙事中描述"冰火不相容"的困难，降低成功率。

**P2实现**：统一 playerSlice 和 pathSlice 的道痕数据结构（二选一，废弃冗余的那个）。在 `context-builder.ts` 中注入玩家当前道痕分布（前三个最高流派），AI据此生成流派相关的叙事选择和风险提示。

### 18.2 道痕机制（Dao Mark Mechanic）

道痕是蛊修体系的核心物理规则。每条道痕是一个"不可逆的记录"——一旦获得不会消失。道痕数量影响：
- 同流派蛊虫亲和度（道痕越多→成功率越高→消耗越低）
- 互斥流派蛊虫惩罚（冰道痕>30时使用炎道蛊虫有额外反噬概率）
- 杀招威力（与流派匹配时伤害系数×1.2~2.0；每一百道痕增加一成杀招威力——原著MC百科确认的量化规则）

**P2实现范围**：道痕作为AI叙事的隐性规则，不向玩家暴露数值。AI在生成选择时根据道痕分布调整选项的 `risk_note` 和成功率。`DaoMarkPanel.tsx`（现有组件）仅在玩家主动打开时显示前三个流派的相对强弱（文字描述，无数字）。P3扩展至蛊仙阶段的道痕量变效应。

### 18.3 本命蛊机制（Lifebound Gu）

**原著设定**：蛊师可绑定一只蛊虫为本命蛊。本命蛊随蛊师境界同步成长（一转凡蛊→六转仙蛊）。本命蛊死亡则蛊师重伤或死亡。绑定后不可更换（除非用极其罕见的蛊虫）。

**P2实现范围**：在 `guSlice` 中新增 `lifeboundGuId: string | null` 字段和 `bindLifeboundGu(guId)` 方法。绑定条件：蛊虫状态为 optimal、蛊虫转数≤蛊师转数、空窍中有空位。本命蛊加成：催动消耗减半、不受饥饿机制影响、反噬概率降低50%。本命蛊死亡触发"濒死"状态（HP=1，触发特殊叙事事件），不直接致死——符合第十七部分确认的"P2仅轻中反噬"原则。

### 18.4 蛊虫食物链（Gu Feeding Ecology）

**原著设定**：蛊虫需要特定食物——月光蛊需月光、酒虫需美酒、金蚕蛊需金叶桑叶。不是所有蛊虫都能用元石喂养。错误喂食或长期饥饿导致蛊虫衰弱/反噬/死亡。

**P2实现范围**：在 `gu-database.json` 的每条蛊虫数据中新增 `feedRequirement: { type: string; rarity: 'common' | 'uncommon' | 'rare'; description: string }` 字段。`MerchantPanel` 的喂养功能从"消耗元石"改为"选择喂养材料"——优先使用 materialBag 中的匹配蛊材，无匹配材料时降级为元石喂养（效果减半，饥饿恢复仅一级而非完全恢复）。AI叙事中在蛊虫饥饿时提示"你的月光蛊微微颤抖——它已经三天没有吸收月光了"。

## 第二十部分：DeepSeek V4 缓存调优与动静分离架构

### 20.1 缓存机制原理（官方文档验证）

依据 DeepSeek API 官方文档（api-docs.deepseek.com/guides/kv_cache/），上下文硬盘缓存对所有用户默认开启，基于"缓存前缀单元"(Cache Prefix Unit) 机制：

**三种落盘方式**：

1. **请求边界持久化**：每次请求的用户输入结束位置和模型输出结束位置各生成一个缓存前缀单元。后续请求若完整匹配该单元即可命中。
2. **公共前缀检测持久化**：系统自动识别多次请求间的公共前缀，将其作为独立单元落盘。2次以上请求出现相同前缀时触发，第3次起命中。
3. **定长切分持久化**：长文本按固定 token 间隔切分出单元，避免长前缀因无终点而永远无法被缓存。

**System Message 行为**：system message 是最长消息前缀的一部分。当系统检测到多次请求的 system message 内容相同时，会将其识别为公共前缀并独立缓存——这正是本方案的核心依据。

**V4定价（2026.4.26降价后）**：
- V4-Flash 缓存命中 ¥0.02/M tokens（全价 ¥0.2/M 的 1/10）
- V4-Pro 缓存命中 ¥0.1/M tokens（5月5日前 2.5折叠加 = ¥0.025/M）

**关键约束**：
- 缓存命中要求后续请求的前缀与某个已持久化的单元**完全一致**（逐字节匹配），不是任意子串匹配
- 缓存系统"尽力而为"，不保证 100% 命中率
- 缓存闲置后几小时到几天内自动清除
- 多轮对话中，第2轮请求可匹配第1轮的输入结束边界单元（如 `system + user1 + assistant + user2` 命中前 `system + user1` 的缓存）

### 20.2 当前代码缓存破坏诊断

#### 破坏点一（`deepseek.ts:41-44`）：随机前缀 + 时间戳

```
const systemWithBreak = `[sid:${Math.random().toString(36).slice(2,8)}] ${systemPrompt}`;
const userWithCacheBreak = `${userMessage}\n#[ts:${Date.now().toString(36)}]`;
```

每次请求的 system message 以唯一的 `[sid:xxxxxx]` 开头。DeepSeek 的公共前缀检测需要 2 次以上请求出现**完整相同的 system message 前缀**才能触发——随机前缀使这永远不可能。这是缓存命中率仅 1.3% 的直接原因。

#### 破坏点二（`context-builder.ts:157`）：元石余额注入 system prompt

`injectEconomyRules(store)` 将 `${currency}`（当前元石余额）注入 system prompt。元石每轮交易都在变化 → system message 每轮不同 → 即使移除随机前缀也无法缓存。

#### 破坏点三（`context-builder.ts:188`）：玩家境界参与 NPC 评分

NPC 上下文注入使用 `playerRealm`（`store.profile?.realm?.grand`）对 NPC 做相关性排序。境界每约 10 回合变化一次 → system message 随之变化 → 缓存失效。

#### 破坏点四（`context-builder.ts:306-308`）：蛊虫知识随库存变化

`injectGuKnowledge(store.inventory)` 根据玩家当前蛊虫库存生成知识注入。获取/失去蛊虫时 system message 变化。

### 20.3 动静分离架构重构

**核心原则**：system prompt 仅包含**完全静态**的内容（在玩家不跨域时不变），所有动态数据移入 user message。

```
┌─ System Prompt（完全静态，缓存友好）──────────────────────┐
│ Layer 1: 世界观核心 + JSON格式硬约束                          │
│ Layer S: 叙事风格指南（黑暗基调/禁用爽文/NPC不降智）          │
│ Layer 2: 字段速查（choices/state_update字段清单）             │
│ World Rules: 蛊界核心规则（蛊虫销毁/道痕战力/气运/喂养/互斥）  │
│ 术语速查: 真元/空窍/道痕/杀招/本命蛊                          │
│ NPC身份: 名称+头衔+静态性格（不含动态好感度/关系）             │
│ Canon/IF模式声明                                              │
│ 物价参考表: 各转数元石基数（不含当前余额）                     │
└────────────────────────────────────────────────────────────┘

┌─ User Message（每轮不同，含所有动态数据）─────────────────┐
│ 【玩家当前状态】完整JSON（HP/真元/属性/道心/flags/蛊虫/势力）  │
│ 【当前元石余额】128块                                         │
│ 【动态NPC关系】古月方正好感+15 / 铁家-30 ...                  │
│ 【当前蛊虫状态摘要】月光蛊(饥饿中)/金蚕蛊(optimal)...         │
│ 【近期关键事件】[第N回合] ...                                 │
│ 【历史摘要】滚动摘要文本                                       │
│ 【最近对话历史】最近5轮消息摘要                                │
│ 【玩家选择】choiceId                                           │
└────────────────────────────────────────────────────────────┘
```

**context-builder.ts 具体重构**：

| 函数 | 改动 | 说明 |
|------|------|------|
| `injectEconomyRules` | 删 `${currency}` 行，物价参考表保留 | 元石余额移入 user message 的 `buildMessages` |
| `injectNPCContext` | 删 `playerRealm` 评分依赖，改为按原著重要性静态排序 | 动态好感度移入 user message |
| `injectGuKnowledge` | 从 system prompt 移除 | 蛊虫状态摘要移入 user message |
| `buildMessages` | 新增 `buildDynamicContext()` 段 | 在 user message 中追加经济/NPC关系/蛊虫状态三个动态段 |
