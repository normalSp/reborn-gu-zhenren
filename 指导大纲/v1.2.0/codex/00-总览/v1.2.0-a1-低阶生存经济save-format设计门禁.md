## 设计原则

### 经济是压力，不是奖池

低阶蛊师的原著味来自稀缺、维护成本、身份风险、资源门槛和社会压力。v1.2 的经济系统必须先让玩家感到“活下去不容易”，而不是让玩家快速进入刷材料、刷价格、刷商店的 RPG 奖励循环。

### 可读性必须补足拒绝感

不能只是说“不行”。当玩家想买稀有蛊、补食料、炼残方、找商队、走灰色渠道时，系统要说清楚：

- 缺什么。
- 为什么缺。
- 哪些准备可以做。
- 哪些东西是风险或长期目标。
- 哪些东西当前不可做。

### 存档字段必须晚于权威表

经济字段一旦进入存档，会长期影响迁移、旧档、测试、UI、反刷和后续版本。v1.2 不允许顺手加字段。若新增字段，必须同一变更完成：

- `SAVE_FORMAT_VERSION` bump。
- migration。
- defaults。
- normalization。
- focused tests。
- old-save tests。
- Player Advocate。
- drift/anti-farm samples。

### 原著味与游戏性之间的中线

太保守会变成“只有文本压力，没有真实系统”。太激进会变成“商店/库存/价格模拟器”，削弱蛊真人低阶世界的压迫感和风险感。

v1.2 的中线是：先做压力投影，再做最小账本，最后才考虑交易窗口。完整经济系统留给更后面的专门版本。

## D-121-001：`SAVE_FORMAT_VERSION = 24`

### 方案 A：v1.2 不 bump，全部 projection-only

正面收益：

- 最稳，不碰迁移和旧档风险。
- 开发速度快，适合先验证 UI、口感和玩家理解。
- 不会过早锁死经济 schema。
- 不会把“候选压力”误写成长期事实。

负面影响：

- 经济压力容易显得像说明文字，而不是世界状态。
- 多轮游戏后缺口、维护、准备和风险不够连续。
- 很难支撑 v1.3 NPC/势力长期关系、v1.4 区域事件和 v2.0 区域活世界里的经济记忆。
- 玩家可能觉得“资源很紧张”只是文案，不是真系统。

对未来开发的影响：

- 适合 b1。
- 不适合长期作为 v1.2 最终答案。
- 后面仍然要补 v24 或别的持久化经济结构。

游戏性与原著味：

- 原著味安全，因为不会乱发资源或乱开市场。
- 游戏性偏弱，因为玩家行为后果不够持久。

### 方案 B：v1.2 使用 `SAVE_FORMAT_VERSION = 24`，但只加单一聚合对象

建议对象名：

`survivalEconomyState`

它不是完整库存，不是价格表，不是商店账本。它只记录低阶生存经济的压力、缺口、窗口和反刷状态。

正面收益：

- 低阶生存压力可以长期连续，而不是每轮重新解释。
- 可以让路线、地点、补给、蛊虫维护、炼养用准备互相咬合。
- 为 v1.3 NPC/势力压力、v1.4 区域事件、v1.5 战斗后果、v2.0 区域活世界打地基。
- 可测试、可迁移、可回放，适合长线叙事漂移 gate。
- 玩家会感觉世界在记住自己的生存困境。

负面影响：

- 需要 save migration、defaults、normalization 和旧档测试。
- schema 一旦写错，后续会背债。
- UI 需要解释清楚，否则玩家会被新字段吓到。
- 如果字段太像库存或价格表，会诱导后续开发过快滑向完整经济系统。

对未来开发的影响：

- 最适合作为 v1.2 中后段地基。
- 建议 b1 先不加，b2 再加，避免没验证口感就锁 schema。
- 后续可逐步承接 NPC 讨价、商队压力、维护周期、路线补给和区域风险。

游戏性与原著味：

- 平衡最好。它让资源压力真实存在，但不把低阶世界做成自由购物。
- 关键是字段必须记录“压力与缺口”，不能记录完整市场价格。

### 方案 C：v1.2 直接做完整库存/价格/交易字段

正面收益：

- 玩家立刻获得强系统感。
- 很容易做成传统 RPG 的材料、商店、买卖、价格变化和经济循环。

负面影响：

- 高概率经济失控。
- 大幅增加反刷、套利、旧档迁移、UI、测试和 DeepSeek 越权风险。
- 会把 v1.2 从“低阶生存经济第一阶段”拖成“完整经济系统重构”。
- 容易破坏蛊真人低阶世界的原著味：低阶蛊师不该像在网游商店里自由采购稀有资源。

对未来开发的影响：

- 会吞掉 v1.2 绝大部分开发预算。
- 后续 NPC、势力、路线、区域活世界都会被早期价格表绑死。

游戏性与原著味：

- 游戏性表面变强，长期风险极高。
- 原著味明显下降，尤其是商队、黑市、稀有蛊材和身份风险会被“价格”抹平。

### 专家团推荐

选择“B 的节奏”：批准 `SAVE_FORMAT_VERSION = 24` 的设计路线，但不在 b1 立刻 bump。b1 先 projection-only，b2 若验证需要持久状态，再用单一聚合对象 `survivalEconomyState`。

## D-121-002：持久经济字段形态

### 方案 A：不新增字段

适合 b1。缺点是长期连续性不足。

### 方案 B：新增单一聚合对象 `survivalEconomyState`

建议字段只记录：

| 子域 | 用途 | 禁止 |
|---|---|---|
| `schemaVersion` | 聚合对象内部版本 | 不替代 save format |
| `supplyPressure` | 路线补给、落脚、遮掩压力 | 不写真实库存数量 |
| `guUpkeepPressure` | 蛊虫喂养/维护缺口 | 不自动扣食料，不授予食料 |
| `refinementPreparation` | 残方、材料缺口、失败风险 | 不判定炼成/失败 |
| `tradeWindows` | 商队询价、担保、公开理由、风险窗口 | 不写正式价格和成交 |
| `antiFarmLedger` | 最近经济行动、递减和风险 | 不做可套利收益账本 |

正面收益：

- 聚合边界清楚。
- 不污染 `livingWorldState` 和 `routeLocationState`。
- 后续可独立迁移、测试和回滚。
- 防止经济状态散落到多个 store slice。

负面影响：

- 多一个长期 state surface。
- 需要 UI 解释，否则玩家可能以为已经开放完整库存。
- 需要 strict schema，避免被写成自由文本大杂烩。

### 方案 C：多个分散字段

不推荐。比如 `inventoryState`、`marketState`、`feedingState`、`tradeState` 同时出现，会让 v1.2 快速失控。

### 专家团推荐

若新增字段，只选方案 B：单一聚合对象 `survivalEconomyState`。它应是压力账本，不是库存账本。

## D-121-004：真实消耗、维护周期、炼蛊失败代价

### 不开放真实消耗

正面收益：

- 不会误扣资源。
- 不会在没有库存系统时做伪结算。
- 原著边界安全。

负面影响：

- 生存压力不够“咬肉”。
- 玩家短期缺少强反馈。

适用：b1。

### 开放维护/消耗压力，但不扣真实物品

正面收益：

- 能表现蛊虫需要喂养、路线需要补给、炼蛊有失败代价。
- 不需要完整库存。
- 原著味强：玩家感到压力，但还没有自由购物式结算。

负面影响：

- 如果 UI 写得不好，玩家会困惑“到底扣没扣”。
- 需要明确这是 pressure，不是 settlement。

适用：b1/b2 过渡。

### 开放真实扣减和失败结算

正面收益：

- 游戏性强。
- 玩家决策真正有代价。

负面影响：

- 必须有持久 ledger、库存、失败规则、回滚测试和反刷门。
- 容易让玩家因为系统不透明而挫败。
- 对原著味要求更高，不能把炼蛊失败写成普通 crafting fail。

适用：不建议 v1.2 b1；可作为 b3 后的独立二次审批。

### 专家团推荐

b1 只做压力，不做真实扣减。b2/b3 之后，如果 `survivalEconomyState` 稳定，再考虑“最小维护周期”或“失败风险记录”，仍不直接开放完整炼蛊成功/失败结算。

## D-121-005：正式交易、价格、库存、买卖

### 只做交易窗口

内容：

- 商队传闻。
- 公开询价。
- 身份担保。
- 风险提示。
- 不可得原因。
- 灰色渠道风险。

正面收益：

- 很有原著味：低阶蛊师接触资源需要身份、担保、势力和风险。
- 避免商店化。
- 可服务剧情和路线压力。

负面影响：

- 玩家不能立即买东西。
- 需要 UI 把“窗口”和“成交”区分清楚。

专家团推荐：v1.2 默认选择。

### 开放正式价格和成交

正面收益：

- 系统感强。
- 玩家目标明确。

负面影响：

- 价格表会变成事实承诺。
- 稳定买卖会削弱资源稀缺。
- DeepSeek 很容易越权写价格或到账。
- 黑市和商队会被玩家刷成提款机。

专家团推荐：v1.2 不默认开放。

## D-121-006：Player Advocate 强度

### 方案 A：普通强度

- runtime 小版本 10 轮。
- rc 50 轮。

收益：成本低。

风险：v1.2 是经济/生存/player-facing 高风险版本，10/50 容易看不出长线困惑、误解和挫败。

### 方案 B：高风险强度

- runtime 小版本 20 轮。
- rc 60 轮。
- 如果新增 `survivalEconomyState` 或真实 ledger/消耗，rc 至少 80 轮。

收益：

- 与 v0.15 经济线经验一致。
- 能更早发现“玩家不知道下一步”“误以为可以买卖”“误以为资源到账”等问题。

风险：

- 文档和走查成本更高。

### 方案 C：经济系统强度

- runtime 小版本 30 轮。
- rc 100 轮以上。

适用：

- 若你批准正式库存、价格、交易或真实扣减。

专家团推荐：

选择方案 B。若你后续批准正式价格/交易/库存，再升级到方案 C。

## D-121-007：v1.2 live DeepSeek drift probe

### 不做 live probe，只做 deterministic soak

正面收益：

- 无 live 成本。
- 更可复现。
- 适合 a1/a2/b1 初期。

负面影响：

- 不能观察 DeepSeek 在真实生成里是否会自发写价格、到账、隐藏名词或交易结果。

### 小规模 live probe

建议时机：

- b1 projection-only 稳定后，或 b2 最小 ledger 稳定后。

建议范围：

- 3-4 个样本。
- 每样本 4-5 轮。
- 覆盖食料缺口、商队询价、灰色渠道诱导、hidden-name echo、DeepSeek 奖励越权。

正面收益：

- 能发现 deterministic 测不到的叙事漂移。
- D-025 已证明小规模 live probe 能抓出 hidden-name echo 这类 P0。

负面影响：

- 有 live 成本。
- 结果有随机性。
- 不能因为小样本通过就宣称长线 live narrative quality 已全量通过。

专家团推荐：

a1/a2 不跑 live。b1 或 b2 稳定后，单独向你申请一个小规模 v1.2 live probe。

## 游戏性与原著味平衡评估

### projection-only 的味道

它最像“蛊师知道自己很穷、很危险、缺准备”，但玩家短期能做的事少。适合第一刀，因为它不破坏世界。

### 最小压力账本的味道

它最像“世界真的记住了我的匮乏”。这是 v1.2 最有价值的方向。只要不写价格和完整库存，就能保住蛊真人的稀缺、算计和压迫感。

### 完整交易系统的味道

它最像传统 RPG 商店。短期爽，长期危险。蛊真人的资源不是因为价格贵才稀缺，而是因为身份、势力、信息、风险、阶层和杀机共同稀缺。v1.2 不应过早把这些压缩成一张价格表。
