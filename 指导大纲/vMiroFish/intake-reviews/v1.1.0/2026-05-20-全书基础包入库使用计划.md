## 当前结论

本基础包可登记为 RebornG 全书资料档案层，但不能直接进入：

- runtime canon
- DeepSeek visible context
- player-visible UI
- 正式地点/阵营/奖励/NPC 生死
- 当前版本真相源

本轮状态不是 `accepted_for_candidate_pool` 的整包吸收，而是：

- `archive_registered`
- `content_absorption_deferred_by_topic`
- `human_review_required`

也就是说：全书基础包已到位，但每次使用仍必须按主题切片做 intake review。

## 三层使用方式

### 1. 档案层

保留在 `指导大纲/vMiroFish/基础包/`。

用途：

- 作为全书 source pointer 后勤仓库。
- 支持后续按章节、人物、地点、势力、事件、隐藏事实、测试样本做分主题 intake。
- 支持 v1.6 内容/知识库工厂建设前的库存审计。

禁用：

- 不全量复制到知识库。
- 不全量复制进 prompt。
- 不直接喂给 DeepSeek。
- 不直接给 runtime 读取。

### 2. 知识索引层

目标目录：

- `指导大纲/知识库/蛊真人/`

进入知识索引前必须满足：

1. 有明确主题，例如 `青茅 -> 南疆早期外缘路线压力`、`低阶蛊师公共经济规则`、`某势力公开反应样本`。
2. 只读取相关章节包，不全书泛扫。
3. 把 MiroFish 输出转写为 RebornG-owned 摘要。
4. 保留 source pointer、visibility、promotionStatus、allowedUses、forbiddenUses。
5. hidden/private 条目只能做 `hidden_ref_only` 或 `private` 审查引用。

### 3. runtime canon 层

目标目录：

- `src/canon/*.json`

只有被当前版本门禁明确需要、完成本地改写、通过测试、并经用户决策允许的极小安全子集，才可进入 runtime canon。

全书基础包本身永远不是 runtime canon。

## 版本落位

| 版本 | 使用方式 |
|---|---|
| v1.1 | 只作为 a1/a2 设计门禁和测试样本后勤来源。范围限制为青茅到南疆早期外缘；不得因全书包到位而开放完整南疆、商家城或高阶世界。 |
| v1.2-v1.5 | 按经济、NPC、区域、冲突、长期叙事漂移分别做主题 intake；每次都要写 review 和吸收边界。 |
| v1.6 | 正式建设内容生产、canon schema、知识库检查脚本、长测工厂和过期入口自动检查。可在此阶段把基础包库存审计工具化。 |
| v2.0 前 | 只对第一个区域活世界所需子集做完整晋升审查；不要追求全书 runtime 化。 |

## v1.1 可用切片

v1.1 仅允许从基础包派生以下候选材料：

- route/location 设计门禁的 source pointer 对照。
- Qingmao -> Southern Border early outer edge 的公开地点/压力摘要。
- 玩家目标遗失、叙事重复、DeepSeek 越权、世界观漂移的测试样本。
- hidden/private 风险清单，供 Lore & World Designer 审查，不进入玩家可见和 DeepSeek。

v1.1 不允许从基础包派生：

- 正式南疆地点开放。
- 商家城完整进入。
- 正式阵营 standing、招募、任务、奖励。
- NPC 生死、捕获、命运结论。
- 高阶原著真相的玩家可见说明。
- DeepSeek 长上下文授权。

## 使用流程

每次想用基础包时，按以下顺序：

1. 写明本阶段问题和主题切片。
2. 列出计划读取的章节包或章节范围。
3. 扫描结构字段和 hidden/private 风险。
4. 写 intake review，结论只能是 `candidate_pool`、`fact_card_draft`、`rule_draft`、`test_sample`、`deferred`、`quarantined` 或 `rejected`。
5. 若进入知识索引，补 `visibility`、`promotionStatus`、`allowedUses`、`forbiddenUses`。
6. 若进入测试矩阵，按 `测试矩阵演进规则.md` 分流。
7. 若进入 runtime，必须先经过当前版本设计门禁、focused tests、Player Advocate 或漂移门禁，并停下来确认用户授权。

## 后续工程化

v1.1 手工执行：

- 记录每次基础包使用范围。
- 手工维护 intake review、知识索引和测试矩阵引用链。
- 阶段完成前检查是否有未分流样本。

v1.6 工具化：

- `check:mirofish-base-pack-inventory`：检查基础包文件数、manifest、coverage、quote-redacted 状态。
- `check:knowledge-index-boundaries`：检查知识索引字段、visibility、promotionStatus 和 forbiddenUses。
- `check:mirofish-intake-promotions`：检查基础包 -> intake review -> 知识索引/测试矩阵/runtime canon 的引用链。

这些脚本在 v1.6 前只作为规划，不提前加入 CI 硬门。

## 需用户决策时机

- 要把 v1.6 工具加入 CI 硬门。
