# v2.0.0 专家团启动会纪要

状态：draft；等待用户决策 D-200 / D-201。
日期：2026-05-22
主题候选：第一个区域活世界入场门禁

## 开会背景

v1.7 做出了低阶区域活世界 projection-first 第一阶段，v1.8 做出了低阶身份路线和同开局差异度 projection-first 地基，v1.9 把 v2.0 前的 readiness、事件 envelope、T3 长测、P2 术语和子代理风险收束成 report-only 门禁。

现在 v2.0 面对的问题不是“能不能继续做一个更漂亮的面板”，而是：

- 一个区域能不能支撑 300+ 回合仍不漂移。
- 玩家行为能不能被路线、经济、社会、冲突、身份和叙事共同承接。
- 同开局能不能长期不一样，但世界事实仍稳定。
- DeepSeek 能不能让叙事鲜活，同时永远不拿到事实裁决权。
- MiroFish 和全书知识库能不能提供 source pointer，而不是变成大上下文。

## 专家团总建议

推荐 v2.0 主线：

`第一个区域活世界：南疆早期低阶外缘小区域`

一句话目标：

> 在一个小而硬的区域内，建立可持久、可回放、可长测的区域事件承接链；让玩家选择、身份候选、低阶经济压力、社会压力、冲突风险和 DeepSeek 叙事表达共同作用，但最终事实仍由本地 WorldCore / engine 裁决。

## 专家团好点子池

| 点子 | 价值 | 风险/处理 |
|---|---|---|
| 最小 `regionalEventLedger` | 让区域事件可回放、可追责、可长测 | 必须单一聚合，不做万能世界状态 |
| `EventEnvelope -> WorldCore -> Ledger` 三段链 | 区分候选、裁决、持久记录 | DeepSeek 不能越过 envelope 直接写 ledger |
| 暂不落 `runFingerprint` | 避免把随机性和事实差异过早写死 | 先用 session-local variant / replay evidence 证明需要 |
| T3 320 轮标准档 | 比 300+ 更稳，便于分 160 live / 160 replay | 执行耗时高，但成本不是主要阻碍 |
| P2 正式凭证词 kill list | 收束“木牌/令牌/登记/member”误导 | 进入 prompt/eval/test，不变成正式身份 |
| 区域事件后果三层 | immediate pressure、pending follow-up、forbidden outcome | 不直接给正式奖励、NPC 生死、地点权限 |
| v2.0 不上 BFF/RAG | 把复杂度留给 WorldCore 与测试门 | 公开测试/API key 安全仍是 v2.x 专项 |
| Agent Lab 延后到 v2.1+ | v2.0 先证明一个区域能活 | 不把长期架构提前塞进 runtime |

## 不推荐现在做

- 直接新增 `regionalLifeState` / `areaLivingState`。
- 直接新增 `identityRouteState` / `professionState`。
- 直接新增 `runFingerprint`。
- 同时开事件账本、身份状态、区域状态和后端。
- 完整南疆、完整商家城、正式商队/散修/城市身份。
- 正式价格、库存、工资、奖励、战斗掉落、NPC 生死。
- DeepSeek visible full-book RAG。
- BFF/backend。
- 子代理写文件或参与 runtime/canon/git。

## 专家团结论

v2.0 可以启动，但必须先让用户批准 D-200 / D-201。当前建议是：

1. 正式批准 v2.0 主线为第一个区域活世界。
2. 正式批准第一核心区域为南疆早期低阶外缘小区域。
3. a1 认真批准或驳回 `SAVE_FORMAT_VERSION = 25` 与最小 `regionalEventLedger`。
4. 暂不新增 `runFingerprint`、正式区域状态和正式身份状态。
5. v2.0 rc 前执行 T3 320 轮 mixed/live/replay 长测。
6. 不启用 DeepSeek RAG、BFF/backend、子代理或 formal outcome 权限。
