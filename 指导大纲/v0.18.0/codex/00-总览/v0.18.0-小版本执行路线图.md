## 推荐执行顺序

1. 用户批准启动范围和 MiroFish request。
2. 做 `a1`，只写设计门禁，尤其是 route/location/save/BFF 边界。
3. 做 `a2`，完成 MiroFish intake、字段表和测试矩阵。
4. 若用户批准 save-format 变更，b2 前写迁移/默认/旧档测试计划并再次停下来确认。
5. b1/b2/b3/b4/b5 每刀都必须更新文档、测试、仪表盘、Git 状态。
6. rc 前做完整 Player Advocate 和全量质量门。

## Git 建议

- a1/a2 可合并为一个 docs commit。
- b1/b2/b3/b4/b5 每阶段一个 runtime commit。
- rc 一个收束 commit。
- 不使用 `git add -A`。
- 不混入当前未跟踪的美术候选、zip、artifacts、bgm、`.cursor` 或 `指导大纲/大方向/`。
