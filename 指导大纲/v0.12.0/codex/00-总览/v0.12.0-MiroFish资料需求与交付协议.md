# v0.12.0 MiroFish 资料需求与交付协议

日期：2026-05-16
状态：已启用，作为 v0.12 之后 canon/IF/NPC/faction/route 设计前置门禁。
定位：MiroFish 只负责产出 quote-redacted 候选材料；RebornG 本地 canon、engine、测试和用户决策仍是最终权威。

## 结论

MiroFish 不并入 RebornG 运行时代码，不作为 canon 真相源，不给 DeepSeek 新权限。

MiroFish 的职责是：

- 从原著材料中抽取候选事实、动机、势力压力、路线压力、隐藏事实引用。
- 交付 quote-redacted JSON 包。
- 提供 source pointers、review status、promotion/block reason。

RebornG 的职责是：

- 审核候选包。
- 决定是否吸收为 fact card、anchor、IF rule、reaction rule 或测试样本。
- 保证 hidden fact 只以 `hidden_ref_only` 进入本地门禁。
- 保证 DeepSeek 只能写叙事、候选、线索、传闻、压力表达和裁决解释。

## 何时必须评估 MiroFish

进入下列工作前，专家团必须先判断是否需要 MiroFish 包：

- 原著事实卡扩展。
- 正史锚点、剧情锚点、IF 分叉。
- NPC 动机、NPC 记忆、人物反应。
- 势力压力、阵营关系、商队/小组/家老反应。
- 路线、补给、追击、逃离、跨区域准备链。
- 方源公开旁证、隐藏事实 probe、隐藏事实保护。
- 任何会影响“像不像蛊真人”的内容系统。

下列工作不需要等待 MiroFish：

- 纯工程修 bug。
- UI 布局修复。
- 测试体系、CI、GitHub 模板。
- 存档迁移和类型整理。
- DeepSeek telemetry、缓存命中、prompt hash 等非 canon 观测。
- 已有 canon/engine 明确覆盖的低风险接线。

## 阻塞等级

每个阶段必须标注 MiroFish 需求的阻塞等级：

| 等级 | 含义 |
|---|---|
| `blocking` | 没有材料包就不应继续实现该阶段 runtime。 |
| `preferred` | 有材料包更稳；没有也可用现有 RebornG canon 先做保守最小刀。 |
| `optional` | 只是补充背景，不影响阶段实现。 |
| `not_needed` | 本阶段不需要 MiroFish。 |

默认规则：

- canon/IF/隐藏事实/方源旁证：`blocking` 或 `preferred`。
- route/supply/pursuit：至少 `preferred`；若涉及原著具体路线或追击细节，则 `blocking`。
- NPC/faction reaction：至少 `preferred`。
- 工程/测试/流程：`not_needed`。

## 请求文件格式

我需要 MiroFish 产包时，必须在 `指导大纲/vMiroFish/requests/` 新增请求文件，包含：

- `requestId`
- `targetPhase`
- `blockingLevel`
- `purpose`
- `scope`
- `requiredFields`
- `forbiddenContent`
- `acceptanceCriteria`
- `suggestedOutputName`
- `handoffMessageForMiroFish`

用户把这份请求转交给负责 MiroFish 的会话。当前无法由本 Codex 线程直接联系另一个 Codex 会话。

## 主动提醒规则

为了避免只靠用户临时想起来沟通 MiroFish，后续按以下节奏执行：

1. 拟定每个大版本大纲时，必须列出该版本可能需要的 MiroFish 包，并标注 `blocking`、`preferred`、`optional`、`not_needed`。
2. 进入每个小版本 startup review 时，必须再次判断当前阶段是否需要 MiroFish 包。
3. 如果需要 MiroFish 包，Codex 必须在回复里明确告诉用户：
   - 需要哪个包。
   - 目标阶段是什么。
   - 是否阻塞当前开发。
   - 请求文件路径是什么。
   - 需要用户转交给哪个 MiroFish 会话。
4. 如果阶段不需要 MiroFish，也要在阶段启动记录中写明 `not_needed`，避免后续误会。
5. 如果包已经到达但未审查，不能进入运行时吸收；只能做文档整理或 intake review。

## 交付包落位

MiroFish 交付物统一放在：

- `指导大纲/vMiroFish/`

推荐文件：

- 主包：`*_export_ready.json`
- 报告：`*_report.json`
- 审校 ledger：`*_review*.json`
- 晋级/阻断说明：`*.md`

如果交付包很大，可以保留在 `vMiroFish` 子目录，但必须有一个说明文档指出主包路径。

## RebornG Intake Review

我吸收前必须在 `指导大纲/vMiroFish/intake-reviews/` 新增审查文件，至少记录：

- 主包路径。
- 总项数、字段分类、source pointer 数。
- 是否 quote-free。
- hidden facts 是否全部 `hidden_ref_only`。
- 是否存在重复 ID、主体归一问题、source pointer 格式问题。
- 可吸收项、延期项、阻断项。
- 是否影响当前阶段。
- 是否需要用户决策。

只有 intake review 结论为“可吸收”的材料，才允许转入 RebornG 本地 canon/rules/tests。

## 吸收使用流程

MiroFish 包通过 intake review 之后，仍然只能按以下顺序吸收：

1. **候选池**：先记录哪些条目进入 `candidate_pool`，哪些延期，哪些阻断。
2. **本地改写**：把候选条目改写成 RebornG-owned summary、rule draft、fact-card draft 或测试样本，不复制原著正文，不照搬隐藏事实 body。
3. **权威归属**：所有 runtime 判定必须来自 RebornG 本地 canon、engine、schema、store action 和测试，不来自 MiroFish 包本身。
4. **可见性门禁**：hidden fact 只能成为 hidden ref 或测试门禁，不能进入 UI、DeepSeek visible context 或玩家可见叙事。
5. **测试绑定**：涉及 route/supply/pursuit、IF、hidden fact、NPC reaction 的吸收项，必须至少补 focused unit test；若进入 UI 或玩家动作，还要补 e2e 或明确为什么暂不需要。
6. **文档同步**：吸收后更新当前版本仪表盘、阶段记录、`PROJECT-STATE.md` 和最新上下文交接。
7. **下一包建议**：阶段结束时说明下一个可能需要的 MiroFish 包，或者写明 `not_needed`。

## 字段要求

MiroFish 包至少应包含：

- 稳定 id。
- `summary` 或等价摘要。
- `sourcePointers`。
- `review.status` 或等价审校状态。
- `reborngGate` 或等价门禁字段。
- `visibility` 或 `playerVisibility`。
- `confidence`。

隐藏事实必须包含：

- `hiddenRefOnly: true`。
- `runtimeVisible: false`。
- `deepSeekVisible: false`。
- `requiresHumanCanonReview: true`。
- `whoKnows` / `whoDoesNotKnow`，如果可抽取。

## 禁止内容

MiroFish 交付包不得包含：

- 原著正文 quote。
- `originalText`。
- `excerpt`。
- 长段原文 paraphrase 到接近复述。
- 直接 runtime 权限。
- 正式奖励、蛊虫、元石、地点解锁、NPC 生死、阵营变更结论。
- “逃离成功”“正史改变”“隐藏事实公开”等结果性结论。
