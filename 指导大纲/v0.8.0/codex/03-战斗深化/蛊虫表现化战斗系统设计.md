# 蛊虫表现化战斗系统设计

更新时间：2026-05-10  
阶段：v0.8.0-pre  
状态：协议与首批数据闸门已落地  

## 设计目标

v0.8.0 的战斗深化不再把蛊虫和杀招写成“描述 + 伤害倍率”。每个蛊虫的文本描述必须能被拆成三类可执行能力：

- 战斗能力：射程、目标、消耗、冷却、状态、反制、地形亲和和棋盘影响。
- 剧情能力：侦察、破障、追踪、治疗、遮掩、威慑、禁忌仪式等可被选择系统调用的解法。
- 视觉能力：GSAP/Motion 可消费的 motif、颜色、动作节拍和表现强度。

玩家看到的不是“攻击蛊/防御蛊/治疗蛊”，而是“月光蛊能切断绳索、直线切过目标，但怕遮蔽和厚甲”“石皮蛊能硬接近战，但会慢下来并怕水道软化”。描述必须成为游戏规则。

## 战斗状态模型

v0.8 采用 5×3 轻地图棋盘作为第一阶段战场：

- 格子：地形、遮蔽、危险、阵位、道痕场、占位单位。
- 单位：玩家、小队成员、敌人、中立单位。
- 阶段：侦察、布阵、玩家回合、敌方回合、结算、结束。
- 轨迹：所有 UI 动画只播放 `BattleResolutionStep`，不在组件内私算胜负或伤害。

`BattleResolutionStep` 是战斗表现的中心协议：移动、蛊虫发动、杀招发动、命中、未命中、状态变化、地形改变、反制、资源消耗、失败原因、结算都要进入 step。

## 蛊虫设计原则

每只蛊必须有独立玩法身份：

- 月光蛊：直线月刃、切割、夜间信号、可破轻遮蔽。
- 小光蛊：增强光道、照明、辅助识别，不作为伤害按钮。
- 石皮蛊：近战硬抗、减速、怕水道软化。
- 青丝蛊：束缚、拉扯、室内优势、怕火和利刃。
- 酒虫：真元质量和战前准备，不作为常规战斗攻击。
- 血颅蛊、妇人心蛊：禁忌场景蛊，只走强剧情门槛，不显示为普通按钮。

底层机制可以复用，但对玩家展示必须独特。允许多个蛊共享 `projectile`、`shield`、`bind` 等底层动词，但必须有不同目标、限制、视觉和剧情用途。

## 杀招设计原则

杀招不是“大招按钮”，而是多蛊组合工程：

- 核心蛊决定杀招能否成立。
- 辅助蛊决定范围、稳定度、特殊效果或反制窗口。
- 蓄势阶段给敌人打断机会。
- 释放阶段输出 `BattleResolutionStep` 序列。
- 维持阶段消耗真元或暴露破绽。
- 失败模式和反噬必须可见、可记录、可测试。

例：`金钟不破` 不是“防御 +X”，而是金钟蛊、金罡蛊、石皮蛊共同形成定点护罩；它强在守点和挡投射物，弱在不能移动、怕共振和持续重击。

## DeepSeek 边界

DeepSeek 可以写：

- 敌方意图候选。
- 第三方介入候选。
- 战斗叙事描述。
- 可选行动文案。

DeepSeek 不能写：

- 正式胜负。
- HP/真元/状态直接变化。
- 蛊虫效果是否命中。
- 正式掉落、蛊虫获得、杀招习得。
- 未登记蛊虫、未登记杀招的数值效果。

AI 文本若提到可执行效果，必须能映射到 `GuExpressionSpec`、`KillerMoveExpressionSpec` 或被降级为传闻/线索。

## 验收口径

- 所有首批可战斗蛊都有表现规格。
- 所有流派名来自 `path-registry.json`。
- 禁忌蛊不能成为普通战斗按钮。
- 每个杀招都有核心蛊、辅助蛊、蓄势、失败、反噬和视觉节拍。
- 战斗 UI 只能消费引擎输出的 `BattleResolutionStep`。
- 剧情选择能识别玩家已持有蛊虫的 `sceneUtilities`。

## v0.8.0-a1 引擎 API 与边界

a1 的运行时入口是 `src/engine/v080-battlefield-combat-engine.ts`。它是新增引擎，不改旧 `combat-engine.ts` / `squad-combat-engine.ts` 行为，也不接 store 或 UI。

公开 API 固定为：

```ts
createBattlefieldCombatState(input): BattlefieldCombatState
listBattlefieldActionTargets(state, action): BattlefieldActionValidation
validateBattlefieldAction(state, action): BattlefieldActionValidation
executeBattlefieldAction(state, action): BattlefieldActionResolution
advanceBattlefieldRound(state): BattlefieldActionResolution
interruptPendingBattlefieldAction(state, pendingActionId, reason): BattlefieldActionResolution
```

数值真相源是 `src/canon/battlefield-combat-rules.json`。该文件覆盖首批蛊虫和杀招现有的所有棋盘 shape，并记录移动、遮蔽、地形亲和、阵位、危险格、冷却、蓄势、反噬和撤退参数。引擎复用 `combat-formulas.ts` 的 seeded RNG、命中、伤害、境界和流派克制公式，禁止直接调用 `Math.random`。

行动边界：

- 蛊虫：只有 `availability === "direct"` 且 `realmScope === "mortal"` 的蛊默认可作为普通战斗行动。
- 禁忌/场景蛊：`scene_gated` 或 `mortal_forbidden` 默认阻断，除非 action 明确带 `sceneGate: true`；即便放行，也仍由本地引擎结算资源、目标和失败原因。
- 被动蛊：不作为主动按钮，由后续被动/战前系统接入。
- 杀招：必须已学会，且核心蛊和辅助蛊齐全；蓄势杀招进入 `pendingActions`，打断和释放都输出可播放 step。
- 叙事：DeepSeek 仍只能提出候选和描述，不能直接写入 HP、状态、胜负、掉落或正式战斗结果。

不入存档原则：

- a1 的 `BattlefieldCombatState` 是运行时状态，不写入正式 save。
- 本阶段不提升 `SAVE_FORMAT_VERSION`。
- 后续若要保存战斗中断点、pending action、持续地形或战斗结果，必须单独设计迁移和兼容测试。

## v0.8.0-a2 UI 接入边界

a2 的 UI 入口是 `BattlefieldCombatOverlay`，默认通过“凡战”按钮或 E2E harness 启动非持久演武战场。它是可玩竖切，不是正式剧情分支：

- store 只保存临时 `battlefield*` 字段，全部排除出存档 partialize，读档时重置。
- UI 的行动卡、目标范围、不可用原因来自 `v080-battlefield-ui-model.ts` 与 a1 validation；组件不得私算命中、伤害、状态、胜负、撤退。
- 执行动作只调用 `executeBattlefieldAction`、`advanceBattlefieldRound`、`interruptPendingBattlefieldAction` 等 a1 API。
- 战斗轨迹只播放 `BattleResolutionStep[]`，日志文案可摘要，但不得新增未由引擎输出的结果事实。
- 普通蛊虫按钮继续只展示 `direct + mortal`，`passive`、`scene_gated`、`mortal_forbidden` 只显示为场景门槛或禁忌说明，不进入常规快捷攻击。
- 杀招卡只暴露已学会且通过核心蛊/辅助蛊校验的候选；蓄势、打断、反噬由引擎输出 step 后再展示。

a2 验收新增：

- 桌面 1440x900 与移动端 390x844 都必须能打开凡战、看到 15 格棋盘、切换行动栏、选择目标、执行动作并阅读轨迹。
- reduced motion 下关闭大幅位移和闪烁，但保留静态高亮、不可用原因和轨迹推进。
- 调试浮层不得遮挡战斗 overlay；全屏战斗面板是当前最高优先级交互层。

## v0.8.0-a3 剧情蛊虫表现化边界

a3 的运行时入口是 `src/engine/v080-narrative-gu-affordances.ts`，它服务剧情选择和叙事上下文，不服务正式战斗结算。

新增真相源：

- `src/canon/narrative-gu-scene-utility-map.json`
  - 把 `GuExpressionSpec.sceneUtilities` 与 `KillerMoveExpressionSpec.sceneUtilities` 映射到固定剧情用途分类。
  - 分类固定为：侦察、追踪、治疗、解毒、破障、遮掩、威慑、禁忌仪式、移动、防护、控制、信号、生存、炼化。
  - 该文件只决定 prompt、UI 标签和本地候选校验，不授予奖励、掉落、胜负或数值改动。

公开能力：

```ts
collectNarrativeGuAffordances(store)
buildNarrativeGuAffordancePromptInject(store)
annotateNarrativeGuChoices(narrative, store)
buildSelectedChoiceNarrativeGuContext(store, choiceId)
validateNarrativeGuUseSuggestion(suggestion, store)
```

执行边界：

- `context-builder` 可以告诉 DeepSeek 玩家有哪些已登记剧情蛊虫用途、限制和风险。
- `response-pipeline` 可以把玩家本轮选择的文本、风险与蛊虫用途 metadata 注入下一轮上下文。
- `narrative-consistency` 可以把不合规选择降级为线索或传闻，并给 UI 标注“缺少蛊虫/禁忌门槛”。
- `state-update-applier` 是最后一道门：只有已持有、已登记、用途匹配、通过 `sceneValidated=true` 与 `sceneTags` 门槛、且在 `gu-use-registry` 允许执行的建议，才进入既有本地执行路径。
- 普通 `sceneUtilities` 不自动改 HP、真元、资源、掉落、状态或胜负。
- 战斗攻击类蛊虫即便在剧情中可显示为解法，也必须转入本地战斗引擎，不能由剧情候选直接结算。
- 禁忌蛊可以显示为“禁忌门槛”，但没有强场景授权时只能是候选、风险或线索。
