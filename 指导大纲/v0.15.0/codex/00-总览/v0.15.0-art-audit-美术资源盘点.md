## 接入状态

### 蛊虫图鉴

结论：青茅 S0 蛊虫图鉴当前覆盖度很好。

- runtime 文件：59 个。
- `GU_IMAGE_MAP` 唯一映射：59 个。
- 未映射 runtime 蛊图：0 个。
- 映射但文件缺失：0 个。

风险：

- 其中 27 个 runtime PNG 与 `doc/art/candidates/gu/s0-qingmao/*-candidate-*.png` 是精确重复，属于“候选图复制入库后保留候选源”的正常重复。
- 若要减少仓库体积，后续可把已入库候选源移入归档目录或只保留 manifest 记录，但不建议现在删除。

### 角色图

结论：原著角色图数量够青茅阶段使用，但需要一次角色映射/预留整理。

- 原著角色 runtime 文件：25 个。
- `CHAR_IMAGE_MAP` 唯一映射：23 个。
- 未映射原著角色文件：2 个，且与正式图精确重复：
  - `public/rebrng/characters/canon/fang-yuan-style-sample.png`
  - `public/rebrng/characters/canon/gu-yue-fang-zheng-style-sample.png`
- 原创角色：9 个，当前未在 `image-maps.ts` 映射，适合作为后续随机 NPC / 原创 NPC 素材池。

建议：

- 方源/方正 style-sample 可作为候选清理项，但先不删；建议在美术专项中让用户确认是否保留样张。
- 原创角色需要建 `ORIGINAL_CHAR_IMAGE_MAP` 或角色素材池 manifest，否则 v1.0 前容易“有图但无法稳定调用”。

## v1.0 支撑度判断

我的判断：当前资源足以支撑“青茅山内部开发测试”和局部演示，但还不足以直接支撑 v1.0 公开发布。

足够的部分：

- 青茅 S0 蛊虫图鉴覆盖度强，59 个 runtime 蛊图全映射。
- 原著角色基础图足够青茅第一阶段使用。
- 已有风格母版 v2，后续图能按同一视觉语言收束。
- 战斗、标题、区域、南疆/商家城已有大量候选素材，可作为后续选择池。

不足的部分：

- 场景 manifest 落后于实际磁盘资源，多个 SVG/PNG 未登记。
- 候选、accepted、rejected、review-only 与 runtime_active 混杂，需要治理。
- 标题屏、OG 图、EdgeOne 落地页 hero 仍是候选，未正式绑定。
- 战斗图多为候选，缺少正式 runtime 入库与剧情节点绑定。
- 原创角色未接入角色素材池，随机 NPC/原创 NPC 还不能稳定调用。
- 发布用音频需要版权/授权再次分层，不能直接把本地 fan pack 当公开发布资源。

## 是否需要专项

建议需要，但不要大规模重构。

建议专项名：

`v0.15-art-audit-1 美术资源治理与 v1.0 发布支撑专项`

定位：

- 不生成新图。
- 不删图。
- 不改玩法。
- 只做资源 manifest、状态归档、重复候选处理方案、v1.0 缺口清单和用户挑选清单。

建议交付：

1. 美术资源总 manifest：统一 `runtime_active / candidate / accepted_candidate / rejected / review-only / blocked / archive_candidate`。
2. 场景 visual-assets 补登记：把当前新增 SVG/PNG 全部纳入状态管理。
3. 重复资源处理清单：先列出，不删除。
4. 多候选用户挑选清单：战斗图、标题图、style-sample。
5. v1.0 发布美术最小清单：标题、OG、青茅主界面、路线/经济场景、战斗样板、角色/蛊图。
6. 音频发布分层：public fallback 与本地 fan pack 分开。

建议是否阻塞 v0.15-b1：

- 不应长期阻塞 v0.15-b1。
- 但在 b1 前做一刀 manifest 对齐很值得，因为 v0.15 正好进入经济/补给/市场视觉表达，场景资源如果继续散着，后面 UI 会越来越难管。
