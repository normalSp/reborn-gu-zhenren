# v0.10.0-b2 低阶凡蛊与战斗内容包

日期：2026-05-13
状态：收束验收完成，已建立候选 canon 包、readiness 引擎、ActionPanel 候选展示、验证型战斗候选登记、NarrativeCombatPanel 边界展示和战后回流无掉落激活测试；尚未激活为奖励池、商店、掉落或完整战斗数值。

## 目标

把 v0.10.0 的青茅区域从“能出发”推进到“有一小组可审查的低阶凡蛊、行为模板和遭遇模板”，但不一次性铺开一百多种蛊虫。

本阶段第一刀只做候选索引：

- 12-18 个低阶凡蛊/表现条目。
- 4-6 类战斗行为模板。
- 3-5 个青茅遭遇模板。
- 全部保持 `candidate_review`，不直接变成可刷奖励或永久数值成长。

## 新增真相源

新增 `src/canon/qingmao-low-rank-content-pack.json`：

- `guCandidates`：16 个低阶凡蛊候选。
- `behaviorTemplates`：6 类 5x3 凡战行为模板。
- `encounterTemplates`：4 个青茅区域遭遇模板。
- `_meta.runtimeActivation = not_active`。
- `_meta.saveFormatImpact = none`。

配套验证：`src/canon/qingmao-low-rank-content-pack.test.ts`。

## Readiness 引擎

第二刀新增 `src/engine/v010-qingmao-combat-pack.ts`：

- `listQingmaoCombatTemplateReadiness()`：读取候选包，检查模板是否仍是 5x3 凡战、奖励策略是否越界、引用蛊虫是否存在。
- `buildQingmaoCombatEventCandidate(templateId, store)`：为通过 readiness 的模板生成标准 `CombatEventCandidate`，再调用现有 `evaluateCombatEncounterEntry` 做本地战斗入口校验。
- 第一批只把 3 个模板标为 `ready_for_local_validation`：族学切磋、前山巡查、狼影压近。
- 林道暗线仍保持 `candidate_only`，等前 2-3 个模板稳定后再接。

该引擎是只读验证层，不会写 store、不扣 AP、不结算奖励、不生成掉落。

## UI 展示与候选登记

第三刀在 `src/components/game/ActionPanel.tsx` 增加“青茅凡战候选”小节，第四刀进一步允许 ready 模板登记到现有 `combatEventCandidates`：

- 显示 readiness 模板名称、风险、奖励策略、推荐蛊虫和本地入口校验提示。
- 只有 `ready_for_local_validation` 模板启用“登记候选”。
- 登记只写入既有 `flags.combatEventCandidates` 队列。
- 不扣 AP，不生成战斗实例，不发奖励，不写掉落。
- `candidate_only` 模板仍阻断。

第五刀细化 `src/components/game/NarrativeCombatPanel.tsx`：

- 登记后的 ready 候选保持 `engineValidation = pending`，避免在玩家点击前被误判为已处理。
- 面板显示入口校验通过/阻断、`rewardPolicy` 和不可直接掉落边界。
- `local_engine_only` 只表示胜负、损耗和回流由本地战斗引擎决定，不直接发放蛊虫、蛊材、声望或地点解锁。
- `candidate_clue_only` 只允许线索/传闻，不能作为正式掉落、奖励池或商店来源。

配套 e2e 已扩展 `tests/e2e/v010-qingmao-region-actions.spec.ts`，桌面和移动端都会检查 readiness 卡片可见且宽度可读，并验证登记后的候选能显示奖励边界且进入现有 5x3 battlefield route。

## 候选内容口径

第一批候选优先引用已有 `gu-expression-specs.json`：

- 古月/月光线：月光蛊、小光蛊、月芒蛊。
- 防护线：白玉蛊、玉皮蛊、石皮蛊。
- 熊家/力道线：熊力蛊、黑豕蛊、白豕蛊。
- 支援/巡查线：酒虫、侦察蛊、巡夜蛊、追踪蛊、治愈蛊。
- 林地/风险线：青丝蛊、毒液蛊。

候选不是直接发放清单。后续要进入运行时，必须再过：

- 战斗数值审查。
- 世界观边界审查。
- 经济/奖励来源审查。
- UI 表达审查。

## 行为模板

本阶段冻结 6 个行为模板作为后续战斗系统接入依据：

- 直线投射/切割。
- 自身护体/硬接。
- 近身力道/破防。
- 侦察/标记/揭示。
- 林地束缚/拉扯。
- 战前准备/续航支持。

这些模板只声明“应该由哪个本地引擎拥有事实”，不在 JSON 中写伤害公式。

## 遭遇模板

第一批遭遇模板：

- 族学切磋。
- 前山巡查。
- 狼影压近。
- 林道暗线。

所有模板的 `rewardPolicy` 仅允许 `local_engine_only` 或 `candidate_clue_only`，不直接写蛊虫、蛊材、声望、地点解锁或战斗胜负。

## SAVE_FORMAT_VERSION 结论

`v0.10.0-b2` 不提升 `SAVE_FORMAT_VERSION`。

原因：

- 只新增静态 canon JSON 和测试。
- 后续登记只复用既有 `flags.combatEventCandidates`，没有新增玩家背包、区域状态、遭遇历史或战斗持久化字段。
- 没有把候选内容写入运行时奖励池。
- 战后回流测试只验证既有 `combatEncounterState`、`lastBattleOutcomeSummary` 和 scene ledger，不新增持久化协议。

- 内容包保持 `candidate_review` 且 `saveFormatImpact = none`。
- 蛊虫候选数量在批准的小包范围内。
- 行为模板和遭遇模板数量在批准范围内。
- 所有候选都引用已有 `gu-expression-specs.json`。
- 所有候选流派都在 `path-registry.json` 中 runtime allowed。

- readiness 引擎只把前三个模板推进本地验证准备。

- 登记候选点击后进入现有 `skirmish_5x3` battlefield route，不新增旧 duel/squad 入口。
- 青茅 `local_engine_only` 候选战后回流不触发 `beastLoot`、材料掉落或奖励池激活；scene ledger 只记录战斗摘要与本地结算结果。
