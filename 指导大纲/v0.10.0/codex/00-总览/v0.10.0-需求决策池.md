# v0.10.0 需求决策池

日期：2026-05-13
状态：用户决策已记录；D-006 不纳入本轮。

## 待用户决策

### D-001 v0.10 主线主题

提出专家：项目总策划、玩法/数值架构师、蛊真人世界观审校
用户决策：批准
目标版本：`v0.10.0-a1`

将 v0.10.0 主线锁定为 `青茅山三寨区域主线可玩化`。

收益：

- 承接 v0.9 的青茅凡战视觉竖切和统一行动协议。
- 不铺太大，能做出一个连续可玩的区域样板。
- 最适合公开测试与玩家理解。

风险：

- 如果过度追求原著完整长线，会迅速膨胀。
- 需要严格区分 canon-near 与 IF 支线。

### D-002 内容规模策略

提出专家：玩法/数值架构师、QA/发布稳定专家
用户决策：批准
目标版本：`v0.10.0-b2`

采用“小内容包高质量”策略：首批只做 12-18 个凡蛊/表现条目、3-5 个战斗遭遇模板、4-6 类战斗行为模板。

收益：

- 可测试、可平衡、可审查。
- 避免一次性处理 100 多种蛊虫导致质量失控。

风险：

- 玩家会暂时感到内容池不够大，需要用清晰版本说明管理预期。

### D-003 世界观口径

提出专家：蛊真人世界观审校、项目总策划
用户决策：批准
目标版本：`v0.10.0-a1`

v0.10 采用 `canon-near + IF 支线`：青茅山原著主线硬边界不乱改，玩家可以以古月/熊家/白家/外来身份参与压力线，但重大原著结果不能随意被系统写死。

收益：

- 保证原著稳定性。
- 支持不同出身，不再把非古月玩家改成古月族人。

风险：

- 需要更多 provenance 字段和 UI 提示，避免玩家误解哪些是原著、哪些是 IF。

### D-004 视觉范围

提出专家：前端创意导演、Motion/Technical Artist
用户决策：批准
目标版本：`v0.10.0-b4`

继续青茅山视觉深化：2-3 个场景变体、截图构图规范、V-003 短录屏脚本；不做全局重皮，不批量生成图。

收益：

- 把 v0.9 的青茅凡战优势延伸到真实区域玩法。
- 能继续向 Antoine / In Pieces 级别的局部表现力靠近，但不牺牲可读性。

风险：

- 美术工作量容易膨胀，需要单张慢产与资产 manifest 审查。

### D-005 DeepSeek 策略

提出专家：DeepSeek/模型调优专家、Codex 操作安全官
用户决策：批准
目标版本：`v0.10.0-a1`

v0.10 不重新评估模型，不主动提出 Pro/Reasoner/其他模型切换。继续使用 `deepseek-v4-flash`，重点优化 prompt 稳定前缀、JSON 通过率、cache hit tokens、retry 成本和世界观违规率。

收益：

- 遵守用户“不再评估其他模型”的决定。
- 把精力放在缓存命中和质量闸门，而不是反复换模型。

风险：

- 如果未来 Flash 出现服务波动，需要单独开“运行故障应急”，不能混成 v0.10 需求。

### D-006 EdgeOne 公测准备

提出专家：QA/发布稳定专家、Codex 操作安全官
用户决策：不用，本轮不纳入范围
目标版本：暂缓；不列入 `v0.10.0-rc` 必选项

原提案：把 EdgeOne smoke、公开测试存档、发布说明、问题反馈池纳入 v0.10 rc。

用户决策：本轮不用。v0.10 rc 改为本地发布候选质量收束；EdgeOne 公测准备如后续需要，另开独立发布阶段或复议 D-006。

原提案收益（暂不采用）：

- 让 v0.10 不只是本地开发，而是能给玩家试。
- 公测反馈可以反哺专家团主动需求池。

原提案风险：

- 云端账号、Token、域名权限不能进入仓库；需要用户在 CodeBuddy/EdgeOne 侧完成授权。

### D-007 版本号与存档策略

提出专家：QA/发布稳定专家、玩法/数值架构师
用户决策：批准原则，具体是否提升存档版本在 a1 决定
目标版本：`v0.10.0-a1`

只有在新增持久化字段时才提升 `SAVE_FORMAT_VERSION`。若 a1 只是文档、纯 engine 或派生 UI 状态，不提升存档版本。

收益：

- 避免频繁迁移存档。
- 保持 v0.9 公开测试存档可读。

风险：

- 区域主线板大概率需要持久化字段，a1 必须先设计迁移默认值。

### D-008 b3 第一刀范围

提出专家：项目总策划、玩法/数值架构师、蛊真人世界观审校
用户决策：批准
目标版本：`v0.10.0-b3`

建议批准“青茅低阶食料/蛊材小循环”，第一刀只做月光蛊、酒虫、白玉蛊缺口显示这三类相关入口。

风险：

### D-009 b3 存档策略

提出专家：QA/发布稳定专家、Codex 操作安全官
用户决策：批准
目标版本：`v0.10.0-b3`

专家团建议第一刀不新增持久化字段，不提升 `SAVE_FORMAT_VERSION`；只复用 `materialBag`、`feedingCredits` 和 scene ledger。

收益：

- 不破坏 v0.9/当前测试存档。
- 第一刀可先验证经济闭环，不被迁移复杂度拖住。

风险：

- 不能长期记录青茅区域资源历史、委托进度或生态压力；这些需要后续单独批准新增状态。

### D-010 b3 收益上限

提出专家：玩法/数值架构师、QA/发布稳定专家
用户决策：批准
目标版本：`v0.10.0-b3`

专家团建议首批青茅行动每次最多给 1-2 份普通/精品以下注册蛊材或 1 个 feeding credit；不掉蛊虫、不掉完整蛊方、不掉仙材。

收益：

- 避免稳定刷钱、刷蛊、刷仙材。
- 能用 20/100/300 回合经济检查发现材料膨胀。

风险：

- 玩家短期获得感偏克制，需要 UI 解释这是“炼养用资源”，不是战斗战利品。

### D-011 b3 炼蛊开放口径

提出专家：蛊真人世界观审校、玩法/数值架构师
用户决策：批准
目标版本：`v0.10.0-b3`

专家团建议 b3 只开放残方进度、炼蛊尝试和失败代价；完整蛊方解锁仍由本地 engine 私有动作决定，DeepSeek 不能直接解锁。

收益：

- 保持蛊方稀缺性和世界观稳定。
- 防止 AI 文案直接制造完整配方和正式奖励。

风险：

- 炼蛊体验会先偏“门槛展示”，完整成就感需要后续小阶段补齐。

### D-012 白玉蛊资源来源

提出专家：蛊真人世界观审校、玩法/数值架构师
用户决策：批准；已随 b3 第一刀执行为缺口展示
目标版本：`v0.10.0-b3`

事实：`白玉蛊` 喂养需要玉石/金道蛊材，而现有 `碎玉片` 在商店配置中不属于青茅山期常规商品。

专家团建议第一刀先显示缺口，不新增青茅山稳定来源；后续可把“碎玉片”作为族学/委托稀有奖励另审。

收益：

- 不让白玉蛊维护成本过早失控。
- 保留稀有奖励和剧情委托的设计空间。

风险：

- 白玉蛊玩家会短期看到缺料提示，需要行动入口说明原因。

### D-013 b3-2 喂养可读闭环

提出专家：玩法/数值架构师、前端创意导演、QA/发布稳定专家
用户决策：沿 D-008 至 D-011 已批准范围执行；已随 b3-2 完成
目标版本：`v0.10.0-b3-2`

把“青茅炼养用资源”入口与现有蛊虫喂养 UI 串起来：玩家获得 `月华草` 或 `美酒` 后，能在蛊虫背包里看懂对应蛊虫可喂养、会扣哪一种材料、扣除后如何进入本地账本/日志。不得把喂养改成元石兜底。

收益：

- 玩家能理解资源入口不是孤立奖励，而是服务炼养用。
- 验证 `materialBag -> feedGuHunger -> hungerCounter/currentState` 的闭环。

风险：

- 如果 UI 过度解释会挤占 ActionPanel 和 GuInventoryPanel 空间；优先做小提示和测试，不做大重构。

### D-014 b3-2 月光蛊残方可读入口

提出专家：蛊真人世界观审校、玩法/数值架构师
用户决策：沿 D-011 已批准口径执行；已随 b3-2 完成，不直接解锁完整蛊方
目标版本：`v0.10.0-b3-2`

围绕 `frag_moonlight_advanced` 显示月光蛊升炼残方的材料缺口、尝试入口和失败代价，让玩家知道“残方”和“完整蛊方”不同。完整蛊方仍只由本地 engine 私有动作产生，DeepSeek 不得直接解锁。

收益：

- 连接资源入口、残方、炼蛊尝试三者。
- 保持蛊方稀缺性。

风险：

- 如果直接送残方会扩大进度；第一步优先做缺口/尝试可读性，不发放新残方。

### D-015 b3 经济反刷基线

提出专家：DeepSeek/模型调优专家、QA/发布稳定专家
用户决策：沿 D-010 已批准收益上限执行；已随 b3 收束完成最低限度基线
目标版本：`v0.10.0-b3`

为青茅资源小循环建立最低限度经济测试：同一场景同一入口不能重复结算，单次奖励不超过上限，战斗候选不会借 b2 内容包直接掉落材料，DeepSeek 回流文本不能追加材料或蛊方。

收益：

- 在做更多资源入口前先压住套利风险。

风险：

- 20/100/300 回合完整经济模拟仍需要后续更成熟的区域状态；当前先做单场景反刷与回流边界。

执行结果：

- `v010-qingmao-resource-loop.test.ts` 覆盖同场景同入口反刷。
- `v010-qingmao-resource-loop.test.ts` 覆盖单次材料奖励维持在 1-2 份范围内。
- `v010-qingmao-resource-loop.test.ts` 覆盖白玉蛊缺口入口不产出材料。
- b2 store 测试继续保证青茅战斗候选战后回流不激活材料掉落或奖励池。

### D-016 b4 场景变体与素材可读性

提出专家：前端创意导演、Motion/Technical Artist、玩法/数值架构师、蛊真人世界观审校
用户决策：沿 D-004 已批准视觉范围执行；已随 b4 第一刀完成
目标版本：`v0.10.0-b4`

把青茅山视觉扩展控制在 3 个场景变体：族学演武坪、前山巡查路、月草坡与酒坊支线。每个变体只负责玩家理解、截图构图和短录屏脚本，不引入新存档字段、奖励、蛊虫、蛊方、声望或地点解锁。

收益：

- 把 v0.9 青茅凡战视觉优势延伸到 v0.10 区域行动。
- 给后续对外截图/短录屏提供素材路线，而不是临时抓屏。
- 让玩家看懂“行动、候选、炼养用资源、缺口展示”之间的关系。

风险：

- 候选图若误写成 active runtime art，可能暗示场景已经最终定稿；因此本阶段全部登记为 `candidate + generic_candidate`。

执行结果：

- 新增 `qingmao-scene-variants.json` 与 `v010-qingmao-scene-variants.ts`。
- 新增 3 张候选 SVG 氛围图，并通过 `npm run check:qingmao-assets`。
- ActionPanel 新增“青茅场景变体”区块，e2e 覆盖桌面、移动端和图片加载。
