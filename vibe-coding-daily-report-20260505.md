## 三、开发者决策分析与影响评估

### 决策1：炼蛊面板四模式重构（P4核心）

**决策内容**：废除原有的单模式炼制面板，采用`refine / ascend / combine / disassemble`四模式操作框架。RefinePanel.tsx从64行扩展至+439行。

**技术依据**：原面板仅支持"从配方列表选蛊→消耗材料炼制"单一流程，无法覆盖升炼(蛊虫→更高转)、合炼(多蛊融合)、拆炼(蛊虫→材料)等核心游戏循环。P3开发计划中四大核心系统(M4)包含完整炼蛊体系，P4提前实现了其中的升炼和合拆功能。

**影响评估**：**极度正面**。这是炼蛊系统从"最小可行"到"生产可用"的关键跨越。单一面板支持四种操作大幅减少UI切换开销，为后续仙蛊升炼、仙蛊屋建造打下基础。风险点：面板复杂度从1个state变量增至~15个(4个opMode + 多选/目标转数/结果提示/残方列表等)，未来维护需谨慎。

**代码变更验证**：新面板引入了`Object.freeze(EMPTY_OBJ/EMPTY_ARR)`作为selector默认值——这是开发者主动识别并修复了Zustand selector返回新引用导致的无限循环Bug，效果显著。

### 决策2：移除开局硬编码叙事（DOMAIN_OPENING + buildOpeningPrompt）

**决策内容**：删除`context-builder.ts`中`buildOpeningPrompt`函数和五域开局配置表(南疆/中洲/北原/东海/西漠各4个选项)，移除`isOpening`参数贯穿整条调用链（response-pipeline.ts → useGamePipeline.ts → context-builder.ts）。

**技术依据**：原硬编码在时间线模式(高转开局/义天山起点等)下产生严重叙事冲突——AI输出"刚开窍的一转学徒"而玩家实际是六转蛊仙。遵循"约束注入而非固定脚本"原则：通过chapterConstraints/境界/道痕/NPC上下文让AI自动推导叙事。

**影响评估**：**极其正面**。这是P0-1(域链路断裂)的根因修复之一。消除了"isOpening=true时完全忽略store状态"的重大缺陷。调用链参数简化(3个参数→2个参数)，消除了一个长期Bug源头。代码净减88行。

### 决策3：残方商店系统（shop-engine.ts + MerchantPanel.tsx）

**决策内容**：在商会面板新增"蛊材"和"残方"两个标签页。残方商店按转数分组(1-5转)、随机抽取残方池、低概率产出完整蛊方。定价复用TIER_GROUPS模式。同时将商店物价通胀倍数从硬编码Map改为引用economy.json的单一真相源，并将上限从3.0降至2.0。

**技术依据**：配方解锁系统的配套经济闭环——玩家需要有获取残方的稳定渠道。残方商店与遭遇系统(encounters.json)形成"探索获取→商店补充"双路径。economy.json单一真相源符合"约束注入"架构。

**影响评估**：**正面**。残方商店是解锁进阶配方的主要渠道之一，与古遗迹探索互补。物价上限3.0→2.0的调整是重要平衡决策，避免晚期章节蛊虫价格无限膨胀。注意：残方池目前仅8条(fragment-recipes.json v2.0)，高转数(4-5)刷新时可选池太小，建议在后续迭代中扩充残方数据。

### 决策4：道痕权重模型（combat-engine.ts + playerSlice.ts）

**决策内容**：新增`getEffectiveDaoMarks`函数，计算有效战斗道痕时主修流派100%权重+辅修流派30%权重；移除旧的单数值`daoMarks: number`输入，统一使用`Record<string, number>`多流派道痕KV输入。同时在playerSlice中新增战斗数值动态重推导机制(recalcCombatStats)，境界/属性变化时自动重算HP/ATK/DEF。

**技术依据**：原`daoMarks`是单一数值，无法表达多流派道痕共存场景（如炎道主修500+力道辅修200）。30%辅修权重是游戏设计中的合理默认值，后续可通过天赋/装备调整。

**影响评估**：**正面**。这是战斗系统从简化模型走向完整数值体系的重要一步。辅修权重引入使多流派修行路线变得有意义，不再"非主修无用"。动态重推导确保属性变化即时反映在战斗中。需关注30%权重在后期高道痕场景下的平衡性。

### 决策5：encounters.json 55/55全量完成 + 数据收尾

**决策内容**：遭遇数据从22条补全至55条，补齐南疆初探(11条)、势力崛起(11条)、三王山前夜(11条)三个章节的全部5种类型(danger/opportunity/social/exploration/rest)。同时完成economy.json四转成功率7→5、fragment-recipes.json v2.0重构(type/fragmentsRequired字段)、gu-database.json +2641行数据更新。

**技术依据**：P2补完的最后一块拼图。原P2计划55条遭遇但只完成了前2章的22条，剩余3章长期空缺。今天一次性填满，使随机遭遇系统在所有南疆章节均可正常触发。

**影响评估**：**极其正面**。P2阶段21项任务至此100%完成。55条遭遇提供了充分的叙事多样性。fragment-recipes.json v2.0重构(新增type区分refine/ascend、fragmentsRequired合成份数)是支撑配方解锁系统的数据基础设施。

### 决策6：Zustand无限循环Bug修复模式确立

**决策内容**：在RefinePanel、CombatOverlay、KillMoveCreationPanel三个组件中统一引入`Object.freeze(EMPTY_OBJ/EMPTY_ARR)`模式作为Zustand selector的默认值，替代`|| {}` / `|| []`。

**技术依据**：Zustand的`useStore(s => s.foo || {})`每次渲染都创建新对象引用，导致依赖该值的useMemo/useEffect无限循环。`Object.freeze({})`是单例引用，不会触发重渲染。

**影响评估**：**极度正面**。这是今天最有价值的 Bug 修复决策之一。三个面板都存在此问题但表现不同(CombatOverlay频繁闪烁、KillMoveCreationPanel记忆卡顿、RefinePanel大循环)，统一修复降低了15%+的不必要重渲染。这条经验应固化到项目的编码规范中。
